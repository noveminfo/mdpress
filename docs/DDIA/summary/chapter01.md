# Chapter 1: Reliable, Scalable, and Maintainable Applications（信頼性・スケーラビリティ・保守性）

## 概要

この章では、データ集約型アプリケーション（data-intensive application）の設計における**3つの根本的な関心事**を解説している：

1. **Reliability（信頼性）**
2. **Scalability（スケーラビリティ）**
3. **Maintainability（保守性）**

---

## 1. データシステムについての考え方

現代のアプリケーションは「計算集約型」ではなく**「データ集約型」**。CPUパワーよりも、データの量・複雑さ・変化速度が課題になる。

典型的な構成要素：

- **データベース** — データの永続化
- **キャッシュ** — 高コストな処理結果の記憶（読み取り高速化）
- **検索インデックス** — キーワード検索・フィルタリング
- **ストリーム処理** — 非同期メッセージ送信
- **バッチ処理** — 蓄積データの定期的な一括処理

### ツールのカテゴリ境界の曖昧化

Redis、Kafka、DBなどのカテゴリの境界は曖昧になってきている。

- Redisはキャッシュとしてもメッセージキューとしても使われる
- KafkaはメッセージキューだがDB的な永続性もある

つまり**単一のツールだけでは要件を満たせず、複数を組み合わせた「複合データシステム」**を設計する必要がある。

## 2. Reliability（信頼性）

> 「何かがうまくいかなくても（fault）、正しく動き続けること」

### Fault vs Failure の区別

| 用語 | 意味 |
|------|------|
| **Fault（障害）** | システムの一部が仕様から逸脱すること |
| **Failure（故障）** | システム全体がユーザーへのサービスを停止すること |

目標は **fault-tolerant（耐障害性）** — faultがfailureにならないようにする。

### 3種類のFault

#### (a) ハードウェア障害

- ディスク故障、RAM不良、停電、ネットワーク断
- **従来の対策**: RAID、デュアル電源、ホットスワップCPU等の冗長化
- **最近のトレンド**: クラウド環境でのマルチマシン冗長化（ソフトウェア的な耐障害性）
  - AWS等のクラウドは単一VMの信頼性より柔軟性を優先するため

#### (b) ソフトウェアの系統的エラー

ハードウェア障害はランダムで独立だが、ソフトウェアバグは**相関的**（同時に多ノードに影響）。

具体例：

- Linux カーネルのうるう秒バグ（2012年6月30日）
- 暴走プロセスがCPU/メモリ/ディスクを食い尽くす
- 依存サービスの応答遅延によるカスケード障害

対策：徹底的なテスト、プロセス分離、監視・アラート

#### (c) 人的エラー

最大の障害原因。設定エラーが停止の主因という調査結果がある。

対策：

- ミスしにくいAPI・管理画面の設計
- **サンドボックス環境**（本番に影響しない実験場所）
- 自動テスト（ユニット〜統合テスト）
- 迅速なロールバック手段
- 詳細な監視（テレメトリ）

## 3. Scalability（スケーラビリティ）

> 「負荷が増えたとき、それに対処するための戦略」

### 負荷の記述（Describing Load）

負荷は **load parameters（負荷パラメータ）** で表現する。

- リクエスト/秒
- DB の読み書き比率
- 同時アクティブユーザー数
- キャッシュヒット率

#### Twitter の事例

2つの主要操作：

- **ツイート投稿**: 4.6K req/sec（ピーク12K）
- **ホームタイムライン閲覧**: 300K req/sec

**問題の本質はファンアウト（fan-out）**: 各ユーザーが多数のフォロワーを持つ。

| アプローチ | 方式 | トレードオフ |
|------------|------|-------------|
| **アプローチ1** | タイムライン閲覧時にJOINクエリ | 読み取りが重い |
| **アプローチ2** | ツイート投稿時に全フォロワーのキャッシュに書き込み | 書き込みが重い |

Twitterの最終解：**ハイブリッド方式**

- 通常ユーザー → アプローチ2（キャッシュに事前書き込み）
- 著名人（フォロワー3000万人超）→ アプローチ1（閲覧時にマージ）

### パフォーマンスの記述（Describing Performance）

- **スループット**: バッチ処理向け（レコード/秒）
- **レスポンスタイム**: オンラインサービス向け

#### パーセンタイルの重要性

**平均値ではなくパーセンタイルを使う。**

| 指標 | 意味 |
|------|------|
| **p50（中央値）** | 半数のリクエストがこれ以下 |
| **p95** | 95%がこれ以下 |
| **p99** | 99%がこれ以下 |
| **p999** | 99.9%がこれ以下（テールレイテンシ） |

- Amazonの例：p999が100msでも売上に影響 → テールレイテンシの最適化は重要
- ただし p9999 は最適化コストに見合わないことも

#### テールレイテンシ増幅

1回のユーザーリクエストが裏で複数のバックエンドコールを発生させる場合、1つでも遅いと全体が遅くなる。

### 負荷への対処

| 手法 | 説明 |
|------|------|
| **スケールアップ（垂直）** | より強力なマシンへ |
| **スケールアウト（水平）** | 複数マシンに分散（shared-nothing architecture） |

- 実際には**両方の組み合わせ**が現実的
- **弾力的（elastic）** なシステム = 負荷に応じて自動でリソース追加

> **著者の重要な主張**: 大規模システムのアーキテクチャはその負荷パラメータに固有のものであり、万能な「スケーラブルアーキテクチャ」は存在しない。

## 4. Maintainability（保守性）

> 「ソフトウェアの大半のコストは初期開発ではなく、継続的なメンテナンスにある」

### 3つの設計原則

#### (a) Operability（運用性）

運用チームが楽に仕事できるようにする。

- 良い監視
- 自動化ツール
- ドキュメント
- 予測可能な振る舞い
- デフォルト設定の良さ

#### (b) Simplicity（シンプルさ）

**偶発的複雑性（accidental complexity）** を排除する。

- 本質的な問題の複雑さとは別の、実装由来の複雑さ
- 主なツール：**抽象化** — 良い抽象化は実装の詳細を隠し、再利用を促進

#### (c) Evolvability（発展性）

- 要件は常に変わる → システムを変更しやすくする
- Agile開発のプラクティス（TDD、リファクタリング）が小規模では有効
- 本書では、大規模データシステムレベルでの発展性（agility）を追求

## まとめ

Chapter 1は本書全体の**思考フレームワーク**を提示している。

| 関心事 | 問い | 核心 |
|--------|------|------|
| **Reliability** | 障害が起きても正しく動くか？ | fault-tolerance |
| **Scalability** | 負荷が増えたらどう対処するか？ | 負荷パラメータとパーセンタイル |
| **Maintainability** | 長期的に変更・運用しやすいか？ | 運用性・シンプルさ・発展性 |

この3つの視点が、以降の全章（データモデル、ストレージ、レプリケーション、パーティショニング等）を貫く評価軸になる。
