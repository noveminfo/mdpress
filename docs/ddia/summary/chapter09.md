# Chapter 9: Consistency and Consensus（一貫性と合意）

## 概要

> 「生きていて間違っているのと、正しくて死んでいるのと、どちらがいいか？」— Jay Kreps

分散システムの障害（Chapter 8）にどう対処するか。本章では、最も重要な抽象概念である**コンセンサス（合意）**を中心に、線形化可能性、順序保証、分散トランザクションを統一的に論じる。

---

## 1. 一貫性保証（Consistency Guarantees）

| 一貫性モデル | 強さ | 特性 |
|------------|------|------|
| **Eventual consistency** | 最弱 | 書き込み停止後、いつかレプリカが収束（convergence） |
| **Causal consistency** | 中間 | 因果関係を保持。ネットワーク遅延に影響されない最強モデル |
| **Linearizability** | 最強 | 単一コピーの幻想。全操作がアトミックに見える |

**注意**: 分散一貫性（レプリカ間の調整）とトランザクション分離（並行トランザクションの競合回避）は独立した概念。

---

## 2. 線形化可能性（Linearizability）

### 定義

```
線形化可能性 = 最新性の保証（recency guarantee）
├── データのコピーが1つしかないかのように振る舞う
├── すべての操作がアトミックに見える
├── 一度新しい値が読まれたら、以降すべての読み取りは新しい値を返す
└── 別名: atomic consistency, strong consistency, immediate consistency
```

### Linearizability vs Serializability

| 概念 | 対象 | 保証 |
|------|------|------|
| **Serializability** | トランザクション（複数オブジェクト） | ある直列順序と等価 |
| **Linearizability** | 単一レジスタ（個々のオブジェクト） | 最新値の保証（recency） |
| **Strict Serializability** | 両方 | 両方を同時に満たす |

### 線形化可能性が必要な場面

```
用途:
├── ロックとリーダー選出 → 全ノードが同じリーダーに合意
├── 一意性制約 → ユーザー名の重複防止（CASに相当）
├── クロスチャネルのタイミング依存 → ファイルストレージ + メッセージキュー間の競合
└── 外部キー制約 → 線形化可能性は不要
```

### 各レプリケーション方式の線形化可能性

| 方式 | 線形化可能性 | 備考 |
|------|------------|------|
| **シングルリーダー** | 可能（条件付き） | リーダーから読む場合。snapshot isolationでは不可 |
| **コンセンサスアルゴリズム** | **可能** | ZooKeeper, etcd |
| **マルチリーダー** | **不可** | 非同期レプリケーション → 競合発生 |
| **リーダーレス** | **おそらく不可** | strict quorumでも競合の可能性 |

### CAPの定理（批判的検討）

```
CAP = "Consistent or Available when Partitioned"

よくある誤解: 「3つから2つ選べ」
現実: ネットワーク分断は選択ではなく不可避
├── 線形化可能性が必要 → 分断時に一部が利用不可
└── 線形化可能性が不要 → 分断時も可用性を維持

CAPの限界:
├── 1つの一貫性モデル（線形化可能性）しか考慮しない
├── 1種類の障害（ネットワーク分断）しか考慮しない
├── 歴史的には重要だが、実用的価値は限定的
└── より精密な結果に取って代わられた
```

**性能とのトレードオフ**: 線形化可能性を捨てる主な理由はフォールトトレランスではなく**性能**。RAMですら線形化可能ではない（CPUキャッシュのため）。

## 3. 順序保証（Ordering Guarantees）

### 因果関係と順序

```
因果的順序 = 半順序（partial order）
├── A が B の原因 → A は B より先
├── 並行（concurrent）→ 順序なし（incomparable）
└── 例: 質問 → 回答、行の作成 → 更新

線形化可能 ⊃ 因果的一貫性
├── 線形化可能 → 全順序（total order）
├── 因果的一貫性 → 半順序（partial order）
└── 因果的一貫性はネットワーク遅延に影響されない最強の一貫性
```

### シーケンス番号による順序付け

| 方式 | 因果関係との整合性 | 問題 |
|------|------------------|------|
| ノード別独立番号（奇数/偶数） | **不整合** | ノード間の速度差 |
| 物理クロックのタイムスタンプ | **不整合** | クロックスキュー |
| ブロック事前割り当て | **不整合** | ブロック間の順序逆転 |
| **Lamportタイムスタンプ** | **整合** | (counter, nodeID) のペア |

### Lamportタイムスタンプ

```
仕組み:
├── 各ノードが (counter, nodeID) を持つ
├── 全リクエストに最大カウンター値を含める
├── 受信ノードは自分のカウンターを max に更新
└── 因果関係と整合した全順序を保証

限界:
├── 「事後」にしか順序を決定できない
├── 「今」一意性制約を判定するには不十分
└── 全ノードへの確認が必要 → 障害に弱い

Lamportタイムスタンプ vs バージョンベクトル:
├── Lamport: 全順序を強制、並行を区別できない
└── バージョンベクトル: 因果関係と並行を区別できる
```

### Total Order Broadcast（全順序ブロードキャスト）

```
2つの安全性:
├── 信頼性のある配信（Reliable delivery）: メッセージが消失しない
└── 全順序配信（Totally ordered delivery）: 全ノードに同じ順序で配信

用途:
├── データベースレプリケーション（state machine replication）
├── 直列化可能なトランザクション（stored procedure的）
├── ロックサービス + fencing tokens（ZooKeeper zxid）
└── ログ: レプリケーションログ、トランザクションログ、WAL
```

### 線形化可能性と全順序ブロードキャストの関係

```
Total Order Broadcast → 線形化可能ストレージ:
├── ログにメッセージを追加（ユーザー名の要求）
├── ログを読み、自分のメッセージが配信されるのを待つ
├── 同じユーザー名の最初のメッセージが自分なら成功
└── ただし読み取りの線形化可能性には追加措置が必要

線形化可能ストレージ → Total Order Broadcast:
├── 線形化可能な整数レジスタ（increment-and-get）
├── メッセージにシーケンス番号を付与
└── ギャップなしの連番 → Lamportとの本質的な違い

核心: 線形化可能 CAS/increment-and-get ≡ Total Order Broadcast ≡ コンセンサス
```

## 4. 分散トランザクションとコンセンサス

### コンセンサスが必要な場面

- **リーダー選出**: split brain防止
- **アトミックコミット**: 全ノードがコミットまたはアボートに合意

### FLP不可能性定理

```
FLP [Fischer, Lynch, Paterson, 1985]:
├── 非同期モデルでは、1ノードのクラッシュ可能性だけでコンセンサスは不可能
├── ただし: タイムアウトやランダム性を許可すれば回避可能
└── 理論的に重要だが、実用的にはコンセンサスは達成可能
```

### 2PC（Two-Phase Commit）

```
Phase 1（Prepare）:
├── コーディネータが全参加者に prepare 送信
├── 各参加者: コミット可能なら "yes"、不可なら "no"
└── "yes" = 後でコミットする約束（もう自力でアボートできない）

Phase 2（Commit/Abort）:
├── 全員 "yes" → コーディネータがcommitを決定 → 全参加者にcommit送信
├── 1つでも "no" → コーディネータがabortを決定 → 全参加者にabort送信
└── コーディネータの決定は取り消し不可

コーディネータ障害:
├── Phase 1前のクラッシュ → 参加者がアボート
├── Phase 2前のクラッシュ → 参加者がin-doubt状態（身動き取れない）
├── コーディネータ回復を待つしかない → ブロッキングプロトコル
└── 3PC: 理論的には解決だが、ネットワーク遅延が無制限だと機能しない
```

### 分散トランザクションの実践

| 種類 | 例 | 特性 |
|------|-----|------|
| **DB内部の分散トランザクション** | 同一DB技術間 | 最適化可能、性能良好 |
| **異種の分散トランザクション** | 異なるDB・MQ間 | 非常に困難 |

```
XAトランザクション:
├── 標準API（X/Open XA）、多くのDBとMQがサポート
├── コーディネータ = 単一障害点（SPOF）
├── in-doubt トランザクション → ロックを保持し続ける
├── 性能低下: 分散トランザクションは10倍以上遅い（MySQL報告）
└── 異種システム間の制約: SSIなどの検出プロトコルが使えない
```

### フォールトトレラントなコンセンサス

```
形式的性質（4つ）:
├── Uniform agreement: 2つのノードが異なる決定をしない
├── Integrity: 同じ値を2回決定しない
├── Validity: ノードが値vを決定 → vは誰かが提案した値
└── Termination: クラッシュしないノードは最終的に値を決定する
    └── ただし過半数が生存している場合のみ

2PC との違い:
├── 2PC: 全参加者の "yes" が必要 → 1ノード障害でブロック
└── コンセンサス: 過半数の合意で十分 → フォールトトレラント
```

### エポック番号とクォーラム

```
合意プロトコルのリーダー管理:
├── エポック番号（Paxos: ballot, Raft: term, VSR: view）
├── 各エポックでリーダーは一意
├── リーダーの決定前に2回の投票:
│   ├── 1回目: リーダー選出
│   └── 2回目: リーダーの提案への投票
├── 2つのクォーラムが重なる → より高いエポックのリーダーが優先
└── 2PCとの本質的な違い: 過半数で十分 + リカバリプロセスあり
```

### コンセンサスの限界

```
コスト:
├── 同期レプリケーション → 性能への影響
├── 厳密な過半数が必要 → 3ノードで1障害、5ノードで2障害まで
├── 固定メンバーシップ前提 → 動的な追加/削除は困難
├── タイムアウト依存 → 誤検知によるリーダー交代 → 性能低下
└── ネットワーク問題に敏感（Raftのedge case等）
```

## 5. メンバーシップと協調サービス

### ZooKeeper / etcd

```
特徴:
├── 少量のデータをメモリに保持（汎用DBではない）
├── フォールトトレラントな全順序ブロードキャストで複製
└── 提供する機能:
    ├── 線形化可能なアトミック操作（CAS → ロック実装）
    ├── 操作の全順序付け（zxid/cversion → fencing token）
    ├── 障害検出（セッション + ハートビート → ephemeral nodes）
    └── 変更通知（watch機能 → ポーリング不要）

用途:
├── リーダー選出
├── パーティション割り当て
├── サービスディスカバリ（DNSは線形化可能ではないがそれでOK）
└── メンバーシップサービス（どのノードが生存しているかの合意）

アーキテクチャ:
├── 固定ノード数（3または5）で過半数投票
├── 大量のクライアントをサポート
└── コンセンサスを「外注」するサービス
```

## まとめ

```
Chapter 9 の構造
├── 線形化可能性（Linearizability）
│   ├── 単一コピーの幻想 + アトミック操作
│   ├── 用途: ロック、リーダー選出、一意性制約
│   ├── レプリケーション方式による実現可能性の違い
│   ├── CAPの批判: 実用的価値は限定的
│   └── 性能コスト: RAMですら線形化可能ではない
│
├── 順序保証（Ordering Guarantees）
│   ├── 因果的順序 = 半順序、線形化可能 = 全順序
│   ├── Lamportタイムスタンプ: 因果整合だが「事後」のみ
│   ├── Total Order Broadcast: 信頼性 + 全順序配信
│   └── 等価性: 線形化可能CAS ≡ TOB ≡ コンセンサス
│
├── 分散トランザクション
│   ├── 2PC: ブロッキング、コーディネータがSPOF
│   ├── XA: 標準だが性能と運用に課題
│   └── 3PC: 理論的には非ブロッキングだが実用困難
│
├── フォールトトレラントなコンセンサス
│   ├── FLP不可能性: 非同期モデルでは不可能
│   ├── 実用アルゴリズム: VSR, Paxos, Raft, Zab
│   ├── エポック番号 + クォーラム投票
│   └── 限界: 過半数必要、固定メンバーシップ、タイムアウト依存
│
└── ZooKeeper / etcd
    ├── コンセンサスの「外注」サービス
    ├── ロック、リーダー選出、障害検出、変更通知
    └── 自前実装は避けるべき

コンセンサスに帰着される問題:
├── 線形化可能な CAS レジスタ
├── アトミックトランザクションコミット
├── Total Order Broadcast
├── ロックとリース
├── メンバーシップ/協調サービス
└── 一意性制約

リーダー障害時の3つの選択肢:
├── 1. 回復を待つ（ブロック） → termination性質を満たさない
├── 2. 人間が手動フェイルオーバー → 「神の行為」による合意
└── 3. コンセンサスアルゴリズムで自動選出 → 推奨
```

### 核心的な洞察

- **線形化可能性**は直感的だが遅い — 因果的一貫性がより実用的な代替
- **CAPの定理**は歴史的に重要だが、「3つから2つ選べ」は**誤解を招く**
- **Lamportタイムスタンプ**は因果関係に整合するが、リアルタイム判定には**不十分**
- **Total Order Broadcast**と**線形化可能CAS**と**コンセンサス**は**等価**
- **2PC**はコンセンサスアルゴリズムだが、コーディネータ障害に弱い**ブロッキングプロトコル**
- **フォールトトレラントなコンセンサス**（Raft, Paxos等）は過半数の投票で前進可能
- ZooKeeper等の**協調サービス**でコンセンサスを「外注」するのが実用的
- Part II の集大成: レプリケーション → パーティショニング → トランザクション → 障害モデル → **一貫性と合意**
