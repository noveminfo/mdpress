# Chapter 12: The Future of Data Systems（データシステムの将来）

## 概要

> 「船長の最高の目的がその船を保全することであったなら、彼はその船を永遠に港に留めておくだろう。」— 聖トマス・アクィナス

本書の最終章では、これまでの技術的議論を統合し、将来のデータシステム設計のあり方を提案する。データ統合（data integration）、データベースのアンバンドリング（unbundling）、正確性（correctness）の追求、そしてデータ倫理（ethics）まで幅広く論じる。

---

## 1. Data Integration（データ統合）

### 専門ツールの組み合わせ

単一のツールですべてのアクセスパターンに対応することは不可能。複数のシステムを組み合わせる必要がある。

```
データの流れの明確化:
├── System of Record（記録システム）に最初に書き込む
├── CDC等で変更を他システムに伝播
├── 全入力を単一システムに集約 → 全順序を決定
└── 冪等（idempotent）な更新で障害からの回復が容易
```

### 導出データ vs 分散トランザクション

| 観点 | 分散トランザクション | ログベース導出データ |
|------|---------------------|---------------------|
| **順序決定** | ロック（2PL） | ログの順序 |
| **一貫性** | 原子コミット → exactly-once | 決定論的リトライ + 冪等性 |
| **タイミング** | 線形化可能（linearizable） | 非同期更新（結果整合性） |
| **障害耐性** | XAは耐障害性が低い | ローカル障害を封じ込め |
| **スケール** | 制限あり | 高スケーラビリティ |

### 全順序の限界

```
全順序（total ordering）が困難な場合:
├── 1. スループットが単一ノードを超える → パーティション間の順序は曖昧
├── 2. 地理分散データセンター → 各DCに別リーダー
├── 3. マイクロサービス → サービス間で順序未定義
└── 4. クライアント側状態 → オフライン時、C/Sで順序が異なる

因果関係の捕捉:
├── 論理タイムスタンプ（Lamport timestamps等）
├── イベントID参照による因果依存の記録
└── 衝突解決アルゴリズム（状態維持には有効、外部副作用には不十分）
```

## 2. Batch and Stream Processing（バッチ＆ストリーム処理）

### 導出状態の維持

| 特性 | バッチ処理 | ストリーム処理 |
|------|-----------|--------------|
| **データセット** | 有界（bounded） | 無限（unbounded） |
| **スタイル** | 関数型（純粋関数、不変入力） | 管理された障害耐性状態 |
| **利点** | 大量の過去データの再処理 | 低遅延での変更反映 |

### Lambda Architecture

```
Lambda Architecture:
├── イベントは不変ログに追記
├── バッチ層: 正確だが遅い結果を生成
├── ストリーム層: 近似的だが高速な結果を生成
└── 両結果をマージしてクエリに応答

問題点:
├── 同じロジックを2つのフレームワークで維持する負担
├── バッチとストリームの出力マージが複雑
└── インクリメンタルバッチ化でバッチ層の単純さが失われる
```

### バッチとストリームの統一

```
統一に必要な機能:
├── 過去イベントのリプレイ能力（例: Kafkaのリプレイ）
├── Exactly-once semantics（障害時も出力が正しい）
└── イベント時間によるウィンドウ処理（処理時間ではなく）

例: Apache Beam → Flink / Google Cloud Dataflow で実行
例: Spark = バッチ上のストリーム（microbatch）
    Flink = ストリーム上のバッチ
```

## 3. Unbundling Databases（データベースのアンバンドリング）

### データベース機能の分解

```
データベースが内部的に提供する機能:
├── セカンダリインデックス
├── マテリアライズドビュー
├── レプリケーションログ
└── 全文検索インデックス

↕ 対応するデータフローの概念:
├── バッチ/ストリーム処理によるインデックス構築
├── マテリアライズドビュー保守
├── CDCによるレプリケーション
└── 検索インデックスの導出
```

### 2つの統合アプローチ

| アプローチ | 目的 | 特徴 |
|-----------|------|------|
| **Federated databases**（連合DB） | 読み取りの統一 | 多様なストレージに統一クエリインターフェース（例: PostgreSQL FDW） |
| **Unbundled databases**（非束縛DB） | 書き込みの統一 | CDC/イベントログで書き込みを同期。Unix哲学的なツール合成 |

### ログベース統合の利点

```
疎結合の2つのレベル:
├── システムレベル:
│   ├── 非同期イベントストリームで個別障害を封じ込め
│   ├── 遅延したコンシューマもバッファリングで対応
│   └── 分散トランザクションはローカル障害を大規模障害に拡大
└── 人間レベル:
    ├── 各チームが独立してコンポーネントを開発・保守
    ├── イベントログが明確なインターフェースを提供
    └── 順序保証と耐久性を備えつつ汎用的
```

### Dataflow設計パターン（"database inside-out"）

```
アプリケーションコードと状態の分離:
├── 従来: DB = 受動的な共有変数、アプリがポーリング
├── Dataflow: 状態変更をストリームとして購読
│   ├── スプレッドシートのように依存データを自動更新
│   ├── ストリームオペレータをパイプで合成
│   └── マイクロサービスの同期RPCをストリーム結合に置換
└── 例: 為替レート
    ├── マイクロサービス: 購入時にレートサービスにRPC
    └── Dataflow: レート更新ストリームを事前購読 → ローカルDB参照
```

### Write Path と Read Path

```
Write Path（書き込み経路）= eager evaluation:
├── データ到着時に事前計算
├── インデックス更新、マテリアライズドビュー更新
└── 読み取りが必要かどうかに関わらず実行

Read Path（読み取り経路）= lazy evaluation:
├── クエリ時にのみ実行
└── 導出データセットで検索・応答

境界の移動:
├── インデックス多 → write path の仕事が増、read path は軽い
├── インデックス無 → write path は軽い、read path は全スキャン（grep的）
└── キャッシュ = よくあるクエリ結果を事前計算（write/read の中間）

例: Twitterのタイムライン（第1章の例）
  通常ユーザ → ファンアウト（write path重い）
  セレブリティ → read時にマージ（read path重い）
```

### Stateful な オフライン対応クライアント

```
クライアント状態の進化:
├── 従来: ステートレスブラウザ、サーバに権限
├── 現在: SPA、モバイルアプリがローカル状態を保持
├── オフラインファースト: ローカルDBで動作、バックグラウンド同期
└── デバイス上の状態 = サーバ状態のキャッシュ/レプリカ

状態変更のプッシュ:
├── SSE（Server-Sent Events）、WebSocket
├── Write path をエンドユーザデバイスまで延伸
├── 初期状態は read path で取得、以降はストリームで更新
└── オフライン中の欠落 = コンシューマオフセットで復帰（Kafka同様）

End-to-end イベントストリーム:
├── React/Redux/Elm 的なクライアント側イベント購読
├── サーバからの状態変更もイベントパイプラインに統合
└── デバイス → イベントログ → ストリーム処理 → 別デバイスのUI
```

### Reads Are Events Too（読み取りもイベント）

```
読み取りをイベントとして扱う:
├── 書き込みも読み取りもストリーム処理経由
├── 読み取りイベント = ストリームテーブル結合（stream-table join）
├── 一回限りの読み取り = 即時結合して忘れる
├── 購読 = 過去・未来のイベントとの永続的結合
└── 読み取りの記録 → 因果依存の追跡、データ出所の把握

Multi-partition データ処理:
├── 複数パーティションにまたがるクエリをストリーム結合で実行
├── 例: Storm の分散RPC（URLを見た人数の計算）
└── 例: 不正検出（IP/メール/住所のレピュテーションスコア結合）
```

## 4. Aiming for Correctness（正確性の追求）

### End-to-End Argument（エンドツーエンド原則）

```
問題: DBのトランザクションだけでは不十分
├── アプリのバグによる不正データは防げない
├── TCP重複抑制はTCP接続内のみ
├── 2PC も end-user → app server 間の重複は防げない
└── ブラウザ → サーバ間のリトライは別リクエスト扱い

解決: エンドツーエンドの操作識別子
├── クライアントがUUID等の一意IDを生成
├── 全経路（ブラウザ → サーバ → DB）でIDを伝播
├── DB側でユニーク制約によりIDの重複を排除
└── Example 12-2: request_id テーブルで冪等性を実現
```

### 制約の強制

```
ユニーク制約にはコンセンサスが必要:
├── 単一リーダーが全決定を行う（単一ノードがボトルネック）
├── パーティショニングで値ベースのスケールアウト可能
└── 非同期マルチマスタレプリケーションでは不可

ログベースメッセージングでのユニーク制約:
├── 1. ユーザ名リクエストをハッシュでパーティション分割しログに追記
├── 2. ストリーム処理が順次読み取り、ローカルDBで重複チェック
├── 3. 成功/拒否メッセージを出力ストリームに発行
└── クライアントは出力ストリームを監視して結果を取得

Multi-partition リクエスト処理（原子コミット不要）:
├── 1. 送金リクエストにUUID付与 → リクエストIDでパーティション分割
├── 2. ストリーム処理が debit/credit の2メッセージを別パーティションに発行
├── 3. 各パーティションのプロセッサがリクエストIDで重複排除して適用
└── 決定論的処理 + エンドツーエンドID = 原子コミットと同等の正確性
```

### Timeliness vs Integrity（適時性 vs 整合性）

| 観点 | Timeliness（適時性） | Integrity（整合性） |
|------|---------------------|---------------------|
| **意味** | ユーザが最新状態を観測できる | データ損失・矛盾がない |
| **違反時** | 結果整合性（一時的） | 永続的な不整合 |
| **重要度** | 煩わしいが致命的ではない | 壊滅的 |
| **CAP** | 線形化可能性（consistency） | — |
| **ACID** | 通常、両方を提供 | 原子性・永続性が保証 |
| **スローガン** | "eventual consistency" | "perpetual inconsistency" |

### Dataflow システムの正確性メカニズム

```
整合性を保つ4つの仕組み:
├── 1. 書き込み内容を単一メッセージで原子的に表現（イベントソーシング）
├── 2. そのメッセージから決定論的導出関数で他の状態を更新
├── 3. クライアント生成リクエストIDで全レベルの重複抑制
└── 4. メッセージの不変性 + 導出データの再処理でバグからの回復

緩やかな制約の解釈:
├── ユーザ名重複 → 後から謝罪して変更を依頼（compensating transaction）
├── 在庫超過 → 追加発注して顧客に遅延を通知
├── 航空会社のオーバーブッキング → 補償プロセス
└── 口座引き落とし超過 → 手数料を請求
  → 適時性なしでも整合性があればビジネスは成立
```

### Coordination-Avoiding Data Systems（協調回避型データシステム）

```
2つの観察:
├── 1. データフローは原子コミット/線形化可能性なしで整合性を維持可能
└── 2. 多くのアプリは一時的な制約違反を許容できる

→ 協調なしで強い整合性保証を持つシステムが可能
  ├── 複数DC間で非同期レプリケーション
  ├── 各DCが独立稼働
  ├── 弱い適時性（線形化不可）だが強い整合性
  └── 必要な箇所のみ同期協調を導入
```

### Trust, but Verify（信頼、しかし検証）

```
現実のデータ破壊:
├── ディスク上の静止データの破壊
├── ネットワーク上のTCPチェックサムを回避する破壊
├── メモリのビットフリップ（rowhammer攻撃）
├── DBソフトのバグ（MySQL、PostgreSQL含む）
└── アプリケーションコードのバグ

監査可能性の設計:
├── イベントソーシング: 入力を不変イベントで記録
├── 状態更新を決定論的に導出 → 再現・検証可能
├── ハッシュでイベントストレージの破壊を検出
├── バッチ/ストリーム処理を再実行して結果を比較
└── エンドツーエンドの整合性チェック

暗号的監査ツール:
├── Merkle tree: データの存在証明を効率的に行う木構造ハッシュ
├── Certificate Transparency: TLS証明書の妥当性検証
└── ブロックチェーン/分散台帳: 相互不信のレプリカ間で整合性検証
```

## 5. Doing the Right Thing（正しいことをする）

### データ倫理の主要テーマ

```
予測分析の問題:
├── アルゴリズムが「リスクが高い」とラベルした人の社会的排除
├── 「アルゴリズムの刑務所」: 就職、旅行、保険等から体系的に排除
├── 機械学習は偏見のマネーロンダリング（入力のバイアスを増幅）
├── 統計的正確性 ≠ 個別の正確性
└── フィードバックループ: 予測が現実を強化する悪循環

プライバシーと監視:
├── "data" を "surveillance" に置換してみる思考実験
├── プライバシー = 何を誰に開示するか選択する自由
├── データ収集 → プライバシー権が個人から企業に移転
├── 同意の形骸化: ユーザはデータの行方を理解していない
├── サービスが社会参加に不可欠 → 拒否は実質不可能
└── データは資産であり有害物質でもある（"toxic asset"）

産業革命との類比:
├── 情報時代のデータ = 産業時代の公害
├── 規制なき初期 → 環境保護法が最終的に社会を改善
├── データ保護法（EU Data Protection Directive等）
└── 自主規制 + ユーザの尊厳・自律性の尊重が必要
```

```
本章の全体像:

│   ├── バッチ/ストリーム処理で変更を伝播
│   └── 非同期・疎結合で障害を封じ込め
├── データベースのアンバンドリング

│   ├── クライアントまでイベントストリームを延伸
│   └── 読み取りもイベントとして扱う
├── 正確性
│   ├── エンドツーエンド操作IDで冪等性を実現

└── 倫理
    ├── 予測分析のバイアスとフィードバックループ
    ├── プライバシーと監視のバランス
    └── エンジニアの倫理的責任
```

### 本書全体を通じた設計原則

| 原則 | 内容 |
|------|------|
| **信頼性（Reliability）** | 障害耐性アルゴリズム、冪等性、エンドツーエンド正確性 |
| **スケーラビリティ（Scalability）** | パーティショニング、非同期処理、協調回避 |
| **保守性（Maintainability）** | データフローの明確化、スキーマ進化、再処理による段階的移行 |
| **正確性（Correctness）** | 整合性保証、監査可能性、エンドツーエンド原則 |
| **倫理（Ethics）** | 人間の尊厳を尊重するシステム設計 |
