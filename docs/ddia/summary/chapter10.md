# Chapter 10: Batch Processing（バッチ処理）

## 概要

> 「初期設計が完成し十分に堅牢になったとき、多くの異なる視点を持つ人々がそれぞれの実験を行い始めるところから、真のテストが始まる。」— Donald Knuth

本書の Part I, II ではリクエスト/レスポンス型のオンラインシステムを中心に議論した。Part III ではデータの**導出（derived data）** に焦点を移す。本章はバッチ処理 — Unixの設計哲学からMapReduce、そしてその先のデータフローエンジンまで — を体系的に扱う。

### 3種類のシステム

| 種類 | 特性 | 性能指標 |
|------|------|---------|
| **Services（オンライン）** | リクエスト → レスポンス | 応答時間 |
| **Batch processing（オフライン）** | 大量入力 → 大量出力 | スループット |
| **Stream processing（準リアルタイム）** | イベント発生直後に処理 | レイテンシ |

## 1. Unixツールによるバッチ処理

### ログ分析の例

```bash
cat /var/log/nginx/access.log |
  awk '{print $7}' |       # URLを抽出
  sort             |       # アルファベット順にソート
  uniq -c          |       # 重複をカウント
  sort -r -n       |       # カウント降順でソート
  head -n 5                # 上位5件
```

### ソート vs インメモリ集約

| アプローチ | 利点 | 適用場面 |
|-----------|------|---------|
| **インメモリハッシュテーブル**（Rubyスクリプト等） | ランダムアクセスが速い | ワーキングセットがメモリに収まる場合 |
| **ソートベース**（Unixパイプライン） | ディスクスピルに対応、自動並列化 | メモリを超える大規模データ |

### Unix哲学

```
4原則:
├── 1. 各プログラムは1つのことをうまくやる
├── 2. 全プログラムの出力が別プログラムの入力になることを想定
├── 3. ソフトウェアは早期に試す（数週間以内）
└── 4. 未熟な助けよりツールを使う

設計の特徴:
├── 統一インターフェース: ファイル（バイト列）
│   ├── ファイル、stdin/stdout、ソケット、デバイスが同じインターフェース
│   └── 慣例: ASCII テキスト、\n 区切り
├── ロジックとワイヤリングの分離
│   ├── stdin/stdout を使えば入出力の接続はシェルが決める
│   └── 疎結合、遅延バインディング、制御の反転
└── 透明性と実験容易性
    ├── 入力は不変（immutable）
    ├── パイプラインの途中で less に出力して確認可能
    └── 中間結果をファイルに保存して再開可能

限界: 単一マシンでしか動作しない → Hadoop の出番
```

## 2. MapReduce と分散ファイルシステム

### HDFS（Hadoop Distributed File System）

```
HDFS:
├── Shared-nothing アーキテクチャ（NAS/SANのShared-diskと対照的）
├── 各マシンのデーモンプロセスがファイルを公開
├── NameNode がファイルブロックの位置を管理
├── ファイルブロックを複数マシンにレプリケート
├── erasure coding（Reed-Solomon等）も可能
└── スケール: 数万台、数百ペタバイト
```

### MapReduce ジョブの実行

```
4ステップ:
├── 1. 入力ファイルをレコードに分割
├── 2. Mapper: 各レコードからキーと値を抽出
├── 3. Sort: キーでソート（暗黙的、常に実行）
└── 4. Reducer: 同じキーの値をイテレートして処理

分散実行:
├── Mapper: 入力ファイルブロックごとに1タスク
│   └── データの近くで計算（computation near data）
├── Shuffle: キーのハッシュでReducerにパーティション分割
│   ├── 各Mapタスクがキーのハッシュでパーティション
│   ├── 各パーティションをソート済みファイルとしてローカルディスクに書き込み
│   └── Reducerがmapperからソート済みファイルをフェッチ
└── Reducer: ソート済みファイルをマージ → 出力をHDFSに書き込み
```

### MapReduce ワークフロー

```
連鎖:
├── 1つのジョブの出力 → 次のジョブの入力（ディレクトリ名で暗黙的に接続）
├── Unixパイプとは異なり、中間データを完全にファイルに書き出す（materialization）
├── ワークフロースケジューラ: Oozie, Azkaban, Luigi, Airflow
├── 推薦システムでは50〜100個のMapReduceジョブが一般的
└── 高レベルツール: Pig, Hive, Cascading, Crunch, FlumeJava
```

## 3. Reduce-Side Joins と Grouping

### Sort-Merge Join

```
例: ユーザーアクティビティログ × ユーザーDB
├── Mapper 1: アクティビティイベント → (userID, event)
├── Mapper 2: ユーザーDB → (userID, date_of_birth)
├── Shuffle: 同じ userID のレコードが同じ Reducer に集まる
├── Secondary sort: ユーザーDBレコードが先、イベントが後
└── Reducer: ユーザー情報をローカル変数に保持 → イベントを処理

利点: ネットワークリクエスト不要、メモリ効率が良い
```

### GROUP BY

```
用途:
├── 各グループ内の集計（COUNT, SUM, TOP-k）
├── セッション化（sessionization）: 同一ユーザーのアクション列を集める
└── 実装: Mapperがグルーピングキーで出力 → ソート → 同じキーが同じReducerに
```

### スキュー（Hot Keys）対策

| 手法 | 説明 |
|------|------|
| **Pig: skewed join** | サンプリングでhot keyを特定 → 複数Reducerにランダム分散 |
| **Crunch: sharded join** | hot keyを明示的に指定 → 複数Reducerに分散 |
| **Hive: skewed join** | hot keyのレコードを別ファイルに保存 → map-side joinを使用 |
| **2段階集約** | 1段目: ランダムReducerで部分集約 → 2段目: 最終集約 |

---

## 4. Map-Side Joins

| Join方式 | 前提条件 | 仕組み |
|----------|---------|--------|
| **Broadcast hash join** | 片方が小さい（メモリに収まる） | 全Mapperが小さい入力をハッシュテーブルにロード |
| **Partitioned hash join** | 両入力が同じ方法でパーティション分割済み | 各パーティションで独立にハッシュjoin |
| **Map-side merge join** | 両入力が同じキーでパーティション分割＋ソート済み | Mapperがマージ操作を実行 |

## 5. バッチワークフローの出力

### 主な用途

```
出力の種類:
├── 検索インデックスの構築（Google原始のMapReduce用途）
│   ├── Mapperがドキュメントをパーティション
│   ├── Reducerがパーティションごとのインデックスを構築
│   └── インデックスファイルは作成後イミュータブル
├── Key-Valueストアの構築（推薦システム、ML等）
│   ├── バッチジョブ内でDBファイルを構築 → HDFSに出力
│   ├── サーバーがHDFSからファイルをコピー → アトミックに切り替え
│   └── 例: Voldemort, Terrapin, ElephantDB, HBase bulk loading
└── 分析レポート（集計、ランキング等）
```

### バッチ処理出力の哲学

```
Unix哲学の継承:
├── 入力はイミュータブル（変更しない）
├── 出力は前の出力を完全に置換
├── 副作用なし
├── バグ → コードをロールバックしてジョブを再実行するだけ（human fault tolerance）
├── map/reduceタスクの失敗 → フレームワークが自動リトライ
├── 同じ入力ファイルを複数の異なるジョブの入力に使える
└── ロジックとワイヤリングの分離（チーム間の責任分離）
```

## 6. Hadoop vs 分散データベース（MPP）

| 観点 | MPP データベース | Hadoop |
|------|----------------|--------|
| **ストレージ** | スキーマ必須、専用フォーマット | 任意のバイト列（schema-on-read） |
| **処理モデル** | SQL特化（モノリシック） | 任意のコード実行可能（多様なモデル共存） |
| **データ取り込み** | 慎重なモデリング → インポート | まずダンプ → 後でモデリング（data lake） |
| **障害処理** | クエリ全体をアボート → 再実行 | タスク単位でリトライ |
| **メモリ使用** | できるだけメモリに保持 | 積極的にディスクに書き込む |

**「寿司の原則」**: "raw data is better" — 生データのままダンプして、後から複数の変換を適用。

### MapReduceが頻繁な障害に設計された理由

```
Googleの環境:
├── オンラインサービスとバッチジョブが同じマシン上で共存
├── 低優先度のバッチジョブは高優先度プロセスにプリエンプトされる
├── 1時間実行のMapReduceタスク → 約5%の確率でプリエンプト
├── 100タスク × 10分 → 50%以上の確率で少なくとも1タスクが終了前に停止
└── → タスク粒度のリトライが合理的（ハードウェア障害ではなくプリエンプション対策）
```

## 7. MapReduceを超えて（Beyond MapReduce）

### 中間状態のマテリアライゼーションの問題

```
MapReduceの問題:
├── 前のジョブの全タスク完了を待つ必要 → stragglerがボトルネック
├── Mapperが冗長（前のReducerの出力をそのまま読むだけの場合が多い）
└── 中間状態をHDFSに書き込む → レプリケーションが過剰
```

### データフローエンジン（Spark, Tez, Flink）

```
MapReduceとの違い:
├── ワークフロー全体を1つのジョブとして扱う
├── Map/Reduceの厳密な交互実行ではなく、柔軟な「オペレータ」の組み合わせ
├── 3つの接続方式:
│   ├── リパーティション＋ソート（sort-merge join用）
│   ├── リパーティションのみ（partitioned hash join用）
│   └── ブロードキャスト（broadcast hash join用）
│
├── 利点:
│   ├── ソートが必要な箇所でのみ実行
│   ├── 不要なMapタスクを排除
│   ├── データ依存関係の宣言 → ローカリティ最適化
│   ├── 中間状態をメモリ/ローカルディスクに保持（HDFSレプリケーション不要）
│   ├── オペレータは入力準備次第で即座に実行開始
│   └── JVMプロセスの再利用（起動オーバーヘッド削減）
│
└── 障害耐性:
    ├── 中間状態をHDFSに書かない → 障害時は再計算
    ├── Spark: RDD（Resilient Distributed Dataset）で血統（ancestry）を追跡
    ├── Flink: オペレータ状態のチェックポイント
    └── 決定論的オペレータが重要（非決定論 → カスケード障害のリスク）
```

### グラフとイテレーティブ処理

```
課題: MapReduceは1パスのみ → 「完了まで繰り返す」を表現できない

Pregel モデル（BSP: Bulk Synchronous Parallel）:
├── 各頂点が隣接頂点にメッセージを送信
├── 各イテレーションで関数が呼ばれ、受信メッセージを処理
├── 頂点はメモリに状態を保持（MapReduceと異なる）
├── メッセージがなければ作業不要
├── アクターモデルに類似（ただしタイミング保証あり）
├── 障害耐性: 定期的なチェックポイント
└── 実装: Apache Giraph, Spark GraphX, Flink Gelly

注意: グラフがメモリに収まるなら単一マシンのほうが速いことが多い
```

### 高レベルAPIと宣言的クエリ

```
進化:
├── 高レベルAPI: Hive, Pig, Cascading, Crunch, Spark DataFrame
│   └── リレーショナル的な操作（join, group, filter, aggregate）
├── コストベースのクエリオプティマイザ（Hive, Spark, Flink）
│   └── 結合アルゴリズムの自動選択、結合順序の最適化
├── 宣言的 + 命令的のハイブリッド
│   ├── 宣言的: 列指向ストレージ活用、ベクトル化実行
│   ├── 命令的: 任意のコード実行（ML、NLP、画像分析等）
│   └── Spark: JVMバイトコード生成、Impala: LLVMネイティブコード生成
└── 結果: バッチ処理フレームワークとMPP DBが収斂しつつある
```

## まとめ

```
Chapter 10 の構造
├── Unix哲学
│   ├── 入力イミュータブル、出力は次の入力
│   ├── 小さなツールの組み合わせで複雑な問題を解決
│   ├── 統一インターフェース（ファイル/パイプ）
│   └── MapReduce → Unixの分散版
│
├── MapReduce
│   ├── Mapper（キー抽出）→ Sort → Reducer（集約）
│   ├── HDFS上で動作、データの近くで計算
│   └── ワークフローの連鎖（中間状態のマテリアライゼーション）
│
├── Joinアルゴリズム
│   ├── Sort-merge join（Reduce-side）
│   ├── Broadcast hash join（Map-side、小+大）
│   ├── Partitioned hash join（Map-side、同パーティション）
│   └── Map-side merge join（同パーティション+同ソート）
│
├── Hadoop vs MPP
│   ├── ストレージの多様性（schema-on-read）
│   ├── 処理モデルの多様性（SQL + 任意コード）
│   └── 障害設計（タスク粒度リトライ vs クエリ全体再実行）
│
├── Beyond MapReduce
│   ├── データフローエンジン（Spark, Tez, Flink）
│   ├── 中間状態のマテリアライゼーション回避
│   ├── グラフ処理（Pregel/BSP）
│   └── 高レベルAPI + クエリオプティマイザ
│
└── 2つの主要課題
    ├── パーティショニング: 関連データを同じ場所に集める
    └── 障害耐性: MapReduce=ディスク書き込み、データフローエンジン=再計算

核心: バッチ処理ジョブは入力を読み出力を生成し、入力を変更しない
    → 出力は入力から導出される（derived）
    → 入力は有界（bounded）→ ジョブはいつか完了する
    → Chapter 11: 入力が無界（unbounded）のストリーム処理へ
```
