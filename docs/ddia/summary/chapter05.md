# Chapter 5: Replication（レプリケーション）

## 概要

> 「うまくいかないかもしれないものと、絶対にうまくいかないはずがないものの最大の違いは、絶対にうまくいかないはずがないものがうまくいかなくなったとき、たいてい手が付けられず修理もできないということだ。」— ダグラス・アダムズ

レプリケーションとは、ネットワークで接続された複数のマシンに**同じデータのコピー**を保持すること。目的は以下の4つ：

| 目的 | 説明 |
|------|------|
| **低レイテンシ** | データをユーザーの近くに配置 |
| **高可用性** | 一部の障害でもシステムが稼働継続 |
| **読み取りスループット** | 読み取りクエリを処理するマシン数のスケールアウト |
| **オフライン動作** | ネットワーク切断時もアプリケーションが動作継続 |

本章では3つのレプリケーションアルゴリズムを議論：**シングルリーダー**、**マルチリーダー**、**リーダーレス**。

---

## 1. リーダーとフォロワー（Leader-based Replication）

```
クライアント → [書き込み] → リーダー → レプリケーションログ → フォロワー1
                                                              → フォロワー2
クライアント → [読み取り] → リーダー or フォロワー
```

- 書き込みはリーダーのみが受け付ける
- フォロワーはリーダーからの変更ストリームを適用（read-only）
- PostgreSQL, MySQL, MongoDB, Kafka, RabbitMQ等で使用

### 同期 vs 非同期レプリケーション

| 方式 | 利点 | 欠点 |
|------|------|------|
| **同期** | フォロワーのデータが常に最新 | フォロワー障害で書き込みブロック |
| **非同期** | リーダーが書き込み継続可能 | リーダー障害時にデータ損失の可能性 |
| **半同期** | 少なくとも2ノードに最新データ保証 | 実用的なバランス |

### 新しいフォロワーのセットアップ

1. リーダーのスナップショットを取得（ロック不要）
2. スナップショットを新しいフォロワーにコピー
3. スナップショット以降の変更をリーダーに要求（ログシーケンス番号 / binlog座標）
4. バックログ処理完了 → キャッチアップ完了

### ノード障害の処理

**フォロワー障害: キャッチアップリカバリ**
- ローカルログから最後に処理したトランザクションを特定 → リーダーから差分を取得

**リーダー障害: フェイルオーバー**
1. リーダーの障害検出（タイムアウト）
2. 新しいリーダーの選出（最も最新のデータを持つレプリカ）
3. システムの再構成

**フェイルオーバーの危険性:**
- 非同期レプリケーション → 未複製の書き込みが失われる
- GitHubの事例: 古いフォロワーが昇格 → 主キーの再利用 → MySQL/Redis間の不整合 → 個人データ漏洩
- **スプリットブレイン**: 2つのノードがリーダーだと認識 → データ損失・破損
- タイムアウトが短すぎると不要なフェイルオーバー

### レプリケーションログの実装方式

| 方式 | 説明 | 例 |
|------|------|-----|
| **ステートメントベース** | SQL文をそのまま転送 | MySQL（5.1以前） |
| **WALシッピング** | ストレージエンジンの書き込みログを転送 | PostgreSQL, Oracle |
| **論理（行ベース）ログ** | 行レベルの変更を論理的に記述 | MySQL binlog（行ベース） |
| **トリガーベース** | アプリケーション層でのレプリケーション | Oracle GoldenGate, Bucardo |

## 2. レプリケーションラグの問題

非同期レプリケーションでは、フォロワーが遅れると**結果整合性**（eventual consistency）の問題が発生。

### 3つの一貫性保証

| 保証 | 問題 | 解決策 |
|------|------|--------|
| **Read-after-write consistency** | 自分の書き込みが読めない | 自分が変更したデータはリーダーから読む、タイムスタンプ追跡 |
| **Monotonic reads** | 時間が戻ったように見える | 同じユーザーは常に同じレプリカから読む（ユーザーIDハッシュ） |
| **Consistent prefix reads** | 因果関係の逆転（答えが質問の前に見える） | 因果関係のある書き込みを同じパーティションに |

### 根本的な解決策
- アプリケーション層での対処は複雑でエラーを起こしやすい
- **トランザクション**がデータベースレベルでの解決策（Chapter 7, 9で詳述）

## 3. マルチリーダーレプリケーション

複数のノードが書き込みを受け付ける（master-master / active-active）。

### ユースケース

| ユースケース | 説明 |
|-------------|------|
| **マルチデータセンター** | 各データセンターにリーダー → 低レイテンシ、DC障害耐性、ネットワーク問題耐性 |
| **オフライン動作クライアント** | 各デバイスがローカルリーダー（カレンダーアプリ等） |
| **共同編集** | Google Docs等、各ユーザーのローカル変更を非同期レプリケーション |

### 書き込みコンフリクトの処理

```
ユーザー1: タイトル A → B  （リーダー1で成功）
ユーザー2: タイトル A → C  （リーダー2で成功）
→ 非同期レプリケーション時にコンフリクト検出
```

**コンフリクト解決戦略:**

| 戦略 | 説明 | 問題点 |
|------|------|--------|
| **コンフリクト回避** | 特定レコードの書き込みを常に同じリーダーへ | リーダー変更時に破綻 |
| **Last Write Wins (LWW)** | タイムスタンプが最も新しい書き込みを採用 | データ損失の危険 |
| **値のマージ** | アルファベット順に並べて連結等 | アプリ依存 |
| **カスタムロジック** | on write（書き込み時）/ on read（読み取り時）で解決 | 複雑 |

**自動コンフリクト解決の研究:**
- **CRDTs** (Conflict-free Replicated Datatypes) — Riak 2.0
- **Mergeable persistent data structures** — Git的な3-wayマージ
- **Operational transformation** — Google Docs, Etherpad

### レプリケーショントポロジー

```
円形（circular）    星形（star）      全対全（all-to-all）
  A → B              A               A ⇄ B
  ↑   ↓            / | \             ↕ ✕ ↕
  D ← C           B  C  D            D ⇄ C
```

- 円形/星形: 1ノード障害で全体に影響
- 全対全: 耐障害性は高いが、因果順序の問題（書き込みの追い越し）

## 4. リーダーレスレプリケーション

リーダーなし。クライアントが複数のレプリカに直接書き込み・読み取り。
Amazon Dynamoで有名に → Riak, Cassandra, Voldemort（Dynamoスタイル）。

### クォーラム（Quorum）

```
n = レプリカ総数
w = 書き込み成功に必要なノード数
r = 読み取りに必要なノード数

条件: w + r > n → 少なくとも1つの最新ノードから読める保証
```

| 設定 | 耐障害性 |
|------|----------|
| n=3, w=2, r=2 | 1ノード障害を許容 |
| n=5, w=3, r=3 | 2ノード障害を許容 |

### データ修復メカニズム

| メカニズム | 説明 |
|-----------|------|
| **Read repair** | 読み取り時に古いレプリカを検出 → 最新値を書き戻す |
| **Anti-entropy process** | バックグラウンドでレプリカ間の差分を検出・修正 |

### クォーラム一貫性の限界

w + r > n でも、以下の場合に古い値が返る可能性：
- **Sloppy quorum**: 書き込みノードと読み取りノードが異なる
- **並行書き込み**: 勝者の決定が不明確
- **書き込みと読み取りの並行**: 一部のレプリカにのみ反映
- **書き込み失敗の非ロールバック**: w未満の成功でもロールバックされない
- **ノード復旧時の古いデータ**: 最新ノード障害後に古いバックアップから復元

### Sloppy Quorum と Hinted Handoff

- **Sloppy quorum**: 本来のノードに到達できない場合、他のノードに一時的に書き込み
- **Hinted handoff**: ネットワーク復旧後、一時ノードから本来のノードにデータを転送
- 書き込み可用性は向上するが、読み取り一貫性の保証は弱まる

### 並行書き込みの検出

**Last Write Wins (LWW):**
- タイムスタンプで勝者を決定 → データ損失の危険
- Cassandraのデフォルト
- 安全な使い方: キーを一度だけ書き込み、以後はイミュータブル（UUIDをキーに）

**"happens-before" 関係と並行性:**
```
操作Aが操作Bの前に発生 ← BがAを知っている/依存している場合
並行（concurrent） ← どちらも相手を知らない場合
```

**バージョン番号によるアルゴリズム（ショッピングカートの例）:**
1. サーバーはキーごとにバージョン番号を管理
2. クライアントは読み取り時にバージョン番号を受信
3. 書き込み時に前回のバージョン番号を送信 + 受信した値をマージ
4. サーバーはバージョン番号で上書き/並行を判断

**バージョンベクトル（Version Vectors）:**
- 複数レプリカの場合、レプリカごと × キーごとにバージョン番号を管理
- Riak 2.0のdotted version vectors
- ベクタークロックとは微妙に異なる

## まとめ

```
レプリケーション方式
├── シングルリーダー — 理解しやすい、コンフリクト解決不要
│   └── 同期/非同期/半同期の選択
├── マルチリーダー — 耐障害性高い、コンフリクト解決が必要
│   └── ユースケース: マルチDC、オフライン、共同編集
└── リーダーレス — 耐障害性高い、弱い一貫性保証
    └── クォーラム（w + r > n）、バージョンベクトル

一貫性モデル
├── Read-after-write consistency（自分の書き込みを読める）
├── Monotonic reads（時間が戻らない）
└── Consistent prefix reads（因果順序が保たれる）
```

### 核心的な洞察
- レプリケーションの難しさは**変更データの処理**にある
- フェイルオーバーは想像以上に危険（スプリットブレイン、データ損失）
- 結果整合性は「いつか一致する」の「いつか」に上限がない
- **トランザクション**はこれらの問題に対するデータベースレベルの解決策
- 並行書き込みの検出には**バージョンベクトル**が正しいデータ構造
