# Chapter 6: Partitioning（パーティショニング）

## 概要

> 「順次処理から脱却し、コンピュータを制限しないようにしなければならない。」— グレース・マレー・ホッパー

Partitioning（パーティショニング）とは、大きなデータセットを小さなサブセットに分割し、異なるノードに配置すること。主な目的は**スケーラビリティ**。

| 用語（システム別） | システム |
|-------------------|---------|
| Partition | 本書の用語 |
| Shard | MongoDB, Elasticsearch, SolrCloud |
| Region | HBase |
| Tablet | Bigtable |
| vnode | Cassandra, Riak |
| vBucket | Couchbase |

**注意**: パーティショニングはレプリケーションと組み合わせて使用される。各パーティションのリーダーが1つのノードに割り当てられ、フォロワーが他のノードに割り当てられる。

---

## 1. キーバリューデータのパーティショニング

目標: データとクエリ負荷をノード間に**均等に**分散。不均等な分散をskew（偏り）と呼び、過度に高い負荷を持つパーティションをhot spot（ホットスポット）と呼ぶ。

### パーティショニング方式

| 方式 | 仕組み | 利点 | 欠点 |
|------|--------|------|------|
| **Key Range** | キーの連続範囲を各パーティションに割り当て | レンジスキャンが効率的 | ホットスポットのリスク（例: タイムスタンプキー） |
| **Hash of Key** | キーのハッシュ値の範囲を割り当て | 負荷の均等分散 | レンジクエリが非効率（全パーティションに送信） |

**Cassandraの妥協案**: 複合プライマリキー — 最初の部分をハッシュでパーティション、残りをSSTable内でソート順にインデックス。

```
例: (user_id, update_timestamp)
→ user_idでパーティション決定
→ 同一user_id内ではtimestampでソート
→ 特定ユーザーの特定期間の更新を効率的に取得可能
```

### ホットスポットの軽減

- 有名人のソーシャルメディア投稿 → 同じキーに大量の読み書き
- 対策: ホットキーに乱数を付加 → 100個のキーに分散
- 代償: 読み取り時に100キーすべてから読んで結合が必要

## 2. パーティショニングとセカンダリインデックス

| 方式 | 別名 | 書き込み | 読み取り |
|------|------|---------|---------|
| **Document-partitioned** | ローカルインデックス | 1パーティションのみ更新 | scatter/gather（全パーティションにクエリ） |
| **Term-partitioned** | グローバルインデックス | 複数パーティション更新の可能性 | 1パーティションから読み取り可能 |

```
Document-partitioned（ローカルインデックス）:
  Partition 0: {color:red → [doc1, doc3], color:yellow → [doc5]}
  Partition 1: {color:red → [doc7], color:blue → [doc8]}
  → "赤い車"の検索 → 両パーティションにクエリ必要

Term-partitioned（グローバルインデックス）:
  Index Partition 0 (a-r): {color:red → [doc1, doc3, doc7]}
  Index Partition 1 (s-z): {color:yellow → [doc5]}
  → "赤い車"の検索 → 1パーティションで完結
```

## 3. パーティションのリバランシング

ノードの追加・削除時にデータとリクエストを移動するプロセス。

### リバランシング戦略

| 戦略 | 説明 | 使用例 |
|------|------|--------|
| **hash mod N** ❌ | `hash(key) mod N` | **使ってはいけない** — N変更時にほぼ全データ移動 |
| **固定数パーティション** | 事前に多数のパーティションを作成（例: 10ノードに1000パーティション） | Riak, Elasticsearch, Couchbase, Voldemort |
| **動的パーティション** | パーティションが大きくなったら分割、小さくなったらマージ | HBase, RethinkDB, MongoDB |
| **ノード比例パーティション** | ノードあたりのパーティション数を固定 | Cassandra（デフォルト256/ノード）, Ketama |

### 自動 vs 手動リバランシング

- **完全自動**: 便利だが予測不能、過負荷ノードの障害検出と組み合わさると**カスケード障害**の危険
- **手動承認**: Couchbase, Riak, Voldemort（自動提案、管理者承認）
- → **人間の介在**があるほうが運用上の驚きを防げる

## 4. リクエストルーティング（サービスディスカバリ）

```
3つのアプローチ:

1. クライアント → 任意のノード → 正しいノードに転送
2. クライアント → ルーティング層 → 正しいノード
3. クライアント → 直接正しいノード（クライアントがパーティション情報を認識）
```

| 実装 | 方式 |
|------|------|
| **ZooKeeper使用** | HBase, Kafka, SolrCloud, Espresso（LinkedIn） |
| **独自config server** | MongoDB（mongos） |
| **Gossipプロトコル** | Cassandra, Riak（外部コーディネーション不要） |
| **ルーティング層（moxi）** | Couchbase |

### 並列クエリ実行

- MPP（超並列処理）データベース: 複雑なクエリをステージに分解して並列実行
- 分析ワークロードで特に有用（Chapter 10で詳述）

## まとめ

```
パーティショニング方式
├── Key Rangeパーティショニング
│   ├── レンジスキャン: 効率的
│   ├── ホットスポットリスク: 高い
│   └── リバランシング: 動的分割/マージ
├── Hashパーティショニング
│   ├── レンジスキャン: 非効率
│   ├── 負荷分散: 均等
│   └── リバランシング: 固定数パーティション
└── ハイブリッド（複合キー）
    └── 例: Cassandra (hash + sort)

セカンダリインデックス
├── ローカル（document-partitioned）→ 書き込み簡単、読み取りscatter/gather
└── グローバル（term-partitioned）→ 読み取り効率的、書き込み複雑

リバランシング
├── hash mod N → ❌ 使うな
├── 固定数パーティション → パーティションをノード間で移動
├── 動的パーティション → サイズに応じて分割/マージ
└── ノード比例 → ノードあたり固定数

リクエストルーティング
├── ZooKeeper等のコーディネーションサービス
├── Gossipプロトコル
└── ルーティング層/ロードバランサー
```

### 核心的な洞察
- パーティショニングの目標はホットスポットを避けた**均等な負荷分散**
- `hash mod N` は使ってはいけない — ノード数変更時にデータ大量移動
- セカンダリインデックスのパーティショニングは**ローカル vs グローバル**のトレードオフ
- 完全自動リバランシングは便利だが**カスケード障害**のリスク
- 各パーティションは独立して動作 → 複数パーティションにまたがる操作が課題（Chapter 7へ）
