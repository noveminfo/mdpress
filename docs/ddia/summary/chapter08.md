# Chapter 8: The Trouble with Distributed Systems（分散システムの問題）

## 概要

> 「ネットワーク遅延が大きい / でもここにデータがある / だから保存して」— Kyle Kingsbury

分散システムでは、単一マシンと異なり**部分的な障害（partial failure）** が発生する。非決定的で、成功か失敗かもわからない場合がある。本章は3つの問題カテゴリ — ネットワーク、クロック、プロセス停止 — を徹底的に悲観的に探索する。

---

## 1. 障害と部分的障害（Faults and Partial Failures）

| 環境 | 障害モデル | 対処法 |
|------|-----------|--------|
| **単一コンピュータ** | 完全動作 or 完全故障（deterministic） | クラッシュさせて再起動 |
| **スーパーコンピュータ（HPC）** | 部分障害 → 全体停止 | チェックポイント＋全クラスタ再起動 |
| **クラウド/インターネットサービス** | 部分障害を許容して継続稼働 | フォールトトレランスをソフトウェアに組み込む |

**核心原則**: 信頼できないコンポーネントから信頼できるシステムを構築する

```
例:
├── エラー訂正コード → ビットエラーのあるチャネルで正確なデータ転送
├── IP（非信頼）→ TCP（信頼性のある転送層）
└── ただし限界はある: 遅延は魔法では消せない
```

---

## 2. 信頼できないネットワーク（Unreliable Networks）

### リクエスト送信時に起きうること

```
1. リクエストが消失（ケーブル抜け等）
2. リクエストがキューで待機中（過負荷）
3. リモートノードが障害
4. リモートノードが一時停止（GC pause等）
5. リモートノードは処理したがレスポンスが消失
6. レスポンスが遅延

→ 送信者にはこれらを区別する方法がない
→ 唯一の手段: タイムアウト
```

### 実際のネットワーク障害

| 事実 | 詳細 |
|------|------|
| 中規模DC | 月に約12件のネットワーク障害 |
| EC2 | 一時的なネットワーク障害が頻発 |
| 冗長化の限界 | 人的ミス（設定ミス等）は冗長化では防げない |
| サメ | 海底ケーブルを噛む |

### タイムアウトと無制限の遅延

```
理想: 2d + r でタイムアウト設定可能（d=最大遅延、r=最大処理時間）
現実: 非同期ネットワークは無制限の遅延
      → タイムアウトは経験的に決定するしかない

遅延の原因（キューイング）:
├── ネットワークスイッチのキュー → 輻輳時にパケットドロップ
├── OS のCPUキュー → 過負荷時に任意の待ち時間
├── VMモニタのバッファリング → VM切り替え時のネットワークデータ蓄積
└── TCP のフロー制御 → 送信側でのキューイング
```

**適応型タイムアウト**: Phi Accrual failure detector（Akka, Cassandra）— 応答時間の分布を観測し動的に調整

### 同期 vs 非同期ネットワーク

| ネットワーク | 帯域確保 | 遅延 | 適用 |
|-------------|---------|------|------|
| **電話網（回線交換）** | 固定帯域を予約 | 有界（bounded） | 音声通話 |
| **インターネット（パケット交換）** | 動的共有 | 無界（unbounded） | バースト的トラフィック |

**なぜパケット交換か**: バースト的データ転送に最適化。回線交換は帯域を無駄にする。可変遅延は自然の法則ではなく、**コスト/利便性のトレードオフ**。

## 3. 信頼できないクロック（Unreliable Clocks）

### 2種類のクロック

| 種類 | 用途 | 特性 | 例 |
|------|------|------|----|
| **Time-of-day clock** | 現在日時 | NTP同期可能、逆行の可能性あり | `System.currentTimeMillis()` |
| **Monotonic clock** | 経過時間の測定 | 常に前進、ノード間比較不可 | `System.nanoTime()` |

### クロック同期の問題

```
問題一覧:
├── 水晶発振器のドリフト: 200 ppm → 30秒ごとの同期で6msのずれ
├── NTPサーバーとの同期: ネットワーク遅延に制限される（最良でも35ms誤差）
├── ファイアウォールによるNTPブロック: 気づかれずに長期間ドリフト
├── 閏秒: 大規模システムのクラッシュ原因に
├── VM環境: CPU共有による時間ジャンプ
└── ユーザーデバイス: 意図的な時刻改ざん
```

### イベント順序付けにクロックを使う危険性

```
例: LWW（Last Write Wins）
  Node 1: x = 1 @ 42.004s
  Node 3: x = 2 @ 42.003s（因果的にはx=1の後）
  → Node 2: x=1が「最新」と誤判定 → x=2が消失

問題:
├── データが静かに消失（エラー報告なし）
├── 順序的な書き込みと真に並行な書き込みを区別不能
└── 同一タイムスタンプの衝突

解決: Logical clocks（論理クロック）— カウンターベース、因果関係のみを追跡
```

### Google Spanner の TrueTime API

```
TrueTime: [earliest, latest] の信頼区間を返す
├── GPSレシーバーまたは原子時計を各DCに配備
├── 不確実性 ≈ 7ms
└── コミット前に信頼区間の長さだけ待機
    → 信頼区間が重ならない = 因果順序が確定
```

## 4. プロセス停止（Process Pauses）

### リース（lease）の危険性

```java
while (true) {
    request = getIncomingRequest();
    if (lease.expiryTimeMillis - System.currentTimeMillis() < 10000) {
        lease = lease.renew();
    }
    if (lease.isValid()) {
        process(request);  // ← ここで15秒停止したら？
    }
}
```

**問題**: `lease.isValid()` と `process(request)` の間に長い停止が入る可能性

### 停止の原因

| 原因 | 詳細 |
|------|------|
| **GC pause** | Stop-the-worldで数分停止の可能性 |
| **VM サスペンド** | ライブマイグレーション時に任意の期間停止 |
| **コンテキストスイッチ** | CPUスケジューリングによる任意の停止 |
| **同期ディスクI/O** | ネットワークファイルシステムの場合さらに悪化 |
| **スワッピング** | メモリ圧迫時のページフォールト → thrashing |
| **SIGSTOP** | Ctrl-Z等による明示的な停止 |

### リアルタイム保証

- **Hard real-time**: RTOS、航空機・ロケット等 — 非常に高コスト
- **一般的なサーバー**: リアルタイム保証は経済的に非現実的
- **GC影響の軽減策**: GC pauseを計画的な短期停止として扱い、その間は他のノードに振り向ける

## 5. 知識、真実、嘘（Knowledge, Truth, and Lies）

### 真実は多数決で決まる

```
シナリオ:
├── 非対称ネットワーク障害: 受信可能だが送信不可 → 他ノードに「死亡」宣告される
├── GC pause: 60秒停止 → 「死亡」宣告 → 復帰しても時すでに遅し
└── 教訓: 単一ノードの自己判断は信頼できない → quorum（多数決）が必要
```

**Quorum**: 過半数（majority）によるノード死亡判定。3ノードなら1障害、5ノードなら2障害まで許容。

### リーダーとロック

「自分がリーダーだ」と信じるノードが実は既に解任されている → **fencing token（フェンシングトークン）** で対策

```
Fencing tokens:
├── ロックサービスが単調増加するトークンを発行
├── Client 1: token 33を取得 → 長い停止 → lease失効
├── Client 2: token 34を取得 → storage serviceに書き込み
├── Client 1: 復帰 → token 33で書き込み試行
└── Storage service: token 34 > 33 → token 33の書き込みを拒否
    → ZooKeeper の zxid/cversion が使える
```

### Byzantine Faults（ビザンチン障害）

| 前提 | 説明 |
|------|------|
| **本書の前提** | ノードは誠実だが障害を起こしうる（non-Byzantine） |
| **Byzantine fault** | ノードが嘘をつく・偽メッセージを送る可能性 |
| **適用場面** | 航空宇宙（放射線）、ブロックチェーン（相互不信） |
| **通常のDC** | Byzantine対策は不要（コストが高すぎる） |

### システムモデル

| タイミングモデル | 仮定 |
|----------------|------|
| **Synchronous** | 遅延・停止・クロック誤差に上限あり（非現実的） |
| **Partially synchronous** | 通常は有界、時に無界（**最も現実的**） |
| **Asynchronous** | タイミング仮定なし（非常に制約的） |

| ノード障害モデル | 仮定 |
|-----------------|------|
| **Crash-stop** | クラッシュしたら永遠に戻らない |
| **Crash-recovery** | クラッシュ後に復帰可能（**最も現実的**） |
| **Byzantine** | 任意の（悪意ある）動作が可能 |

### Safety と Liveness

| 性質 | 定義 | 例 |
|------|------|----|
| **Safety** | 「悪いことが起きない」— 違反したら取り消せない | 一意性、単調増加 |
| **Liveness** | 「良いことがいずれ起きる」— 将来に希望あり | 可用性（応答が返る） |

**原則**: Safety は常に保持すべき（全ノードクラッシュでも）。Liveness は条件付き（過半数が生存し、ネットワークが回復すれば）。

## まとめ

```
分散システムの3大問題
├── ネットワーク
│   ├── パケット消失・遅延・重複
│   ├── レスポンスなし → 原因不明
│   ├── タイムアウト → 誤検知のリスク
│   └── 可変遅延 = コスト/利便性のトレードオフ
│
├── クロック
│   ├── Time-of-day: NTP同期、逆行の可能性
│   ├── Monotonic: 経過時間用、ノード間比較不可
│   ├── LWW はデータロスの危険
│   ├── Spanner TrueTime: 信頼区間ベース
│   └── Logical clocks が安全な代替手段
│
└── プロセス停止
    ├── GC, VM, ディスクI/O, スワップ等
    ├── 停止中は「死亡」宣告される可能性
    ├── リアルタイム保証は高コスト
    └── Fencing tokensで安全性確保

対処の原則
├── 部分障害は不可避 → ソフトウェアで許容
├── 障害検出はタイムアウト頼み → 誤検知を前提に設計
├── 単一ノードの判断を信じない → quorum（多数決）
├── Fencing tokensでstale leaderを防止
├── 最も現実的なモデル: Partially synchronous + Crash-recovery
└── Safety properties は常に保持、Liveness は条件付き
```

### 核心的な洞察

- 分散システムの問題は**ネットワーク・クロック・プロセス停止**の3つに集約される
- 「応答がない」とき、**ネットワーク障害かノード障害かを区別できない**
- クロックは**思ったほど信頼できない** — LWWは危険
- プロセスは**任意の時点で任意の期間停止**しうる
- 真実は**quorum（多数決）** で決まる — 単一ノードの自己判断は信用できない
- Fencing token で**古いリーダーのstale write**を防止
- 理論的にはこれらの問題は解決可能 → **Chapter 9でアルゴリズムを議論**
