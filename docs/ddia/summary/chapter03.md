# Chapter 3: Storage and Retrieval（ストレージと検索）

## 概要

> 「きちんと整理する人は、探すのが面倒なだけだ。」— ドイツのことわざ

データベースは根本的に2つのことを行う：**データを格納する**ことと、**後でそれを返す**こと。本章ではこれをデータベースの視点から議論し、ストレージエンジンの内部構造を探求する。

ストレージエンジンは大きく2つのカテゴリに分類される：

1. **OLTP（トランザクション処理）向け** — ログ構造化 vs ページ指向
2. **OLAP（分析処理）向け** — 列指向ストレージ

---

## 1. データベースを支えるデータ構造

### 最もシンプルなデータベース

```bash
db_set () { echo "$1,$2" >> database; }
db_get () { grep "^$1," database | sed -e "s/^$1,//" | tail -n 1; }
```

- `db_set`: O(1) — append-only（追記のみ）で非常に高速
- `db_get`: O(n) — ファイル全体をスキャンするため遅い
- **インデックス**が必要 → 読み取りを高速化するが、書き込みにオーバーヘッドが生じる

### 重要なトレードオフ
> 適切に選ばれたインデックスは読み取りを高速化するが、**すべてのインデックスは書き込みを遅くする**。

---

## 2. Hash Indexes（ハッシュインデックス）

- インメモリのハッシュマップで、キー → ファイル内のバイトオフセット をマッピング
- **Bitcask**（Riakのデフォルトエンジン）がこの方式を使用
- 各キーの値が頻繁に更新されるワークロードに適する

### セグメントとコンパクション

| 概念 | 説明 |
|------|------|
| セグメント | ログを一定サイズのファイルに分割 |
| コンパクション | 重複キーを捨て、最新の値のみ保持 |
| マージ | 複数のセグメントを結合して新しいファイルに書き込み |

### 実装上の考慮事項
- **ファイルフォーマット**: バイナリが効率的（CSVではなく）
- **削除**: tombstone（墓標）レコードを追記
- **クラッシュリカバリ**: ハッシュマップのスナップショットをディスクに保存
- **並行性制御**: 書き込みスレッドは1つ、読み取りは並行可能

### Append-onlyの利点
- シーケンシャル書き込みはランダム書き込みより高速
- 並行性とクラッシュリカバリがシンプル
- フラグメンテーションの問題を回避

### 制限
- すべてのキーがRAMに収まる必要がある
- 範囲クエリが効率的でない

## 3. SSTables and LSM-Trees

### SSTable（Sorted String Table）

ログセグメントのキーバリューペアを**キーでソート**して格納する方式。

#### SSTableの利点（ハッシュインデックスとの比較）
1. **マージが効率的** — mergesort風のアルゴリズムで、メモリより大きくてもOK
2. **疎なインデックスで十分** — すべてのキーをメモリに保持する必要がない
3. **ブロック圧縮が可能** — ディスクスペースとI/O帯域幅を節約

### SSTableの構築（memtable方式）

```
書き込み → memtable（インメモリ赤黒木/AVL木）
    ↓ 閾値到達
SSTableとしてディスクに書き出し
    ↓ バックグラウンド
マージ＆コンパクション
```

- クラッシュ対策: 別のappend-onlyログで書き込みを即座に記録

### LSM-Tree（Log-Structured Merge-Tree）

- LevelDB、RocksDB、Cassandra、HBase、Luceneなどで使用
- Patrick O'Neilらが提唱
- **Bloom filter（ブルームフィルタ）**: 存在しないキーの検索を高速化

#### コンパクション戦略

| 戦略 | 説明 | 使用例 |
|------|------|--------|
| Size-tiered | 新しく小さいSSTableを古く大きいSSTableにマージ | HBase |
| Leveled | キー範囲を小さなSSTableに分割、レベル別に管理 | LevelDB, RocksDB |

---

## 4. B-Trees（Bツリー）

最も広く使われているインデックス構造。1970年に導入。

### 基本設計

| 特性 | LSM-Tree | B-Tree |
|------|----------|--------|
| 単位 | 可変サイズのセグメント | 固定サイズのページ（通常4KB） |
| 書き込み | append-only | その場で上書き |
| 構造 | マージされるSSTableのカスケード | バランス木 |

- **分岐係数（branching factor）**: 通常数百 → 3〜4レベルで256TBを格納可能
- **ページ分割**: ページが満杯になると2つに分割

### 信頼性の確保
- **WAL（Write-Ahead Log）**: すべての変更をまずログに書き込み、クラッシュ時に復元
- **ラッチ（latch）**: 並行アクセス時のデータ一貫性を保護

### 最適化手法
- Copy-on-write方式（LMDB）
- キーの省略（内部ページ）
- リーフページの兄弟ポインタ
- フラクタルツリー

---

## 5. B-Tree vs LSM-Tree 比較

| 観点 | LSM-Tree | B-Tree |
|------|----------|--------|
| 書き込み | 一般的に高速（シーケンシャル） | Write amplification が大きい |
| 読み取り | 複数構造を確認（遅い場合あり） | 一般的に高速 |
| 圧縮 | より良く圧縮、低フラグメンテーション | ページ内に未使用スペースが残る |
| 予測可能性 | コンパクションが干渉する可能性 | より予測可能なレスポンス |
| トランザクション | キーが複数箇所に存在しうる | 各キーが1箇所（ロックに適する） |

## 6. その他のインデックス構造

### セカンダリインデックス
- キーが一意でない → 行識別子のリスト or キーに識別子を付加

### 値の格納方法

| 方式 | 説明 |
|------|------|
| Heap file | インデックスはオフセットへの参照のみ |
| Clustered index | インデックス内にデータ本体を格納（InnoDB主キー） |
| Covering index | 一部の列をインデックスに含める |

### 複合インデックス
- **Concatenated index**: フィールドを連結して1つのキーに（電話帳方式）
- **Multi-dimensional index**: R-tree等を使った空間クエリ（地理情報、色検索など）

### 全文検索とあいまいインデックス
- 編集距離（edit distance）による類似検索
- Lucene: 有限状態オートマトン → レーベンシュタインオートマトン

### インメモリデータベース
- RAM価格の低下 → すべてをメモリに保持することが現実的に
- **パフォーマンスの理由**: ディスクI/O回避ではなく、ディスク用エンコードのオーバーヘッド回避
- 例: VoltDB, MemSQL, Redis, RAMCloud
- Anti-caching: メモリ不足時にLRUデータをディスクに退避

## 7. トランザクション処理か分析か？（OLTP vs OLAP）

| 特性 | OLTP | OLAP |
|------|------|------|
| 読み取りパターン | キーで少数レコード取得 | 大量レコードの集計 |
| 書き込みパターン | ユーザー入力、低レイテンシ | バルクインポート（ETL） |
| 使用者 | エンドユーザー | ビジネスアナリスト |
| データの意味 | 最新状態 | イベントの履歴 |
| サイズ | GB〜TB | TB〜PB |

### Data Warehousing（データウェアハウジング）

```
OLTP DB1 ─┐
OLTP DB2 ─┤─→ ETL（Extract-Transform-Load）─→ Data Warehouse ─→ 分析クエリ
OLTP DB3 ─┘
```

- OLTPシステムへの負荷を避け、分析専用DBで自由にクエリ
- 分析に最適化されたストレージエンジンを使用可能

### Star Schema（スタースキーマ）

```
          dim_date
              ↑
dim_store ← fact_sales → dim_product
              ↓
          dim_customer
```

- **Fact table（ファクトテーブル）**: イベント（購入等）を記録、非常に大きい
- **Dimension table（ディメンションテーブル）**: 誰・何・どこ・いつ・なぜを表現
- **Snowflake schema**: ディメンションをさらにサブディメンションに分解（より正規化）

## 8. Column-Oriented Storage（列指向ストレージ）

### 基本アイデア
- 行指向: 1行のすべての値をまとめて格納
- **列指向: 各列のすべての値をまとめて格納** → 必要な列だけ読み取れる

### Column Compression（列の圧縮）

#### Bitmap Encoding（ビットマップエンコーディング）
- 各異なる値に対して1つのビットマップ（行ごとに1ビット）
- 疎なビットマップにはランレングスエンコーディングを適用

#### クエリへの応用
- `WHERE x IN (30, 68, 69)` → 3つのビットマップのbitwise OR
- `WHERE x = 31 AND y = 3` → 2つのビットマップのbitwise AND

### Vectorized Processing（ベクタ化処理）
- 圧縮データをL1キャッシュに収め、タイトなループで処理
- SIMD命令の活用で高速化

### ソート順序
- 行全体でソート（列を独立にソートしてはいけない）
- 最初のソートキーの圧縮効果が最も高い
- **複数のソート順序**: レプリカごとに異なるソートで格納（C-Store/Vertica）

### 列指向ストレージへの書き込み
- B-tree的なin-place更新は不可能
- **LSM-tree方式**: memtable → ディスク上の列ファイルとマージ

### Materialized Views & Data Cubes
- **Materialized view**: クエリ結果の実コピーをディスクに保存
- **Data cube (OLAP cube)**: ディメンションごとの集計をグリッドで事前計算
- 高速だが柔軟性に欠ける → 生データの保持が依然重要

## まとめ：ストレージエンジンの全体像

```
ストレージエンジン
├── OLTP向け
│   ├── ログ構造化（Log-structured）
│   │   ├── Hash Index（Bitcask）
│   │   └── LSM-Tree（LevelDB, RocksDB, Cassandra）
│   │       ├── memtable + SSTable
│   │       ├── Bloom filter
│   │       └── コンパクション戦略（size-tiered / leveled）
│   └── ページ指向（Page-oriented）
│       └── B-Tree
│           ├── WAL（クラッシュリカバリ）
│           ├── ラッチ（並行性制御）
│           └── 最適化（copy-on-write, キー省略等）
└── OLAP向け
    └── 列指向ストレージ
        ├── 列の圧縮（ビットマップ、ランレングス）
        ├── ベクタ化処理
        ├── ソート順序の最適化
        └── マテリアライズドビュー / データキューブ
```

### 核心的な洞察
- **ログ構造化**: ランダム書き込み → シーケンシャル書き込みに変換 → 高い書き込みスループット
- **B-Tree**: 各キーが1箇所 → トランザクション分離に適する、読み取りが予測可能
- **列指向**: 必要な列だけ読む → 分析クエリで圧倒的に高速
- ストレージエンジンの内部を理解すれば、アプリケーションに最適なツールを選択でき、チューニングパラメータの効果を理解できる
