# Chapter 4: Encoding and Evolution（エンコーディングと進化）

## 概要

> 「すべては変わり、何も留まらない。」— ヘラクレイトス

アプリケーションは時間とともに必ず変化する。データフォーマットやスキーマが変更されるとき、新旧のコードとデータが**同時に共存**できる必要がある。本章では、データをバイト列に変換する**エンコーディング方式**と、プロセス間の**データフローのモード**を議論する。

### 互換性の2方向

| 種類 | 意味 | 難易度 |
|------|------|--------|
| **Backward compatibility**（後方互換性） | 新しいコードが古いデータを読める | 比較的容易 |
| **Forward compatibility**（前方互換性） | 古いコードが新しいデータを読める | より困難 |

---

## 1. データのエンコーディングフォーマット

### メモリとディスク/ネットワークの2つの表現

```
メモリ上: オブジェクト、構造体、ハッシュテーブル、ポインタ
   ↕ encoding（serialization, marshalling）/ decoding（parsing, deserialization）
バイト列: ファイル、ネットワーク送信用の自己完結的なシーケンス
```

### 言語固有のフォーマット

Java `Serializable`、Ruby `Marshal`、Python `pickle` など。

**問題点:**
- 特定のプログラミング言語に縛られる
- 任意のクラスをインスタンス化 → **セキュリティリスク**
- バージョニング・互換性が考慮されていない
- パフォーマンスが悪いケースがある

→ **一般的に悪い選択**。一時的な用途以外では使わないこと。

### JSON, XML, CSV

テキスト形式で人間に読みやすいが、いくつかの問題がある：

| 問題 | 詳細 |
|------|------|
| 数値のあいまいさ | XMLでは数値と数字文字列を区別不能。JSONでは整数と浮動小数点の区別なし。2^53を超える整数の精度問題（Twitter IDの例） |
| バイナリ文字列非対応 | Base64エンコードで回避（データサイズ33%増加） |
| スキーマの複雑さ | XML Schema/JSON Schemaは強力だが複雑。CSVにはスキーマなし |

→ 組織間のデータ交換には十分。内部用途にはバイナリ形式を検討。

### バイナリエンコーディング

#### MessagePack（JSON用バイナリ）

Example 4-1のレコード:
```json
{"userName": "Martin", "favoriteNumber": 1337, "interests": ["daydreaming", "hacking"]}
```

| フォーマット | サイズ |
|-------------|--------|
| JSON（テキスト） | 81 bytes |
| MessagePack | 66 bytes |
| Thrift BinaryProtocol | 59 bytes |
| Thrift CompactProtocol | 34 bytes |
| Protocol Buffers | 33 bytes |
| **Avro** | **32 bytes** |

## 2. Thrift and Protocol Buffers

スキーマを定義し、フィールド名の代わりに**フィールドタグ番号**を使用。

```
Thrift IDL:                          Protocol Buffers:
struct Person {                       message Person {
  1: required string userName,          required string user_name = 1;
  2: optional i64 favoriteNumber,       optional int64 favorite_number = 2;
  3: optional list<string> interests    repeated string interests = 3;
}                                     }
```

### スキーマ進化のルール

| 操作 | 前方互換性 | 後方互換性 |
|------|-----------|-----------|
| フィールド追加（optional/default付き） | OK（古いコードは無視） | OK（新しいコードはデフォルト使用） |
| フィールド追加（required） | OK | **NG**（古いデータにフィールドなし） |
| フィールド削除（optional） | 後方互換性と逆 | 前方互換性と逆 |
| タグ番号の変更 | **NG** | **NG** |
| フィールド名の変更 | OK | OK |

**重要**: フィールドタグ番号は絶対に変更してはならない。名前は変更可能。

## 3. Avro

Hadoop用に2009年に開発。**タグ番号を使わない**独自のアプローチ。

```
Avro IDL:
record Person {
    string               userName;
    union { null, long }  favoriteNumber = null;
    array<string>         interests;
}
```

### Writer's Schema と Reader's Schema

- エンコード時: writer's schema を使用
- デコード時: reader's schema を使用
- **2つのスキーマが同一である必要はない** — 互換性があればよい
- フィールドは**名前で**マッチング（タグ番号ではない）

### スキーマ進化ルール
- デフォルト値のあるフィールドのみ追加・削除可能
- `null` をデフォルトにするには `union { null, T }` が必要
- フィールド名の変更はエイリアスで対応（後方互換のみ）

### Writer's Schema の伝達方法

| コンテキスト | 方法 |
|-------------|------|
| 大量レコードのファイル | ファイル先頭にスキーマを1回だけ含める |
| DBの個別レコード | レコードにバージョン番号を含め、DBからスキーマを取得 |
| ネットワーク通信 | 接続確立時にスキーマバージョンをネゴシエーション |

### Avroの利点: 動的スキーマ生成
- RDBのテーブルからAvroスキーマを自動生成可能
- スキーマ変更時にタグ番号の手動割り当てが不要
- Thrift/Protocol Buffersでは手動対応が必要

## 4. スキーマベースのエンコーディングの利点

1. **コンパクト** — フィールド名を省略できる
2. **ドキュメントとしてのスキーマ** — デコードに必須なので常に最新
3. **互換性チェック** — デプロイ前にスキーマの互換性を検証可能
4. **コード生成** — 静的型付け言語でのコンパイル時型チェック

## 5. データフローのモード

データは3つの主要な方法でプロセス間を流れる：

### 5.1 データベース経由のデータフロー

```
書き込みプロセス → encode → DB → decode → 読み取りプロセス
（writer）                                    （reader）
```

- 「未来の自分」へのメッセージ送信と考えられる
- **前方互換性も後方互換性も必要**（新旧のコードが同時にアクセス）
- 未知のフィールドの保持が重要（古いコードが新しいフィールドを消さないように）
- **"data outlives code"**（データはコードより長生きする）— 5年前のデータが元のエンコーディングのまま存在

### 5.2 サービス経由のデータフロー: REST and RPC

#### サービスとマイクロサービス
- サーバーがAPIを公開、クライアントがリクエスト
- SOA → マイクロサービスアーキテクチャへの進化
- 各サービスは独立してデプロイ・進化可能

#### REST vs SOAP

| 特性 | REST | SOAP |
|------|------|------|
| 哲学 | HTTPの原則を活用 | HTTPから独立しようとする |
| データフォーマット | JSON（主に） | XML |
| API記述 | OpenAPI/Swagger | WSDL |
| 使いやすさ | シンプル、curlで実験可能 | ツール依存、手動構築困難 |
| 普及度 | パブリックAPIで主流 | 大企業の既存システム |

#### RPCの問題点

ローカル関数呼び出しとネットワークリクエストの根本的な違い：

1. **予測不可能性** — ネットワーク問題でリクエスト/レスポンスが消失
2. **タイムアウト** — 結果なしで返る可能性（成功/失敗が不明）
3. **べき等性** — リトライで重複実行の可能性
4. **レイテンシの変動** — ミリ秒から数秒まで変動
5. **パラメータのエンコード** — ポインタは送れない
6. **言語間の型変換** — 異なる言語間のデータ型の不一致

#### 現代のRPCフレームワーク
- gRPC（Protocol Buffers）、Thrift、Avro RPC、Finagle、Rest.li
- futures/promises で非同期処理を明示化
- gRPCはストリームをサポート
- サービスディスカバリ機能

#### RPCの互換性
- サーバーを先に更新、クライアントを後に更新と仮定
- **リクエスト: 後方互換性**が必要
- **レスポンス: 前方互換性**が必要
- APIバージョニング: URLにバージョン番号、HTTPヘッダー、APIキーとの紐付け

### 5.3 メッセージパッシングによるデータフロー

RPCとデータベースの中間に位置する。

#### メッセージブローカーの利点
1. 受信者が利用不可でもバッファとして機能
2. クラッシュしたプロセスへの自動再配信
3. 送信者が受信者のIPアドレス/ポートを知る必要がない
4. 1つのメッセージを複数の受信者に送信可能
5. 送信者と受信者の論理的なデカップリング

#### メッセージブローカー
- RabbitMQ、ActiveMQ、NATS、Apache Kafka
- プロデューサー → queue/topic → コンシューマー
- 通常は一方向通信（非同期）

#### 分散アクターフレームワーク

| フレームワーク | メッセージエンコーディング |
|---------------|------------------------|
| **Akka** | デフォルトはJavaシリアライゼーション。Protocol Buffersに置換可能 |
| **Orleans** | デフォルトは独自フォーマット（ローリングアップグレード非対応）。カスタムプラグイン可能 |
| **Erlang OTP** | レコードスキーマの変更が驚くほど困難 |

## まとめ

```
エンコーディングフォーマット
├── 言語固有（Java Serializable等）→ 避けるべき
├── テキスト形式（JSON, XML, CSV）→ 組織間交換に適する
└── バイナリスキーマ駆動
    ├── Thrift — フィールドタグ + BinaryProtocol/CompactProtocol
    ├── Protocol Buffers — フィールドタグ + 可変長整数
    └── Avro — タグ番号なし、writer/readerスキーマ方式

データフローのモード
├── データベース経由 — 書き込みがencode、読み取りがdecode
├── RPC/REST API — クライアントがrequest encode、サーバーがresponse encode
└── メッセージパッシング — 送信者がencode、受信者がdecode
```

### 核心的な洞察
- **ローリングアップグレード**を可能にするために、前方・後方互換性が不可欠
- スキーマベースのバイナリエンコーディングは、schema-on-readと同等の柔軟性を持ちながら、より強い保証を提供
- RPCはローカル関数呼び出しと根本的に異なる — この違いを隠そうとすべきでない
- **"data outlives code"** — データの互換性は長期的視点で考える必要がある
