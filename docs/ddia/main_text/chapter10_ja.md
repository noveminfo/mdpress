# Chapter 10: Batch Processing（バッチ処理）

> *「システムが1人の人間に強く影響されすぎると成功しえない。初期設計が完成し十分に堅牢になったとき、多くの異なる視点を持つ人々がそれぞれの実験を始めるところから、真のテストが始まる。」*
> — Donald Knuth

本書の最初の2部では、リクエストとクエリ、そしてそれに対応するレスポンスや結果について多く語った。このデータ処理スタイルは多くの現代的なデータシステムで想定されている: 何かを要求するか命令を送ると、しばらくしてシステムが（うまくいけば）答えを返す。データベース、キャッシュ、検索インデックス、ウェブサーバー、その他多くのシステムがこの方式で動作する。

このようなオンラインシステムでは、ウェブブラウザがページをリクエストするにせよ、サービスがリモートAPIを呼び出すにせよ、リクエストは人間のユーザーによってトリガーされ、ユーザーはレスポンスを待っていると一般に想定する。あまり長く待たせるべきではないので、これらのシステムの応答時間には多くの注意を払う。

ウェブと増加するHTTP/RESTベースのAPIにより、リクエスト/レスポンス型のインタラクションは非常に一般的になり、当然のことと思いがちだ。しかし、それがシステムを構築する唯一の方法ではなく、他のアプローチにもメリットがあることを忘れてはならない。3種類の異なるシステムを区別しよう:

**サービス（オンラインシステム）**: クライアントからのリクエストまたは命令の到着を待つ。受信すると、できるだけ早く処理してレスポンスを返す。応答時間が通常の主要な性能指標であり、可用性も非常に重要だ。

**バッチ処理システム（オフラインシステム）**: 大量の入力データを受け取り、ジョブを実行して処理し、出力データを生成する。ジョブはしばらくかかる（数分から数日）ので、通常はユーザーがジョブの完了を待っていない。バッチジョブは定期的に実行されることが多い（例えば1日1回）。バッチジョブの主要な性能指標は通常スループット（ある大きさの入力データセットを処理するのにかかる時間）だ。

**ストリーム処理システム（準リアルタイムシステム）**: オンラインとオフライン/バッチ処理の中間に位置する（ニアリアルタイムまたはニアライン処理とも呼ばれる）。バッチ処理システムのように入力を消費し出力を生成するが、ストリームジョブはイベントが発生した直後に操作する。バッチジョブは固定された入力データセットに対して操作する。この違いにより、ストリーム処理システムは同等のバッチシステムより低いレイテンシを持つことができる。

本章で見るように、バッチ処理は信頼性が高く、スケーラブルで、保守しやすいアプリケーションを構築する上で重要な構成要素だ。例えば、2004年に発表されたバッチ処理アルゴリズム **MapReduce** は、（やや熱狂的に）「Google を大規模にスケーラブルにするアルゴリズム」と呼ばれた。その後、Hadoop、CouchDB、MongoDB を含むさまざまなオープンソースデータシステムに実装された。

MapReduce は、何年も前にデータウェアハウス向けに開発された並列処理システムと比較するとかなり低レベルなプログラミングモデルだが、コモディティハードウェアで達成可能な処理規模の面で大きな前進だった。MapReduce の重要性は現在低下しているが、それでも理解する価値がある。バッチ処理がなぜ、どのように有用かの明確な図を提供してくれるからだ。

実際、バッチ処理は非常に古い形式のコンピューティングだ。プログラマブルなデジタルコンピュータが発明されるはるか前から、1890年の米国国勢調査で使用された Hollerith マシンのようなパンチカード集計機が、大量の入力から集計統計を計算する半機械化された形式のバッチ処理を実装していた。

## Unixツールによるバッチ処理（Batch Processing with Unix Tools）

### 簡単なログ分析（Simple Log Analysis）

ウェブサーバーがリクエストを処理するたびにログファイルに1行を追加する例を考える。nginx のデフォルトアクセスログ形式では:

```
216.58.210.78 - - [27/Feb/2015:17:55:11 +0000] "GET /css/typography.css HTTP/1.1"
200 3377 "http://martin.kleppmann.com/" "Mozilla/5.0 ..."
```

ウェブサイトで最も人気のある5つのページを見つけるには、Unix シェルで以下のようにする:

```bash
cat /var/log/nginx/access.log |
  awk '{print $7}' |
  sort            |
  uniq -c         |
  sort -r -n      |
  head -n 5
```

awk で各行から7番目のフィールド（リクエストURL）を抽出し、sort でアルファベット順にソートし、uniq -c で重複をカウントし、sort -r -n でカウント降順にソートし、head -n 5 で上位5件を出力する。

このコマンドチェーンは不慣れな人には不明瞭に見えるかもしれないが、非常に強力だ。ギガバイトのログファイルを数秒で処理でき、分析を簡単に修正できる。

### コマンドチェーン vs カスタムプログラム

同じことを Ruby で書くこともできる:

```ruby
counts = Hash.new(0)
File.open('/var/log/nginx/access.log') do |file|
  file.each do |line|
    url = line.split[6]
    counts[url] += 1
  end
end
top5 = counts.map{|url, count| [count, url] }.sort.reverse[0...5]
top5.each{|count, url| puts "#{count} #{url}" }
```

Ruby スクリプトはインメモリのハッシュテーブルに URL のカウントを保持する。Unixパイプラインはハッシュテーブルを持たず、代わりにURL リストのソートに依存する。

どちらのアプローチが優れているかは、異なるURLの数に依存する。ワーキングセットがメモリに収まるなら、インメモリハッシュテーブルが適切だ。ワーキングセットが利用可能なメモリより大きい場合、ソートベースのアプローチが有利だ — チャンクをメモリ内でソートしてセグメントファイルとしてディスクに書き出し、複数のソート済みセグメントをマージできる。GNU Coreutils の sort ユーティリティは、メモリ超過データセットを自動的にディスクにスピルし、複数CPUコアでソートを自動並列化する。

### Unix哲学（The Unix Philosophy）

Doug McIlroy（Unix パイプの発明者）が1964年にパイプを最初に記述した: 「庭のホースのようにプログラムを接続する方法があるべきだ — データを別の方法でマッサージする必要があるときに別のセグメントをねじ込む。」

Unix 哲学は1978年に以下のように記述された:

1. 各プログラムは1つのことをうまくやる。新しい仕事には、古いプログラムに新しい「機能」を追加して複雑にするのではなく、新たに作る。
2. すべてのプログラムの出力が、まだ未知の別のプログラムの入力になることを期待する。
3. ソフトウェア（OSですら）を早期に試すべく設計・構築する。できれば数週間以内に。
4. プログラミングタスクを軽減するために、未熟な助けよりツールを使う。

このアプローチ — 自動化、ラピッドプロトタイピング、インクリメンタルイテレーション、実験への友好性、大きなプロジェクトの管理可能なチャンクへの分割 — は、今日のアジャイルやDevOps ムーブメントに驚くほど似ている。

**統一インターフェース**: Unixでは、そのインターフェースがファイル（正確にはファイルディスクリプタ）だ。ファイルは単なるバイトの順序列。ファイルシステム上の実際のファイル、別のプロセスへの通信チャネル（Unixソケット、stdin、stdout）、デバイスドライバ、TCP 接続を表すソケットなど、非常に異なるものが同じインターフェースを共有できる。

**ロジックとワイヤリングの分離**: プログラムが stdin と stdout を使えば、シェルユーザーが入出力を好きなように接続できる。プログラムは入力がどこから来ているか、出力がどこに行くかを知る必要がない。これは疎結合、遅延バインディング、制御の反転の一形態と言える。

**透明性と実験容易性**: Unixコマンドへの入力ファイルは通常イミュータブルとして扱われ、コマンドを何度でも実行できる。パイプラインの任意の時点で出力を less に渡して確認できる。中間段階の出力をファイルに書き込み、後の段階の入力として使える。

しかし、Unixツールの最大の限界は単一マシンでしか動作しないことだ — そこが Hadoop のようなツールの出番だ。

## MapReduce と分散ファイルシステム（MapReduce and Distributed Filesystems）

MapReduce は Unix ツールに少し似ているが、潜在的に数千台のマシンに分散している。Unixツールと同様に、かなり鈍器的で力任せだが、驚くほど効果的なツールだ。単一の MapReduce ジョブは単一の Unix プロセスに匹敵する: 1つ以上の入力を受け取り、1つ以上の出力を生成する。

Unixツールが stdin と stdout を入出力として使うのに対し、MapReduce ジョブは分散ファイルシステム上のファイルを読み書きする。Hadoop の MapReduce 実装では、そのファイルシステムは **HDFS（Hadoop Distributed File System）** と呼ばれ、Google File System (GFS) のオープンソース再実装だ。

HDFS は shared-nothing 原則に基づく（NAS や SAN の shared-disk アプローチとは対照的）。HDFS は各マシンでデーモンプロセスが動作し、他のノードがそのマシンに保存されたファイルにアクセスできるネットワークサービスを公開する。**NameNode** と呼ばれる中央サーバーがどのファイルブロックがどのマシンに保存されているかを追跡する。マシンとディスクの障害に耐えるため、ファイルブロックは複数のマシンにレプリケートされる。

HDFS は十分にスケールしている: 最大のHDFSデプロイメントは数万台のマシン上で動作し、合計ストレージ容量は数百ペタバイトに達する。

### MapReduce ジョブの実行（MapReduce Job Execution）

MapReduce は分散ファイルシステム上の大規模データセットを処理するコードを書くためのプログラミングフレームワークだ。データ処理のパターンはログ分析の例と非常に類似している:

1. 入力ファイルのセットを読み取り、レコードに分割する。
2. **mapper** 関数を呼び出して各入力レコードからキーと値を抽出する。
3. すべてのキーバリューペアをキーでソートする。
4. **reducer** 関数を呼び出してソート済みのキーバリューペアをイテレートする。

ステップ2（map）と4（reduce）がカスタムデータ処理コードを書く箇所だ。ステップ3のソートは MapReduce では暗黙的 — 書く必要はなく、mapper の出力は reducer に渡される前に常にソートされる。

**Mapper**: 各入力レコードに対して1回呼ばれ、キーと値を抽出する。各入力に対して任意の数のキーバリューペアを生成できる（ゼロも含む）。1つの入力レコードから次への状態は保持しないので、各レコードは独立して処理される。

**Reducer**: MapReduce フレームワークがmapper によって生成されたキーバリューペアを取り、同じキーに属するすべての値を集め、その値コレクションのイテレータ付きで reducer を呼び出す。reducer は出力レコードを生成できる。

### 分散実行（Distributed Execution of MapReduce）

Unixコマンドパイプラインとの主な違いは、MapReduce が多数のマシンにわたって計算を並列化でき、並列性を明示的に扱うコードを書く必要がないことだ。mapper と reducer は一度に1つのレコードだけを操作し、入力がどこから来ているか、出力がどこに行くかを知る必要がない。

MapReduce の並列化は**パーティショニング**に基づく: ジョブへの入力は通常 HDFS 上のディレクトリであり、入力ディレクトリ内の各ファイルまたはファイルブロックが別々のパーティションとして、別々の map タスクで処理できると見なされる。

MapReduce スケジューラは、入力ファイルのレプリカを保存しているマシンの1つで各 mapper を実行しようとする。この原則は**データの近くで計算する（putting the computation near the data）** として知られ、入力ファイルのネットワーク越しコピーを節約する。

reduce 側の計算もパーティション分割される。map タスクの数は入力ファイルブロック数で決まるが、reduce タスクの数はジョブ作成者が設定する。同じキーを持つすべてのキーバリューペアが同じ reducer に届くように、フレームワークはキーのハッシュを使ってどの reduce タスクが特定のキーバリューペアを受け取るべきかを決定する。

mapper の出力を reducer ごとにパーティション分割し、ソートし、データパーティションを mapper から reducer にコピーするプロセスは**シャッフル（shuffle）** として知られる。

### MapReduce ワークフロー（MapReduce Workflows）

単一の MapReduce ジョブで解ける問題の範囲は限られている。そのため、MapReduce ジョブをワークフローに連鎖させ、1つのジョブの出力が次のジョブの入力になるようにすることが非常に一般的だ。Hadoop MapReduce フレームワークにはワークフローの特別なサポートがないため、この連鎖はディレクトリ名で暗黙的に行われる。

バッチジョブの出力はジョブが成功裏に完了したときにのみ有効と見なされる（MapReduce は失敗したジョブの部分的出力を破棄する）。ジョブ実行間の依存関係を処理するため、Hadoop 向けのさまざまなワークフロースケジューラが開発された: Oozie、Azkaban、Luigi、Airflow、Pinball など。推薦システムを構築する際には50から100のMapReduceジョブで構成されるワークフローが一般的だ。

Pig、Hive、Cascading、Crunch、FlumeJava などの高レベルツールも、自動的に適切に接続された複数のMapReduceステージのワークフローを設定する。

## Reduce-Side Joins と Grouping

バッチ処理の文脈で joins について語るとき、データセット内のある関連性のすべての出現を解決することを意味する。

### ユーザーアクティビティイベントの分析例

典型的な例: ユーザーのアクティビティイベントのログ（ファクトテーブル）とユーザーデータベース（ディメンション）の結合。アクティビティイベントにはユーザーIDのみ含まれ、完全なプロファイル情報はない。

最も単純な実装は各ユーザーIDに対してリモートのユーザーデータベースにクエリすることだが、性能が非常に悪い。より良いアプローチは、ユーザーデータベースのコピー（ETLプロセスで抽出）をユーザーアクティビティイベントのログと同じ分散ファイルシステムに配置し、MapReduce を使って関連するすべてのレコードを同じ場所に集めて効率的に処理することだ。

### Sort-Merge Join

mapper の目的は各入力レコードからキーと値を抽出すること。1セットの mapper がアクティビティイベントを処理し（ユーザーIDをキー、イベントを値として抽出）、もう1セットの mapper がユーザーデータベースを処理する（ユーザーIDをキー、生年月日を値として抽出）。

MapReduce フレームワークが mapper の出力をキーでパーティション分割しソートすると、同じユーザーIDを持つすべてのアクティビティイベントとユーザーレコードが reducer の入力で隣接する。MapReduce ジョブは、reducer が常にユーザーデータベースのレコードを最初に見て、その後にアクティビティイベントをタイムスタンプ順に見るようにレコードのソートを調整することもできる — この技法は**セカンダリソート（secondary sort）** として知られる。

reducer は実際の結合ロジックを簡単に実行できる: ユーザーIDごとに1回呼ばれ、セカンダリソートにより最初の値はユーザーデータベースからの生年月日レコードだと期待される。reducer は生年月日をローカル変数に保存し、同じユーザーIDのアクティビティイベントをイテレートする。

このアルゴリズムは **sort-merge join** として知られる。mapper の出力がキーでソートされ、reducer が結合の両側からのソート済みレコードリストをマージするからだ。

### 関連データを同じ場所に集める

sort-merge join では、mapper とソートプロセスにより、特定のユーザーIDの結合操作に必要なすべてのデータが同じ場所（reducer の単一呼び出し）に集められる。すべての必要なデータが事前に整列されているため、reducer は高スループットと低メモリオーバーヘッドでレコードを処理できるシンプルな単一スレッドのコードでよい。

mapper が reducer に「メッセージを送る」と考えることもできる: mapper がキーバリューペアを発行すると、キーは値が配信されるべき宛先アドレスのように機能する。

MapReduce プログラミングモデルは、計算の物理的なネットワーク通信の側面をアプリケーションロジックから分離する。MapReduce がすべてのネットワーク通信を処理するため、アプリケーションコードは部分的障害を心配する必要がない: MapReduce は失敗したタスクをアプリケーションロジックに影響を与えずに透過的にリトライする。

### GROUP BY

結合以外に、「関連データを同じ場所に集める」パターンの一般的な使用法は、あるキーでレコードをグルーピングすること（SQL の GROUP BY 句のように）だ。典型的なユースケース: レコード数のカウント（COUNT(*)）、特定フィールドの合計（SUM）、ランキング関数による上位k件の選択。

もう1つの一般的な用途は、特定のユーザーセッションのすべてのアクティビティイベントを照合すること — **セッション化（sessionization）** と呼ばれるプロセスだ。

### スキューの処理（Handling Skew）

「同じキーを持つすべてのレコードを同じ場所に集める」パターンは、単一のキーに非常に大量のデータが関連付けられている場合に破綻する。ソーシャルネットワークでは、ほとんどのユーザーが数百人程度と繋がっているが、少数のセレブリティは数百万のフォロワーを持つことがある。このような不均衡にアクティブなデータベースレコードは**リンチピンオブジェクト（linchpin objects）** や**ホットキー（hot keys）** として知られる。

対策としていくつかのアルゴリズムがある: Pig の skewed join（サンプリングでホットキーを特定し、ランダムに複数 reducer に分散）、Crunch の sharded join（ホットキーを明示的に指定）、Hive の skewed join optimization（ホットキーのレコードを別ファイルに保存し、map-side join を使用）。グルーピングと集約の場合は2段階で行える: 第1段階でランダム reducer に送って部分集約、第2段階で最終集約。

## Map-Side Joins

前節の結合アルゴリズムは reducer で実際の結合ロジックを実行するため、**reduce-side join** として知られる。reduce-side アプローチは入力データについて仮定を置く必要がないという利点があるが、すべてのソート、reducer へのコピー、reducer 入力のマージがかなり高コストになりうるという欠点がある。

入力データについて特定の仮定を置ける場合、**map-side join** を使ってより高速に結合できる。このアプローチでは reducer もソートもない cut-down MapReduce ジョブを使用する。

### Broadcast Hash Join

大きなデータセットと小さなデータセットの結合で最もシンプル。小さなデータセットが各 mapper のメモリに完全にロードできる必要がある。各 mapper が起動時に小さな入力を分散ファイルシステムからインメモリハッシュテーブルに読み込み、大きな入力のレコードをスキャンしながらハッシュテーブルで検索する。Pig では "replicated join"、Hive では "MapJoin" として知られる。

### Partitioned Hash Join

結合の両入力が同じ方法でパーティション分割されている場合、各パーティションに対して独立にハッシュ結合を適用できる。各 mapper はそのパーティションの入力のみを読めばよいため、ハッシュテーブルに少量のデータをロードするだけで済む。Hive では bucketed map joins として知られる。

### Map-Side Merge Join

入力データセットが同じ方法でパーティション分割されているだけでなく、同じキーでソートもされている場合に適用できる。この場合、入力がメモリに収まるかどうかは関係ない。mapper が通常 reducer が行うのと同じマージ操作を実行できる: 両方の入力ファイルをキーの昇順で段階的に読み、同じキーのレコードをマッチングする。

## バッチワークフローの出力（The Output of Batch Workflows）

バッチ処理はトランザクション処理でも分析でもない。分析に近いが、MapReduce ジョブのワークフローは分析目的のSQL クエリと同じではない。バッチプロセスの出力はレポートではなく、何らかの別の構造であることが多い。

### 検索インデックスの構築

Google の MapReduce の元来の使用目的は検索エンジンのインデックス構築で、5〜10 の MapReduce ジョブのワークフローとして実装されていた。mapper がドキュメントセットを必要に応じてパーティション分割し、各 reducer がそのパーティションのインデックスを構築し、インデックスファイルが分散ファイルシステムに書き込まれる。

### Key-Valueストアとしてのバッチプロセス出力

バッチ処理のもう1つの一般的な用途は、機械学習システム（分類器、推薦システム等）の構築だ。これらのバッチジョブの出力は通常何らかのデータベースだ。

最も明白な選択は mapper や reducer 内からデータベースに直接書き込むことだが、これは悪いアイデアだ:
- 各レコードごとのネットワークリクエストはバッチタスクの通常のスループットより桁違いに遅い
- すべての mapper/reducer が同時に同じ出力データベースに書き込むと、データベースが過負荷になりうる
- MapReduce の all-or-nothing 保証が失われる（外部への副作用は隠せない）

はるかに良い解決策は、バッチジョブ内でまったく新しいデータベースを構築し、分散ファイルシステム上のジョブの出力ディレクトリにファイルとして書き出すことだ。これらのデータファイルは一度書き込まれたらイミュータブルで、読み取り専用クエリを処理するサーバーにバルクロードできる。Voldemort、Terrapin、ElephantDB、HBase bulk loading がこの方式をサポートする。

### バッチプロセス出力の哲学

Unix哲学の継承: プログラムは入力を読み出力を書く。入力は変更されず、以前の出力は新しい出力で完全に置き換えられ、他の副作用はない。これにより:
- バグを導入した場合、コードの前のバージョンにロールバックしてジョブを再実行するだけ（**human fault tolerance**）
- ロールバックの容易さにより、機能開発をより速く進められる（不可逆性の最小化）
- map/reduce タスクの失敗時、MapReduce フレームワークが自動的にリスケジュールしてリトライ（入力がイミュータブルで失敗タスクの出力が破棄されるため安全）
- 同じファイルセットを複数の異なるジョブの入力として使える

## Hadoop と分散データベースの比較（Comparing Hadoop to Distributed Databases）

Hadoop は分散版の Unix のようなものだ。HDFS がファイルシステムで、MapReduce は Unix プロセスの風変わりな実装（map フェーズと reduce フェーズの間に常に sort ユーティリティを実行する）だ。

MapReduce 論文が発表されたとき、それはある意味でまったく新しくなかった。本章で議論したすべての処理と並列結合アルゴリズムは、10年以上前に **MPP（Massively Parallel Processing）データベース** で既に実装されていた。

**ストレージの多様性**: データベースはデータを特定のモデル（リレーショナルやドキュメント）に構造化することを要求するが、分散ファイルシステム上のファイルは任意のバイト列だ。Hadoop はデータをHDFSに無差別にダンプし、後で処理方法を考える可能性を開いた。これは**データレイク（data lake）** として知られる概念で、「寿司の原則（sushi principle）」— 「生データの方が良い（raw data is better）」— と呼ばれることもある。

**処理モデルの多様性**: MPP データベースはモノリシックで緊密に統合されたソフトウェアだが、Hadoop プラットフォームのオープン性により、さまざまなアプローチの実装が可能になった。OLTP データベース（HBase）、MPPスタイルの分析データベース（Impala）、MapReduce など、非常に異なるアプローチが同じクラスタ上で共存できる。

**頻繁な障害に対する設計**: MPP データベースはクエリ実行中にノードがクラッシュするとクエリ全体をアボートして再実行する。MapReduce はタスク単位で障害に耐える。Google の環境では、1時間実行の MapReduce タスクは高優先度プロセスにリソースを譲るために約5%の確率でプリエンプトされる。100タスク×10分のジョブでは、少なくとも1タスクが完了前に停止する確率は50%以上だ。MapReduce がタスクの頻繁な予期しない終了に耐えるよう設計されたのは、ハードウェアが特に信頼できないからではなく、プロセスを任意に終了できる自由がコンピューティングクラスタのリソース利用率を向上させるからだ。

## MapReduce を超えて（Beyond MapReduce）

MapReduce は有用な学習ツールだが、分散システムの多くの可能なプログラミングモデルの1つに過ぎない。

### 中間状態のマテリアライゼーション（Materialization of Intermediate State）

1つのジョブの出力が別のジョブへの入力としてのみ使われる場合、分散ファイルシステム上のファイルは単なる**中間状態（intermediate state）** だ。この中間状態をファイルに書き出すプロセスを**マテリアライゼーション（materialization）** と呼ぶ。

MapReduce の中間状態の完全マテリアライゼーションには欠点がある:
- 先行ジョブの全タスクが完了するまで次のジョブを開始できない（stragglerがボトルネック）
- mapper がしばしば冗長（前の reducer の出力をそのまま読んで次のパーティション/ソートに備えるだけ）
- 中間状態を分散ファイルシステムに保存すると、ファイルが複数ノードにレプリケートされる（一時データには過剰）

### データフローエンジン（Dataflow Engines）

これらの問題を解決するため、**Spark**、**Tez**、**Flink** などの新しい実行エンジンが開発された。共通点: ワークフロー全体を独立したサブジョブに分割するのではなく、1つのジョブとして扱う。

これらのシステムは**データフローエンジン**として知られ、複数の処理ステージを通じたデータの流れを明示的にモデル化する。MapReduce のように1レコードずつユーザー定義関数を呼び出すが、map と reduce の厳密な交互実行の役割ではなく、より柔軟な方法で**オペレータ（operators）** を組み立てられる:

- リパーティション＋ソート（sort-merge join 用）
- リパーティションのみ（partitioned hash join 用）
- ブロードキャスト（broadcast hash join 用）

利点:
- ソートは本当に必要な場所でのみ実行
- 不要な map タスクの排除
- ジョイン・データ依存関係の明示的宣言 → ローカリティ最適化
- 中間状態をメモリまたはローカルディスクに保持（HDFSレプリケーション不要）
- オペレータは入力準備次第で即座に実行開始
- JVMプロセスの再利用（起動オーバーヘッド削減）

**障害耐性**: Spark は **RDD（Resilient Distributed Dataset）** 抽象でデータの系譜を追跡し、Flink はオペレータ状態のチェックポイントを取る。再計算にはオペレータの決定論性が重要だ — 非決定論的な振る舞い（ハッシュテーブルのイテレーション順序、乱数、システムクロック等）は排除する必要がある。

### グラフとイテレーティブ処理（Graphs and Iterative Processing）

グラフ全体に対してオフライン処理や分析を行う必要がある場合 — 例えば機械学習や **PageRank** のようなランキングアルゴリズム — MapReduce では「完了まで繰り返す」を表現できない（1パスしか実行しないため）。外部スケジューラで反復的にバッチプロセスを実行する方法は機能するが、MapReduce は反復的な性質を考慮しないため非効率だ。

**Pregel 処理モデル**: バッチ処理グラフの最適化として、**BSP（Bulk Synchronous Parallel）** 計算モデルが普及した。Apache Giraph、Spark の GraphX API、Flink の Gelly API で実装されている。

Pregel では、1つの頂点が別の頂点にメッセージを送信でき、典型的にはグラフのエッジに沿って送信される。各イテレーションで関数が各頂点に対して呼び出され、送信されたすべてのメッセージが渡される。MapReduce との違いは、Pregel モデルでは頂点がイテレーション間でメモリに状態を保持すること。メッセージがグラフのある部分で送信されていなければ、作業は不要だ。

障害耐性はイテレーションの終わりに全頂点の状態を定期的にチェックポイントすることで達成される。

注意点: グラフがメモリに収まる単一コンピュータ上では、単一マシン（場合によっては単一スレッド）のアルゴリズムが分散バッチプロセスを上回る可能性が高い。

### 高レベル API と言語（High-Level APIs and Languages）

Hive、Pig、Cascading、Crunch、Spark DataFrame、Flink などの高レベル言語と API は、リレーショナルスタイルのビルディングブロック（結合、グルーピング、フィルタリング、集約）で計算を表現する。

**宣言的クエリ言語への移行**: 結合をリレーショナルオペレータとして指定すると、フレームワークが結合入力のプロパティを分析し、どの結合アルゴリズムが最も適切かを自動的に決定できる。Hive、Spark、Flink にはこれを行えるコストベースのクエリオプティマイザがあり、結合の順序を変更して中間状態を最小化することもできる。

同時に、MapReduce の遺産であるバッチ処理システムは任意のコードを容易に実行する自由を保持している。パーシング、自然言語分析、画像分析、数値・統計アルゴリズムなど、既存の大規模なライブラリエコシステムを活用できる。

宣言的な側面を高レベル API に組み込み、実行中にクエリオプティマイザがそれらを活用することで、バッチ処理フレームワークは MPP データベースに似たものになり始めている（同等の性能を達成できる）。同時に、任意のコードを実行し任意のフォーマットのデータを読む拡張性により、柔軟性の優位性を維持している。

## まとめ（Summary）

本章ではバッチ処理のトピックを探索した。awk、grep、sort などの Unix ツールから出発し、それらのツールの設計哲学が MapReduce やより新しいデータフローエンジンにどのように引き継がれているかを見た。設計原則のいくつか: 入力はイミュータブル、出力はまだ未知の別のプログラムへの入力になることを意図、複雑な問題は「1つのことをうまくやる」小さなツールの組み合わせで解決。

Unix の世界では、あるプログラムを別のプログラムと組み合わせることを可能にする統一インターフェースがファイルとパイプだ。MapReduce では、そのインターフェースが分散ファイルシステムだ。データフローエンジンは中間状態を分散ファイルシステムにマテリアライズすることを避けるために独自のパイプ的データ転送メカニズムを追加するが、ジョブの初期入力と最終出力は依然としてHDFSであることが多い。

分散バッチ処理フレームワークが解決すべき2つの主要な問題:

**パーティショニング**: MapReduce では、mapper は入力ファイルブロックに従ってパーティション分割される。mapper の出力はリパーティション、ソート、マージされて設定可能な数の reducer パーティションになる。このプロセスの目的は、関連するすべてのデータ — 例えば同じキーを持つすべてのレコード — を同じ場所に集めることだ。

**障害耐性**: MapReduce は頻繁にディスクに書き込むため、ジョブ全体を再起動せずに個々の失敗タスクから簡単に回復できるが、障害のない場合の実行が遅くなる。データフローエンジンは中間状態のマテリアライゼーションを減らしメモリにより多く保持するため、ノード障害時により多くのデータを再計算する必要がある。決定論的なオペレータが再計算の必要量を削減する。

分散バッチ処理エンジンは意図的に制限されたプログラミングモデルを持つ: コールバック関数（mapper や reducer）はステートレスで、指定された出力以外に外部から見える副作用を持たないと仮定される。この制限により、クラッシュやネットワーク問題に直面しても、タスクを安全にリトライでき、失敗タスクからの出力は破棄される。

バッチ処理ジョブの特徴は、入力データを読んで出力データを生成し、入力を変更しないことだ — つまり、出力は入力から**導出（derived）** される。決定的に重要なのは、入力データが**有界（bounded）** であること: 既知の固定サイズを持つ。有界であるため、ジョブは入力全体の読み取りをいつ終えたかを知り、処理が完了するとジョブは終了する。

次章ではストリーム処理に転じる。そこでは入力が**無界（unbounded）** — つまり、ジョブはあるが、入力は終わりのないデータストリームだ。ジョブは決して完了しない。なぜなら、いつでもまだ新しい作業が来る可能性があるからだ。
