# Chapter 11: Stream Processing（ストリーム処理）

> 動作する複雑なシステム（complex system）は、必ず動作する単純なシステム（simple system）から発展したものである。逆の命題もまた真であるように見える：ゼロから設計された複雑なシステムは決して動作せず、動作させることもできない。
> — John Gall, *Systemantics* (1975)

Chapter 10 ではバッチ処理（batch processing）について議論した — 一連のファイルを入力として読み込み、新しい出力ファイルを生成する技法である。出力は導出データ（derived data）の一形態であり、必要であればバッチ処理を再度実行することで再作成できるデータセットである。検索インデックス、レコメンデーションシステム、分析などの構築にこの単純だが強力なアイデアがどう使えるかを見てきた。

しかし、Chapter 10 を通じて1つの大きな前提が残っていた：すなわち、入力が**有界（bounded）** — つまり既知で有限のサイズ — であり、バッチ処理は入力の読み取りがいつ完了したかを知ることができるという前提である。例えば、MapReduce の中核であるソート操作は、出力の生成を開始する前に入力全体を読み取る必要がある：最後の入力レコードが最も小さいキーを持つかもしれず、したがって最初の出力レコードとなるべきものかもしれないため、早期に出力を開始することはできない。

現実には、多くのデータは**無界（unbounded）** である。なぜなら、データは時間の経過とともに徐々に到着するからだ：ユーザーは昨日も今日もデータを生成し、明日も引き続き生成するだろう。事業を停止しない限り、このプロセスは決して終わらず、データセットは意味のある形で「完全」になることはない [1]。したがって、バッチプロセッサはデータを固定期間のチャンク（chunk）に人為的に分割しなければならない：例えば、毎日の終わりに1日分のデータを処理する、あるいは毎時間の終わりに1時間分のデータを処理する、といった具合である。

日次バッチ処理の問題は、入力の変更が出力に反映されるのが1日遅れになることであり、多くのせっかちなユーザーにとっては遅すぎる。遅延を減らすために、処理をより頻繁に実行できる — 例えば毎秒の終わりに1秒分のデータを処理する — あるいはさらに進んで、固定の時間区切りを完全に放棄し、イベントが発生するたびに処理を行うこともできる。これが**ストリーム処理（stream processing）** の背後にあるアイデアである。

一般に「ストリーム（stream）」とは、時間の経過とともにインクリメンタルに利用可能になるデータを指す。このコンセプトは多くの場面に登場する：Unix の stdin と stdout、プログラミング言語（遅延リスト）[2]、ファイルシステム API（Java の FileInputStream など）、TCP 接続、インターネット上での音声・映像配信など。

本章では、イベントストリーム（event stream）をデータ管理メカニズムとして見ていく：前章で見たバッチデータに対する、無界でインクリメンタルに処理される対応物である。まずストリームがどのように表現、保存、ネットワーク上で転送されるかを議論する。「データベースとストリーム」ではストリームとデータベースの関係を調査する。そして最後に「ストリームの処理」では、それらのストリームを継続的に処理するアプローチとツール、そしてアプリケーション構築への活用方法を探る。

## イベントストリームの伝送（Transmitting Event Streams）

バッチ処理の世界では、ジョブの入出力はファイル（おそらく分散ファイルシステム上の）である。ストリーミングにおける同等のものはどのようなものか？

入力がファイル（バイト列）の場合、最初の処理ステップは通常、それをレコードの列にパースすることである。ストリーム処理の文脈では、レコードは一般に**イベント（event）** と呼ばれるが、本質的に同じものである：ある時点で何かが起きたことの詳細を含む、小さく自己完結的で不変のオブジェクトである。イベントには通常、それがいつ発生したかを示すタイムスタンプ（timestamp）が含まれる（「単調クロック対実時間クロック」288ページ参照）。

例えば、起きたことはユーザーが取ったアクション（ページの閲覧や購入など）かもしれない。温度センサーからの定期的な測定値やCPU使用率メトリクスのように、マシンから発生するものかもしれない。「Unix ツールによるバッチ処理」（391ページ）の例では、Web サーバーログの各行が1つのイベントである。

イベントはテキスト文字列、JSON、あるいは Chapter 4 で議論したようなバイナリ形式でエンコードできる。このエンコーディングにより、例えばファイルへの追記、リレーショナルテーブルへの挿入、ドキュメントデータベースへの書き込みなどの形でイベントを保存できる。また、処理のためにネットワーク経由で別のノードにイベントを送信することもできる。

バッチ処理では、ファイルは一度書き込まれ、その後複数のジョブによって読み取られる可能性がある。同様に、ストリーミング用語では、イベントは**プロデューサー（producer）**（パブリッシャーまたは送信者とも呼ばれる）によって一度生成され、その後複数の**コンシューマー（consumer）**（サブスクライバーまたは受信者）によって処理される可能性がある [3]。ファイルシステムではファイル名が関連レコードの集合を識別する；ストリーミングシステムでは、関連イベントは通常**トピック（topic）** またはストリーム（stream）にグループ化される。

原理的には、プロデューサーとコンシューマーを接続するにはファイルやデータベースで十分である：プロデューサーは生成するすべてのイベントをデータストアに書き込み、各コンシューマーは定期的にデータストアをポーリング（poll）して、前回実行以降に出現したイベントを確認する。これは本質的にバッチ処理が毎日の終わりに1日分のデータを処理するときに行っていることと同じである。

しかし、低遅延での継続的処理に向かう場合、データストアがこの種の使用法向けに設計されていなければ、ポーリングはコストが高くなる。ポーリング頻度を上げるほど、新しいイベントを返すリクエストの割合が低くなり、したがってオーバーヘッドが大きくなる。代わりに、新しいイベントが出現したときにコンシューマーに通知する方が良い。

データベースはこの種の通知メカニズムを伝統的にあまりうまくサポートしてこなかった：リレーショナルデータベースは一般にトリガー（trigger）を持っており、変更（例えばテーブルへの行の挿入）に反応できるが、できることが非常に限られており、データベース設計の中ではある種の後付けであった [4, 5]。代わりに、イベント通知の配信を目的とした専用ツールが開発されてきた。

### メッセージングシステム（Messaging Systems）

新しいイベントをコンシューマーに通知するための一般的なアプローチは、メッセージングシステム（messaging system）を使用することである：プロデューサーがイベントを含むメッセージを送信し、それがコンシューマーにプッシュされる。これらのシステムについては「メッセージパッシングデータフロー」（136ページ）で以前触れたが、ここではより詳細に掘り下げる。

Unix パイプや TCP 接続のようなプロデューサーとコンシューマー間の直接通信チャネルは、メッセージングシステムを実装するシンプルな方法であろう。しかし、ほとんどのメッセージングシステムはこの基本モデルを拡張している。特に、Unix パイプと TCP は正確に1人の送信者と1人の受信者を接続するのに対し、メッセージングシステムは複数のプロデューサーノードが同じトピックにメッセージを送信し、複数のコンシューマーノードがトピック内のメッセージを受信することを可能にする。

このパブリッシュ/サブスクライブ（publish/subscribe）モデルの中で、異なるシステムは幅広いアプローチを取っており、すべての目的に対して唯一の正解はない。システムを区別するために、以下の2つの問いが特に有用である：

1. **プロデューサーがコンシューマーの処理速度を超えてメッセージを送信した場合、何が起きるか？** 大まかに言って3つの選択肢がある：システムはメッセージをドロップ（drop）するか、キュー（queue）にバッファリングするか、バックプレッシャー（backpressure、フロー制御とも呼ばれる；つまりプロデューサーのさらなるメッセージ送信をブロックする）を適用するかである。例えば、Unix パイプと TCP はバックプレッシャーを使用する：小さな固定サイズのバッファがあり、それが満杯になると、受信者がバッファからデータを取り出すまで送信者はブロックされる（「ネットワークの輻輳とキューイング」282ページ参照）。メッセージがキューにバッファリングされる場合、そのキューが成長するにつれて何が起きるかを理解することが重要である。キューがメモリに収まらなくなったらシステムはクラッシュするのか、それともメッセージをディスクに書き込むのか？もしそうなら、ディスクアクセスはメッセージングシステムのパフォーマンスにどう影響するか？ [6]

2. **ノードがクラッシュしたり一時的にオフラインになった場合、メッセージは失われるか？** データベースと同様に、耐久性（durability）にはディスクへの書き込みやレプリケーション（replication）の何らかの組み合わせが必要であり（「レプリケーションと耐久性」のサイドバー227ページ参照）、これにはコストが伴う。時にメッセージを失うことが許容できるなら、同じハードウェアでより高いスループットとより低いレイテンシを得ることができるだろう。

メッセージ損失が許容可能かどうかは、アプリケーションに大きく依存する。例えば、定期的に送信されるセンサー読み取り値やメトリクスの場合、時折のデータポイントの欠落はおそらく重要ではない。更新された値がすぐ後に送信されるからである。しかし、大量のメッセージがドロップされると、メトリクスが不正確であることがすぐには明らかにならない可能性があるので注意が必要である [7]。イベントをカウントしている場合は、確実に配信されることがより重要である。なぜなら、失われたメッセージはそのまま不正確なカウンターを意味するからである。

Chapter 10 で探究したバッチ処理システムの優れた特性は、強力な信頼性保証を提供することであった：失敗したタスクは自動的にリトライされ、失敗したタスクの部分出力は自動的に破棄される。これは、あたかも障害が発生しなかったかのような出力を意味し、プログラミングモデルの簡素化に役立つ。本章の後半では、ストリーミングの文脈でどのように同様の保証を提供できるかを検討する。

### プロデューサーからコンシューマーへの直接メッセージング（Direct messaging from producers to consumers）

いくつかのメッセージングシステムは、仲介ノードを経由せずにプロデューサーとコンシューマー間の直接ネットワーク通信を使用する：

- **UDP マルチキャスト（UDP multicast）** は金融業界で株式市場フィードなどのストリームに広く使用されている。低レイテンシが重要な場面である [8]。UDP 自体は信頼性がないが、アプリケーションレベルのプロトコルで失われたパケットを回復できる（プロデューサーは送信したパケットを記憶し、要求に応じて再送信できなければならない）。
- **ZeroMQ** [9] や **nanomsg** などのブローカーレスメッセージングライブラリは、TCP または IP マルチキャスト上でパブリッシュ/サブスクライブメッセージングを実装する、同様のアプローチを取る。
- **StatsD** [10] や **Brubeck** [7] は、ネットワーク上のすべてのマシンからメトリクスを収集し監視するために、信頼性のない UDP メッセージングを使用する。（StatsD プロトコルでは、カウンターメトリクスはすべてのメッセージが受信された場合にのみ正確である；UDP の使用によりメトリクスはせいぜい近似値となる [11]。）
- コンシューマーがネットワーク上でサービスを公開している場合、プロデューサーは直接 HTTP または RPC リクエストを行って（「サービスを介したデータフロー：REST と RPC」131ページ参照）、メッセージをコンシューマーにプッシュできる。これは**Webhook** [12] の背後にあるアイデアであり、あるサービスのコールバック URL が別のサービスに登録され、イベントが発生するたびにその URL にリクエストを行うパターンである。

これらの直接メッセージングシステムは、設計対象の状況ではうまく動作するが、一般的にメッセージ損失の可能性をアプリケーションコードが認識している必要がある。許容できる障害はかなり限定的である：たとえプロトコルがネットワークで失われたパケットを検出し再送信できるとしても、一般にプロデューサーとコンシューマーが常時オンラインであることを前提としている。

コンシューマーがオフラインの場合、到達不能な間に送信されたメッセージを見逃す可能性がある。一部のプロトコルはプロデューサーに失敗したメッセージ配信のリトライを許可するが、プロデューサーがクラッシュしてリトライすべきだったメッセージのバッファを失った場合、このアプローチは破綻する可能性がある。

### メッセージブローカー（Message brokers）

広く使用されている代替案は、**メッセージブローカー（message broker）**（メッセージキューとも呼ばれる）を介してメッセージを送信することであり、これは本質的にメッセージストリームの処理に最適化されたデータベースの一種である [13]。サーバーとして動作し、プロデューサーとコンシューマーがクライアントとしてそれに接続する。プロデューサーはブローカーにメッセージを書き込み、コンシューマーはブローカーからそれらを読み取って受信する。

ブローカーにデータを集中させることで、これらのシステムは接続・切断・クラッシュするクライアントをより容易に許容でき、耐久性の問題はブローカーに移される。一部のメッセージブローカーはメッセージをメモリにのみ保持し、他のブローカーは（設定に応じて）ブローカーのクラッシュ時に失われないようにディスクに書き込む。遅いコンシューマーに対しては、一般に無制限のキューイングを許可する（メッセージのドロップやバックプレッシャーとは対照的に）が、この選択も設定に依存する場合がある。

キューイングの結果として、コンシューマーは一般に非同期（asynchronous）である：プロデューサーがメッセージを送信するとき、通常はブローカーがメッセージをバッファリングしたことの確認を待つだけで、メッセージがコンシューマーによって処理されるのを待たない。コンシューマーへの配信は不確定な将来の時点で行われる — 多くの場合1秒未満だが、キューのバックログがある場合はかなり遅くなることもある。

### メッセージブローカーとデータベースの比較（Message brokers compared to databases）

一部のメッセージブローカーは XA や JTA を使用した2相コミットプロトコル（two-phase commit protocol）にも参加できる（「分散トランザクションの実践」360ページ参照）。この機能によりデータベースと非常に似た性質を持つが、メッセージブローカーとデータベースの間にはなお重要な実用上の違いがある：

- **データベース**は通常、明示的に削除されるまでデータを保持する。一方、ほとんどの**メッセージブローカー**はメッセージがコンシューマーに正常に配信されると自動的に削除する。このようなメッセージブローカーは長期データストレージには適さない。
- メッセージを素早く削除するため、ほとんどのメッセージブローカーはワーキングセット（working set）がかなり小さい — つまりキューが短いことを前提としている。ブローカーがコンシューマーの遅さのために多くのメッセージをバッファリングする必要がある場合（メモリに収まらなくなるとディスクにスピルする可能性がある）、各メッセージの処理にかかる時間が長くなり、全体的なスループットが低下する可能性がある [6]。
- データベースはセカンダリインデックス（secondary index）やデータを検索するための様々な方法をサポートすることが多い。一方、メッセージブローカーは特定のパターンに一致するトピックのサブセットにサブスクライブする方法をサポートすることが多い。メカニズムは異なるが、どちらも本質的にはクライアントが知りたいデータの部分を選択する方法である。
- データベースへのクエリでは、結果は通常データのある時点のスナップショット（point-in-time snapshot）に基づく；別のクライアントがその後データベースに書き込みを行ってクエリ結果を変更しても、最初のクライアントはその以前の結果が古くなったことを知らない（クエリを繰り返すか変更をポーリングしない限り）。対照的に、メッセージブローカーは任意のクエリをサポートしないが、データが変更されたとき（つまり新しいメッセージが利用可能になったとき）にクライアントに通知する。

これはメッセージブローカーの伝統的な見方であり、**JMS** [14] や **AMQP** [15] などの標準に組み込まれ、RabbitMQ、ActiveMQ、HornetQ、Qpid、TIBCO Enterprise Message Service、IBM MQ、Azure Service Bus、Google Cloud Pub/Sub [16] などのソフトウェアに実装されている。

### 複数のコンシューマー（Multiple consumers）

複数のコンシューマーが同じトピックのメッセージを読む場合、2つの主要なメッセージングパターンが使用される：

**ロードバランシング（Load balancing）**
各メッセージは1つのコンシューマーに配信され、コンシューマーはトピック内のメッセージ処理の作業を共有できる。ブローカーはメッセージを任意にコンシューマーに割り当てることができる。このパターンは、メッセージの処理コストが高く、処理を並列化するためにコンシューマーを追加したい場合に有用である。（AMQP では、同じキューから消費する複数のクライアントを持つことでロードバランシングを実装でき、JMS ではこれを共有サブスクリプション（shared subscription）と呼ぶ。）

**ファンアウト（Fan-out）**
各メッセージはすべてのコンシューマーに配信される。ファンアウトにより、複数の独立したコンシューマーがそれぞれ同じメッセージのブロードキャストに「チューニング」でき、互いに影響を与えない — 同じ入力ファイルを読む複数の異なるバッチジョブがあるのと同様のストリーミング版である。（この機能は JMS のトピックサブスクリプション、AMQP のエクスチェンジバインディングで提供される。）

2つのパターンは組み合わせることができる：例えば、2つの別々のコンシューマーグループがそれぞれトピックにサブスクライブし、各グループが全体としてすべてのメッセージを受信するが、各グループ内では1つのノードだけが各メッセージを受信する。

### 確認応答と再配信（Acknowledgments and redelivery）

コンシューマーはいつでもクラッシュする可能性があるため、ブローカーがメッセージをコンシューマーに配信したが、コンシューマーがそれを処理しなかった、あるいはクラッシュ前に部分的にしか処理しなかったということが起こりうる。メッセージが失われないことを保証するために、メッセージブローカーは**確認応答（acknowledgment）** を使用する：クライアントはメッセージの処理が完了したことをブローカーに明示的に伝え、それによりブローカーはキューからメッセージを削除できる。

クライアントへの接続がブローカーが確認応答を受信する前にクローズまたはタイムアウトした場合、メッセージは処理されなかったと見なし、別のコンシューマーに再配信する。（実際にはメッセージが完全に処理されたが確認応答がネットワークで失われた可能性もあることに注意。このケースの処理にはアトミックコミットプロトコル（atomic commit protocol）が必要であり、「分散トランザクションの実践」360ページで議論した。）

ロードバランシングと組み合わせると、この再配信動作はメッセージの順序に興味深い影響を与える。コンシューマーは一般にプロデューサーが送信した順序でメッセージを処理する。しかし、コンシューマー2がメッセージ m3 を処理中にクラッシュし、同時にコンシューマー1がメッセージ m4 を処理しているとする。未確認のメッセージ m3 はその後コンシューマー1に再配信され、結果としてコンシューマー1はメッセージを m4, m3, m5 の順序で処理する。したがって、m3 と m4 はプロデューサー1が送信した順序とは異なる順序で配信される。

メッセージブローカーがメッセージの順序を保持しようとしても（JMS と AMQP の両標準で要求される通り）、ロードバランシングと再配信の組み合わせは必然的にメッセージの再順序付けにつながる。この問題を回避するには、コンシューマーごとに別々のキューを使用する（つまりロードバランシング機能を使用しない）ことができる。メッセージが互いに完全に独立している場合、メッセージの再順序付けは問題ではないが、メッセージ間に因果的依存関係がある場合は重要になりうる。これについては本章の後半で見ていく。

## パーティション化ログ（Partitioned Logs）

ネットワーク上でパケットを送信したりネットワークサービスにリクエストを行ったりすることは、通常は永続的な痕跡を残さない一時的な操作である。パケットキャプチャやロギングを使用して永続的に記録することは可能であるが、通常そのようには考えない。メッセージをディスクに永続的に書き込むメッセージブローカーでさえ、コンシューマーに配信した後はすぐに削除する。これは一時的なメッセージングの発想に基づいて構築されているためである。

データベースとファイルシステムは反対のアプローチを取る：データベースやファイルに書き込まれたものは、誰かが明示的に削除を選択するまで、通常は永続的に記録されることが期待される。

この発想の違いは、導出データ（derived data）がどのように作成されるかに大きな影響を与える。Chapter 10 で議論したバッチ処理の重要な特徴は、入力を損傷するリスクなく処理ステップを繰り返し実験しながら実行できることであった（入力は読み取り専用であるため）。これは AMQP/JMS スタイルのメッセージングには当てはまらない：メッセージの受信は破壊的であり、確認応答によりブローカーから削除されるため、同じコンシューマーを再度実行しても同じ結果を期待できない。

メッセージングシステムに新しいコンシューマーを追加すると、通常は登録された時点以降に送信されたメッセージのみを受信し始める；それ以前のメッセージはすでに失われており、回復できない。ファイルやデータベースとは対照的に、任意のタイミングで新しいクライアントを追加でき、任意の過去に書き込まれたデータを読むことができる（アプリケーションによって明示的に上書きまたは削除されていない限り）。

データベースの永続的なストレージアプローチとメッセージングの低レイテンシ通知機能を組み合わせたハイブリッドを持つことはできないのか？これが**ログベースメッセージブローカー（log-based message broker）** の背後にあるアイデアである。

### ログを使ったメッセージ保存（Using logs for message storage）

ログ（log）とは、ディスク上のレコードの追記専用（append-only）シーケンスに過ぎない。Chapter 3 ではログ構造化ストレージエンジンとライトアヘッドログ（write-ahead log）の文脈で、Chapter 5 ではレプリケーションの文脈でログについて以前議論した。

同じ構造をメッセージブローカーの実装に使用できる：プロデューサーはログの末尾にメッセージを追加することで送信し、コンシューマーはログを順次読み取ることでメッセージを受信する。コンシューマーがログの末尾に達すると、新しいメッセージが追加されたという通知を待つ。Unix ツール `tail -f`（ファイルにデータが追加されるのを監視する）は、本質的にこのように動作する。

単一ディスクが提供できる以上のスループットにスケールするために、ログをパーティション分割（partition）できる（Chapter 6 の意味で）。異なるパーティションは異なるマシンにホストでき、各パーティションは他のパーティションから独立して読み書きできる別個のログとなる。トピック（topic）は、同じ種類のメッセージを運ぶパーティションのグループとして定義できる。

各パーティション内で、ブローカーはすべてのメッセージに単調増加するシーケンス番号、つまり**オフセット（offset）** を割り当てる。このようなシーケンス番号はパーティションが追記専用であるため理にかなっている。したがって、パーティション内のメッセージは完全に順序付けられている（totally ordered）。異なるパーティション間には順序保証はない。

**Apache Kafka** [17, 18]、**Amazon Kinesis Streams** [19]、Twitter の **DistributedLog** [20, 21] はこのように動作するログベースメッセージブローカーである。**Google Cloud Pub/Sub** はアーキテクチャ的には似ているが、ログ抽象化ではなく JMS スタイルの API を公開している [16]。これらのメッセージブローカーはすべてのメッセージをディスクに書き込むにもかかわらず、複数マシンにわたるパーティショニングにより毎秒数百万メッセージのスループットを達成し、メッセージのレプリケーションによるフォールトトレランスを実現している [22, 23]。

### ログと従来型メッセージングの比較（Logs compared to traditional messaging）

ログベースのアプローチはファンアウトメッセージングを自明にサポートする。なぜなら、複数のコンシューマーがログに影響を与えずに独立して読み取れるからである — メッセージを読んでもログから削除されない。コンシューマーのグループ間でロードバランシングを達成するために、個々のメッセージをコンシューマークライアントに割り当てるのではなく、ブローカーはパーティション全体をコンシューマーグループ（consumer group）内のノードに割り当てることができる。

各クライアントは、割り当てられたパーティション内のすべてのメッセージを消費する。典型的には、コンシューマーにログパーティションが割り当てられると、パーティション内のメッセージを順次、単純なシングルスレッドで読み取る。この粗粒度のロードバランシングアプローチにはいくつかの欠点がある：

- トピックの消費作業を共有するノード数は、そのトピックのログパーティション数が上限となる。なぜなら、同じパーティション内のメッセージは同じノードに配信されるからである。^i
- 単一のメッセージの処理が遅い場合、そのパーティション内の後続メッセージの処理が滞る（ヘッドオブラインブロッキング（head-of-line blocking）の一形態；「パフォーマンスの記述」13ページ参照）。

したがって、メッセージの処理コストが高くメッセージ単位で処理を並列化したい場面で、かつメッセージ順序がそれほど重要でない場合は、JMS/AMQP スタイルのメッセージブローカーが望ましい。一方、メッセージスループットが高く、各メッセージの処理が高速で、メッセージ順序が重要な場面では、ログベースのアプローチが非常にうまく機能する。

### コンシューマーオフセット（Consumer offsets）

パーティションを順次消費することで、どのメッセージが処理済みかを容易に判断できる：コンシューマーの現在のオフセットより小さいオフセットを持つすべてのメッセージは既に処理済みであり、より大きいオフセットを持つすべてのメッセージはまだ見ていない。したがって、ブローカーは個々のメッセージごとに確認応答を追跡する必要がない — コンシューマーオフセット（consumer offset）を定期的に記録するだけでよい。この記録管理のオーバーヘッド削減と、バッチングやパイプラインの機会が、ログベースシステムのスループット向上に役立つ。

このオフセットは実際のところ、シングルリーダーデータベースレプリケーションで一般的に見られるログシーケンス番号（log sequence number）と非常に似ている（「新しいフォロワーの設定」155ページ参照）。データベースレプリケーションでは、ログシーケンス番号によりフォロワーが切断後にリーダーに再接続し、書き込みをスキップすることなくレプリケーションを再開できる。ここでもまったく同じ原理が使用されている：メッセージブローカーはリーダーデータベースのように振る舞い、コンシューマーはフォロワーのように振る舞う。

コンシューマーノードが障害を起こした場合、コンシューマーグループ内の別のノードが障害を起こしたコンシューマーのパーティションを割り当てられ、最後に記録されたオフセットからメッセージの消費を開始する。コンシューマーが後続のメッセージを処理したがまだオフセットを記録していなかった場合、それらのメッセージは再起動時に2度目の処理が行われる。この問題への対処方法については本章の後半で議論する。

### ディスク容量の使用（Disk space usage）

ログに追記し続けるだけでは、最終的にディスク容量が不足する。ディスク容量を回収するために、ログは実際にはセグメント（segment）に分割され、時々古いセグメントが削除されるかアーカイブストレージに移動される。（ディスク容量を解放するより洗練された方法については後ほど議論する。）

これは、遅いコンシューマーがメッセージの速度に追いつけず、そのコンシューマーオフセットが削除されたセグメントを指すほど遅れた場合、一部のメッセージを見逃すことを意味する。実質的に、ログは満杯になると古いメッセージを破棄する固定サイズのバッファを実装している。これは**サーキュラーバッファ（circular buffer）** または**リングバッファ（ring buffer）** とも呼ばれる。ただし、このバッファはディスク上にあるため、かなり大きくできる。

概算をしてみよう。執筆時点で、典型的な大容量ハードドライブは6TBの容量と150MB/sの順次書き込みスループットを持っている。最速の速度でメッセージを書き込む場合、ドライブを満たすのに約11時間かかる。したがって、ディスクは11時間分のメッセージをバッファできる。この比率は、多くのハードドライブやマシンを使用しても同じままである。実際のデプロイメントではディスクの完全な書き込み帯域幅をめったに使用しないため、ログは通常数日分、あるいは数週間分のメッセージのバッファを保持できる。

メッセージをどれだけ長く保持するかに関わらず、ログのスループットはほぼ一定に保たれる。なぜなら、すべてのメッセージはどのみちディスクに書き込まれるからである [18]。この動作は、デフォルトでメッセージをメモリに保持し、キューが大きくなりすぎた場合にのみディスクに書き込むメッセージングシステムとは対照的である：そのようなシステムはキューが短いときは高速だが、ディスクへの書き込みを開始すると大幅に遅くなるため、スループットは保持する履歴の量に依存する。

### コンシューマーがプロデューサーに追いつけない場合（When consumers cannot keep up with producers）

「メッセージングシステム」の冒頭で、コンシューマーがプロデューサーのメッセージ送信速度に追いつけない場合の3つの選択肢について議論した：メッセージのドロップ、バッファリング、またはバックプレッシャーの適用である。この分類において、ログベースのアプローチは大きいが固定サイズのバッファ（利用可能なディスク容量に制限される）によるバッファリングの一形態である。

コンシューマーが大きく遅れて、必要なメッセージがディスク上に保持されているものより古い場合、それらのメッセージを読むことができない — つまりブローカーはバッファサイズが収容できる範囲を超えて遡る古いメッセージを実質的にドロップする。コンシューマーがログの先頭からどれだけ遅れているかを監視し、大幅に遅れた場合にアラートを発生させることができる。バッファが大きいため、メッセージの見逃しが始まる前に人間のオペレーターが遅いコンシューマーを修正し追いつかせるのに十分な時間がある。

コンシューマーが遅れすぎてメッセージの見逃しが始まったとしても、影響を受けるのはそのコンシューマーだけである；他のコンシューマーのサービスを妨害しない。この事実は大きな運用上の利点である：本番ログを開発、テスト、またはデバッグ目的で実験的に消費できるが、本番サービスの妨害をあまり心配する必要がない。コンシューマーがシャットダウンまたはクラッシュすると、リソースの消費を停止する — 残るのはコンシューマーオフセットだけである。

この動作は、コンシューマーがシャットダウンされたキューを削除するよう注意しなければならない従来型メッセージブローカーとも対照的である — さもなければ、不必要にメッセージを蓄積し続け、まだアクティブなコンシューマーからメモリを奪うことになる。

### 古いメッセージの再生（Replaying old messages）

以前述べたように、AMQP および JMS スタイルのメッセージブローカーでは、メッセージの処理と確認応答は破壊的操作であり、ブローカー上でメッセージが削除される。一方、ログベースメッセージブローカーでは、メッセージの消費はファイルからの読み取りにより近い：ログを変更しない読み取り専用の操作である。

処理の副作用は、コンシューマーの出力以外では、コンシューマーオフセットが前に進むことだけである。しかしオフセットはコンシューマーの制御下にあるため、必要であれば容易に操作できる：例えば、昨日のオフセットでコンシューマーのコピーを起動し、出力を別の場所に書き込むことで、直近1日分のメッセージを再処理できる。これは何度でも繰り返すことができ、処理コードを変更しながら行える。

この側面により、ログベースメッセージングは前章のバッチ処理により近くなる：導出データ（derived data）は繰り返し可能な変換プロセスを通じて入力データから明確に分離される。これにより、より多くの実験とエラーやバグからのより容易な回復が可能になり、組織内のデータフロー統合のための優れたツールとなる [24]。

## データベースとストリーム（Databases and Streams）

メッセージブローカーとデータベースの間にいくつかの比較を描いてきた。それらは伝統的に別のカテゴリのツールと考えられてきたが、ログベースメッセージブローカーがデータベースからアイデアを取り入れてメッセージングに適用することに成功しているのを見てきた。逆方向に進むこともできる：メッセージングとストリームからアイデアを取り入れ、データベースに適用するのである。

以前述べたように、イベントとはある時点で何かが起きたことの記録である。起きたことはユーザーアクション（例えば検索クエリの入力）やセンサーの読み取り値かもしれないが、データベースへの書き込みでもありうる。何かがデータベースに書き込まれたという事実はイベントであり、キャプチャ、保存、処理が可能である。この観察は、データベースとストリームの間の接続がディスク上のログの物理的な保存だけにとどまらず — かなり根本的なものであることを示唆している。

実際、レプリケーションログ（replication log）（「レプリケーションログの実装」158ページ参照）はデータベース書き込みイベントのストリームであり、リーダーがトランザクションを処理する際に生成される。フォロワーはその書き込みのストリームを自身のデータベースのコピーに適用し、同じデータの正確なコピーを得る。レプリケーションログ内のイベントは発生したデータ変更を記述する。

「全順序ブロードキャスト」（348ページ）では状態機械レプリケーション原理（state machine replication principle）にも出会った。これは次のように述べる：すべてのイベントがデータベースへの書き込みを表し、すべてのレプリカが同じイベントを同じ順序で処理すれば、レプリカはすべて同じ最終状態に到達する。（イベントの処理は決定論的操作であると仮定される。）これもまたイベントストリームの一例に過ぎない！

このセクションでは、まず異種データシステムで生じる問題を見て、その後イベントストリームからのアイデアをデータベースに持ち込むことでどのように解決できるかを探る。

### システム間の同期の維持（Keeping Systems in Sync）

本書を通じて見てきたように、すべてのデータの保存、クエリ、処理のニーズを満たせる単一のシステムは存在しない。実際には、ほとんどの非自明なアプリケーションは要件を満たすためにいくつかの異なる技術を組み合わせる必要がある：例えば、ユーザーリクエストを処理するための OLTP データベース、よくあるリクエストを高速化するためのキャッシュ（cache）、検索クエリを処理するための全文インデックス（full-text index）、分析のためのデータウェアハウス（data warehouse）を使用する。これらはそれぞれデータの独自のコピーを持ち、それぞれの目的に最適化された独自の表現で保存される。

同じまたは関連するデータが複数の異なる場所に現れるため、互いに同期を保つ必要がある：データベースでアイテムが更新された場合、キャッシュ、検索インデックス、データウェアハウスでも更新が必要である。データウェアハウスではこの同期は通常 ETL プロセス（「データウェアハウジング」91ページ参照）によって行われ、多くの場合データベースの完全なコピーを取り、変換し、データウェアハウスにバルクロードする — つまりバッチ処理である。同様に、「バッチワークフローの出力」（411ページ）では、検索インデックス、レコメンデーションシステム、その他の導出データシステムがバッチ処理を使用してどのように作成されるかを見た。

定期的なフルデータベースダンプが遅すぎる場合、時に使用される代替手段は**デュアルライト（dual writes）** であり、これはデータが変更されたときにアプリケーションコードが各システムに明示的に書き込むものである：例えば、まずデータベースに書き込み、次に検索インデックスを更新し、次にキャッシュエントリを無効化する（あるいはこれらの書き込みを並行して実行する）。

しかし、デュアルライトにはいくつかの深刻な問題がある。その1つはレース条件（race condition）である。2つのクライアントが同時にアイテム X を更新しようとする：クライアント1は値を A に設定し、クライアント2は値を B に設定したい。両方のクライアントはまず新しい値をデータベースに書き込み、次に検索インデックスに書き込む。タイミングが悪いと、リクエストがインターリーブされる：データベースはまずクライアント1からの値 A への書き込みを見て、次にクライアント2からの値 B への書き込みを見るため、データベースの最終値は B となる。検索インデックスはまずクライアント2からの書き込みを見て、次にクライアント1からの書き込みを見るため、検索インデックスの最終値は A となる。2つのシステムはエラーが発生していないにもかかわらず、永久に不整合になる。

「同時書き込みの検出」（184ページ）で議論したバージョンベクター（version vector）のような追加の並行性検出メカニズムがなければ、同時書き込みが発生したことにすら気付かない — 一方の値がもう一方を静かに上書きするだけである。

デュアルライトのもう1つの問題は、書き込みの一方が成功し他方が失敗する可能性があることである。これは並行性の問題ではなくフォールトトレランスの問題であるが、2つのシステムが互いに不整合になるという同様の効果がある。両方が成功するか両方が失敗するかを保証することは、アトミックコミット問題（atomic commit problem）であり、解決コストが高い（「アトミックコミットと2相コミット（2PC）」354ページ参照）。

レプリケートされたデータベースが単一リーダーで1つだけある場合、そのリーダーが書き込みの順序を決定するため、データベースのレプリカ間では状態機械レプリケーションアプローチが機能する。しかし、データベースにリーダーがあり検索インデックスにもリーダーがあるが、どちらも他方に従わないため、コンフリクト（conflict）が発生しうる（「マルチリーダーレプリケーション」168ページ参照）。

本当に単一のリーダーだけがあれば状況は良くなる — 例えばデータベース — そして検索インデックスをデータベースのフォロワーにできれば。しかし、これは実際に可能なのか？

### Change Data Capture（変更データキャプチャ）

ほとんどのデータベースのレプリケーションログの問題は、それらが長い間データベースの内部実装の詳細であり、パブリック API ではないと考えられてきたことである。クライアントはデータモデルとクエリ言語を通じてデータベースにクエリすることが想定されており、レプリケーションログを解析してデータを抽出しようとすることは想定されていない。

数十年もの間、多くのデータベースには書き込まれた変更のログを取得する文書化された方法が単純に存在しなかった。このため、データベースに行われたすべての変更を取得し、検索インデックス、キャッシュ、データウェアハウスなどの異なるストレージ技術にレプリケートすることは困難であった。

より最近では、**Change Data Capture（CDC、変更データキャプチャ）** への関心が高まっている。これはデータベースに書き込まれたすべてのデータ変更を観察し、他のシステムにレプリケートできる形で抽出するプロセスである。CDC は変更が書き込まれた直後にストリームとして利用可能になる場合に特に興味深い。

例えば、データベースの変更をキャプチャし、同じ変更を検索インデックスに継続的に適用できる。変更のログが同じ順序で適用されれば、検索インデックスのデータがデータベースのデータと一致することを期待できる。検索インデックスやその他の導出データシステムは変更ストリーム（change stream）のコンシューマーに過ぎない。

#### CDC の実装（Implementing change data capture）

ログコンシューマーを導出データシステム（derived data system）と呼ぶことができる（Part III の序論で議論した通り）：検索インデックスやデータウェアハウスに保存されたデータは、記録システム（system of record）内のデータの別のビューに過ぎない。Change Data Capture は、記録システムに対して行われたすべての変更が導出データシステムにも反映され、導出システムがデータの正確なコピーを持つことを保証するメカニズムである。

本質的に、Change Data Capture は1つのデータベースをリーダー（変更がキャプチャされるもの）とし、他のものをフォロワーに変える。ログベースメッセージブローカーは、メッセージの順序を保持する（再順序付けの問題を回避する）ため、ソースデータベースから変更イベントを転送するのに適している。

データベーストリガー（database trigger）を使用して Change Data Capture を実装できる（「トリガーベースレプリケーション」161ページ参照）。データテーブルへのすべての変更を観察するトリガーを登録し、変更ログテーブルに対応するエントリを追加する。しかし、トリガーは脆弱でパフォーマンスオーバーヘッドが大きい傾向がある。レプリケーションログの解析（parsing）はより堅牢なアプローチとなりうるが、スキーマ変更の処理などの課題も伴う。

LinkedIn の **Databus** [25]、Facebook の **Wormhole** [26]、Yahoo! の **Sherpa** [27] はこのアイデアを大規模に使用している。**Bottled Water** は API を使用してライトアヘッドログ（WAL）をデコードすることで PostgreSQL 向けの CDC を実装している [28]。**Maxwell** と **Debezium** は binlog を解析することで MySQL 向けに同様のことを行っている [29, 30, 31]。**Mongoriver** は MongoDB の oplog を読み取り [32, 33]、**GoldenGate** は Oracle 向けに同様の機能を提供する [34, 35]。

メッセージブローカーと同様に、Change Data Capture は通常**非同期（asynchronous）** である：記録システムのデータベースはコンシューマーに変更が適用されるのを待たずにコミットする。この設計には、遅いコンシューマーを追加しても記録システムにあまり影響を与えないという運用上の利点があるが、レプリケーションラグ（replication lag）のすべての問題が適用されるという欠点がある（「レプリケーションラグの問題」161ページ参照）。

#### 初期スナップショット（Initial snapshot）

データベースに対して行われたすべての変更のログがあれば、ログを再生（replay）することでデータベースの状態全体を再構築できる。しかし多くの場合、すべての変更を永遠に保持するにはディスク容量が多すぎ、再生にも時間がかかりすぎるため、ログを切り詰める（truncate）必要がある。

例えば、新しい全文インデックスの構築にはデータベース全体の完全なコピーが必要である — 最近の変更のログだけを適用するだけでは不十分であり、最近更新されていないアイテムが欠落する。完全なログ履歴がない場合は、「新しいフォロワーの設定」（155ページ）で以前議論したように、一貫性のあるスナップショット（consistent snapshot）から開始する必要がある。

データベースのスナップショットは変更ログ内の既知のポジションまたはオフセットに対応しなければならない。これにより、スナップショットの処理後にどの時点から変更の適用を開始すべきかがわかる。一部の CDC ツールはこのスナップショット機能を統合しているが、手動操作に委ねるものもある。

#### ログコンパクション（Log compaction）

限られたログ履歴しか保持できない場合、新しい導出データシステムを追加するたびにスナップショットプロセスを経なければならない。しかし、**ログコンパクション（log compaction）** は良い代替手段を提供する。

ログコンパクションについては以前「ハッシュインデックス」（72ページ）でログ構造化ストレージエンジン（log-structured storage engine）の文脈で議論した。原理はシンプルである：ストレージエンジンが定期的に同じキーのログレコードを探し、重複を破棄し、各キーの最新の更新のみを保持する。このコンパクションとマージのプロセスはバックグラウンドで実行される。

ログ構造化ストレージエンジンでは、特別な null 値（**トゥームストーン（tombstone）**）による更新はキーが削除されたことを示し、ログコンパクション中にそれが除去される原因となる。しかしキーが上書きも削除もされない限り、ログに永遠に残る。このようなコンパクトされたログに必要なディスク容量は、データベースの現在の内容にのみ依存し、データベースで過去に行われた書き込み回数には依存しない。同じキーが頻繁に上書きされる場合、以前の値は最終的にガベージコレクション（garbage collection）され、最新の値のみが保持される。

同じアイデアがログベースメッセージブローカーと Change Data Capture の文脈でも機能する。CDC システムがすべての変更がプライマリキーを持ち、キーに対するすべての更新がそのキーの以前の値を置き換えるように設定されている場合、特定のキーの最新の書き込みのみを保持すれば十分である。

これにより、検索インデックスのような導出データシステムを再構築したい場合はいつでも、ログコンパクトされたトピックのオフセット0から新しいコンシューマーを開始し、ログ内のすべてのメッセージを順次スキャンできる。ログはデータベース内のすべてのキーの最新の値を含むことが保証されている（いくつかの古い値も含まれるかもしれない）— つまり、CDC ソースデータベースの別のスナップショットを取ることなく、データベース内容の完全なコピーを取得できる。

このログコンパクション機能は Apache Kafka でサポートされている。本章の後半で見るように、メッセージブローカーを一時的なメッセージングだけでなく永続ストレージとしても使用できるようにする。

#### 変更ストリームの API サポート（API support for change streams）

データベースは変更ストリーム（change stream）をファーストクラスのインターフェースとしてサポートし始めている。これは典型的な後付けでリバースエンジニアリングされた CDC の取り組みとは異なる。例えば、**RethinkDB** はクエリ結果が変更されたときの通知にサブスクライブするクエリを許可する [36]。**Firebase** [37] と **CouchDB** [38] は変更フィード（change feed）に基づくデータ同期を提供し、これはアプリケーションにも利用可能である。**Meteor** は MongoDB の oplog を使用してデータ変更をサブスクライブし、ユーザーインターフェースを更新する [39]。

**VoltDB** はデータベースからストリームの形でデータを継続的にエクスポートするトランザクションを許可する [40]。データベースはリレーショナルデータモデルにおいて出力ストリームをテーブルとして表現し、トランザクションがタプル（tuple）を挿入できるが、クエリはできない。ストリームはコミットされたトランザクションがこの特別なテーブルに書き込んだタプルのログで構成され、コミットされた順序で並ぶ。外部コンシューマーはこのログを非同期に消費し、導出データシステムを更新するために使用できる。

**Kafka Connect** [41] は、幅広いデータベースシステム向けの Change Data Capture ツールを Kafka と統合する取り組みである。変更イベントのストリームが Kafka に入ると、検索インデックスなどの導出データシステムの更新や、本章の後半で議論するストリーム処理システムへのフィードに使用できる。

### Event Sourcing（イベントソーシング）

ここまで議論してきたアイデアと、ドメイン駆動設計（DDD: Domain-Driven Design）コミュニティで開発された技法である**イベントソーシング（event sourcing）** [42, 43, 44] の間にはいくつかの類似点がある。イベントソーシングはストリーミングシステムにとって有用で関連性の高いアイデアを取り入れているため、簡潔に議論する。

Change Data Capture と同様に、イベントソーシングはアプリケーション状態（application state）へのすべての変更を変更イベントのログとして保存することを含む。最大の違いは、イベントソーシングがアイデアを異なる抽象レベルで適用することである：

- **Change Data Capture** では、アプリケーションはデータベースをミュータブル（mutable）な方法で使用し、レコードを自由に更新・削除する。変更のログは低レベルで（例えばレプリケーションログの解析により）データベースから抽出され、データベースから抽出された書き込みの順序が実際に書き込まれた順序と一致することを保証する。データベースに書き込むアプリケーションは CDC が行われていることを認識する必要がない。
- **イベントソーシング（event sourcing）** では、アプリケーションロジックはイベントログ（event log）に書き込まれるイミュータブル（immutable）なイベントの上に明示的に構築される。この場合、イベントストアは追記専用（append-only）であり、更新や削除は推奨されないか禁止される。イベントは低レベルの状態変更ではなく、アプリケーションレベルで起きた事柄を反映するように設計される。

イベントソーシングはデータモデリングのための強力な技法である：アプリケーションの視点からは、ミュータブルなデータベースへのアクション結果を記録するよりも、ユーザーのアクションをイミュータブルなイベントとして記録する方がより意味がある。イベントソーシングにより、アプリケーションの時間経過に伴う進化が容易になり、何かがなぜ起きたかを事後に理解しやすくなることでデバッグに役立ち、アプリケーションバグに対するガードとなる。

例えば、「学生がコース登録をキャンセルした」というイベントを保存することは、単一のアクションの意図をニュートラルな形で明確に表現する。一方で「登録テーブルから1エントリが削除され、学生フィードバックテーブルに1つのキャンセル理由が追加された」という副作用の記述は、データが後でどのように使用されるかについて多くの仮定を埋め込んでいる。新しいアプリケーション機能が導入された場合 — 例えば「その席はウェイティングリストの次の人に提供される」— イベントソーシングのアプローチにより、新しい副作用を既存のイベントから容易にチェーンできる。

イベントソーシングはクロニクルデータモデル（chronicle data model）[45] に類似しており、イベントログとスターキーマ（star schema）のファクトテーブル（fact table）（「スターとスノーフレーク：分析用スキーマ」93ページ参照）の間にも類似点がある。

**Event Store** [46] のような専用データベースがイベントソーシングを使用するアプリケーションをサポートするために開発されているが、一般にこのアプローチは特定のツールに依存しない。従来型のデータベースやログベースメッセージブローカーを使用してこのスタイルのアプリケーションを構築することもできる。

#### イベントログから現在の状態を導出する（Deriving current state from the event log）

イベントログ自体はあまり有用ではない。なぜなら、ユーザーは一般にシステムの現在の状態を見ることを期待し、変更の履歴を見ることを期待しないからである。例えば、ショッピングサイトでは、ユーザーはカートの現在の内容を見ることを期待し、カートに対して行われたすべての変更の追記専用リストを見ることは期待しない。

したがって、イベントソーシングを使用するアプリケーションは、イベントのログ（システムに書き込まれたデータを表す）を取り、ユーザーに表示するのに適したアプリケーション状態に変換する必要がある（システムからデータが読み取られる方法 [47]）。この変換は任意のロジックを使用できるが、決定論的（deterministic）であるべきであり、同じイベントログから同じアプリケーション状態を再度導出できるようにする。

Change Data Capture と同様に、イベントログの再生により現在のシステム状態を再構築できる。しかし、ログコンパクションは異なる方法で処理される必要がある：

- CDC イベントによるレコードの更新は通常、レコードの完全な新バージョンを含むため、プライマリキーの現在の値はそのプライマリキーの最新のイベントによって完全に決定され、ログコンパクションは同じキーの以前のイベントを破棄できる。
- 一方、イベントソーシングでは、イベントはより高いレベルでモデル化される：イベントは通常、アクションの結果として発生した状態更新のメカニクスではなく、ユーザーアクションの意図を表現する。この場合、後のイベントは通常以前のイベントをオーバーライドしないため、最終状態を再構築するには完全なイベント履歴が必要である。ログコンパクションは同じ方法では不可能である。

イベントソーシングを使用するアプリケーションは通常、イベントのログから導出された現在の状態のスナップショットを保存するメカニズムを持っている。これにより完全なログを繰り返し再処理する必要がなくなる。しかしこれは読み取りとクラッシュからの回復を高速化するためのパフォーマンス最適化に過ぎない；意図は、システムがすべてのrawイベントを永遠に保存し、必要に応じて完全なイベントログを再処理できることである。「不変性の限界」でこの仮定について議論する。

#### コマンドとイベント（Commands and events）

イベントソーシングの哲学は、イベントとコマンドを慎重に区別する [48]。ユーザーからのリクエストが最初に到着したとき、それは最初は**コマンド（command）** である：この時点ではまだ失敗する可能性がある（例えば何らかの整合性条件に違反しているため）。アプリケーションはまずコマンドを実行できるかどうかをバリデーションしなければならない。バリデーションが成功しコマンドが受け入れられると、それは**イベント（event）** となり、永続的でイミュータブルとなる。

例えば、ユーザーが特定のユーザー名を登録しようとしたり、飛行機や劇場の座席を予約しようとしたりする場合、アプリケーションはそのユーザー名や座席がまだ使われていないかチェックする必要がある。（この例については以前「フォールトトレラントな合意」364ページで議論した。）チェックが成功すると、アプリケーションは特定のユーザーIDによって特定のユーザー名が登録された、あるいは特定の顧客によって特定の座席が予約されたことを示すイベントを生成できる。

イベントが生成された時点で、それは**事実（fact）** となる。たとえ顧客が後で予約を変更またはキャンセルすることを決定しても、以前は特定の座席の予約を持っていたという事実は真のまま残り、変更やキャンセルは後で追加される別のイベントである。

イベントストリームのコンシューマーはイベントを拒否することが許されていない：コンシューマーがイベントを見る時点で、それはすでにログのイミュータブルな部分であり、他のコンシューマーによって既に見られている可能性がある。したがって、コマンドのバリデーションはイベントになる前に同期的に行われる必要がある — 例えば、コマンドのバリデーションとイベントの公開をアトミックに行うシリアライザブルトランザクション（serializable transaction）を使用する。

あるいは、座席を予約するユーザーリクエストを2つのイベントに分割することもできる：まず仮予約（tentative reservation）、次に予約がバリデーションされた後の別の確認イベント（confirmation event）（「全順序ブロードキャストを使用したリニアライザブルストレージの実装」350ページで議論した通り）。この分割により、バリデーションを非同期プロセスで行うことができる。

### 状態、ストリーム、不変性（State, Streams, and Immutability）

Chapter 10 ではバッチ処理が入力ファイルの不変性（immutability）から恩恵を受けることを見た。既存の入力ファイルに対して損傷を恐れずに実験的な処理ジョブを実行できる。この不変性の原理こそが、イベントソーシングと Change Data Capture を非常に強力にしているものでもある。

データベースはアプリケーションの現在の状態を保存するものとして通常考えられる — この表現は読み取りに最適化されており、通常はクエリの処理に最も便利である。状態の性質は変化するものであり、したがってデータベースはデータの挿入だけでなく更新と削除もサポートする。これはどのように不変性と両立するのか？

状態が変化するとき、その状態は時間の経過とともにそれを変異させたイベントの結果である。例えば、現在利用可能な座席のリストは処理された予約の結果であり、現在の口座残高は口座への入出金の結果であり、Web サーバーの応答時間グラフは発生したすべての Web リクエストの個別の応答時間の集約である。

状態がどのように変化しても、常にそれらの変化を引き起こしたイベントのシーケンスが存在した。物事が行われたり取り消されたりしても、それらのイベントが発生したという事実は真のまま残る。重要なアイデアは、ミュータブルな状態とイミュータブルなイベントの追記専用ログは互いに矛盾しない：それらは同じコインの裏表であるということである。すべての変更のログ、すなわち変更ログ（changelog）は、時間経過における状態の進化を表している。

数学的に言えば、アプリケーション状態はイベントストリームを時間について積分（integrate）した結果であり、変更ストリーム（change stream）は状態を時間で微分（differentiate）した結果である。この類推には限界がある（例えば状態の2階微分は意味があるように見えない）が、データについて考えるための有用な出発点である。

変更ログを永続的に保存すれば、状態を再現可能（reproducible）にする効果がある。イベントのログを記録システム（system of record）と見なし、ミュータブルな状態をそこから導出されたものと見なせば、システムを通じたデータの流れについて推論しやすくなる。Pat Helland はこう述べている [52]：

> トランザクションログ（transaction log）はデータベースに対して行われたすべての変更を記録する。高速な追記がログを変更する唯一の方法である。この視点からすると、データベースの内容はログ内の最新のレコード値のキャッシングを保持している。**真実はログである。データベースはログのサブセットのキャッシュである。** そのキャッシュされたサブセットは、たまたまログからの各レコードとインデックス値の最新値である。

ログコンパクション（「ログコンパクション」456ページで議論）は、ログとデータベース状態の区別を橋渡しする1つの方法である：各レコードの最新バージョンのみを保持し、上書きされたバージョンを破棄する。

#### イミュータブルなイベントの利点（Advantages of immutable events）

データベースにおける不変性は古いアイデアである。例えば、会計士は何世紀にもわたって金融の簿記（bookkeeping）に不変性を使用してきた。取引が発生すると、それは追記専用の元帳（ledger）に記録される。これは本質的に、お金、商品、またはサービスが手を変えたことを記述するイベントのログである。利益と損失や貸借対照表（balance sheet）などの勘定は、元帳内の取引を合算することで導出される [53]。

間違いがあった場合、会計士は元帳内の誤った取引を消去したり変更したりしない — 代わりに、間違いを補償する別の取引を追加する。例えば、誤った請求の返金である。誤った取引は元帳に永遠に残る。監査上の理由で重要かもしれないからである。誤った元帳から導出された誤った数値がすでに公表されている場合、次の会計期間の数値に修正が含まれる。このプロセスは会計において完全に正常なことである [54]。

このような監査可能性は金融システムで特に重要であるが、このような厳格な規制の対象ではない他の多くのシステムにとっても有益である。「バッチ処理出力の哲学」（413ページ）で議論したように、不正なデータをデータベースに書き込むバグのあるコードを誤ってデプロイした場合、コードがデータを破壊的に上書きできるのであれば、回復はずっと困難になる。イミュータブルなイベントの追記専用ログがあれば、何が起きたかの診断と問題からの回復がずっと容易になる。

イミュータブルなイベントは現在の状態よりも多くの情報をキャプチャする。例えば、ショッピングサイトで顧客がカートにアイテムを追加し、その後それを削除するかもしれない。注文処理の視点からは2番目のイベントが1番目のイベントを打ち消すが、分析目的では顧客が特定のアイテムを検討していたが決定に至らなかったことを知ることが有用かもしれない。おそらく将来それを購入するか、あるいは代替品を見つけたのかもしれない。この情報はイベントログに記録されるが、カートから削除されたときにアイテムを削除するデータベースでは失われるだろう [42]。

#### 同一イベントログから複数のビューを導出する（Deriving several views from the same event log）

さらに、ミュータブルな状態をイミュータブルなイベントログから分離することで、同じイベントのログから複数の異なる読み取り指向の表現を導出できる。これはストリームの複数のコンシューマーを持つのと同じように機能する：例えば、分析データベース **Druid** はこのアプローチを使用して Kafka から直接取り込み [55]、**Pistachio** は Kafka をコミットログとして使用する分散キーバリューストアであり [56]、**Kafka Connect** シンクは Kafka から様々な異なるデータベースやインデックスにデータをエクスポートできる [41]。検索サーバーなど、他の多くのストレージやインデクシングシステムも同様に分散ログから入力を取ることが理にかなうだろう（「システム間の同期の維持」452ページ参照）。

イベントログからデータベースへの明示的な変換ステップを持つことで、アプリケーションの進化が時間の経過とともに容易になる：既存のデータを何らかの新しい方法で表示する新機能を導入したい場合、イベントログを使用して新機能用の別の読み取り最適化ビューを構築し、既存のシステムを変更することなく並行して実行できる。新旧のシステムを並行して実行することは、既存のシステムで複雑なスキーママイグレーション（schema migration）を実行するよりも多くの場合容易である。古いシステムが不要になったら、単にシャットダウンしてリソースを回収すればよい [47, 57]。

データのクエリやアクセスの方法を心配する必要がなければ、データの保存は通常かなり単純である；スキーマ設計、インデクシング、ストレージエンジンの複雑さの多くは、特定のクエリやアクセスパターンをサポートしたいという欲求の結果である（Chapter 3 参照）。このため、データが書き込まれる形式と読み取られる形式を分離し、複数の異なる読み取りビューを許可することで、多くの柔軟性が得られる。このアイデアは **CQRS（Command Query Responsibility Segregation）** [42, 58, 59] と呼ばれることがある。

データベースとスキーマ設計の伝統的なアプローチは、データはクエリされるのと同じ形式で書き込まれなければならないという誤謬に基づいている。正規化と非正規化に関する議論（「多対一と多対多の関係」33ページ参照）は、書き込み最適化されたイベントログから読み取り最適化されたアプリケーション状態に変換できるのであれば、大部分が無関係になる：変換プロセスがイベントログとの整合性を保つメカニズムを提供するため、読み取り最適化ビューでデータを非正規化することは完全に合理的である。

「負荷の記述」（11ページ）では Twitter のホームタイムライン（home timeline）について議論した。これは特定のユーザーがフォローしている人々による最近の書き込みツイートのキャッシュ（メールボックスのようなもの）である。これは読み取り最適化状態のもう1つの例である：ホームタイムラインは高度に非正規化されている。あなたのツイートはフォローしているすべての人のタイムラインに複製されるためである。しかし、ファンアウトサービス（fan-out service）はこの複製された状態を新しいツイートと新しいフォロー関係で同期し続け、複製を管理可能に保つ。

#### 並行性制御（Concurrency control）

イベントソーシングと Change Data Capture の最大の欠点は、イベントログのコンシューマーが通常非同期であるため、ユーザーがログに書き込みを行い、その後ログ導出ビューから読み取ると、自分の書き込みがまだ読み取りビューに反映されていないことがありうることである。この問題と潜在的な解決策については以前「自分の書き込みの読み取り」（162ページ）で議論した。

1つの解決策は、ログへのイベントの追加と同期的に読み取りビューの更新を行うことであろう。これには書き込みをアトミックな単位に結合するトランザクションが必要であり、イベントログと読み取りビューを同じストレージシステムに保持するか、異なるシステム間の分散トランザクション（distributed transaction）が必要になる。あるいは、「全順序ブロードキャストを使用したリニアライザブルストレージの実装」（350ページ）で議論したアプローチを使用することもできる。

他方、イベントログから現在の状態を導出することは、並行性制御のいくつかの側面を簡素化する。マルチオブジェクトトランザクション（multi-object transaction）の必要性の多くは（「単一オブジェクトと複数オブジェクトの操作」228ページ参照）、単一のユーザーアクションが複数の異なる場所でデータを変更する必要があることに起因する。イベントソーシングでは、イベントをユーザーアクションの自己完結的な記述として設計できる。ユーザーアクションは1つの場所での単一の書き込みのみを必要とする — すなわちログへのイベントの追加 — これはアトミックにすることが容易である。

イベントログとアプリケーション状態が同じ方法でパーティション分割されている場合（例えば、パーティション3の顧客のイベントを処理するにはアプリケーション状態のパーティション3の更新のみが必要）、単純なシングルスレッドのログコンシューマーは書き込みの並行性制御を必要としない — 構成上、一度に1つのイベントのみを処理する（「実際のシリアル実行」252ページも参照）。ログはパーティション内のイベントの直列順序を定義することで、並行性の非決定性を除去する [24]。イベントが複数の状態パーティションに影響する場合は、もう少し作業が必要であり、Chapter 12 で議論する。

#### 不変性の限界（Limitations of immutability）

イベントソーシングモデルを使用しない多くのシステムも、それでも不変性に依存している：様々なデータベースはポイントインタイムスナップショット（point-in-time snapshot）をサポートするために内部的にイミュータブルなデータ構造やマルチバージョン（multi-version）データを使用している（「インデックスとスナップショット分離」241ページ参照）。Git、Mercurial、Fossil などのバージョン管理システムもファイルのバージョン履歴を保持するためにイミュータブルなデータに依存している。

すべての変更のイミュータブルな履歴を永遠に保持することはどの程度現実的か？答えはデータセットのチャーン（churn）の量に依存する。一部のワークロードは主にデータを追加し、更新や削除はめったに行わない；これらはイミュータブルにすることが容易である。他のワークロードは比較的小さなデータセットに対して高い割合の更新と削除を行う；これらの場合、イミュータブルな履歴は禁止的に大きくなる可能性があり、フラグメンテーション（fragmentation）が問題になりうるため、コンパクションとガベージコレクションのパフォーマンスが運用上の堅牢性にとって重要となる [60, 61]。

パフォーマンス上の理由に加えて、すべての不変性にもかかわらず、管理上の理由でデータの削除が必要な状況もある。例えば、プライバシー規制により、ユーザーがアカウントを閉鎖した後に個人情報を削除することが求められる場合がある。データ保護法により、誤った情報の削除が求められる場合がある。あるいは、機密情報の偶発的な漏洩を封じ込める必要がある場合もある。

これらの状況では、以前のデータが削除されたと見なされるべきであることを示す別のイベントをログに追加するだけでは不十分である — データが最初から書き込まれなかったかのように、履歴を実際に書き換える必要がある。例えば、Datomic はこの機能を**エクシジョン（excision）** [62] と呼び、Fossil バージョン管理システムにも**シャニング（shunning）** [63] と呼ばれる同様のコンセプトがある。

データを真に削除することは驚くほど難しい [64]。コピーは多くの場所に存在しうるからである：例えば、ストレージエンジン、ファイルシステム、SSD はしばしば上書きではなく新しい場所に書き込む [52]。バックアップは偶発的な削除や破損を防ぐために意図的にイミュータブルであることが多い。削除は「データの取得を不可能にする」というよりも「データの取得をより困難にする」ということに近い。それでも、「法規制と自主規制」（542ページ）で見るように、時に試みなければならない。

## ストリームの処理（Processing Streams）

本章ではこれまで、ストリームの出所（ユーザーアクティビティイベント、センサー、データベースへの書き込み）と、ストリームの転送方法（直接メッセージング、メッセージブローカー経由、イベントログ内）について話してきた。

残るのは、ストリームを取得した後に何ができるかの議論である — すなわち処理できる。大まかに3つの選択肢がある：

1. イベント内のデータを取り、データベース、キャッシュ、検索インデックス、または類似のストレージシステムに書き込み、他のクライアントからクエリできるようにする。これは、システムの他の部分で起きている変更とデータベースを同期させる良い方法である — 特にストリームコンシューマーがデータベースに書き込む唯一のクライアントである場合。ストレージシステムへの書き込みは「バッチワークフローの出力」（411ページ）で議論したことのストリーミング版である。
2. イベントを何らかの方法でユーザーにプッシュする — 例えばメールアラートやプッシュ通知の送信、またはリアルタイムダッシュボードへのイベントのストリーミングと可視化。この場合、人間がストリームの最終的なコンシューマーである。
3. 1つ以上の入力ストリームを処理して1つ以上の出力ストリームを生成する。ストリームは最終的に出力（選択肢1または2）に到達する前に、このような処理ステージのパイプラインを通過する場合がある。

本章の残りでは、選択肢3について議論する：他の導出ストリーム（derived stream）を生成するためのストリームの処理である。このようにストリームを処理するコードの一部分は**オペレーター（operator）** または**ジョブ（job）** として知られる。これは Chapter 10 で議論した Unix プロセスや MapReduce ジョブと密接に関連しており、データフローのパターンも類似している：ストリームプロセッサは入力ストリームを読み取り専用の形で消費し、追記専用の形で別の場所に出力を書き込む。

ストリームプロセッサにおけるパーティショニングと並列化のパターンも、Chapter 10 で見た MapReduce やデータフローエンジンのものと非常に類似しているため、それらのトピックはここでは繰り返さない。レコードの変換やフィルタリングなどの基本的なマッピング操作も同様に機能する。

バッチジョブとの1つの重要な違いは、ストリームが決して終わらないことである。この違いには多くの含意がある：本章の冒頭で議論したように、無界のデータセットではソートは意味をなさず、したがってソートマージ結合（sort-merge join）（「Reduce 側の結合とグループ化」403ページ参照）は使用できない。フォールトトレランスのメカニズムも変更しなければならない：数分間実行されたバッチジョブでは、失敗したタスクを最初から単純に再起動できるが、数年間実行されてきたストリームジョブでは、クラッシュ後に最初から再起動することは現実的な選択肢ではないかもしれない。

### ストリーム処理の用途（Uses of Stream Processing）

ストリーム処理は、組織が特定のことが起きた場合にアラートを受けたいという監視目的で長い間使用されてきた。例えば：

- **不正検出システム**は、クレジットカードの使用パターンが予期せず変化したかどうかを判断し、盗難の可能性が高い場合はカードをブロックする必要がある。
- **取引システム**は、金融市場の価格変動を調査し、指定されたルールに従って取引を実行する必要がある。
- **製造システム**は、工場内の機械の状態を監視し、故障があれば迅速に問題を特定する必要がある。
- **軍事・情報システム**は、潜在的な攻撃者の活動を追跡し、攻撃の兆候がある場合に警報を発する必要がある。

これらの種類のアプリケーションには非常に高度なパターンマッチングと相関分析が必要である。しかし、ストリーム処理の他の用途も時間の経過とともに出現してきた。このセクションでは、これらのアプリケーションのいくつかを簡潔に比較対照する。

#### 複合イベント処理（Complex event processing）

**複合イベント処理（CEP: Complex Event Processing）** は、イベントストリームを分析するために1990年代に開発されたアプローチであり、特に特定のイベントパターンの検索を必要とする種類のアプリケーション向けである [65, 66]。正規表現（regular expression）が文字列内の特定の文字パターンを検索できるのと同様に、CEP ではストリーム内の特定のイベントパターンを検索するルールを指定できる。

CEP システムはしばしば SQL のような高レベルの宣言的クエリ言語（declarative query language）やグラフィカルユーザーインターフェースを使用して、検出すべきイベントパターンを記述する。これらのクエリは、入力ストリームを消費し、必要なマッチングを実行する状態機械（state machine）を内部的に維持する処理エンジンに送信される。マッチが見つかると、エンジンは検出されたイベントパターンの詳細を含む**複合イベント（complex event）**（名前の由来）を発行する [67]。

これらのシステムでは、クエリとデータの関係が通常のデータベースとは逆転している。通常、データベースはデータを永続的に保存し、クエリを一時的なものとして扱う：クエリが来ると、データベースはクエリに一致するデータを検索し、完了するとクエリを忘れる。CEP エンジンはこれらの役割を逆転させる：クエリが長期的に保存され、入力ストリームからのイベントがイベントパターンに一致するクエリを探して継続的にそれらを通過する [68]。

CEP の実装には **Esper** [69]、**IBM InfoSphere Streams** [70]、Apama、TIBCO StreamBase、SQLstream などがある。Samza のような分散ストリームプロセッサもストリーム上の宣言的クエリに対する SQL サポートを獲得しつつある [71]。

#### ストリーム分析（Stream analytics）

ストリーム処理が使用されるもう1つの領域は、ストリーム上の分析（analytics）である。CEP とストリーム分析の境界は曖昧であるが、一般的なルールとして、分析は特定のイベントシーケンスの発見にはあまり関心がなく、多数のイベントにわたる集約（aggregation）と統計的指標（statistical metrics）により指向している — 例えば：

- 特定の種類のイベントの発生率の測定（時間間隔あたりの発生頻度）
- ある期間にわたる値の移動平均（rolling average）の計算
- 現在の統計を以前の時間間隔と比較する（例えばトレンドの検出や、先週の同じ時間と比較して異常に高いまたは低いメトリクスのアラート）

このような統計は通常、固定の時間間隔にわたって計算される — 例えば、直近5分間のサービスへの1秒あたりの平均クエリ数とその期間の99パーセンタイル応答時間を知りたいかもしれない。数分間の平均化は1秒ごとの無関係な変動を平滑化しつつ、トラフィックパターンの変化をタイムリーに把握させてくれる。集約する時間間隔は**ウィンドウ（window）** として知られ、ウィンドウの詳細については「時間についての推論」で見ていく。

ストリーム分析システムは、Bloom フィルター（「パフォーマンスの最適化」79ページで遭遇した）による集合メンバーシップの判定、**HyperLogLog** [72] によるカーディナリティ推定、各種パーセンタイル推定アルゴリズム（「実務におけるパーセンタイル」16ページ参照）などの確率的アルゴリズム（probabilistic algorithm）を使用することがある。確率的アルゴリズムは近似的な結果を生成するが、ストリームプロセッサで正確なアルゴリズムよりも大幅に少ないメモリで済むという利点がある。この近似アルゴリズムの使用により、ストリーム処理システムは常に非可逆（lossy）で不正確であると信じる人もいるが、それは誤りである：ストリーム処理について本質的に近似的なことは何もなく、確率的アルゴリズムは単なる最適化に過ぎない [73]。

分析を念頭に設計された多くのオープンソース分散ストリーム処理フレームワークがある：例えば **Apache Storm**、**Spark Streaming**、**Flink**、Concord、**Samza**、**Kafka Streams** [74]。ホステッドサービスには **Google Cloud Dataflow** と **Azure Stream Analytics** がある。

#### マテリアライズドビューの維持（Maintaining materialized views）

「データベースとストリーム」で見たように、データベースへの変更のストリームを使用して、キャッシュ、検索インデックス、データウェアハウスなどの導出データシステムをソースデータベースと最新の状態に保つことができる。これらの例を**マテリアライズドビュー（materialized view）** の維持（「集約：データキューブとマテリアライズドビュー」101ページ参照）の特定のケースと見なすことができる：あるデータセットの代替ビューを導出して効率的にクエリできるようにし、基礎となるデータが変更されるたびにそのビューを更新する [50]。

同様に、イベントソーシングではイベントのログを適用することでアプリケーション状態が維持される；ここでもアプリケーション状態はマテリアライズドビューの一種である。ストリーム分析のシナリオとは異なり、ある時間ウィンドウ内のイベントのみを考慮するだけでは通常不十分である：マテリアライズドビューの構築には、ログコンパクション（「ログコンパクション」456ページ参照）によって破棄された陳腐化イベントを除き、任意の期間にわたるすべてのイベントが潜在的に必要である。実質的に、時間の始まりまで遡るウィンドウが必要である。

原理的には、任意のストリームプロセッサをマテリアライズドビューの維持に使用できるが、イベントを永遠に維持する必要性は、限られた期間のウィンドウで主に動作する一部の分析指向フレームワークの前提に反する。**Samza** と **Kafka Streams** はこの種の使用法をサポートしており、Kafka のログコンパクションサポートの上に構築されている [75]。

#### ストリーム上の検索（Search on streams）

複数のイベントから成るパターンの検索を可能にする CEP に加えて、複雑な基準（全文検索クエリなど）に基づいて個々のイベントを検索する必要がある場合もある。

例えば、メディアモニタリングサービスはメディアアウトレットからのニュース記事や放送のフィードにサブスクライブし、企業、製品、または関心のあるトピックに言及するニュースを検索する。これは検索クエリを事前に策定し、ニュースアイテムのストリームをそのクエリと継続的にマッチングすることで行われる。一部の Web サイトにも類似の機能がある：例えば、不動産サイトのユーザーは、検索条件に一致する新しい物件が市場に出たときに通知を受けるよう依頼できる。Elasticsearch の **Percolator** 機能 [76] はこの種のストリーム検索を実装するための1つの選択肢である。

従来の検索エンジンはまずドキュメントをインデックス化し、次にインデックスに対してクエリを実行する。対照的に、ストリームの検索は処理を逆さまにする：CEP と同様に、クエリが保存され、ドキュメントがクエリを通過する。最も単純なケースでは、すべてのドキュメントをすべてのクエリに対してテストできるが、クエリの数が多い場合は遅くなる可能性がある。プロセスを最適化するために、ドキュメントだけでなくクエリもインデックス化し、マッチする可能性のあるクエリのセットを絞り込むことが可能である [77]。

#### メッセージパッシングと RPC（Message passing and RPC）

「メッセージパッシングデータフロー」（136ページ）では、RPC の代替としてのメッセージパッシングシステムについて議論した — すなわち、例えばアクターモデル（actor model）で使用されるような、サービスが通信するためのメカニズムである。これらのシステムもメッセージとイベントに基づいているが、通常はストリームプロセッサとは考えられない：

- **アクターフレームワーク（actor framework）** は主に、通信するモジュールの並行性と分散実行を管理するメカニズムであるのに対し、**ストリーム処理**は主にデータ管理技法である。
- アクター間の通信はしばしば一時的（ephemeral）で一対一（one-to-one）であるのに対し、イベントログは永続的（durable）でマルチサブスクライバー（multi-subscriber）である。
- アクターは任意の方法で通信でき（循環的なリクエスト/レスポンスパターンを含む）、ストリームプロセッサは通常、すべてのストリームが特定の1つのジョブの出力であり、明確に定義された入力ストリームのセットから導出される非循環パイプライン（acyclic pipeline）に設定される。

とはいえ、RPC ライクなシステムとストリーム処理の間にはいくつかの交差領域がある。例えば、Apache Storm には**分散 RPC（distributed RPC）** と呼ばれる機能があり、ユーザークエリをイベントストリームも処理するノードのセットにファームアウトでき；これらのクエリは入力ストリームからのイベントとインターリーブされ、結果は集約されてユーザーに返送できる [78]。（「マルチパーティションデータ処理」514ページも参照。）

アクターフレームワークを使用してストリームを処理することも可能である。しかし、多くのそのようなフレームワークはクラッシュ時のメッセージ配信を保証しないため、追加のリトライロジックを実装しない限り処理はフォールトトレラントではない。

### 時間についての推論（Reasoning About Time）

ストリームプロセッサは、特に分析目的で使用される場合に時間を扱う必要がしばしばある。分析は「直近5分間の平均」のような時間ウィンドウ（time window）を頻繁に使用する。「直近5分間」の意味は明確で曖昧さがないように思えるかもしれないが、残念ながらこの概念は驚くほどトリッキーである。

バッチ処理では、処理タスクが大量の過去のイベントのコレクションを素早く処理する。時間による何らかの分類が必要な場合、バッチ処理は各イベントに埋め込まれたタイムスタンプを参照する必要がある。バッチ処理を実行しているマシンのシステムクロックを見ても意味がない。なぜなら、プロセスが実行される時刻はイベントが実際に発生した時刻とは関係がないからである。

バッチ処理は数分以内に1年分の過去のイベントを読み取るかもしれない；ほとんどの場合、関心のあるタイムラインは1年の履歴であり、数分間の処理ではない。さらに、イベント内のタイムスタンプを使用することで処理が決定論的（deterministic）になる：同じ入力に対して同じ処理を再実行すれば同じ結果が得られる（「フォールトトレランス」422ページ参照）。

一方、多くのストリーム処理フレームワークは、ウィンドウの決定に処理マシンのローカルシステムクロック（**処理時刻（processing time）**）を使用する [79]。このアプローチはシンプルであるという利点があり、イベント作成とイベント処理の間の遅延が無視できるほど短い場合は妥当である。しかし、何らかの重大な処理遅延がある場合 — つまり処理がイベントが実際に発生した時刻よりもかなり後に行われる可能性がある場合 — このアプローチは破綻する。

#### イベント時刻と処理時刻（Event time versus processing time）

処理が遅延する理由は多くある：キューイング、ネットワーク障害（「信頼性のないネットワーク」277ページ参照）、メッセージブローカーやプロセッサにおける競合によるパフォーマンス問題、ストリームコンシューマーの再起動、障害からの回復中やコードのバグ修正後の過去のイベントの再処理（「古いメッセージの再生」451ページ参照）。

さらに、メッセージの遅延はメッセージの予測不能な順序にもつながりうる。例えば、ユーザーがまず1つの Web リクエストを行い（Web サーバー A が処理）、次に2番目のリクエストを行った（サーバー B が処理）とする。A と B は処理したリクエストを記述するイベントを発行するが、B のイベントが A のイベントより先にメッセージブローカーに到着する。ストリームプロセッサはまず B のイベント、次に A のイベントを見ることになるが、実際には逆の順序で発生していた。

類推が役立つなら、スター・ウォーズ（Star Wars）映画を考えてほしい：エピソード IV は1977年、エピソード V は1980年、エピソード VI は1983年に公開され、その後エピソード I, II, III がそれぞれ1999年、2002年、2005年に、そしてエピソード VII が2015年に公開された [80]。^ii 映画を公開順に観た場合、映画を処理した順序はナラティブの順序と一致しない。（エピソード番号はイベントタイムスタンプのようなものであり、映画を観た日付は処理時刻である。）人間としてはこのような不連続性に対処できるが、ストリーム処理アルゴリズムはこのようなタイミングと順序の問題に対応するよう明確に記述される必要がある。

イベント時刻と処理時刻を混同すると不正確なデータにつながる。例えば、リクエストの発生率を測定するストリームプロセッサ（1秒あたりのリクエスト数をカウント）があるとする。ストリームプロセッサを再デプロイすると、1分間シャットダウンされ、復帰時にイベントのバックログを処理するかもしれない。処理時刻に基づいてレートを測定すると、バックログの処理中に突然の異常なリクエストスパイクがあったように見えるが、実際にはリクエストの真のレートは安定していたのである。

#### 準備ができたかを知る（Knowing when you're ready）

イベント時刻の観点でウィンドウを定義する際のトリッキーな問題は、特定のウィンドウのすべてのイベントを受信したかどうか、あるいはまだ来るイベントがあるかどうかを確信できないことである。

例えば、1分間のウィンドウにイベントをグループ化して1分あたりのリクエスト数をカウントしているとする。時間帯の37分目のタイムスタンプを持ついくつかのイベントをカウントしており、時間は進み；今ではほとんどの入力イベントが38分目と39分目に該当する。37分目のウィンドウが完了したと宣言し、そのカウンター値を出力するのはいつか？

しばらく新しいイベントが来ない場合にタイムアウトしてウィンドウの準備ができたと宣言できるが、一部のイベントがどこかの別のマシンにバッファリングされていたり、ネットワーク中断のために遅延している可能性がある。ウィンドウが既に完了と宣言された後に到着するこのような**遅延イベント（straggler event）** を処理できる必要がある。大まかに2つの選択肢がある [1]：

1. 遅延イベントを無視する。通常の状況ではおそらくイベントの小さな割合であるため。ドロップされたイベントの数をメトリクスとして追跡し、大量のデータをドロップし始めた場合にアラートを発する。
2. 遅延イベントを含む更新された値の**補正（correction）** を公開する。以前の出力の撤回（retract）も必要になる場合がある。

場合によっては、「今後タイムスタンプ t より前のメッセージはもうない」ことを示す特別なメッセージを使用でき、コンシューマーがウィンドウをトリガーするために使用できる [81]。しかし、異なるマシン上の複数のプロデューサーがそれぞれ独自の最小タイムスタンプ閾値でイベントを生成している場合、コンシューマーは各プロデューサーを個別に追跡する必要がある。プロデューサーの追加と削除はこの場合より複雑になる。

#### 誰のクロックを使っているのか？（Whose clock are you using, anyway?）

イベントにタイムスタンプを割り当てることは、イベントがシステム内の複数のポイントでバッファリングされうる場合、さらに困難になる。例えば、使用メトリクスのイベントをサーバーに報告するモバイルアプリを考える。アプリはデバイスがオフラインの間に使用されるかもしれず、その場合はデバイス上でローカルにイベントをバッファリングし、次にインターネット接続が利用可能になったとき（数時間あるいは数日後かもしれない）にサーバーに送信する。このストリームのコンシューマーにとって、イベントは極端に遅延した遅延イベントとして現れる。

この文脈では、イベントのタイムスタンプは本来、モバイルデバイスのローカルクロックによるユーザーインタラクション発生時刻であるべきである。しかし、ユーザー制御のデバイスのクロックは、偶発的にまたは意図的に誤った時刻に設定されている可能性があるため、信頼できないことが多い（「クロックの同期と正確性」289ページ参照）。イベントがサーバーに受信された時刻（サーバーのクロックによる）はより正確である可能性が高いが、ユーザーインタラクションを記述するという点ではあまり意味がない。なぜならサーバーは自分の管理下にあるからである。

誤ったデバイスクロックを調整するための1つのアプローチは、3つのタイムスタンプを記録することである [82]：

- イベントが発生した時刻（デバイスクロックによる）
- イベントがサーバーに送信された時刻（デバイスクロックによる）
- イベントがサーバーに受信された時刻（サーバークロックによる）

2番目のタイムスタンプを3番目から差し引くことで、デバイスクロックとサーバークロックの間のオフセットを推定できる（ネットワーク遅延が必要なタイムスタンプ精度と比較して無視できると仮定して）。そのオフセットをイベントタイムスタンプに適用することで、イベントが実際に発生した真の時刻を推定できる（イベント発生からサーバーへの送信までの間にデバイスクロックオフセットが変化しなかったと仮定して）。

この問題はストリーム処理に固有のものではない — バッチ処理も時間についての推論のまったく同じ問題に悩まされる。ストリーミングの文脈でより目立つだけであり、時間の経過をより強く意識するためである。

#### ウィンドウの種類（Types of windows）

イベントのタイムスタンプがどのように決定されるべきかがわかれば、次のステップは時間期間にわたるウィンドウをどのように定義すべきかを決めることである。ウィンドウは集約（例えばイベントのカウントやウィンドウ内の値の平均計算）に使用できる。いくつかの種類のウィンドウが一般的に使用されている [79, 83]：

**タンブリングウィンドウ（Tumbling window）**
タンブリングウィンドウは固定長を持ち、すべてのイベントは正確に1つのウィンドウに属する。例えば、1分間のタンブリングウィンドウがある場合、タイムスタンプが10:03:00から10:03:59のすべてのイベントは1つのウィンドウにグループ化され、10:04:00から10:04:59のイベントは次のウィンドウに、という具合である。各イベントのタイムスタンプを取り、最も近い分に切り捨てることで、そのイベントが属するウィンドウを決定する1分間のタンブリングウィンドウを実装できる。

**ホッピングウィンドウ（Hopping window）**
ホッピングウィンドウも固定長を持つが、いくつかの平滑化を提供するためにウィンドウの重複を許可する。例えば、ホップサイズ1分の5分間ウィンドウは、10:03:00から10:07:59のイベントを含み、次のウィンドウは10:04:00から10:08:59のイベントを含む、という具合である。まず1分間のタンブリングウィンドウを計算し、次にいくつかの隣接するウィンドウにわたって集約することで、このホッピングウィンドウを実装できる。

**スライディングウィンドウ（Sliding window）**
スライディングウィンドウは、互いにある間隔以内に発生するすべてのイベントを含む。例えば、5分間のスライディングウィンドウは10:03:39と10:08:12のイベントをカバーする。これらは5分未満の間隔であるため（タンブリングおよびホッピング5分ウィンドウでは固定境界を使用するため、これら2つのイベントを同じウィンドウに入れないことに注意）。時間でソートされたイベントのバッファを保持し、ウィンドウから期限切れになった古いイベントを削除することで、スライディングウィンドウを実装できる。

**セッションウィンドウ（Session window）**
他のウィンドウタイプとは異なり、セッションウィンドウは固定の期間を持たない。代わりに、同じユーザーの時間的に近接して発生するすべてのイベントをグループ化し、ユーザーが一定期間非アクティブだった場合（例えば30分間イベントがなかった場合）にウィンドウが終了するように定義される。セッション化（sessionization）は Web サイト分析の一般的な要件である（「GROUP BY」406ページ参照）。

### ストリーム結合（Stream Joins）

Chapter 10 ではバッチジョブがキーでデータセットを結合する方法と、そのような結合がデータパイプラインの重要な部分を形成する方法について議論した。ストリーム処理は無界のデータセットのインクリメンタル処理へとデータパイプラインを一般化するため、ストリーム上の結合にもまったく同じニーズがある。

しかし、ストリーム上では新しいイベントがいつでも出現しうるという事実が、結合をバッチジョブよりも困難にする。状況をより良く理解するために、3つの異なる種類の結合を区別しよう：ストリーム-ストリーム結合（stream-stream join）、ストリーム-テーブル結合（stream-table join）、テーブル-テーブル結合（table-table join）[84]。以下のセクションでそれぞれを例で説明する。

#### ストリーム-ストリーム結合（ウィンドウ結合）（Stream-stream join / window join）

Web サイトに検索機能があり、検索された URL の最近のトレンドを検出したいとする。誰かが検索クエリを入力するたびに、クエリと返された結果を含むイベントをログに記録する。誰かが検索結果の1つをクリックするたびに、クリックを記録する別のイベントをログに記録する。検索結果の各 URL のクリック率（click-through rate）を計算するには、同じセッション ID で接続された検索アクションとクリックアクションのイベントを結び付ける必要がある。広告システムでも類似の分析が必要である [85]。

ユーザーが検索を中止すればクリックは来ないかもしれず、来たとしても検索からクリックまでの時間は非常に可変的である：多くの場合数秒だが、数日から数週間になることもある（ユーザーが検索を実行し、ブラウザのタブを忘れ、後でタブに戻って結果をクリックする場合）。可変的なネットワーク遅延のため、クリックイベントが検索イベントより先に到着する可能性さえある。結合に適切なウィンドウを選択できる — 例えば、最大1時間以内に発生した場合にクリックと検索を結合するよう選択できる。

検索の詳細をクリックイベントに埋め込むことは、イベントを結合するのと同等ではないことに注意：そうすると、ユーザーが検索結果をクリックしたケースについてしか分からず、ユーザーが結果のいずれもクリックしなかった検索については分からない。検索品質を測定するには正確なクリック率が必要であり、そのためには検索イベントとクリックイベントの両方が必要である。

この種の結合を実装するには、ストリームプロセッサが状態を維持する必要がある：例えば、直近1時間に発生したすべてのイベントをセッション ID でインデックス化して保持する。検索イベントまたはクリックイベントが発生するたびに、適切なインデックスに追加され、ストリームプロセッサは同じセッション ID の別のイベントが既に到着しているかどうかを確認するために他方のインデックスも確認する。一致するイベントがあれば、どの検索結果がクリックされたかを示すイベントを発行する。検索イベントが一致するクリックイベントを見ることなく期限切れになった場合、どの検索結果がクリックされなかったかを示すイベントを発行する。

#### ストリーム-テーブル結合（ストリームエンリッチメント）（Stream-table join / stream enrichment）

「ユーザーアクティビティイベントの分析の例」（404ページ）では、バッチジョブが2つのデータセット — ユーザーアクティビティイベントのセットとユーザープロフィールのデータベース — を結合する例を見た。ユーザーアクティビティイベントをストリームとして考え、ストリームプロセッサで継続的に同じ結合を実行することは自然である：入力はユーザー ID を含むアクティビティイベントのストリームであり、出力はユーザー ID がそのユーザーに関するプロフィール情報で拡充されたアクティビティイベントのストリームである。このプロセスはアクティビティイベントをデータベースからの情報で**エンリッチメント（enrichment）** すると呼ばれることがある。

この結合を実行するには、ストリームプロセスが一度に1つのアクティビティイベントを見て、そのイベントのユーザー ID をデータベースで検索し、プロフィール情報をアクティビティイベントに追加する必要がある。データベース検索はリモートデータベースにクエリすることで実装できるが、「ユーザーアクティビティイベントの分析の例」（404ページ）で議論したように、そのようなリモートクエリは遅くなりがちでデータベースの過負荷のリスクがある [75]。

別のアプローチは、データベースのコピーをストリームプロセッサにロードし、ネットワークラウンドトリップなしでローカルにクエリできるようにすることである。この技法は「Map 側の結合」（408ページ）で議論したハッシュ結合に非常に似ている：データベースのローカルコピーは、十分に小さければインメモリハッシュテーブル（in-memory hash table）であるかもしれないし、ローカルディスク上のインデックスであるかもしれない。

バッチジョブとの違いは、バッチジョブはデータベースのポイントインタイムスナップショット（point-in-time snapshot）を入力として使用するのに対し、ストリームプロセッサは長時間実行されるものであり、データベースの内容は時間の経過とともに変化する可能性が高いため、ストリームプロセッサのデータベースのローカルコピーを最新に保つ必要があることである。この問題は Change Data Capture で解決できる：ストリームプロセッサはアクティビティイベントのストリームに加えて、ユーザープロフィールデータベースの変更ログ（changelog）にもサブスクライブできる。プロフィールが作成または変更されると、ストリームプロセッサはローカルコピーを更新する。こうして、2つのストリームの結合を得る：アクティビティイベントとプロフィール更新である。

ストリーム-テーブル結合は実際にはストリーム-ストリーム結合と非常に類似している；最大の違いは、テーブル変更ログストリームの場合、結合は「時間の始まり」まで遡るウィンドウ（概念的に無限のウィンドウ）を使用し、レコードの新しいバージョンが古いバージョンを上書きすることである。ストリーム入力の場合、結合はウィンドウをまったく維持しないかもしれない。

#### テーブル-テーブル結合（マテリアライズドビューの維持）（Table-table join / materialized view maintenance）

「負荷の記述」（11ページ）で議論した Twitter のタイムラインの例を考える。ユーザーがホームタイムラインを見たいとき、フォローしているすべての人を反復し、最近のツイートを見つけてマージするのはコストが高すぎると述べた。

代わりに、タイムラインキャッシュ（timeline cache）— ツイートが送信されるときに書き込まれるユーザーごとの「受信箱」のようなもの — が必要であり、タイムラインの読み取りが単一のルックアップとなるようにする。このキャッシュのマテリアライゼーションと維持には以下のイベント処理が必要である：

- ユーザー u が新しいツイートを送信すると、u をフォローしているすべてのユーザーのタイムラインに追加される。
- ユーザーがツイートを削除すると、すべてのユーザーのタイムラインから削除される。
- ユーザー u1 がユーザー u2 のフォローを開始すると、u2 の最近のツイートが u1 のタイムラインに追加される。
- ユーザー u1 がユーザー u2 のフォローを解除すると、u2 のツイートが u1 のタイムラインから削除される。

このキャッシュ維持をストリームプロセッサで実装するには、ツイート（送信と削除）とフォロー関係（フォローとフォロー解除）のイベントストリームが必要である。ストリームプロセスは各ユーザーのフォロワーのセットを含むデータベースを維持する必要がある。これにより、新しいツイートが到着したときにどのタイムラインを更新する必要があるかがわかる [86]。

このストリーム処理の別の見方は、2つのテーブル（tweets と follows）を結合するクエリのマテリアライズドビューを維持しているということであり、以下のようなものである：

```sql
SELECT follows.follower_id AS timeline_id,
       array_agg(tweets.* ORDER BY tweets.timestamp DESC)
FROM tweets
JOIN follows ON follows.followee_id = tweets.sender_id
GROUP BY follows.follower_id
```

ストリームの結合はそのクエリ内のテーブルの結合に直接対応する。タイムラインは事実上このクエリの結果のキャッシュであり、基礎となるテーブルが変更されるたびに更新される。^iii

#### 結合の時間依存性（Time-dependence of joins）

ここで説明した3種類の結合（ストリーム-ストリーム、ストリーム-テーブル、テーブル-テーブル）には多くの共通点がある：いずれもストリームプロセッサが1つの結合入力に基づいて何らかの状態（検索およびクリックイベント、ユーザープロフィール、フォロワーリスト）を維持し、他方の結合入力からのメッセージでその状態をクエリすることを必要とする。

状態を維持するイベントの順序は重要である（まずフォローしてからフォロー解除するのか、その逆なのかは重要である）。パーティション化されたログでは、単一パーティション内のイベントの順序は保持されるが、異なるストリームやパーティション間での順序保証は通常ない。

これは疑問を提起する：異なるストリーム上のイベントがほぼ同時に発生した場合、どの順序で処理されるのか？ストリーム-テーブル結合の例では、ユーザーがプロフィールを更新した場合、どのアクティビティイベントが古いプロフィール（プロフィール更新前に処理されたもの）と結合され、どれが新しいプロフィール（プロフィール更新後に処理されたもの）と結合されるのか？別の言い方をすれば：状態が時間とともに変化し、何らかの状態と結合する場合、結合にはどの時点を使用するのか？ [45]

このような時間依存性は多くの場面で発生しうる。例えば、物品を販売する場合、請求書に正しい税率を適用する必要があるが、これは国や州、製品の種類、販売日に依存する（税率は時々変わるため）。売上を税率テーブルと結合する場合、おそらく販売時点の税率と結合したいだろう。過去のデータを再処理している場合、これは現在の税率とは異なるかもしれない。

ストリーム間のイベントの順序が不確定な場合、結合は**非決定的（nondeterministic）** になる [87]。これは、同じ入力に対して同じジョブを再実行しても必ずしも同じ結果が得られないことを意味する：入力ストリーム上のイベントはジョブを再実行したときに異なる方法でインターリーブされるかもしれない。

データウェアハウスでは、この問題は**緩やかに変化するディメンション（SCD: Slowly Changing Dimension）** として知られ、結合されるレコードの特定のバージョンに一意な識別子を使用することでしばしば対処される：例えば、税率が変わるたびに新しい識別子が与えられ、請求書には販売時点の税率の識別子が含まれる [88, 89]。この変更により結合は決定的になるが、テーブル内のレコードのすべてのバージョンを保持する必要があるため、ログコンパクションが不可能になるという結果がある。

### 耐障害性（Fault Tolerance）

本章の最後のセクションでは、ストリームプロセッサがどのように障害を許容できるかを考える。Chapter 10 ではバッチ処理フレームワークが障害をかなり容易に許容できることを見た：MapReduce ジョブのタスクが失敗した場合、別のマシンで単純に再起動でき、失敗したタスクの出力は破棄される。この透過的なリトライが可能なのは、入力ファイルがイミュータブルであり、各タスクが HDFS 上の別のファイルに出力を書き込み、出力はタスクが正常に完了したときにのみ可視化されるからである。

特に、障害に対するバッチアプローチは、バッチジョブの出力が、たとえ実際にはいくつかのタスクが失敗していたとしても、何も問題がなかったかのように同じであることを保証する。すべての入力レコードが正確に1回処理されたかのように見える — レコードはスキップされず、2度処理されることもない。タスクの再起動によりレコードが実際には複数回処理されるかもしれないが、出力における可視的な効果はまるで1回だけ処理されたかのようなものである。この原理は **exactly-once semantics（正確に1回のセマンティクス）** として知られるが、**effectively-once** がより適切な用語であろう [90]。

同じ障害耐性の問題がストリーム処理でも生じるが、処理はそれほど単純ではない：タスクが完了するまで出力を可視化しないで待つことは選択肢ではない。なぜならストリームは無限であり、処理が完了することは決してないからである。

#### マイクロバッチングとチェックポインティング（Microbatching and checkpointing）

1つの解決策は、ストリームを小さなブロックに分割し、各ブロックをミニチュアのバッチ処理として扱うことである。このアプローチは**マイクロバッチング（microbatching）** と呼ばれ、**Spark Streaming** [91] で使用されている。バッチサイズは通常約1秒であり、これはパフォーマンスの妥協の結果である：より小さいバッチはより大きなスケジューリングと調整のオーバーヘッドを生じ、より大きいバッチはストリームプロセッサの結果が可視化されるまでのより長い遅延を意味する。

マイクロバッチングはまた、バッチサイズに等しいタンブリングウィンドウ（tumbling window）を暗黙的に提供する（イベントタイムスタンプではなく処理時刻によるウィンドウ）；より大きなウィンドウを必要とするジョブは、あるマイクロバッチから次のマイクロバッチへ状態を明示的に引き継ぐ必要がある。

**Apache Flink** で使用されるバリアントアプローチは、状態のローリングチェックポイント（rolling checkpoint）を定期的に生成し、永続ストレージに書き込むことである [92, 93]。ストリームオペレーターがクラッシュした場合、最新のチェックポイントから再起動し、最後のチェックポイントとクラッシュの間に生成された出力を破棄できる。チェックポイントはメッセージストリーム内のバリアー（barrier）によってトリガーされ、マイクロバッチ間の境界と似ているが、特定のウィンドウサイズを強制しない。

ストリーム処理フレームワークの範囲内では、マイクロバッチングとチェックポインティングのアプローチはバッチ処理と同じ exactly-once セマンティクスを提供する。しかし、出力がストリームプロセッサを離れると（例えばデータベースへの書き込み、外部メッセージブローカーへのメッセージ送信、メール送信によって）、フレームワークは失敗したバッチの出力を破棄できなくなる。この場合、失敗したタスクの再起動により外部の副作用が2度発生し、マイクロバッチングやチェックポインティングだけではこの問題を防ぐのに不十分である。

#### アトミックコミットの再考（Atomic commit revisited）

障害がある場合に exactly-once 処理の外観を与えるためには、イベントの処理に関するすべての出力と副作用が、処理が成功した場合にのみ効力を発するようにする必要がある。それらの効果には、下流のオペレーターや外部メッセージングシステム（メールやプッシュ通知を含む）に送信されたメッセージ、データベースへの書き込み、オペレーター状態への変更、入力メッセージの確認応答（ログベースメッセージブローカーでのコンシューマーオフセットの前進を含む）が含まれる。

これらすべてがアトミックに（atomically）発生する必要がある。つまり、すべてが起きるか、何も起きないかのいずれかであるべきであり、互いに同期が外れるべきではない。このアプローチに聞き覚えがあるなら、分散トランザクションと2相コミット（two-phase commit）の文脈で「正確に1回のメッセージ処理」（360ページ）で議論したためである。

Chapter 9 では XA などの分散トランザクションの従来の実装における問題点を議論した。しかし、より制限された環境では、このようなアトミックコミット機能を効率的に実装することが可能である。このアプローチは **Google Cloud Dataflow** [81, 92] と **VoltDB** [94] で使用されており、Apache Kafka にも同様の機能を追加する計画がある [95, 96]。XA とは異なり、これらの実装は異種技術間のトランザクションを提供しようとせず、代わりにストリーム処理フレームワーク内で状態変更とメッセージングの両方を管理することで内部に保つ。トランザクションプロトコルのオーバーヘッドは、単一のトランザクション内で複数の入力メッセージを処理することで償却できる。

#### 冪等性（Idempotence）

目標は、失敗したタスクの部分出力を破棄し、2度効力を発することなく安全にリトライできるようにすることである。分散トランザクションはその目標を達成する1つの方法であるが、別の方法は**冪等性（idempotence）** に依存することである [97]。

冪等な操作（idempotent operation）とは、複数回実行しても1回だけ実行したのと同じ効果を持つ操作である。例えば、キーバリューストアでキーを固定値に設定することは冪等である（同じ値を再度書き込んでも同一の値で上書きするだけ）のに対し、カウンターをインクリメントすることは冪等ではない（インクリメントを再度実行すると値が2度インクリメントされる）。

操作が本質的に冪等でなくても、少しの追加メタデータで冪等にできることが多い。例えば、Kafka からメッセージを消費する場合、すべてのメッセージは永続的で単調増加するオフセットを持つ。外部データベースに値を書き込む際、最後の書き込みをトリガーしたメッセージのオフセットを値とともに含めることができる。こうして、更新が既に適用されているかどうかを判断し、同じ更新を再度実行することを避けられる。

Storm の Trident における状態処理は同様のアイデアに基づいている [78]。冪等性に依存するにはいくつかの仮定を含む：失敗したタスクの再起動は同じメッセージを同じ順序で再生しなければならず（ログベースメッセージブローカーはこれを行う）、処理は決定論的（deterministic）でなければならず、他のノードが同じ値を並行して更新してはならない [98, 99]。

あるプロセッシングノードから別のノードへのフェイルオーバー時には、実際には死んでいるが実際には生きているノードからの干渉を防ぐために**フェンシング（fencing）** が必要になる場合がある（「リーダーとロック」301ページ参照）。これらすべての注意事項にもかかわらず、冪等な操作は小さなオーバーヘッドで exactly-once セマンティクスを達成する効果的な方法となりうる。

#### 障害後の状態再構築（Rebuilding state after a failure）

状態を必要とするストリーム処理 — 例えば、ウィンドウ集約（カウンター、平均、ヒストグラム）や結合に使用されるテーブルやインデックス — は、障害後にその状態を回復できることを保証しなければならない。

1つの選択肢は、リモートデータストアに状態を保持しレプリケートすることであるが、「ストリーム-テーブル結合（ストリームエンリッチメント）」（473ページ）で議論したように、個々のメッセージごとにリモートデータベースにクエリする必要があると遅くなる可能性がある。別の方法は、ストリームプロセッサにローカルに状態を保持し、定期的にレプリケートすることである。ストリームプロセッサが障害から回復する際、新しいタスクはレプリケートされた状態を読み取り、データを失うことなく処理を再開できる。

例えば、**Flink** は定期的にオペレーター状態のスナップショットをキャプチャし、HDFS などの永続ストレージに書き込む [92, 93]。**Samza** と **Kafka Streams** は状態変更をログコンパクション付きの専用 Kafka トピックに送信することでレプリケートする。これは Change Data Capture と類似している [84, 100]。**VoltDB** は各入力メッセージを複数のノードで冗長的に処理することで状態をレプリケートする（「実際のシリアル実行」252ページ参照）。

場合によっては、状態をレプリケートする必要がないこともある。入力ストリームから再構築できるからである。例えば、状態がかなり短いウィンドウにわたる集約で構成される場合、そのウィンドウに対応する入力イベントを単純に再生する方が十分に速いかもしれない。状態が Change Data Capture によって維持されるデータベースのローカルレプリカである場合、データベースはログコンパクトされた変更ストリーム（「ログコンパクション」456ページ参照）から再構築することもできる。

しかし、これらすべてのトレードオフは基礎となるインフラストラクチャのパフォーマンス特性に依存する：一部のシステムではネットワーク遅延がディスクアクセスレイテンシより低く、ネットワーク帯域幅がディスク帯域幅に匹敵する場合がある。すべての状況に普遍的に理想的なトレードオフは存在せず、ストレージとネットワーク技術の進化とともに、ローカル状態対リモート状態の利点も変化しうる。

## まとめ（Summary）

本章ではイベントストリーム、それらが果たす目的、およびそれらの処理方法について議論した。いくつかの点で、ストリーム処理は Chapter 10 で議論したバッチ処理と非常に似ているが、固定サイズの入力ではなく無界（never-ending）のストリーム上で継続的に行われる。この観点から、メッセージブローカーとイベントログはファイルシステムのストリーミング版として機能する。

2種類のメッセージブローカーの比較に時間を費やした：

**AMQP/JMS スタイルのメッセージブローカー**
ブローカーが個々のメッセージをコンシューマーに割り当て、コンシューマーは正常に処理された個々のメッセージに対して確認応答する。確認応答されるとメッセージはブローカーから削除される。このアプローチは RPC の非同期形式として（例えばタスクキューにおいて）適切であり、メッセージ処理の正確な順序が重要でなく、処理済みの古いメッセージに戻って読み直す必要がない場合に適している。

**ログベースメッセージブローカー**
ブローカーがパーティション内のすべてのメッセージを同じコンシューマーノードに割り当て、常に同じ順序でメッセージを配信する。パーティショニングによって並列性が達成され、コンシューマーは処理済みの最後のメッセージのオフセットをチェックポインティングすることで進行状況を追跡する。ブローカーはメッセージをディスクに保持するため、必要に応じて戻って古いメッセージを再読み取りすることが可能である。

ログベースのアプローチはデータベースのレプリケーションログ（Chapter 5 参照）やログ構造化ストレージエンジン（Chapter 3 参照）に類似している。このアプローチは入力ストリームを消費し導出状態や導出出力ストリームを生成するストリーム処理システムに特に適していることを見た。

ストリームの出所に関しては、いくつかの可能性を議論した：ユーザーアクティビティイベント、定期的な測定値を提供するセンサー、データフィード（例えば金融の市場データ）は自然にストリームとして表現される。データベースへの書き込みもストリームとして考えることが有用であることを見た：変更ログ（changelog）— つまりデータベースに対するすべての変更の履歴 — を Change Data Capture によって暗黙的に、またはイベントソーシングによって明示的にキャプチャできる。ログコンパクションにより、ストリームがデータベースの内容の完全なコピーを保持することが可能になる。

データベースをストリームとして表現することは、システム統合のための強力な機会を開く。変更のログを消費して導出システムに適用することで、検索インデックス、キャッシュ、分析システムなどの導出データシステムを継続的に最新の状態に保つことができる。ゼロから始めて変更のログを最初から現在まで消費することで、既存のデータに対する新しいビューを構築することさえできる。

ストリームとしての状態の維持とメッセージの再生の機能は、様々なストリーム処理フレームワークにおけるストリーム結合とフォールトトレランスを可能にする技法の基盤でもある。ストリーム処理のいくつかの目的を議論した：イベントパターンの検索（複合イベント処理）、ウィンドウ集約の計算（ストリーム分析）、導出データシステムの最新維持（マテリアライズドビュー）。

次に、ストリームプロセッサにおける時間の推論の困難さについて議論した。処理時刻とイベントタイムスタンプの区別、およびウィンドウが完了したと思った後に到着する遅延イベントへの対処の問題を含む。

ストリーム処理に現れる3種類の結合を区別した：

**ストリーム-ストリーム結合（Stream-stream joins）**
両方の入力ストリームがアクティビティイベントで構成され、結合オペレーターがある時間ウィンドウ内で発生した関連イベントを検索する。例えば、同じユーザーが30分以内に取った2つのアクションをマッチングする。2つの結合入力は実際には同じストリーム（セルフジョイン）である場合もある。

**ストリーム-テーブル結合（Stream-table joins）**
一方の入力ストリームがアクティビティイベントで構成され、他方がデータベース変更ログである。変更ログはデータベースのローカルコピーを最新に保つ。各アクティビティイベントに対して、結合オペレーターがデータベースにクエリし、エンリッチされたアクティビティイベントを出力する。

**テーブル-テーブル結合（Table-table joins）**
両方の入力ストリームがデータベース変更ログである。この場合、一方のすべての変更が他方の最新状態と結合される。結果は2つのテーブルの結合のマテリアライズドビューに対する変更のストリームである。

最後に、ストリームプロセッサで exactly-once セマンティクスを達成するための技法について議論した。バッチ処理と同様に、失敗したタスクの部分出力を破棄する必要がある。しかし、ストリーム処理は長時間実行され継続的に出力を生成するため、すべての出力を単に破棄することはできない。代わりに、マイクロバッチング、チェックポインティング、トランザクション、または冪等な書き込みに基づくより細粒度の回復メカニズムを使用できる。
