# 第3章：ストレージと検索

> *Wer Ordnung hält, ist nur zu faul zum Suchen.*
> （きちんと整理する人は、探すのが面倒なだけだ。）
> ――ドイツのことわざ

最も基本的なレベルにおいて、データベースは2つのことを行う必要がある。データを受け取ったらそれを格納し、後で求められたらそのデータを返すことである。

第2章ではデータモデルとクエリ言語について議論した。すなわち、アプリケーション開発者がデータベースにデータを渡すフォーマットと、後でデータを要求するメカニズムについてである。本章では同じことをデータベースの視点から議論する。与えられたデータをどのように格納し、要求されたときにどうやって見つけ出すかを考える。

アプリケーション開発者として、データベースがストレージと検索を内部的にどのように扱っているかを気にすべきなのはなぜか？ おそらく自分でストレージエンジンをゼロから実装することはないだろうが、利用可能な多くの選択肢の中からアプリケーションに適したストレージエンジンを選ぶ必要がある。ストレージエンジンを自分のワークロードに合わせてチューニングするには、内部で何が行われているかについて大まかなイメージを持つ必要がある。

特に、トランザクション処理に最適化されたストレージエンジンと分析に最適化されたストレージエンジンの間には大きな違いがある。この区別は後ほど「トランザクション処理か分析か？」のセクションで探求し、「列指向ストレージ」のセクションでは分析に最適化されたストレージエンジンのファミリーについて議論する。

しかしまず、おそらくあなたが馴染みのある種類のデータベースで使われているストレージエンジンの話から本章を始める。伝統的なリレーショナルデータベースと、多くのいわゆるNoSQLデータベースである。2つのストレージエンジンのファミリーを検討する：log-structured（ログ構造化）ストレージエンジンと、B-treeのようなpage-oriented（ページ指向）ストレージエンジンである。

## データベースを支えるデータ構造

世界で最もシンプルなデータベースを考えてみよう。2つのBash関数で実装される：

```bash
#!/bin/bash
db_set () {
    echo "$1,$2" >> database
}
db_get () {
    grep "^$1," database | sed -e "s/^$1,//" | tail -n 1
}
```

これら2つの関数はkey-value store（キーバリューストア）を実装している。`db_set key value` を呼び出すとデータベースにキーと値が格納される。キーと値は（ほぼ）何でもよい——たとえば値はJSONドキュメントでもよい。次に `db_get key` を呼び出すと、そのキーに関連付けられた最新の値を検索して返す。

そしてそれは動作する：

```
$ db_set 123456 '{"name":"London","attractions":["Big Ben","London Eye"]}'
$ db_set 42 '{"name":"San Francisco","attractions":["Golden Gate Bridge"]}'
$ db_get 42
{"name":"San Francisco","attractions":["Golden Gate Bridge"]}
```

基盤となるストレージフォーマットは非常にシンプルで、各行がカンマで区切られたキーバリューペアを含むテキストファイルである（おおよそCSVファイルのようなもの、エスケープの問題は無視する）。`db_set` を呼ぶたびにファイルの末尾に追記されるため、キーを何回更新しても古い値は上書きされない——最新の値を見つけるにはファイル内のキーの最後の出現箇所を見る必要がある（そのため `db_get` では `tail -n 1` を使っている）：

```
$ db_set 42 '{"name":"San Francisco","attractions":["Exploratorium"]}'
$ db_get 42
{"name":"San Francisco","attractions":["Exploratorium"]}
$ cat database
123456,{"name":"London","attractions":["Big Ben","London Eye"]}
42,{"name":"San Francisco","attractions":["Golden Gate Bridge"]}
42,{"name":"San Francisco","attractions":["Exploratorium"]}
```

`db_set` 関数は、これほどシンプルなものとしてはかなり良いパフォーマンスを持つ。なぜならファイルへの追記は一般的に非常に効率的だからだ。`db_set` がやっていることと同様に、多くのデータベースは内部的にlog（ログ）を使用する。ログとはappend-only（追記のみ）のデータファイルである。実際のデータベースにはさらに多くの問題がある（並行性制御、ログが際限なく大きくならないようにディスクスペースを回収すること、エラーや部分的に書き込まれたレコードの処理など）が、基本原理は同じである。ログは信じられないほど有用で、本書の残りで何度も出てくる。

> logという語は、アプリケーションが何が起きているかを説明するテキストを出力するアプリケーションログを指すのによく使われる。本書ではlogはより一般的な意味で使用する：append-onlyのレコードの列。人間に読めるものである必要はなく、バイナリで他のプログラムだけが読むことを意図したものかもしれない。

一方で、`db_get` 関数はデータベースに大量のレコードがある場合、ひどいパフォーマンスになる。キーを検索するたびに、`db_get` はデータベースファイル全体を最初から最後までスキャンし、キーの出現箇所を探さなければならない。アルゴリズムの用語で言えば、検索のコストはO(n)である：データベースのレコード数nが2倍になれば、検索にかかる時間も2倍になる。それはよくない。

データベース内の特定のキーに対する値を効率的に見つけるには、別のデータ構造が必要である：index（インデックス）である。本章ではさまざまなインデックス構造を見て、それらを比較する。それらの背後にある一般的なアイデアは、道しるべの役割を果たし、欲しいデータの場所を見つける手助けをする追加のメタデータを脇に保持しておくことである。同じデータを複数の異なる方法で検索したい場合は、データの異なる部分に対して複数の異なるインデックスが必要かもしれない。

インデックスはプライマリデータから派生した追加の構造である。多くのデータベースではインデックスの追加と削除が可能で、これはデータベースの内容には影響しない。クエリのパフォーマンスにのみ影響する。追加の構造を維持するにはオーバーヘッドがかかり、特に書き込みにおいてそうである。書き込みに関しては、単純にファイルに追記するパフォーマンスに勝つのは難しい。なぜならそれは最もシンプルな書き込み操作だからだ。いかなる種類のインデックスも通常は書き込みを遅くする。データが書き込まれるたびにインデックスも更新する必要があるからだ。

これはストレージシステムにおける重要なトレードオフである：適切に選ばれたインデックスは読み取りクエリを高速化するが、すべてのインデックスは書き込みを遅くする。このため、データベースは通常デフォルトですべてにインデックスを付けるのではなく、アプリケーションの典型的なクエリパターンの知識を使って、アプリケーション開発者やデータベース管理者がインデックスを手動で選択することを求める。必要以上のオーバーヘッドを導入せずに、アプリケーションに最大の恩恵をもたらすインデックスを選べるのである。

### Hash Indexes（ハッシュインデックス）

キーバリューデータ用のインデックスから始めよう。これはインデックスを付けられる唯一の種類のデータではないが、非常に一般的であり、より複雑なインデックスの有用な構成要素である。

キーバリューストアは、ほとんどのプログラミング言語にある辞書型にかなり似ており、通常はhash map（ハッシュマップ、ハッシュテーブル）として実装される。ハッシュマップは多くのアルゴリズムの教科書で説明されている [1, 2] ので、ここではその動作の詳細には入らない。すでにインメモリのデータ構造としてハッシュマップがあるのだから、ディスク上のデータのインデックスにもそれを使えばよいのではないか？

データストレージが前述の例のようにファイルへの追記だけで構成されているとしよう。すると最もシンプルなインデックス戦略はこうなる：すべてのキーがデータファイル内のバイトオフセット——値が見つかる場所——にマッピングされたインメモリハッシュマップを保持する（Figure 3-1に示すとおり）。新しいキーバリューペアをファイルに追記するたびに、書き込んだデータのオフセットを反映するようにハッシュマップも更新する（これは新しいキーの挿入と既存のキーの更新の両方で機能する）。値を検索したいときは、ハッシュマップを使ってデータファイル内のオフセットを見つけ、その場所にシークし、値を読む。

*Figure 3-1. CSV風のフォーマットでキーバリューペアのログを格納し、インメモリハッシュマップでインデックスを付ける。*

これは単純に聞こえるかもしれないが、実行可能なアプローチである。実際、これは本質的にBitcask（Riakのデフォルトストレージエンジン）[3] がやっていることである。Bitcaskは高パフォーマンスの読み書きを提供するが、ハッシュマップは完全にメモリ上に保持されるため、すべてのキーが利用可能なRAMに収まるという要件がある。値は利用可能なメモリよりも多くのスペースを使用できる。ディスクからたった1回のディスクシークで読み込めるからだ。データファイルのその部分がすでにファイルシステムキャッシュにある場合、読み取りにディスクI/Oはまったく不要である。

Bitcaskのようなストレージエンジンは、各キーの値が頻繁に更新される状況に適している。たとえば、キーが猫の動画のURLで、値がその再生回数（誰かが再生ボタンを押すたびにインクリメントされる）かもしれない。この種のワークロードでは書き込みが多いが、異なるキーの数はそれほど多くない——キーあたりの書き込み数は大きいが、すべてのキーをメモリに保持するのは現実的である。

ここまでの説明では、ファイルに追記するだけである——ではディスクスペースがいつか不足するのをどう避けるのか？ 良い解決策は、ログを一定サイズのsegment（セグメント）に分割することである。セグメントファイルが一定のサイズに達したら閉じ、以降の書き込みを新しいセグメントファイルに行う。次にこれらのセグメントに対してcompaction（コンパクション）を実行できる（Figure 3-2に示すとおり）。コンパクションとは、ログ内の重複キーを捨て、各キーの最新の更新のみを保持することを意味する。

*Figure 3-2. キーバリュー更新ログのコンパクション（各猫動画の再生回数をカウント）。各キーの最新の値のみを保持する。*

さらに、コンパクションはセグメントをしばしば大幅に小さくするため（1つのセグメント内でキーが平均して何回も上書きされると仮定して）、コンパクションを実行すると同時に複数のセグメントをマージすることもできる（Figure 3-3に示すとおり）。セグメントは書き込まれた後は変更されないので、マージされたセグメントは新しいファイルに書き込まれる。凍結されたセグメントのマージとコンパクションはバックグラウンドスレッドで行うことができ、その間も古いセグメントファイルを使って通常通り読み書きリクエストの処理を続けられる。マージプロセスが完了したら、読み取りリクエストを新しいマージ済みセグメントに切り替え、古いセグメントファイルは単純に削除できる。

*Figure 3-3. コンパクションとセグメントマージを同時に実行する。*

各セグメントはここで独自のインメモリハッシュテーブルを持ち、キーをファイルオフセットにマッピングする。キーの値を見つけるには、まず最新のセグメントのハッシュマップを確認し、キーが存在しなければ2番目に新しいセグメントを確認し、以下同様に行う。マージプロセスはセグメント数を小さく保つので、検索で多くのハッシュマップを確認する必要はない。

このシンプルなアイデアを実際に動作させるには多くの詳細が必要になる。簡潔に、実際の実装で重要な問題のいくつかを挙げる：

**ファイルフォーマット**
CSVはログに最適なフォーマットではない。文字列の長さをバイト数でまずエンコードし、その後に生の文字列を続けるバイナリフォーマットの方が高速でシンプルである（エスケープが不要）。

**レコードの削除**
キーとその関連する値を削除したい場合、データファイルにtombstone（墓標）と呼ばれる特別な削除レコードを追記する必要がある。ログセグメントがマージされるとき、tombstoneはマージプロセスに削除されたキーのそれ以前の値をすべて破棄するよう指示する。

**クラッシュリカバリ**
データベースが再起動すると、インメモリハッシュマップは失われる。原理的には、各セグメントファイルを最初から最後まで読み、各キーの最新の値のオフセットを記録していくことで、各セグメントのハッシュマップを復元できる。しかし、セグメントファイルが大きい場合、これには長い時間がかかり、サーバーの再起動が苦痛になる。Bitcaskは各セグメントのハッシュマップのスナップショットをディスクに保存することでリカバリを高速化しており、それをメモリに素早くロードできる。

**部分的に書き込まれたレコード**
データベースはいつでもクラッシュする可能性があり、ログへのレコードの追記途中を含む。Bitcaskファイルにはチェックサムが含まれており、そのような破損したログの部分を検出して無視できる。

**並行性制御**
書き込みはログに厳密に順序通りに追記されるため、一般的な実装の選択は書き込みスレッドを1つだけ持つことである。データファイルのセグメントはappend-onlyであり、それ以外は不変であるため、複数のスレッドで並行して読み取ることができる。

append-onlyのログは一見無駄に見える。なぜ古い値を新しい値で上書きして、ファイルをその場で更新しないのか？ しかしappend-onlyの設計にはいくつかの理由で優れている：

- 追記とセグメントマージはsequential write（シーケンシャル書き込み）操作であり、一般的にrandom write（ランダム書き込み）よりもはるかに高速で、特に磁気回転ディスクのハードドライブでそうである。ある程度、フラッシュベースのSSDでもシーケンシャル書き込みの方が好ましい [4]。この問題は「B-TreeとLSM-Treeの比較」でさらに議論する。
- セグメントファイルがappend-onlyまたは不変であれば、並行性とクラッシュリカバリはずっとシンプルになる。たとえば、値の上書き中にクラッシュが発生し、ファイルに古い値の一部と新しい値の一部がつなぎ合わされた状態になるケースを心配する必要がない。
- 古いセグメントのマージにより、データファイルが時間とともに断片化する問題が回避される。

しかし、ハッシュテーブルインデックスにも制限がある：

- ハッシュテーブルはメモリに収まる必要があるため、非常に多数のキーがある場合は困る。原理的にはディスク上にハッシュマップを維持することもできるが、残念ながらディスク上のハッシュマップを良いパフォーマンスで動作させるのは難しい。大量のランダムアクセスI/Oが必要で、満杯になったときの拡張にコストがかかり、ハッシュの衝突には面倒なロジックが必要である [5]。
- 範囲クエリが効率的でない。たとえば、kitty00000からkitty99999の間のすべてのキーを簡単にスキャンすることはできない——ハッシュマップで各キーを個別に検索する必要がある。

### SSTables and LSM-Trees（SSTableとLSM-Tree）

Figure 3-3では、各ログ構造化ストレージセグメントはキーバリューペアの列である。これらのペアは書き込まれた順序で並んでおり、ログ内で後の値は同じキーの前の値に優先する。それ以外に、ファイル内のキーバリューペアの順序は問題ではない。

ここでセグメントファイルのフォーマットに簡単な変更を加えよう：キーバリューペアの列がキーでソートされていることを要件とする。一見するとこの要件はシーケンシャル書き込みの能力を壊すように見えるが、それについてはすぐに説明する。

このフォーマットをSorted String Table、略してSSTable（ソート済み文字列テーブル）と呼ぶ。また、各マージ済みセグメントファイル内でキーが1回だけ出現することも要件とする（コンパクションプロセスがすでにそれを保証している）。SSTableにはハッシュインデックス付きのログセグメントと比較していくつかの大きな利点がある：

1. **セグメントのマージがシンプルで効率的であり**、ファイルが利用可能なメモリより大きくてもそうである。アプローチはmergesort（マージソート）アルゴリズムで使われるものに似ている（Figure 3-4に示す）：入力ファイルを並行して読み始め、各ファイルの最初のキーを見て、（ソート順で）最小のキーを出力ファイルにコピーし、繰り返す。これにより、やはりキーでソートされた新しいマージ済みセグメントファイルが生成される。

*Figure 3-4. 複数のSSTableセグメントをマージし、各キーの最新の値のみを保持する。*

同じキーが複数の入力セグメントに現れた場合はどうなるか？ 各セグメントはデータベースのある期間中に書き込まれたすべての値を含んでいることを思い出してほしい。これは、一つの入力セグメントのすべての値が他のセグメントのすべての値より新しいということを意味する（常に隣接するセグメントをマージすると仮定して）。複数のセグメントが同じキーを含む場合、最新のセグメントの値を保持し、古いセグメントの値を破棄できる。

2. **ファイル内の特定のキーを見つけるために、すべてのキーのインデックスをメモリに保持する必要がなくなる。** Figure 3-5を例に見てみよう：キー handiwork を探しているが、セグメントファイル内でそのキーの正確なオフセットを知らないとする。しかし、キー handbag と handsome のオフセットは知っており、ソートのおかげで handiwork はこの2つの間に現れるはずだとわかる。つまり handbag のオフセットにジャンプし、そこからスキャンして handiwork を見つけられる（またはそのキーがファイルに存在しなければ見つからない）。

*Figure 3-5. インメモリインデックスを持つSSTable。*

一部のキーのオフセットを教えるインメモリインデックスはまだ必要だが、sparse（疎）でよい：セグメントファイルの数キロバイトごとに1つのキーで十分である。なぜなら数キロバイトは非常に素早くスキャンできるからだ。^i

3. **読み取りリクエストは要求された範囲内の複数のキーバリューペアをスキャンする必要があるため**、それらのレコードをブロックにまとめ、ディスクに書き込む前に圧縮することが可能である（Figure 3-5の網掛け部分で示す）。疎なインメモリインデックスの各エントリは、圧縮されたブロックの先頭を指す。ディスクスペースの節約に加え、圧縮はI/O帯域幅の使用も削減する。

^i すべてのキーと値が固定サイズであれば、セグメントファイルに対して二分探索を使い、インメモリインデックスを完全に回避できる。しかし実際にはそれらは通常可変長であり、インデックスなしでは1つのレコードがどこで終わり次のレコードがどこから始まるかを判断するのは難しい。

#### SSTableの構築と維持

ここまではよいが、そもそもデータをキーでソートされた状態にするにはどうすればよいのか？ 書き込みは任意の順序で来る可能性がある。

ディスク上でソート済み構造を維持することは可能だが（「B-Tree」を参照）、メモリ上で維持する方がずっと簡単である。red-black tree（赤黒木）やAVL tree（AVL木）[2] のような、よく知られた木構造データ構造がたくさんある。これらのデータ構造を使えば、キーを任意の順序で挿入し、ソートされた順序で読み返すことができる。

ストレージエンジンを次のように動作させることができる：

- 書き込みが来たら、インメモリのバランス木データ構造（たとえば赤黒木）に追加する。このインメモリの木はmemtable（メムテーブル）と呼ばれることがある。
- memtableがある閾値（通常数メガバイト）より大きくなったら、SSTableファイルとしてディスクに書き出す。木はすでにキーバリューペアをキーでソートされた状態で維持しているため、これは効率的に行える。新しいSSTableファイルはデータベースの最新のセグメントとなる。SSTableがディスクに書き出されている間、書き込みは新しいmemtableインスタンスに続けることができる。
- 読み取りリクエストを処理するには、まずmemtableでキーを探し、次に最新のディスク上のセグメント、次にその次に古いセグメント、と順に探す。
- 時折、バックグラウンドでマージとコンパクションのプロセスを実行し、セグメントファイルを結合し、上書きまたは削除された値を破棄する。

このスキームは非常にうまく機能する。ただ1つの問題がある：データベースがクラッシュすると、まだディスクに書き出されていないmemtable内の最新の書き込みが失われる。この問題を回避するため、前のセクションと同様に、すべての書き込みが即座に追記される別のログをディスク上に保持できる。そのログはソートされていないが、それは問題ではない。なぜならその唯一の目的はクラッシュ後にmemtableを復元することだからだ。memtableがSSTableに書き出されるたびに、対応するログは破棄できる。

#### SSTableからLSM-treeを作る

ここで説明したアルゴリズムは、本質的にLevelDB [6] とRocksDB [7] で使われているものである。これらは他のアプリケーションに組み込まれるよう設計されたキーバリューストレージエンジンライブラリである。とりわけLevelDBはBitcaskの代替としてRiakで使用できる。同様のストレージエンジンがCassandraとHBase [8] で使われており、どちらもGoogleのBigtable論文 [9]（SSTableとmemtableという用語を導入した）に触発されている。

元々このインデックス構造はPatrick O'Neilらによって、ログ構造化ファイルシステム [11] に関する以前の研究に基づいて、LSM-Tree（Log-Structured Merge-Tree）[10] という名前で説明された。ソートされたファイルのマージとコンパクションのこの原理に基づくストレージエンジンは、しばしばLSMストレージエンジンと呼ばれる。

Lucene——ElasticsearchとSolrで使われる全文検索のインデックスエンジン——は、その用語辞書の格納に同様の方法を使っている [12, 13]。全文インデックスはキーバリューインデックスよりもはるかに複雑だが、似たアイデアに基づいている：検索クエリ内の単語が与えられたら、その単語に言及するすべてのドキュメント（Webページ、製品説明など）を見つける。これはキーが単語（term、用語）で、値がその単語を含むすべてのドキュメントのIDのリスト（postings list、ポスティングリスト）であるキーバリュー構造で実装される。Luceneでは、この用語からポスティングリストへのマッピングはSSTable風のソートされたファイルに保持され、必要に応じてバックグラウンドでマージされる [14]。

#### パフォーマンスの最適化

常にそうであるように、ストレージエンジンを実際に良いパフォーマンスで動作させるには多くの詳細が必要になる。たとえば、LSM-treeアルゴリズムはデータベースに存在しないキーを検索するとき遅くなる可能性がある：memtableを確認し、次にセグメントを最も古いものまで遡って確認し（それぞれでディスクからの読み取りが必要かもしれない）、キーが存在しないことを確認するまで行わなければならない。この種のアクセスを最適化するために、ストレージエンジンはしばしば追加のBloom filter（ブルームフィルタ）[15] を使用する。（ブルームフィルタは集合の内容を近似するためのメモリ効率の良いデータ構造である。キーがデータベースに存在しないかどうかを教えることができ、存在しないキーに対する多くの不必要なディスク読み取りを節約する。）

SSTableがどのようにコンパクションおよびマージされるかの順序とタイミングを決定する異なる戦略もある。最も一般的なオプションはsize-tiered compaction（サイズ階層型コンパクション）とleveled compaction（レベル化コンパクション）である。LevelDBとRocksDBはleveled compactionを使い（LevelDBの名前の由来）、HBaseはsize-tieredを使い、Cassandraは両方をサポートする [16]。size-tieredコンパクションでは、より新しくて小さなSSTableが、より古くて大きなSSTableに順次マージされる。leveled compactionでは、キー範囲がより小さなSSTableに分割され、古いデータは別の「レベル」に移動され、コンパクションをより漸進的に進め、ディスクスペースの使用を削減できる。

多くの微妙な点があるとはいえ、LSM-treeの基本的なアイデア——バックグラウンドでマージされるSSTableのカスケードを保持する——はシンプルで効果的である。データセットが利用可能なメモリよりもはるかに大きくてもうまく機能し続ける。データはソートされた順序で格納されるため、範囲クエリを効率的に実行でき（ある最小値から最大値までのすべてのキーをスキャン）、ディスク書き込みがシーケンシャルであるため、LSM-treeは驚くほど高い書き込みスループットをサポートできる。

### B-Trees（Bツリー）

ここまで議論してきたログ構造化インデックスは普及が進んでいるが、最も一般的なインデックスの種類ではない。最も広く使われているインデックス構造はまったく異なるものである：B-tree（Bツリー）。

1970年に導入され [17]、10年も経たないうちに「ユビキタス」と呼ばれた [18] B-treeは、時の試練に非常によく耐えてきた。ほぼすべてのリレーショナルデータベースで標準的なインデックス実装として残っており、多くの非リレーショナルデータベースでも使われている。

SSTableと同様に、B-treeはキーバリューペアをキーでソートされた状態で保持し、効率的なキーバリュー検索と範囲クエリを可能にする。しかし類似点はそこまでである：B-treeは非常に異なる設計哲学を持つ。

先ほど見たログ構造化インデックスは、データベースを可変サイズのセグメント（通常数メガバイト以上）に分解し、常にセグメントをシーケンシャルに書き込む。対照的に、B-treeはデータベースを固定サイズのblock（ブロック）またはpage（ページ）——伝統的に4KB（時にはそれ以上）——に分解し、一度に1ページを読み書きする。この設計は基盤となるハードウェアにより密接に対応している。ディスクも固定サイズのブロックで構成されているからだ。

各ページはアドレスまたは位置を使って特定でき、これにより1つのページが別のページを参照できる——ポインタに似ているが、メモリ上ではなくディスク上である。これらのページ参照を使って、ページのツリーを構築できる（Figure 3-6に示すとおり）。

*Figure 3-6. B-treeインデックスを使ったキーの検索。*

1つのページがB-treeのルートとして指定される。インデックスでキーを検索したいときは常にここから始める。ページにはいくつかのキーと子ページへの参照が含まれる。各子ページは連続したキー範囲を担当し、参照間のキーはそれらの範囲の境界がどこにあるかを示す。

Figure 3-6の例では、キー251を探している。200と300の境界の間のページ参照をたどる必要があるとわかる。それにより200〜300の範囲をさらにサブ範囲に分解する似たようなページに行き着く。最終的に個々のキーを含むページ（leaf page、リーフページ）に到達し、各キーの値がインラインで含まれているか、値が見つかるページへの参照が含まれている。

B-treeの1ページ内の子ページへの参照の数はbranching factor（分岐係数）と呼ばれる。たとえばFigure 3-6では分岐係数は6である。実際には、分岐係数はページ参照と範囲境界を格納するために必要なスペースの量に依存するが、通常は数百である。

B-treeで既存のキーの値を更新したい場合、そのキーを含むリーフページを検索し、そのページの値を変更し、ページをディスクに書き戻す（そのページへの参照はすべて有効なまま）。新しいキーを追加したい場合、新しいキーの範囲を含むページを見つけ、そのページに追加する。ページに新しいキーを収容する十分な空きスペースがない場合、ページは2つの半分の充填率のページに分割され、親ページは新しいキー範囲の区分を反映するよう更新される（Figure 3-7を参照）。^ii

*Figure 3-7. ページの分割によるB-treeの成長。*

このアルゴリズムによりツリーはbalanced（バランスされた）状態が保たれる：n個のキーを持つB-treeの深さは常にO(log n)である。ほとんどのデータベースは3〜4レベルの深さのB-treeに収まるため、探しているページを見つけるために多くのページ参照をたどる必要はない。（4KBページで分岐係数500の4レベルのツリーは最大256TBを格納できる。）

^ii B-treeへの新しいキーの挿入はかなり直感的だが、削除は（ツリーのバランスを保ちながら）やや複雑である [2]。

#### B-treeを信頼性のあるものにする

B-treeの基本的な書き込み操作は、ディスク上のページを新しいデータで上書きすることである。上書きしてもページの位置は変わらないことが前提とされる。すなわち、ページが上書きされてもそのページへのすべての参照はそのまま有効である。これはLSM-treeのようなログ構造化インデックスとは対照的である。LSM-treeはファイルに追記するだけ（そして最終的に古いファイルを削除する）であり、ファイルをその場で変更することは決してない。

ディスク上のページの上書きは実際のハードウェア操作として考えることができる。磁気ハードドライブでは、ディスクヘッドを正しい位置に移動し、回転するプラッターの正しい位置が来るのを待ち、適切なセクターを新しいデータで上書きすることを意味する。SSDでは、ストレージチップのかなり大きなブロックを一度に消去して書き直す必要があるため、やや複雑である [19]。

さらに、一部の操作は複数の異なるページの上書きを必要とする。たとえば、挿入によりページが満杯になってページを分割する場合、分割された2つのページと、2つの子ページへの参照を更新する親ページも書き込む必要がある。これは危険な操作である。一部のページだけが書き込まれた後にデータベースがクラッシュすると、破損したインデックスになりかねない（たとえば、どの親の子でもない孤立したページが残る可能性がある）。

クラッシュに対してデータベースを回復力のあるものにするために、B-tree実装にはディスク上の追加のデータ構造を含めるのが一般的である：WAL（Write-Ahead Log、先行書き込みログ、redo logとも呼ばれる）。これはappend-onlyのファイルであり、すべてのB-treeの変更はツリー自体のページに適用される前にまずこのログに書き込まれなければならない。データベースがクラッシュ後に復旧するとき、このログを使ってB-treeを一貫した状態に復元する [5, 20]。

ページをその場で更新することの追加の複雑さとして、複数のスレッドが同時にB-treeにアクセスする場合、注意深い並行性制御が必要である——さもなければスレッドが一貫性のない状態のツリーを見る可能性がある。これは通常、ツリーのデータ構造をlatch（ラッチ、軽量ロック）で保護することで行われる。ログ構造化アプローチはこの点でよりシンプルである。すべてのマージをバックグラウンドで、着信クエリに干渉せずに行い、古いセグメントを新しいセグメントに時折アトミックに入れ替えるからだ。

#### B-treeの最適化

B-treeは長い歴史があるため、長年にわたり多くの最適化が開発されているのは驚くことではない。いくつかだけ挙げる：

- ページの上書きとクラッシュリカバリのためのWALの維持の代わりに、一部のデータベース（LMDBなど）はcopy-on-write（コピーオンライト）方式を使用する [21]。変更されたページは別の場所に書き込まれ、新しい場所を指すツリー内の親ページの新しいバージョンが作成される。このアプローチは並行性制御にも有用であり、「スナップショット分離とリピータブルリード」（237ページ）で見る。
- ページ内のキー全体を格納するのではなく、キーを省略することでページ内のスペースを節約できる。特にツリーの内部のページでは、キーはキー範囲間の境界として機能するだけの十分な情報を提供すれば良い。より多くのキーをページに詰め込むことで、ツリーはより高い分岐係数を持ち、したがってより少ないレベル数になる。^iii
- 一般的にページはディスク上のどこにでも配置できる。キー範囲が近いページがディスク上でも近くにある必要はない。クエリがソートされた順序でキー範囲の大部分をスキャンする必要がある場合、そのページごとのレイアウトは非効率になりうる。読み取られるページごとにディスクシークが必要になるかもしれないからだ。多くのB-tree実装はリーフページがディスク上で順番に並ぶようにツリーを配置しようとする。しかし、ツリーが成長するにつれてその順序を維持するのは難しい。対照的に、LSM-treeはマージ中にストレージの大きなセグメントを一度に書き直すため、シーケンシャルなキーをディスク上で近くに保つのが容易である。
- ツリーに追加のポインタが付加されることがある。たとえば、各リーフページが左右の兄弟ページへの参照を持つことで、親ページにジャンプバックせずにキーを順番にスキャンできる。
- fractal tree（フラクタルツリー）[22] のようなB-treeの変種は、ディスクシークを減らすためにログ構造化のアイデアを一部借りている（フラクタルとは無関係である）。

^iii この変種はB+ tree（B+ツリー）として知られることがあるが、この最適化はあまりにも一般的であるため、他のB-tree変種と区別されないことが多い。

### Comparing B-Trees and LSM-Trees（B-TreeとLSM-Treeの比較）

B-tree実装は一般的にLSM-tree実装よりも成熟しているが、LSM-treeもそのパフォーマンス特性から興味深い。経験則として、LSM-treeは通常書き込みが速く、B-treeは読み取りが速いと考えられている [23]。LSM-treeの読み取りは通常遅い。コンパクションの異なる段階にあるいくつかの異なるデータ構造とSSTableを確認する必要があるからだ。

しかし、ベンチマークはしばしば決定的でなく、ワークロードの詳細に敏感である。自分の特定のワークロードでシステムをテストして、有効な比較を行う必要がある。本セクションでは、ストレージエンジンのパフォーマンスを測定するときに考慮する価値のあることをいくつか簡潔に議論する。

#### LSM-treeの利点

B-treeインデックスはすべてのデータを少なくとも2回書き込む必要がある：1回はWALに、もう1回はツリーのページ自体に（そしてページが分割されるときにはおそらくもう1回）。ページ内の数バイトしか変わらなくても、ページ全体を一度に書き込むオーバーヘッドもある。一部のストレージエンジンは電源障害時に部分的に更新されたページになるのを避けるため、同じページを2回上書きすることさえある [24, 25]。

ログ構造化インデックスもSSTableの繰り返しのコンパクションとマージにより、データを複数回書き直す。データベースへの1回の書き込みが、データベースの寿命にわたってディスクへの複数回の書き込みを引き起こすこの効果は、write amplification（書き込み増幅）として知られている。SSDでは特に懸念される。SSDは摩耗するまでにブロックを上書きできる回数が限られているからだ。

書き込みが多いアプリケーションでは、パフォーマンスのボトルネックはデータベースがディスクに書き込める速度かもしれない。この場合、書き込み増幅は直接的なパフォーマンスコストを持つ：ストレージエンジンがディスクに書き込む量が増えるほど、利用可能なディスク帯域幅内で処理できる書き込み/秒は減る。

さらに、LSM-treeはB-treeよりも高い書き込みスループットを維持できる傾向がある。一つには書き込み増幅が低い場合があるため（ただしストレージエンジンの設定とワークロードに依存する）、もう一つにはツリー内の複数のページを上書きするのではなく、コンパクトなSSTableファイルをシーケンシャルに書き込むためである [26]。この違いは特に磁気ハードドライブで重要で、シーケンシャル書き込みはランダム書き込みよりもはるかに高速である。

LSM-treeはより良く圧縮でき、B-treeよりもディスク上のファイルが小さくなることが多い。B-treeストレージエンジンはフラグメンテーションにより一部のディスクスペースが未使用のまま残る：ページが分割されたり行が既存のページに収まらなかったりすると、ページ内の一部のスペースが未使用のままになる。LSM-treeはページ指向でなく、定期的にSSTableを書き直してフラグメンテーションを除去するため、ストレージオーバーヘッドが低い。特にleveled compactionを使用する場合[27]。

多くのSSDでは、ファームウェアが内部的にログ構造化アルゴリズムを使って基盤のストレージチップ上でランダム書き込みをシーケンシャル書き込みに変換するため、ストレージエンジンの書き込みパターンの影響はそれほど顕著ではない [19]。しかし、書き込み増幅の削減とフラグメンテーションの低減はSSDでも依然として有利である：データをよりコンパクトに表現することで、利用可能なI/O帯域幅内でより多くの読み書きリクエストを処理できる。

#### LSM-treeの欠点

ログ構造化ストレージの欠点は、コンパクションプロセスが進行中の読み書きのパフォーマンスに干渉する場合があることである。ストレージエンジンはコンパクションを漸進的に、並行アクセスに影響を与えずに行おうとするが、ディスクのリソースは限られているため、リクエストがディスクが高価なコンパクション操作を完了するまで待たなければならないことが容易に起こりうる。スループットと平均レスポンスタイムへの影響は通常小さいが、高いパーセンタイル（「パフォーマンスの記述」、13ページ参照）では、ログ構造化ストレージエンジンへのクエリのレスポンスタイムがかなり高くなることがあり、B-treeの方が予測可能な場合がある [28]。

コンパクションに関するもう1つの問題は高い書き込みスループット時に生じる：ディスクの有限の書き込み帯域幅は、初期書き込み（memtableのロギングとディスクへのフラッシュ）とバックグラウンドで実行されるコンパクションスレッドの間で共有される必要がある。空のデータベースへの書き込みでは、ディスク帯域幅の全体を初期書き込みに使えるが、データベースが大きくなるほど、コンパクションに必要なディスク帯域幅が増える。

書き込みスループットが高くコンパクションが注意深く設定されていない場合、コンパクションが着信書き込みの速度に追いつけなくなることがある。この場合、ディスク上の未マージセグメントの数が増え続け、ディスクスペースがなくなるまで続き、読み取りもより多くのセグメントファイルを確認する必要があるため遅くなる。通常、SSTableベースのストレージエンジンはコンパクションが追いつかなくても着信書き込みの速度を制限しないため、この状況を検出するには明示的な監視が必要である [29, 30]。

B-treeの利点は、各キーがインデックス内の正確に1箇所に存在することである。一方、ログ構造化ストレージエンジンは異なるセグメントに同じキーの複数のコピーを持つ可能性がある。この側面により、B-treeは強いトランザクションセマンティクスを提供したいデータベースにとって魅力的である：多くのリレーショナルデータベースでは、トランザクション分離はキー範囲のロックを使って実装されており、B-treeインデックスではそれらのロックをツリーに直接付与できる [5]。第7章でこの点をさらに詳しく議論する。

B-treeはデータベースのアーキテクチャに深く根付いており、多くのワークロードに対して一貫した良いパフォーマンスを提供するため、近いうちに消えることはないだろう。新しいデータストアでは、ログ構造化インデックスの人気が高まっている。自分のユースケースにどちらの種類のストレージエンジンが良いかを決める簡単なルールはないため、経験的にテストする価値がある。

### Other Indexing Structures（その他のインデックス構造）

ここまでキーバリューインデックスのみを議論してきたが、これはリレーショナルモデルのprimary key index（主キーインデックス）のようなものである。主キーはリレーショナルテーブル内の1行、ドキュメントデータベース内の1ドキュメント、またはグラフデータベース内の1頂点を一意に識別する。データベース内の他のレコードは主キー（またはID）でその行/ドキュメント/頂点を参照でき、インデックスはそのような参照を解決するために使用される。

secondary index（セカンダリインデックス）を持つことも非常に一般的である。リレーショナルデータベースでは、`CREATE INDEX` コマンドを使って同じテーブルに複数のセカンダリインデックスを作成でき、JOINの効率的な実行にしばしば不可欠である。たとえば、第2章のFigure 2-1では、各テーブルで同じユーザーに属するすべての行を見つけられるように、user_id列にセカンダリインデックスを持つのが最も一般的だろう。

セカンダリインデックスはキーバリューインデックスから容易に構築できる。主な違いはキーが一意ではないことである。すなわち同じキーを持つ多くの行（ドキュメント、頂点）があるかもしれない。これは2つの方法で解決できる：インデックスの各値を一致する行識別子のリストにするか（全文インデックスのポスティングリストのように）、行識別子をキーに付加して各キーを一意にするかである。いずれにしても、B-treeとログ構造化インデックスの両方がセカンダリインデックスとして使用できる。

#### インデックス内に値を格納する

インデックスのキーはクエリが検索するものだが、値は2つのもののうちの1つでありうる：問題の実際の行（ドキュメント、頂点）そのもの、または別の場所に格納された行への参照である。後者の場合、行が格納される場所はheap file（ヒープファイル）として知られ、データを特定の順序なしに格納する（append-onlyかもしれないし、削除された行を追跡して後で新しいデータで上書きするかもしれない）。ヒープファイルアプローチは一般的である。複数のセカンダリインデックスが存在する場合にデータの重複を避けられるからだ：各インデックスはヒープファイル内の位置を参照するだけで、実際のデータは1箇所に保持される。

キーを変更せずに値を更新する場合、ヒープファイルアプローチはかなり効率的である：新しい値が古い値より大きくなければ、レコードをその場で上書きできる。新しい値が大きい場合は、十分なスペースがあるヒープ内の新しい場所に移動する必要があるかもしれず、その場合すべてのインデックスをレコードの新しいヒープ位置を指すよう更新するか、古いヒープ位置にforwarding pointer（転送ポインタ）を残す必要がある [5]。

場合によっては、インデックスからヒープファイルへの追加のホップが読み取りのパフォーマンスペナルティとして大きすぎることがあり、インデックス内に直接インデックス対象の行を格納するのが望ましいことがある。これはclustered index（クラスタードインデックス）として知られている。たとえば、MySQLのInnoDBストレージエンジンではテーブルの主キーは常にクラスタードインデックスであり、セカンダリインデックスは（ヒープファイルの場所ではなく）主キーを参照する [31]。SQL Serverでは、テーブルごとに1つのクラスタードインデックスを指定できる [32]。

クラスタードインデックス（インデックス内にすべての行データを格納）と非クラスタードインデックス（データへの参照のみを格納）の妥協点はcovering index（カバリングインデックス）またはindex with included columns（含まれる列を持つインデックス）として知られ、テーブルの列の一部をインデックス内に格納する [33]。これにより一部のクエリをインデックスだけで回答できる（この場合、インデックスはクエリを「カバーする」と言う）[32]。

あらゆるデータの重複と同様に、クラスタードインデックスとカバリングインデックスは読み取りを高速化できるが、追加のストレージが必要で、書き込みにオーバーヘッドを追加する可能性がある。データベースは重複によるアプリケーションへの不整合の露出を防ぐために、トランザクション保証を維持する追加の努力も必要である。

#### Multi-column indexes（複合インデックス）

ここまで議論したインデックスは単一のキーを値にマッピングするだけである。テーブルの複数の列（またはドキュメントの複数のフィールド）を同時にクエリする必要がある場合、それでは不十分である。

最も一般的な複合インデックスの種類はconcatenated index（連結インデックス）と呼ばれ、複数のフィールドを1つのキーに結合する（インデックス定義がフィールドの連結順序を指定する）。これは昔ながらの紙の電話帳のようなもので、（姓、名）から電話番号へのインデックスを提供する。ソート順のおかげで、特定の姓を持つすべての人、または特定の姓と名の組み合わせを持つすべての人を見つけるのにインデックスを使える。しかし特定の名だけを持つすべての人を見つけたい場合、このインデックスは役に立たない。

Multi-dimensional index（多次元インデックス）は複数の列を同時にクエリするより一般的な方法であり、地理空間データにとって特に重要である。たとえば、レストラン検索Webサイトには各レストランの緯度と経度を含むデータベースがあるかもしれない。ユーザーが地図上のレストランを見ているとき、現在表示されている長方形の地図エリア内のすべてのレストランを検索する必要がある。これには次のような2次元の範囲クエリが必要である：

```sql
SELECT * FROM restaurants WHERE latitude > 51.4946 AND latitude < 51.5079
                            AND longitude > -0.1162 AND longitude < -0.1004;
```

標準的なB-treeまたはLSM-treeインデックスはこの種のクエリに効率的に回答できない。緯度の範囲にあるすべてのレストラン（ただし任意の経度）か、経度の範囲にあるすべてのレストラン（ただし北極から南極まで）のどちらかを返すことはできるが、両方を同時にはできない。

1つの選択肢は、space-filling curve（空間充填曲線）を使って2次元の位置を単一の数値に変換し、通常のB-treeインデックスを使うことである [34]。より一般的には、R-treeのような特殊なspatial index（空間インデックス）が使用される。たとえばPostGISは、PostgreSQLのGeneralized Search Tree（GiST）インデックス機構を使ってR-treeとして地理空間インデックスを実装している [35]。ここではR-treeの詳細を説明する余裕はないが、それらに関する文献は豊富にある。

興味深いアイデアは、多次元インデックスは地理的な位置だけのためのものではないということだ。たとえば、eコマースサイトでは（赤、緑、青）のディメンションで3次元インデックスを使って特定の色の範囲の製品を検索したり、気象観測データベースでは（日付、気温）の2次元インデックスを使って2013年中に気温が25〜30℃だったすべての観測を効率的に検索したりできる。1次元インデックスでは、2013年のすべてのレコードをスキャンして（気温に関係なく）気温でフィルタリングするか、その逆をするしかない。2Dインデックスはタイムスタンプと気温を同時に絞り込める。この技法はHyperDex [36] で使用されている。

#### Full-text search and fuzzy indexes（全文検索とあいまいインデックス）

ここまで議論したすべてのインデックスは、正確なデータがあることを前提とし、キーの正確な値またはソート順を持つキーの範囲のクエリを可能にする。これらが許さないのは、スペルミスした単語のような、似たキーの検索である。そのようなfuzzy（あいまいな）クエリには異なる技法が必要である。

たとえば、全文検索エンジンは一般的に、ある単語の検索を同義語を含むように拡張したり、単語の文法的変化を無視したり、同じドキュメント内で互いに近い単語の出現を検索したり、テキストの言語分析に依存するさまざまなその他の機能をサポートする。タイポに対処するため、Luceneはテキスト内で一定のedit distance（編集距離）以内の単語を検索できる（編集距離1は1文字が追加、削除、または置換されたことを意味する）[37]。

「SSTableからLSM-treeを作る」で述べたように、Luceneはその用語辞書にSSTable風の構造を使用している。この構造は、ソートされたファイル内でキーを探すべきオフセットをクエリに教える小さなインメモリインデックスを必要とする。LevelDBではこのインメモリインデックスはキーの一部の疎なコレクションだが、Luceneではインメモリインデックスはキー内の文字に対するfinite state automaton（有限状態オートマトン）であり、trie [38] に似ている。このオートマトンはLevenshtein automaton（レーベンシュタインオートマトン）に変換でき、与えられた編集距離内の単語の効率的な検索をサポートする [39]。

その他のあいまい検索技法は文書分類や機械学習の方向に進む。詳しくは情報検索のテキストブック [例えば40] を参照。

#### Keeping everything in memory（すべてをメモリに保持する）

本章でこれまで議論したデータ構造は、すべてディスクの制限に対する回答であった。メインメモリと比べると、ディスクは扱いにくい。磁気ディスクでもSSDでも、読み書きに良いパフォーマンスを得るにはディスク上のデータを注意深く配置する必要がある。しかしディスクには2つの大きな利点があるため、この扱いにくさを許容している：耐久性がある（電源が切れても内容が失われない）こと、そしてRAMよりもギガバイトあたりのコストが低いことである。

RAMが安くなるにつれて、ギガバイトあたりのコストの議論は薄れている。多くのデータセットはそれほど大きくないため、複数のマシンに分散されてでも完全にメモリに保持することが現実的である。これがin-memory database（インメモリデータベース）の開発につながった。

一部のインメモリキーバリューストア（Memcachedなど）はキャッシュ用途のみを意図しており、マシンが再起動した場合にデータが失われても許容される。しかし他のインメモリデータベースは耐久性を目指しており、特殊なハードウェア（バッテリーバックアップRAM）の使用、変更ログのディスクへの書き込み、定期的なスナップショットのディスクへの書き込み、またはインメモリ状態の他のマシンへのレプリケーションによって達成できる。

インメモリデータベースが再起動するとき、ディスクから、またはレプリカからネットワーク経由で状態をリロードする必要がある（特殊なハードウェアを使わない限り）。ディスクへの書き込みを行っているにもかかわらず、これは依然としてインメモリデータベースである。ディスクは耐久性のためのappend-onlyログとしてのみ使用され、読み取りは完全にメモリから提供されるからだ。ディスクへの書き込みには運用上の利点もある：ディスク上のファイルは外部ユーティリティで簡単にバックアップ、検査、分析できる。

VoltDB、MemSQL、Oracle TimesTenなどの製品はリレーショナルモデルを持つインメモリデータベースであり、ベンダーはディスク上のデータ構造の管理に関連するすべてのオーバーヘッドを除去することで大きなパフォーマンス改善を提供できると主張している [41, 42]。RAMCloudは耐久性を持つオープンソースのインメモリキーバリューストアである（メモリ上のデータにもディスク上のデータにもログ構造化アプローチを使用する）[43]。RedisとCouchbaseは非同期でのディスクへの書き込みにより弱い耐久性を提供する。

直感に反するかもしれないが、インメモリデータベースのパフォーマンスの利点は、ディスクから読み取る必要がないという事実によるものではない。十分なメモリがあれば、ディスクベースのストレージエンジンでもディスクから読み取る必要がないかもしれない。オペレーティングシステムが最近使用されたディスクブロックをメモリにキャッシュするからだ。むしろ、インメモリデータ構造をディスクに書き込める形式にエンコードするオーバーヘッドを回避できるため、高速になりうる [44]。

パフォーマンスに加えて、インメモリデータベースのもう1つの興味深い分野は、ディスクベースのインデックスでは実装が困難なデータモデルの提供である。たとえばRedisは優先度キューやセットなどのさまざまなデータ構造に対するデータベース風のインターフェースを提供する。すべてのデータをメモリに保持するため、実装は比較的シンプルである。

最近の研究では、インメモリデータベースアーキテクチャを利用可能なメモリを超えるデータセットのサポートに拡張でき、ディスク中心のアーキテクチャのオーバーヘッドを持ち込まずに済むことが示されている [45]。いわゆるanti-caching（アンチキャッシング）アプローチは、メモリが十分でないときに最も最近使われていないデータをメモリからディスクに退避し、将来再びアクセスされたときにメモリに戻すことで機能する。これはオペレーティングシステムが仮想メモリとスワップファイルで行うことと似ているが、データベースはOSよりもメモリをより効率的に管理できる。メモリページ全体ではなく個々のレコードの粒度で作業できるからだ。このアプローチでもインデックスは完全にメモリに収まる必要がある（本章冒頭のBitcaskの例のように）。

不揮発性メモリ（NVM）技術がより広く採用されるようになれば、ストレージエンジンの設計にさらなる変更が必要になるだろう [46]。現時点ではこれは新しい研究分野だが、将来注目する価値がある。

## Transaction Processing or Analytics?（トランザクション処理か分析か？）

ビジネスデータ処理の初期の頃、データベースへの書き込みは通常、商取引に対応していた。販売の実行、サプライヤーへの発注、従業員への給与の支払いなどである。データベースが金銭のやり取りを伴わない領域にも拡張されるにつれても、transaction（トランザクション）という用語は定着し、論理的な単位を形成する読み書きのグループを指すようになった。

> トランザクションは必ずしもACID（atomicity、consistency、isolation、durability）の特性を持つ必要はない。トランザクション処理とは、クライアントに低レイテンシの読み書きを許可することを意味するだけである——定期的に（例えば1日1回）実行されるバッチ処理ジョブとは対照的に。ACID特性は第7章で、バッチ処理は第10章で議論する。

データベースが多くの異なる種類のデータに使われるようになっても——ブログ投稿へのコメント、ゲーム内のアクション、アドレス帳の連絡先など——基本的なアクセスパターンはビジネストランザクションの処理と似たままであった。アプリケーションは通常、インデックスを使って何らかのキーで少数のレコードを検索する。レコードはユーザーの入力に基づいて挿入または更新される。これらのアプリケーションはインタラクティブであるため、このアクセスパターンはOLTP（Online Transaction Processing、オンライントランザクション処理）として知られるようになった。

しかし、データベースはdata analytics（データ分析）にもますます使われるようになり、そのアクセスパターンは大きく異なる。分析クエリは通常膨大な数のレコードをスキャンする必要があり、各レコードの少数の列だけを読み、生データをユーザーに返すのではなく集計統計（COUNT、SUM、AVERAGEなど）を計算する。たとえばデータが売上トランザクションのテーブルであれば、分析クエリは次のようなものかもしれない：

- 1月の各店舗の総売上はいくらだったか？
- 最新のプロモーション中に通常よりどれだけ多くのバナナを売ったか？
- どのブランドのベビーフードがブランドXのおむつと最もよく一緒に購入されるか？

これらのクエリはビジネスアナリストによって書かれることが多く、企業の経営陣がより良い意思決定を行うのに役立つレポートに反映される（ビジネスインテリジェンス）。トランザクション処理とは異なるこのデータベース使用パターンを区別するために、OLAP（Online Analytic Processing、オンライン分析処理）[47] と呼ばれるようになった。^iv OLTPとOLAPの違いは必ずしも明確ではないが、典型的な特性のいくつかをTable 3-1に示す。

^iv OLAPの「online」の意味は不明確である。おそらくクエリが事前定義されたレポートのためだけでなく、アナリストが探索的クエリのためにOLAPシステムをインタラクティブに使用するという事実を指しているのだろう。

| 特性 | トランザクション処理システム（OLTP） | 分析システム（OLAP） |
|------|------|------|
| 主な読み取りパターン | キーで取得、クエリあたり少数のレコード | 大量のレコードに対する集計 |
| 主な書き込みパターン | ユーザー入力によるランダムアクセス、低レイテンシの書き込み | バルクインポート（ETL）またはイベントストリーム |
| 主な使用者 | Webアプリケーション経由のエンドユーザー/顧客 | 意思決定支援のための社内アナリスト |
| データが表すもの | データの最新状態（現在の時点） | 時間の経過とともに起きたイベントの履歴 |
| データセットサイズ | ギガバイト〜テラバイト | テラバイト〜ペタバイト |

当初は、トランザクション処理と分析クエリの両方に同じデータベースが使われていた。SQLはこの点で非常に柔軟であることがわかった。OLTP型のクエリにもOLAP型のクエリにもうまく機能する。それにもかかわらず、1980年代後半から1990年代初頭にかけて、企業がOLTPシステムを分析目的に使うのをやめ、代わりに別のデータベースで分析を実行する傾向が生まれた。この別のデータベースはdata warehouse（データウェアハウス）と呼ばれた。

### Data Warehousing（データウェアハウジング）

企業は数十の異なるトランザクション処理システムを持つかもしれない。顧客向けWebサイトを動かすシステム、実店舗のPOS（レジ）システムを制御するシステム、倉庫の在庫を追跡するシステム、車両のルートを計画するシステム、サプライヤーを管理するシステム、従業員を管理するシステムなどである。これらのシステムはそれぞれ複雑で、維持には人々のチームが必要なため、互いにほぼ自律的に運用されることになる。

これらのOLTPシステムは通常、高可用性でトランザクションを低レイテンシで処理することが期待される。ビジネスの運営にしばしば不可欠だからだ。そのためデータベース管理者はOLTPデータベースを注意深く守る。ビジネスアナリストにOLTPデータベースでアドホックな分析クエリを実行させることには通常消極的である。これらのクエリはしばしば高価で、データセットの大部分をスキャンし、並行して実行されているトランザクションのパフォーマンスを損なう可能性があるからだ。

対照的にdata warehouse（データウェアハウス）は、アナリストがOLTP操作に影響を与えずに心ゆくまでクエリできる別のデータベースである [48]。データウェアハウスには、企業内のさまざまなOLTPシステムのデータの読み取り専用コピーが含まれる。データはOLTPデータベースから抽出され（定期的なデータダンプまたは継続的な更新ストリームを使用して）、分析に適したスキーマに変換され、クリーンアップされてからデータウェアハウスにロードされる。データをウェアハウスに取り込むこのプロセスはETL（Extract-Transform-Load）として知られ、Figure 3-8に示されている。

*Figure 3-8. データウェアハウスへのETLの簡略図。*

データウェアハウスは現在ほぼすべての大企業に存在するが、小企業ではほとんど聞かれない。これはおそらく、ほとんどの小企業にはそれほど多くの異なるOLTPシステムがなく、データ量が少ない——従来のSQLデータベースでクエリしたり、スプレッドシートで分析したりできるほど少ない——からだろう。大企業では、小企業ではシンプルなことを行うために多くの重労働が必要になる。

OLTPシステムに直接クエリするのではなく別のデータウェアハウスを使うことの大きな利点は、データウェアハウスを分析アクセスパターンに最適化できることである。本章前半で議論したインデックスアルゴリズムはOLTPにはうまく機能するが、分析クエリへの回答にはあまり適していないことがわかった。本章の残りでは、分析に最適化されたストレージエンジンを見る。

#### OLTPデータベースとデータウェアハウスの分岐

データウェアハウスのデータモデルは最も一般的にリレーショナルであるが、SQLは一般的に分析クエリによく適合するからだ。SQLクエリを生成し、結果を視覚化し、アナリストがデータを探索できるようにする（ドリルダウンやスライス＆ダイスなどの操作を通じて）多くのグラフィカルなデータ分析ツールがある。

表面上、データウェアハウスとリレーショナルOLTPデータベースは似ている。どちらもSQLクエリインターフェースを持つからだ。しかし、システムの内部は非常に異なって見えることがある。非常に異なるクエリパターンに最適化されているからだ。多くのデータベースベンダーは、トランザクション処理か分析のワークロードのいずれかのサポートに注力しており、両方ではない。

Microsoft SQL ServerやSAP HANAのような一部のデータベースは、同じ製品内でトランザクション処理とデータウェアハウジングの両方をサポートしている。しかし、共通のSQLインターフェースを通じてアクセス可能であるものの、2つの別々のストレージとクエリエンジンになる傾向が強まっている [49, 50, 51]。

Teradata、Vertica、SAP HANA、ParAccelなどのデータウェアハウスベンダーは、通常システムを高価な商用ライセンスのもとで販売している。Amazon RedShiftはParAccelのホステッド版である。より最近では、多数のオープンソースのSQL-on-Hadoopプロジェクトが登場した。まだ若いが、商用データウェアハウスシステムとの競合を目指している。Apache Hive、Spark SQL、Cloudera Impala、Facebook Presto、Apache Tajo、Apache Drill [52, 53] などがある。これらの一部はGoogleのDremel [54] からのアイデアに基づいている。

### Stars and Snowflakes: Schemas for Analytics（スターとスノーフレーク：分析のためのスキーマ）

第2章で探求したように、トランザクション処理の領域ではアプリケーションのニーズに応じてさまざまな異なるデータモデルが使われる。一方、分析ではデータモデルの多様性ははるかに少ない。多くのデータウェアハウスはstar schema（スタースキーマ、dimensional modeling（ディメンショナルモデリング）[55] とも呼ばれる）というかなり定型的なスタイルで使用される。

Figure 3-9のスキーマ例は、食料品小売業者にあるかもしれないデータウェアハウスを示している。スキーマの中心にはfact table（ファクトテーブル）と呼ばれるテーブルがある（この例ではfact_salesと呼ばれる）。ファクトテーブルの各行は、特定の時点で発生したイベントを表す（ここでは、各行は顧客の製品の購入を表す）。小売販売ではなくWebサイトトラフィックを分析していれば、各行はユーザーのページビューやクリックかもしれない。

*Figure 3-9. データウェアハウスで使用されるスタースキーマの例。*

通常、ファクトは個々のイベントとして取り込まれる。これにより後の分析の柔軟性が最大化されるからだ。しかし、これはファクトテーブルが非常に大きくなりうることを意味する。Apple、Walmart、eBayのような大企業は、データウェアハウスに数十ペタバイトのトランザクション履歴を持つかもしれず、その大部分はファクトテーブルにある [56]。

ファクトテーブルの一部の列は属性であり、製品が販売された価格やサプライヤーからの仕入れコスト（利益率の計算を可能にする）などである。ファクトテーブルの他の列はdimension table（ディメンションテーブル）と呼ばれる他のテーブルへのforeign key（外部キー）参照である。ファクトテーブルの各行がイベントを表すため、ディメンションはそのイベントの誰が、何を、どこで、いつ、どのように、なぜを表す。

たとえば、Figure 3-9では、ディメンションの1つは販売された製品である。dim_productテーブルの各行は、販売されている1種類の製品を表し、SKU（Stock-Keeping Unit）、説明、ブランド名、カテゴリ、脂肪含有量、パッケージサイズなどを含む。fact_salesテーブルの各行は、その特定の取引でどの製品が販売されたかを示すために外部キーを使用する（簡単のため、顧客が複数の異なる製品を一度に購入した場合、ファクトテーブルでは別々の行として表現される）。

日付と時刻さえも通常ディメンションテーブルで表現される。これにより日付に関する追加情報（祝日など）をエンコードでき、クエリで祝日と非祝日の売上を区別できるからだ。

「スタースキーマ」という名前は、テーブルの関係を視覚化すると、ファクトテーブルが中央にあり、ディメンションテーブルに囲まれているという事実に由来する。これらのテーブルへの接続は星の光線のようである。

このテンプレートの変種としてsnowflake schema（スノーフレークスキーマ）が知られており、ディメンションがさらにサブディメンションに分解される。たとえば、ブランドと製品カテゴリに別々のテーブルがあり、dim_productテーブルの各行がブランドとカテゴリをdim_productテーブル内の文字列として格納するのではなく、外部キーとして参照するかもしれない。スノーフレークスキーマはスタースキーマよりも正規化されているが、アナリストにとって扱いやすいため、スタースキーマの方がしばしば好まれる [55]。

典型的なデータウェアハウスでは、テーブルは非常に幅広いことが多い。ファクトテーブルは100を超える列を持つことが多く、時には数百になる [51]。ディメンションテーブルも非常に幅広くなりうる。分析に関連するすべてのメタデータを含むからだ。たとえば、dim_storeテーブルには各店舗でどのサービスが提供されているか、店内にベーカリーがあるか、平方フィート数、店舗が最初に開店した日付、最後に改装された時期、最寄りの高速道路からの距離などの詳細が含まれるかもしれない。

## Column-Oriented Storage（列指向ストレージ）

ファクトテーブルに数兆の行とペタバイトのデータがある場合、それを効率的に格納してクエリすることは困難な問題となる。ディメンションテーブルは通常はるかに小さい（数百万行）ので、このセクションでは主にファクトの格納に集中する。

ファクトテーブルは100列を超えることが多いが、典型的なデータウェアハウスクエリは一度に4〜5列しかアクセスしない（分析で`SELECT *` クエリが必要になることはまれである）[51]。Example 3-1のクエリを見てみよう：大量の行にアクセスする（2013年中に果物またはキャンディを購入した全発生）が、fact_salesテーブルの3列（date_key、product_sk、quantity）にしかアクセスしない。クエリは他のすべての列を無視する。

Example 3-1. 曜日ごとに人々がフレッシュフルーツとキャンディのどちらを買いやすいかの分析

```sql
SELECT
    dim_date.weekday, dim_product.category,
    SUM(fact_sales.quantity) AS quantity_sold
FROM fact_sales
    JOIN dim_date    ON fact_sales.date_key   = dim_date.date_key
    JOIN dim_product ON fact_sales.product_sk = dim_product.product_sk
WHERE
    dim_date.year = 2013 AND
    dim_product.category IN ('Fresh fruit', 'Candy')
GROUP BY
    dim_date.weekday, dim_product.category;
```

このクエリを効率的に実行するにはどうすればよいか？

ほとんどのOLTPデータベースではストレージは行指向で配置される：テーブルの1行のすべての値が隣り合って格納される。ドキュメントデータベースも同様で、ドキュメント全体が通常1つの連続バイトシーケンスとして格納される。Figure 3-1のCSV例でこれが見られる。

Example 3-1のようなクエリを処理するには、fact_sales.date_keyやfact_sales.product_skにインデックスがあれば、特定の日付や製品のすべての売上が見つかる場所を教えてくれるかもしれない。しかしその後、行指向ストレージエンジンはそれらの行すべて（100以上の属性で構成される各行）をディスクからメモリにロードし、パースし、要求された条件を満たさないものをフィルタリングしなければならない。それには長い時間がかかりうる。

column-oriented storage（列指向ストレージ）の背後にあるアイデアはシンプルである：1つの行のすべての値をまとめて格納するのではなく、各列のすべての値をまとめて格納する。各列が別々のファイルに格納されていれば、クエリはそのクエリで使用される列だけを読み取ってパースすればよく、大量の作業を節約できる。この原理はFigure 3-10に示されている。

> 列ストレージはリレーショナルデータモデルで理解するのが最も容易だが、非リレーショナルデータにも同様に適用される。たとえばParquet [57] は、GoogleのDremel [54] に基づくドキュメントデータモデルをサポートする列指向ストレージフォーマットである。

*Figure 3-10. リレーショナルデータを行ごとではなく列ごとに格納する。*

列指向ストレージのレイアウトは、各列ファイルが行を同じ順序で含むことに依存している。したがって行全体を再構築する必要がある場合、各列ファイルのk番目のエントリを取り出し、それらをまとめてテーブルのk番目の行を形成できる。

### Column Compression（列の圧縮）

クエリに必要な列だけをディスクからロードすることに加えて、データを圧縮することでディスクスループットへの要求をさらに減らすことができる。幸いなことに、列指向ストレージは圧縮に非常に適していることが多い。

Figure 3-10の各列の値の列を見てみよう：しばしばかなり反復的に見え、これは圧縮にとって良い兆候である。列のデータに応じて異なる圧縮技法が使える。データウェアハウスで特に効果的な技法の一つはbitmap encoding（ビットマップエンコーディング）であり、Figure 3-11に示されている。

*Figure 3-11. 単一の列の圧縮されたビットマップインデックス付きストレージ。*

しばしば、列内のdistinct value（異なる値）の数は行数と比べて少ない（たとえば小売業者は数十億の売上トランザクションを持つかもしれないが、異なる製品は10万しかない）。n個の異なる値を持つ列を、n個の別々のビットマップに変換できる：各異なる値に1つのビットマップで、各行に1ビット。行がその値を持つ場合ビットは1、持たない場合は0である。

nが非常に小さい場合（たとえば国の列は約200の異なる値を持つかもしれない）、それらのビットマップは行あたり1ビットで格納できる。しかしnがより大きい場合、ほとんどのビットマップに多くのゼロがある（sparse、疎であると言う）。その場合、ビットマップにさらにrun-length encoding（ランレングスエンコーディング）を適用できる（Figure 3-11の下部に示す）。これにより列のエンコーディングは驚くほどコンパクトにできる。

このようなビットマップインデックスは、データウェアハウスで一般的な種類のクエリに非常に適している。たとえば：

`WHERE product_sk IN (30, 68, 69)`:
product_sk = 30、product_sk = 68、product_sk = 69の3つのビットマップをロードし、3つのビットマップのbitwise OR（ビット単位のOR）を計算する。これは非常に効率的に実行できる。

`WHERE product_sk = 31 AND store_sk = 3`:
product_sk = 31とstore_sk = 3のビットマップをロードし、bitwise AND（ビット単位のAND）を計算する。これが機能するのは、列が行を同じ順序で含むため、ある列のビットマップのk番目のビットが別の列のビットマップのk番目のビットと同じ行に対応するからである。

異なる種類のデータに対するさまざまな他の圧縮スキームもあるが、ここでは詳細には入らない。概要は [58] を参照。

> **列指向ストレージとカラムファミリー**: CassandraとHBaseにはcolumn family（カラムファミリー）の概念があり、Bigtable [9] から受け継いだものである。しかし、それらを列指向と呼ぶのは非常に誤解を招く：各カラムファミリー内で、行のすべての列を行キーとともにまとめて格納し、列圧縮は使用しない。したがって、Bigtableモデルは依然としてほぼ行指向である。

#### Memory bandwidth and vectorized processing（メモリ帯域幅とベクタ化処理）

数百万行をスキャンする必要があるデータウェアハウスクエリにとって、大きなボトルネックはディスクからメモリへのデータ転送帯域幅である。しかし、それだけがボトルネックではない。分析データベースの開発者は、メインメモリからCPUキャッシュへの帯域幅の効率的な使用、ブランチミスプレディクションとCPU命令処理パイプラインのバブルの回避、そして最新のCPUのSIMD（Single-Instruction-Multi-Data）命令の活用にも気を配っている [59, 60]。

ディスクからロードする必要があるデータ量の削減に加えて、列指向ストレージレイアウトはCPUサイクルの効率的な使用にも優れている。たとえば、クエリエンジンはCPUのL1キャッシュに収まる圧縮された列データのチャンクを取り、タイトなループ（関数呼び出しなし）で反復できる。CPUはそのようなループを、処理される各レコードに対して多くの関数呼び出しや条件を必要とするコードよりもはるかに高速に実行できる。列圧縮により、同じ量のL1キャッシュに列のより多くの行を収めることができる。前述のビット単位のANDやORのような演算は、そのような圧縮された列データのチャンクに対して直接操作するよう設計できる。この技法はvectorized processing（ベクタ化処理）として知られている [58, 49]。

### Sort Order in Column Storage（列ストレージのソート順序）

列ストアでは、行がどの順序で格納されているかは必ずしも重要ではない。挿入された順序で格納するのが最も簡単で、新しい行の挿入は各列ファイルに追記するだけで済む。しかし、以前SSTableでやったように順序を課し、それをインデックスメカニズムとして使うことを選択できる。

各列を独立にソートしても意味がないことに注意してほしい。それでは列内のどのアイテムが同じ行に属するかわからなくなるからだ。k番目の列の1つのk番目のアイテムが別の列のk番目のアイテムと同じ行に属するとわかっているからこそ、行を再構築できるのだ。

むしろ、列ごとに格納されていても、データは行全体で一度にソートされる必要がある。データベースの管理者は、一般的なクエリの知識を使って、テーブルをソートする列を選択できる。たとえばクエリがしばしば日付範囲（先月など）を対象にする場合、date_keyを最初のソートキーにするのが合理的かもしれない。するとクエリオプティマイザは先月の行だけをスキャンでき、すべての行をスキャンするよりもはるかに高速になる。

最初の列で同じ値を持つ行のソート順序は、2番目の列で決定できる。たとえばFigure 3-10でdate_keyが最初のソートキーなら、product_skを2番目のソートキーにするのが合理的かもしれない。同じ日の同じ製品のすべての売上がストレージ内でグループ化されるからだ。これは特定の日付範囲内で製品ごとにグループ化またはフィルタリングする必要があるクエリに役立つ。

ソート順序のもう1つの利点は、列の圧縮に役立つことである。主ソートの列にあまり多くの異なる値がなければ、ソート後、同じ値が行内で何度も連続して繰り返される長い列が生じる。Figure 3-11で使ったようなシンプルなランレングスエンコーディングで、テーブルに数十億行があっても、その列を数キロバイトに圧縮できる。

この圧縮効果は最初のソートキーで最も強い。2番目と3番目のソートキーはより混在し、繰り返し値の長い連続はそれほどない。ソート優先度のさらに下の列は本質的にランダムな順序で現れるため、おそらくそれほどうまく圧縮されない。しかし最初のいくつかの列がソートされているだけでも、全体としては利点がある。

#### Several different sort orders（複数の異なるソート順序）

このアイデアの巧みな拡張がC-Storeで導入され、商用データウェアハウスVerticaで採用された [61, 62]。異なるクエリは異なるソート順序の恩恵を受けるので、同じデータを複数の異なる方法でソートして格納してはどうか？ いずれにしても、1台のマシンが故障した場合にデータを失わないよう、データは複数のマシンにレプリケーションする必要がある。その冗長なデータを異なる方法でソートして格納すれば、クエリを処理するとき、そのクエリパターンに最も適したバージョンを使用できる。

列指向ストアでの複数のソート順序を持つことは、行指向ストアでの複数のセカンダリインデックスを持つことにやや似ている。しかし大きな違いは、行指向ストアはすべての行を1箇所に保持し（ヒープファイルまたはクラスタードインデックス内）、セカンダリインデックスは一致する行へのポインタを含むだけであることだ。列ストアには通常、別の場所のデータへのポインタはなく、値を含む列のみがある。

### Writing to Column-Oriented Storage（列指向ストレージへの書き込み）

これらの最適化はデータウェアハウスで意味がある。ロードの大部分はアナリストが実行する大規模な読み取り専用クエリだからだ。列指向ストレージ、圧縮、ソートはこれらの読み取りクエリを高速化するのに役立つ。しかし、書き込みをより困難にするという欠点がある。

圧縮された列ではB-treeが使うようなupdate-in-place（その場での更新）アプローチは不可能である。ソートされたテーブルの中間に行を挿入したい場合、すべての列ファイルを書き直す必要がある可能性が高い。行は列内の位置で識別されるため、挿入はすべての列を一貫して更新する必要がある。

幸いなことに、本章の前半ですでに良い解決策を見ている：LSM-treeである。すべての書き込みはまずインメモリストアに行き、ソートされた構造に追加され、ディスクへの書き込みに備えられる。インメモリストアが行指向か列指向かは問題ではない。十分な書き込みが蓄積されたら、ディスク上の列ファイルとマージされ、新しいファイルに一括で書き込まれる。これは本質的にVerticaがやっていることである [62]。

クエリはディスク上の列データとメモリ内の最近の書き込みの両方を調べ、2つを組み合わせる必要がある。しかし、クエリオプティマイザはこの区別をユーザーから隠す。アナリストの観点からは、挿入、更新、または削除で変更されたデータはすぐに後続のクエリに反映される。

### Aggregation: Data Cubes and Materialized Views（集計：データキューブとマテリアライズドビュー）

すべてのデータウェアハウスが必ずしも列ストアであるわけではない：従来の行指向データベースやその他のアーキテクチャも使用される。しかし、列指向ストレージはアドホックな分析クエリに対して大幅に高速になりうるため、急速に人気が高まっている [51, 63]。

データウェアハウスのもう1つの側面として簡潔に言及する価値があるのはmaterialized aggregate（マテリアライズド集計）である。先に議論したように、データウェアハウスクエリはしばしばCOUNT、SUM、AVG、MIN、MAXなどのSQLの集計関数を含む。同じ集計が多くの異なるクエリで使用される場合、毎回生データを処理するのは無駄である。最もよく使われるカウントや合計の一部をキャッシュしてはどうか？

このようなキャッシュを作成する1つの方法がmaterialized view（マテリアライズドビュー）である。リレーショナルデータモデルでは、標準的な（仮想の）ビューのように定義されることが多い。あるクエリの結果であるテーブルのようなオブジェクト。違いは、マテリアライズドビューはクエリ結果の実際のコピーがディスクに書き込まれるのに対し、virtual view（仮想ビュー）はクエリを書くためのショートカットにすぎないことである。仮想ビューから読み取ると、SQLエンジンはそれをビューの基礎となるクエリにオンザフライで展開し、展開されたクエリを処理する。

基礎データが変更されると、マテリアライズドビューはデータの非正規化コピーであるため更新される必要がある。データベースはこれを自動的に行えるが、そのような更新は書き込みをより高価にするため、マテリアライズドビューはOLTPデータベースではあまり使われない。読み取り負荷が高いデータウェアハウスではより意味がある（実際に読み取りパフォーマンスを改善するかどうかは個々のケースに依存する）。

マテリアライズドビューの一般的な特殊ケースはdata cube（データキューブ）またはOLAP cube [64] として知られている。これは異なるディメンションでグループ化された集計のグリッドである。Figure 3-12に例を示す。

*Figure 3-12. データキューブの2つのディメンション、合計によるデータの集計。*

今のところ、各ファクトが2つのディメンションテーブル（Figure 3-12では日付と製品）への外部キーだけを持つと想像してほしい。日付を1つの軸に、製品をもう1つの軸に2次元テーブルを描くことができる。各セルはその日付と製品の組み合わせを持つすべてのファクトの属性（例：net_price）の集計（例：SUM）を含む。次に各行または列に沿って同じ集計を適用し、1つのディメンションで削減された要約を得ることができる（日付に関係なく製品ごとの売上、または製品に関係なく日付ごとの売上）。

一般にファクトは2つ以上のディメンションを持つことが多い。Figure 3-9では、日付、製品、店舗、プロモーション、顧客の5つのディメンションがある。5次元のハイパーキューブがどのようなものか想像するのはかなり困難だが、原理は同じである：各セルは特定の日付-製品-店舗-プロモーション-顧客の組み合わせの売上を含む。これらの値は各ディメンションに沿って繰り返し要約できる。

マテリアライズドデータキューブの利点は、特定のクエリが事前に計算されているため非常に高速になることである。たとえば、昨日の店舗ごとの総売上を知りたい場合、適切なディメンションに沿った合計を見るだけでよい——数百万行をスキャンする必要はない。

欠点は、データキューブが生データのクエリと同じ柔軟性を持たないことである。たとえば、100ドルを超えるアイテムからの売上がどの程度の割合を占めるかを計算する方法はない。価格はディメンションの1つではないからだ。ほとんどのデータウェアハウスはしたがって、できるだけ多くの生データを保持し、データキューブのような集計は特定のクエリのパフォーマンスブーストとしてのみ使用しようとする。

## Summary（まとめ）

本章では、データベースがストレージと検索をどのように処理するかの基本に迫ろうとした。データベースにデータを格納すると何が起こるか、後でデータをクエリしたときにデータベースは何をするのか？

高いレベルでは、ストレージエンジンが2つの大きなカテゴリに分類されることを見た：トランザクション処理（OLTP）に最適化されたものと、分析（OLAP）に最適化されたものである。これらのユースケースにおけるアクセスパターンには大きな違いがある：

- **OLTPシステム**は通常ユーザー向けであり、膨大な量のリクエストを受ける可能性がある。負荷を処理するために、アプリケーションは通常各クエリで少数のレコードにのみアクセスする。アプリケーションは何らかのキーを使ってレコードを要求し、ストレージエンジンはインデックスを使って要求されたキーのデータを見つける。ここではディスクシーク時間がしばしばボトルネックになる。
- **データウェアハウスと同様の分析システム**はあまりよく知られていない。主にビジネスアナリストが使用し、エンドユーザーではないからだ。OLTPシステムよりもはるかに少ないクエリ量を処理するが、各クエリは通常非常に要求が厳しく、短時間で数百万のレコードをスキャンする必要がある。ここではディスク帯域幅（シーク時間ではなく）がしばしばボトルネックであり、列指向ストレージがこの種のワークロードに対してますます人気のあるソリューションとなっている。

OLTPの面では、2つの主要な流派のストレージエンジンを見た：

- **ログ構造化の流派**：ファイルへの追記と古いファイルの削除のみを許可し、書き込まれたファイルを更新しない。Bitcask、SSTable、LSM-tree、LevelDB、Cassandra、HBase、Luceneなどがこのグループに属する。
- **その場での更新の流派**：ディスクを上書き可能な固定サイズのページの集合として扱う。B-treeがこの哲学の最大の例であり、すべての主要なリレーショナルデータベースと多くの非リレーショナルデータベースで使用されている。

ログ構造化ストレージエンジンは比較的最近の発展である。その重要なアイデアは、ディスク上のランダムアクセス書き込みをシーケンシャル書き込みに体系的に変換し、ハードドライブとSSDのパフォーマンス特性により高い書き込みスループットを実現することである。

OLTPの仕上げとして、より複雑なインデックス構造と、すべてのデータをメモリに保持するよう最適化されたデータベースについて簡潔にツアーした。

次にストレージエンジンの内部から離れて、典型的なデータウェアハウスの高レベルアーキテクチャを見た。この背景は、分析ワークロードがOLTPとなぜこれほど異なるかを示した：クエリが大量の行にわたってシーケンシャルにスキャンする必要がある場合、インデックスはあまり関連性がない。代わりに、クエリがディスクから読み取る必要があるデータ量を最小限にするために、データを非常にコンパクトにエンコードすることが重要になる。列指向ストレージがこの目標の達成にどのように役立つかを議論した。

アプリケーション開発者として、ストレージエンジンの内部についてのこの知識を持っていれば、特定のアプリケーションに最も適したツールを知る上ではるかに有利な立場にある。データベースのチューニングパラメータを調整する必要がある場合、この理解により、より高いまたは低い値がどのような効果を持つか想像できるようになる。

本章では特定のストレージエンジンのチューニングのエキスパートにすることはできなかったが、選択したデータベースのドキュメントを理解できるだけの十分な語彙とアイデアを提供できたことを願っている。
