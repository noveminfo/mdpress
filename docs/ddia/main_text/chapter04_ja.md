# 第4章：エンコーディングと進化

> *すべては変わり、何も留まらない。*
> ――ヘラクレイトス（プラトン『クラテュロス』より、紀元前360年頃）

アプリケーションは時間の経過とともに必然的に変化する。新しい製品が発売され、ユーザーの要件がより良く理解され、ビジネスの状況が変わるにつれて、機能が追加または変更される。第1章ではevolvability（進化可能性）のアイデアを紹介した：変化への適応を容易にするシステムを構築することを目指すべきである（「進化可能性：変更を容易にする」21ページ参照）。

ほとんどの場合、アプリケーションの機能の変更にはそれが格納するデータの変更も伴う：新しいフィールドやレコード型を取り込む必要があるかもしれないし、既存のデータを新しい方法で提示する必要があるかもしれない。

第2章で議論したデータモデルには、そのような変更に対処する異なる方法がある。リレーショナルデータベースは一般的にデータベース内のすべてのデータが1つのスキーマに準拠していることを前提とする：そのスキーマは（スキーママイグレーション、すなわちALTER文を通じて）変更できるが、任意の時点で効力を持つスキーマは正確に1つである。一方、schema-on-read（「スキーマレス」）データベースはスキーマを強制しないため、異なる時点で書き込まれた古いフォーマットと新しいフォーマットのデータが混在しうる（「ドキュメントモデルにおけるスキーマの柔軟性」39ページ参照）。

データフォーマットやスキーマが変更されると、それに対応するアプリケーションコードの変更が必要になることが多い（たとえば、レコードに新しいフィールドを追加し、アプリケーションコードがそのフィールドの読み書きを開始する）。しかし大規模なアプリケーションでは、コードの変更を瞬時に行うことはできないことが多い：

- サーバーサイドアプリケーションではrolling upgrade（ローリングアップグレード、staged rolloutとも呼ばれる）を実施したいことがある。新しいバージョンを一度に少数のノードにデプロイし、新しいバージョンがスムーズに動作しているか確認し、徐々にすべてのノードに展開していく。これによりサービスのダウンタイムなしに新バージョンをデプロイでき、より頻繁なリリースとより良い進化可能性が促進される。
- クライアントサイドアプリケーションではユーザーの意向に左右される。ユーザーはしばらくの間アップデートをインストールしないかもしれない。

これは、古いバージョンと新しいバージョンのコード、そして古いフォーマットと新しいフォーマットのデータが、すべてシステム内で同時に共存しうることを意味する。システムがスムーズに動作し続けるためには、両方向の互換性を維持する必要がある：

**Backward compatibility（後方互換性）**
新しいコードが古いコードによって書かれたデータを読める。

**Forward compatibility（前方互換性）**
古いコードが新しいコードによって書かれたデータを読める。

後方互換性は通常、実現するのが難しくない：新しいコードの作者として、古いコードが書いたデータのフォーマットを知っているので、それを明示的に処理できる（必要であれば単に古いコードを保持して古いデータを読めばよい）。前方互換性はより巧妙になりうる。なぜなら古いコードが新しいバージョンのコードによる追加を無視する必要があるからだ。

本章では、JSON、XML、Protocol Buffers、Thrift、Avroを含むデータのエンコーディングのいくつかのフォーマットを見る。特に、スキーマの変更をどのように扱い、新旧のデータとコードの共存が必要なシステムをどのようにサポートするかを見る。次にそれらのフォーマットがデータの格納と通信にどのように使われるかを議論する：Webサービス、REST（Representational State Transfer）、RPC（remote procedure call）、そしてアクターやメッセージキューのようなメッセージパッシングシステムにおいて。

## データのエンコーディングフォーマット

プログラムは通常、（少なくとも）2つの異なる表現でデータを扱う：

1. メモリ上では、データはオブジェクト、構造体、リスト、配列、ハッシュテーブル、木などで保持される。これらのデータ構造はCPUによる効率的なアクセスと操作のために最適化されている（通常はポインタを使用する）。
2. データをファイルに書き込んだりネットワーク上で送信したりする場合、何らかの自己完結的なバイト列（たとえばJSONドキュメント）としてencode（エンコード）する必要がある。ポインタは他のプロセスにとって意味がないため、このバイト列表現はメモリ上で通常使われるデータ構造とはかなり異なって見える。^i

したがって、2つの表現の間で何らかの変換が必要である。インメモリ表現からバイト列への変換はencoding（エンコーディング、serialization〈シリアライゼーション〉またはmarshalling〈マーシャリング〉とも呼ばれる）と呼ばれ、その逆はdecoding（デコーディング、parsing〈パース〉、deserialization〈デシリアライゼーション〉、unmarshalling〈アンマーシャリング〉）と呼ばれる。^ii

> **用語の衝突**: Serializationという用語は残念ながらトランザクションの文脈でも使われ（第7章参照）、まったく異なる意味を持つ。語の多義化を避けるため、本書ではencodingという語を使う。serializationの方がより一般的な用語かもしれないが。

これは非常に一般的な問題であるため、無数のライブラリとエンコーディングフォーマットから選択できる。簡単な概要を見てみよう。

^i メモリマップドファイルのような特殊なケースや、圧縮データに直接操作する場合（「列の圧縮」97ページで説明）を除く。

^ii encodingは暗号化（encryption）とは無関係であることに注意。本書では暗号化について議論しない。

### Language-Specific Formats（言語固有のフォーマット）

多くのプログラミング言語にはインメモリオブジェクトをバイト列にエンコードするための組み込みサポートがある。たとえばJavaには `java.io.Serializable` [1]、Rubyには `Marshal` [2]、Pythonにはpickleモジュール [3] がある。Kryo for Java [4] のような多くのサードパーティライブラリも存在する。

これらのエンコーディングライブラリは非常に便利である。最小限の追加コードでインメモリオブジェクトを保存・復元できるからだ。しかし、いくつかの根深い問題も抱えている：

- エンコーディングはしばしば特定のプログラミング言語に紐付いており、別の言語でデータを読むのは非常に困難である。そのようなエンコーディングでデータを格納・送信すると、潜在的に非常に長い期間にわたって現在のプログラミング言語にコミットすることになり、他の組織のシステム（異なる言語を使用しているかもしれない）との統合の可能性を排除する。
- データを同じオブジェクト型で復元するために、デコードプロセスは任意のクラスをインスタンス化できる必要がある。これはセキュリティ上の問題の頻繁な原因である [5]：攻撃者がアプリケーションに任意のバイト列をデコードさせることができれば、任意のクラスをインスタンス化でき、それによりリモートで任意のコードを実行するなどのひどいことができてしまう [6, 7]。
- データのバージョニングはこれらのライブラリでは往々にして後付けの考慮事項である：データの迅速で簡単なエンコーディングを意図しているため、前方互換性と後方互換性の不便な問題をしばしば無視する。
- 効率性（エンコードまたはデコードにかかるCPU時間、およびエンコードされた構造のサイズ）も往々にして後付けの考慮事項である。たとえばJavaの組み込みシリアライゼーションは、パフォーマンスの悪さと肥大化したエンコーディングで悪名高い [8]。

これらの理由から、非常に一時的な目的以外では言語の組み込みエンコーディングを使用するのは一般的に悪いアイデアである。

### JSON, XML, and Binary Variants（JSON、XML、バイナリ変種）

多くのプログラミング言語で書き読みできる標準化されたエンコーディングに移行すると、JSONとXMLが明白な候補である。これらは広く知られ、広くサポートされ、ほぼ同程度に嫌われている。XMLは冗長すぎて不必要に複雑であるとしばしば批判される [9]。JSONの人気は主にWebブラウザでの組み込みサポート（JavaScriptのサブセットであることによる）とXMLと比べた相対的なシンプルさによる。CSVはもう1つの人気のある言語非依存フォーマットだが、それほど強力ではない。

JSON、XML、CSVはテキスト形式であり、したがってある程度人間に読みやすい（ただし構文は論争の的になりやすい話題である）。表面的な構文の問題に加えて、いくつかの微妙な問題も抱えている：

- **数値のエンコーディングに多くのあいまいさがある。** XMLとCSVでは（外部スキーマを参照しない限り）数値と数字で構成される文字列を区別できない。JSONは文字列と数値を区別するが、整数と浮動小数点数を区別せず、精度も指定しない。これは大きな数値を扱う場合に問題となる。たとえば2^53を超える整数はIEEE 754の倍精度浮動小数点数で正確に表現できないため、浮動小数点数を使う言語（JavaScriptなど）でパースすると不正確になる。Twitterでは各ツイートの識別に64ビット数を使用しており、2^53を超える数が生じる。TwitterのAPIが返すJSONにはツイートIDがJSON数値と10進文字列の両方として2回含まれている。JavaScript アプリケーションで数値が正しくパースされない事実への回避策である [10]。
- **JSONとXMLはUnicode文字列（人間が読めるテキスト）に対する良いサポートを持つが、バイナリ文字列（文字エンコーディングのないバイト列）をサポートしない。** バイナリ文字列は有用な機能であるため、人々はBase64を使ってバイナリデータをテキストとしてエンコードすることでこの制限を回避する。スキーマがその値をBase64エンコードとして解釈すべきことを示す。これは機能するが、やや場当たり的であり、データサイズが33%増加する。
- **XMLとJSONの両方にオプションのスキーマサポートがある** [11, 12]。これらのスキーマ言語はかなり強力であり、したがって学習と実装がかなり複雑である。XMLスキーマの使用はかなり広まっているが、多くのJSONベースのツールはスキーマの使用を気にしない。データの正しい解釈（数値やバイナリ文字列など）はスキーマの情報に依存するため、XML/JSONスキーマを使用しないアプリケーションは適切なエンコード/デコードロジックをハードコードする必要がある可能性がある。
- **CSVにはスキーマがないため**、各行と各列の意味を定義するのはアプリケーション次第である。アプリケーションの変更で新しい行や列が追加された場合、その変更を手動で処理する必要がある。CSVはまたかなりあいまいなフォーマットでもある（値にカンマや改行文字が含まれている場合どうなるか？）。エスケープルールは正式に仕様化されている [13] が、すべてのパーサーがそれを正しく実装しているわけではない。

これらの欠点にもかかわらず、JSON、XML、CSVは多くの目的に十分である。特にデータ交換フォーマット（すなわち1つの組織から別の組織にデータを送信するため）として人気が続くだろう。このような状況では、人々がフォーマットに合意している限り、フォーマットがどれほど美しいか効率的かはあまり重要ではないことが多い。異なる組織が何かに合意する困難さが、他のほとんどの懸念を上回る。

### Binary encoding（バイナリエンコーディング）

組織の内部でのみ使用されるデータに対しては、最小公倍数的なエンコーディングフォーマットを使用するプレッシャーが少ない。たとえば、よりコンパクトでパースが高速なフォーマットを選択できる。小さなデータセットでは利点は無視できるが、テラバイト規模になると、データフォーマットの選択は大きな影響を持ちうる。

JSONはXMLより冗長ではないが、バイナリフォーマットと比較するとどちらもまだ多くのスペースを使う。この観察は、JSON用（MessagePack、BSON、BJSON、UBJSON、BISON、Smileなど）とXML用（WBXMLやFast Infosetなど）の大量のバイナリエンコーディングの開発につながった。これらのフォーマットはさまざまなニッチで採用されているが、JSONとXMLのテキスト版ほど広く採用されているものはない。

これらのフォーマットの一部はデータ型のセットを拡張する（整数と浮動小数点数の区別やバイナリ文字列のサポートの追加など）が、それ以外はJSON/XMLのデータモデルを変更しない。特に、スキーマを規定しないため、エンコードされたデータ内にすべてのオブジェクトフィールド名を含める必要がある。つまり、Example 4-1のJSONドキュメントのバイナリエンコーディングでは、userName、favoriteNumber、interestsという文字列をどこかに含める必要がある。

Example 4-1. 本章でいくつかのバイナリフォーマットでエンコードするレコード例

```json
{
    "userName": "Martin",
    "favoriteNumber": 1337,
    "interests": ["daydreaming", "hacking"]
}
```

2. 2番目のバイト0xa8は、後に続くのが文字列（上位4ビット = 0xa0）で8バイト長（下位4ビット = 0x08）であることを示す。

### Thrift and Protocol Buffers（ThriftとProtocol Buffers）

Apache Thrift [15] とProtocol Buffers（protobuf）[16] は、同じ原理に基づくバイナリエンコーディングライブラリである。Protocol Buffersは元々Googleで開発され、ThriftはもともとFacebookで開発され、どちらも2007〜08年にオープンソース化された [17]。

ThriftとProtocol Buffersの両方とも、エンコードされるデータのスキーマを必要とする。Example 4-1のデータをThriftでエンコードするには、Thriftのインターフェース定義言語（IDL）でスキーマを次のように記述する：

```
struct Person {
    1: required string     userName,
    2: optional i64        favoriteNumber,
    3: optional list<string> interests
}
```

Protocol Buffersの同等のスキーマ定義は非常に似ている：

```
message Person {
    required string user_name       = 1;
    optional int64  favorite_number = 2;
    repeated string interests       = 3;
}
```

ThriftとProtocol Buffersにはそれぞれコード生成ツールが付属しており、ここに示されたようなスキーマ定義を受け取り、さまざまなプログラミング言語でそのスキーマを実装するクラスを生成する [18]。アプリケーションコードはこの生成されたコードを呼び出して、スキーマのレコードをエンコードまたはデコードできる。

このスキーマでエンコードされたデータはどのように見えるか？ 紛らわしいことに、Thriftには2つの異なるバイナリエンコーディングフォーマットがあり^iii、BinaryProtocolとCompactProtocolと呼ばれる。まずBinaryProtocolを見てみよう。そのフォーマットでExample 4-1をエンコードすると59バイトになり、Figure 4-2に示される [19]。

*Figure 4-2. Example 4-1のレコードをThriftのBinaryProtocolでエンコードしたもの。*

Figure 4-1と同様に、各フィールドには型アノテーション（文字列か整数かリストかなどを示す）があり、必要に応じて長さの表示（文字列の長さ、リストのアイテム数）がある。データ内に現れる文字列（"Martin"、"daydreaming"、"hacking"）もASCII（正確にはUTF-8）としてエンコードされる。

Figure 4-1と比較した大きな違いは、**フィールド名（userName、favoriteNumber、interests）がない**ことである。代わりに、エンコードされたデータにはfield tag（フィールドタグ）が含まれ、これは数値（1、2、3）である。スキーマ定義に現れる数値がそれである。フィールドタグはフィールドのエイリアスのようなものである——フィールド名を完全に綴ることなく、どのフィールドについて話しているかを言うコンパクトな方法だ。

#### Field tags and schema evolution（フィールドタグとスキーマの進化）

先に述べたように、スキーマは時間の経過とともに必然的に変更される必要がある。これをschema evolution（スキーマの進化）と呼ぶ。ThriftとProtocol Buffersは後方互換性と前方互換性を保ちながらスキーマの変更をどのように扱うのか？

例からわかるように、エンコードされたレコードはエンコードされたフィールドの連結にすぎない。各フィールドはそのタグ番号（サンプルスキーマの数値1、2、3）で識別され、データ型（文字列や整数など）でアノテーションされる。フィールド値が設定されていない場合、エンコードされたレコードから単に省略される。ここからわかるように、**フィールドタグはエンコードされたデータの意味にとって極めて重要**である。スキーマ内のフィールドの名前は変更できる。エンコードされたデータはフィールド名を参照しないからだ。しかしフィールドのタグは変更できない。既存のすべてのエンコードされたデータが無効になるからだ。

スキーマに新しいフィールドを追加できる。ただし各フィールドに新しいタグ番号を与えることが条件である。古いコード（追加された新しいタグ番号を知らない）が新しいコードで書かれたデータを読もうとした場合、認識しないタグ番号を持つ新しいフィールドは単に無視できる。データ型のアノテーションにより、パーサーはスキップすべきバイト数を決定できる。これにより前方互換性が維持される：古いコードが新しいコードで書かれたレコードを読める。

では後方互換性はどうか？ 各フィールドが一意のタグ番号を持つ限り、新しいコードは常に古いデータを読める。タグ番号が同じ意味を持ち続けるからだ。唯一の詳細は、新しいフィールドを追加する場合、それをrequiredにしてはいけないということだ。フィールドをrequiredにして追加した場合、新しいコードが古いコードで書かれたデータを読むとそのチェックが失敗する。古いコードは追加された新しいフィールドを書き込んでいないからだ。したがって後方互換性を維持するために、スキーマの初回デプロイ後に追加するすべてのフィールドはoptionalであるかデフォルト値を持つ必要がある。

フィールドの削除はフィールドの追加と同じだが、後方互換性と前方互換性の懸念が逆転する。つまり、optionalなフィールドのみ削除できる（requiredフィールドは決して削除できない）。同じタグ番号を再利用してはならない（どこかにその古いタグ番号を含むデータがまだ書き込まれているかもしれず、そのフィールドは新しいコードによって無視されなければならないため）。

#### Datatypes and schema evolution（データ型とスキーマの進化）

フィールドのデータ型の変更はどうか？ それは可能かもしれないが——詳細はドキュメントを確認——値の精度が失われたり切り捨てられたりするリスクがある。たとえば、32ビット整数を64ビット整数に変更したとしよう。新しいコードは古いコードが書いたデータを容易に読める。パーサーが欠けているビットをゼロで埋められるからだ。しかし古いコードが新しいコードで書かれたデータを読む場合、古いコードはまだ値を保持するために32ビットの変数を使用する。デコードされた64ビットの値が32ビットに収まらない場合、切り捨てられる。

Protocol Buffersの興味深い詳細は、リストまたは配列のデータ型を持たないが、代わりにフィールドにrepeatedマーカーを持つことだ（requiredとoptionalに加えた3番目の選択肢）。Figure 4-4に見られるように、repeatedフィールドのエンコーディングはまさにその名のとおり：同じフィールドタグがレコード内に複数回現れるだけである。これにはoptional（単一値）フィールドをrepeated（複数値）フィールドに変更しても問題ないという素晴らしい効果がある。古いデータを読む新しいコードはゼロまたは1つの要素を持つリストを見る（フィールドが存在したかどうかによる）。新しいデータを読む古いコードはリストの最後の要素のみを見る。

Thriftには専用のlistデータ型があり、リスト要素のデータ型でパラメータ化される。これはProtocol Buffersと同じ単一値から複数値への進化を許可しないが、ネストされたリストをサポートする利点がある。

### Avro

Apache Avro [20] は、Protocol BuffersやThriftとは興味深い形で異なるもう1つのバイナリエンコーディングフォーマットである。2009年にHadoopのサブプロジェクトとして開始され、ThriftがHadoopのユースケースに適さなかった結果として生まれた [21]。

Avroもエンコードされるデータの構造を指定するためにスキーマを使用する。2つのスキーマ言語がある：1つ（Avro IDL）は人間による編集を意図しており、もう1つ（JSONベース）はより容易に機械で読み取れる。

この例のスキーマをAvro IDLで書くと次のようになる：

```
record Person {
    string               userName;
    union { null, long }  favoriteNumber = null;
    array<string>         interests;
}
```

そのスキーマの等価なJSON表現は以下のとおり：

```json
{
    "type": "record",
    "name": "Person",
    "fields": [
        {"name": "userName",        "type": "string"},
        {"name": "favoriteNumber",  "type": ["null", "long"], "default": null},
        {"name": "interests",       "type": {"type": "array", "items": "string"}}
    ]
}
```

まず気づくべきは、**スキーマにタグ番号がない**ことである。このスキーマを使ってExample 4-1のレコードをエンコードすると、Avroのバイナリエンコーディングはわずか32バイト長である——これまで見たすべてのエンコーディングの中で最もコンパクトである。エンコードされたバイト列の内訳はFigure 4-5に示されている。

バイト列を調べると、フィールドやそのデータ型を識別するものが何もないことがわかる。エンコーディングは単に値を連結したものから構成される。文字列は長さプレフィックスの後にUTF-8バイトが続くだけだが、エンコードされたデータ内にそれが文字列であることを示すものは何もない。整数かもしれないし、まったく別のものかもしれない。整数は可変長エンコーディング（ThriftのCompactProtocolと同じ）を使ってエンコードされる。

*Figure 4-5. Example 4-1のレコードをAvroでエンコードしたもの。*

バイナリデータをパースするには、スキーマに現れる順序でフィールドを処理し、スキーマを使って各フィールドのデータ型を知る。つまり、バイナリデータはそのデータを書いたコードとまったく同じスキーマを読み取りコードが使用している場合にのみ正しくデコードできる。リーダーとライターの間でスキーマの不一致があると、誤ってデコードされたデータを意味する。

では、Avroはどのようにスキーマの進化をサポートするのか？

#### The writer's schema and the reader's schema（ライターのスキーマとリーダーのスキーマ）

Avroでは、アプリケーションがデータをエンコードしたい（ファイルやデータベースに書き込む、ネットワーク経由で送信するなど）とき、知っているバージョンのスキーマを使ってデータをエンコードする——たとえば、そのスキーマはアプリケーションにコンパイルされているかもしれない。これはwriter's schema（ライターのスキーマ）として知られている。

アプリケーションがデータをデコードしたい（ファイルやデータベースから読み取る、ネットワークから受信するなど）とき、そのデータがあるスキーマに従っていることを期待する。これはreader's schema（リーダーのスキーマ）として知られている。アプリケーションコードが依存するスキーマである——コードはアプリケーションのビルドプロセスでそのスキーマから生成されているかもしれない。

Avroの重要なアイデアは、**ライターのスキーマとリーダーのスキーマが同一である必要はない**——互換性があるだけでよいということだ。データがデコード（読み取り）されるとき、Avroライブラリはライターのスキーマとリーダーのスキーマを並べて比較し、ライターのスキーマからリーダーのスキーマへデータを変換することで差異を解決する。Avro仕様 [20] はこの解決がどのように機能するかを正確に定義しており、Figure 4-6に図示されている。

たとえば、ライターのスキーマとリーダーのスキーマでフィールドの順序が異なっていても問題ない。スキーマの解決はフィールド名でフィールドをマッチングするからだ。データを読み取るコードがライターのスキーマにはあるがリーダーのスキーマにはないフィールドに遭遇した場合、それは無視される。データを読み取るコードがあるフィールドを期待するが、ライターのスキーマにその名前のフィールドがない場合、リーダーのスキーマで宣言されたデフォルト値で埋められる。

*Figure 4-6. Avroのリーダーがライターのスキーマとリーダーのスキーマの違いを解決する。*

#### Schema evolution rules（スキーマ進化のルール）

Avroでは、前方互換性はスキーマの新しいバージョンをライターとして、古いバージョンをリーダーとして使えることを意味する。逆に、後方互換性はスキーマの新しいバージョンをリーダーとして、古いバージョンをライターとして使えることを意味する。

互換性を維持するには、**デフォルト値を持つフィールドのみ追加または削除できる**。（AvroスキーマのfavoriteNumberフィールドにはnullのデフォルト値がある。）たとえば、デフォルト値付きのフィールドを追加したとする。この新しいフィールドは新しいスキーマにはあるが古いスキーマにはない。新しいスキーマを使うリーダーが古いスキーマで書かれたレコードを読むと、欠けているフィールドにデフォルト値が埋め込まれる。

デフォルト値のないフィールドを追加した場合、新しいリーダーは古いライターが書いたデータを読めなくなるため、後方互換性が壊れる。デフォルト値のないフィールドを削除した場合、古いリーダーは新しいライターが書いたデータを読めなくなるため、前方互換性が壊れる。

一部のプログラミング言語ではnullは任意の変数に対する許容されるデフォルトだが、Avroではそうではない：フィールドにnullを許可したい場合、union型を使用する必要がある。たとえば `union { null, long, string } field;` はフィールドが数値、文字列、またはnullであることを示す。nullは union のブランチの1つである場合にのみデフォルト値として使用できる。^iv これはすべてをデフォルトでnull許可にするよりやや冗長だが、何がnullになれて何がなれないかについて明示的であることによりバグの防止に役立つ [22]。

したがって、AvroにはProtocol BuffersやThriftのようなoptionalとrequiredのマーカーがない（代わりにunion型とデフォルト値がある）。フィールドのデータ型の変更はAvroが型を変換できる限り可能である。フィールド名の変更は可能だが少し巧妙である：リーダーのスキーマはフィールド名のエイリアスを含むことができ、古いライターのスキーマのフィールド名をエイリアスとマッチングできる。これはフィールド名の変更が後方互換であるが前方互換ではないことを意味する。同様に、union型にブランチを追加することは後方互換であるが前方互換ではない。

^iv 正確には、デフォルト値はunionの最初のブランチの型でなければならないが、これはAvroの特定の制限であり、union型の一般的な特徴ではない。

#### But what is the writer's schema?（しかしライターのスキーマとは何か？）

ここまで曖昧にしてきた重要な疑問がある：リーダーは特定のデータがどのライターのスキーマでエンコードされたかをどのように知るのか？ スキーマ全体をすべてのレコードに含めることはできない。スキーマはエンコードされたデータよりはるかに大きい可能性が高く、バイナリエンコーディングからのすべてのスペース節約が無駄になるからだ。

答えはAvroが使われる文脈に依存する。いくつかの例を挙げる：

**大量のレコードを含む大きなファイル**
Avroの一般的な用途は——特にHadoopの文脈で——すべて同じスキーマでエンコードされた数百万のレコードを含む大きなファイルの格納である（この種の状況は第10章で議論する）。この場合、そのファイルのライターはファイルの先頭にライターのスキーマを1回だけ含めればよい。Avroはこれを行うためのファイルフォーマット（object container files）を規定している。

**個別に書かれたレコードを持つデータベース**
データベースでは、異なるレコードが異なる時点で異なるライターのスキーマを使って書かれているかもしれない——すべてのレコードが同じスキーマを持つとは仮定できない。最もシンプルな解決策は、エンコードされたすべてのレコードの先頭にバージョン番号を含め、データベースにスキーマバージョンのリストを保持することである。リーダーはレコードを取得し、バージョン番号を抽出し、そのバージョン番号のライターのスキーマをデータベースから取得できる。そのライターのスキーマを使ってレコードの残りをデコードできる（たとえばEspresso [23] はこのように動作する）。

**ネットワーク接続経由でレコードを送信**
双方向のネットワーク接続で2つのプロセスが通信している場合、接続確立時にスキーマバージョンをネゴシエーションし、接続の存続期間中そのスキーマを使用できる。Avro RPCプロトコル（「サービス経由のデータフロー：RESTとRPC」131ページ参照）はこのように動作する。

スキーマバージョンのデータベースはいずれの場合でも有用である。ドキュメントとして機能し、スキーマの互換性を確認する機会を提供するからだ [24]。バージョン番号としてはシンプルなインクリメンタル整数を使うか、スキーマのハッシュを使うことができる。

#### Dynamically generated schemas（動的に生成されるスキーマ）

Protocol BuffersやThriftと比較した、Avroのアプローチの1つの利点は、スキーマにタグ番号が含まれないことである。しかしなぜこれが重要なのか？ スキーマにいくつかの番号を保持することの何が問題なのか？

違いは、Avroがdynamically generated schema（動的に生成されるスキーマ）に対してよりフレンドリーであることだ。たとえば、リレーショナルデータベースの内容をファイルにダンプしたいとし、テキスト形式（JSON、CSV、SQL）の前述の問題を避けるためにバイナリフォーマットを使いたいとする。Avroを使えば、リレーショナルスキーマからAvroスキーマ（前に見たJSON表現）をかなり容易に生成し、そのスキーマを使ってデータベースの内容をエンコードし、Avroオブジェクトコンテナファイルにすべてダンプできる [25]。各データベーステーブルに対してレコードスキーマを生成し、各列がそのレコードのフィールドになる。データベースの列名がAvroのフィールド名にマッピングされる。

データベーススキーマが変更された場合（たとえば1つの列が追加され1つの列が削除された場合）、更新されたデータベーススキーマから新しいAvroスキーマを生成し、新しいAvroスキーマでデータをエクスポートすればよい。データエクスポートプロセスはスキーマの変更に注意を払う必要がない——実行するたびにスキーマ変換を行えばよいだけだ。新しいデータファイルを読む人はレコードのフィールドが変わったことを確認するが、フィールドが名前で識別されるため、更新されたライターのスキーマは古いリーダーのスキーマとマッチングできる。

対照的に、この目的でThriftやProtocol Buffersを使っていた場合、フィールドタグはおそらく手動で割り当てる必要がある。データベーススキーマが変更されるたびに、管理者がデータベースの列名からフィールドタグへのマッピングを手動で更新しなければならない（これを自動化することは可能かもしれないが、スキーマジェネレータは以前に使用されたフィールドタグを割り当てないよう非常に注意する必要がある）。この種の動的に生成されるスキーマは、ThriftやProtocol Buffersの設計目標では単になかったが、Avroでは設計目標だった。

#### Code generation and dynamically typed languages（コード生成と動的型付け言語）

ThriftとProtocol Buffersはコード生成に依存する：スキーマが定義された後、選択したプログラミング言語でそのスキーマを実装するコードを生成できる。これはJava、C++、C#のような静的型付け言語で有用である。デコードされたデータに対して効率的なインメモリ構造を使用でき、データ構造にアクセスするプログラムを書く際にIDEでの型チェックとオートコンプリートが可能になるからだ。

JavaScript、Ruby、Pythonのような動的型付けプログラミング言語では、コードを生成する意味はあまりない。満たすべきコンパイル時の型チェッカーがないからだ。コード生成はこれらの言語ではしばしば嫌がられる。それ以外の点で明示的なコンパイルステップを回避するからだ。さらに、動的に生成されるスキーマ（データベーステーブルから生成されたAvroスキーマなど）の場合、コード生成はデータへのアクセスに対する不必要な障害である。

Avroは静的型付けプログラミング言語に対するオプションのコード生成を提供するが、コード生成なしでもまったく同様に使用できる。オブジェクトコンテナファイル（ライターのスキーマを埋め込んでいる）がある場合、Avroライブラリを使って単にそれを開き、JSONファイルを見るのと同じようにデータを見ることができる。ファイルはすべての必要なメタデータを含んでいるため自己記述的である。

このプロパティはApache Pig [26] のような動的型付けのデータ処理言語と組み合わせて特に有用である。Pigでは、Avroファイルを開き、分析を開始し、スキーマについて考えることすらなく、派生データセットをAvroフォーマットの出力ファイルに書き込むことができる。

### The Merits of Schemas（スキーマの利点）

見てきたように、Protocol Buffers、Thrift、Avroはすべてスキーマを使用してバイナリエンコーディングフォーマットを記述する。これらのスキーマ言語はXML SchemaやJSON Schemaよりはるかにシンプルであり、後者はより詳細なバリデーションルール（「このフィールドの文字列値はこの正規表現にマッチしなければならない」「このフィールドの整数値は0から100の間でなければならない」など）をサポートする。Protocol Buffers、Thrift、Avroは実装がシンプルで使用がシンプルであるため、かなり広範なプログラミング言語のサポートに成長した。

これらのエンコーディングが基づくアイデアは決して新しいものではない。たとえば、ASN.1と多くの共通点がある。ASN.1は1984年に最初に標準化されたスキーマ定義言語である [27]。さまざまなネットワークプロトコルを定義するために使われ、そのバイナリエンコーディング（DER）は今でもSSL証明書（X.509）のエンコードに使われている [28]。ASN.1はProtocol BuffersやThriftと同様にタグ番号を使ってスキーマの進化をサポートする [29]。しかし非常に複雑でドキュメントも不十分なため、ASN.1はおそらく新しいアプリケーションにとって良い選択ではない。

多くのデータシステムもデータに対して何らかの独自のバイナリエンコーディングを実装している。たとえば、ほとんどのリレーショナルデータベースにはネットワークプロトコルがあり、それを通じてデータベースにクエリを送信し応答を受け取れる。これらのプロトコルは一般的に特定のデータベースに固有であり、データベースベンダーがデータベースのネットワークプロトコルからの応答をインメモリデータ構造にデコードするドライバ（ODBCやJDBC APIを使用するなど）を提供する。

以上から、JSON、XML、CSVのようなテキスト形式のデータフォーマットは広く使われているが、スキーマに基づくバイナリエンコーディングも実行可能な選択肢であることがわかる。いくつかの優れた特性がある：

- さまざまな「バイナリJSON」変種よりはるかに**コンパクト**にできる。エンコードされたデータからフィールド名を省略できるからだ。
- **スキーマはドキュメントの価値ある形態**であり、デコードにスキーマが必要であるため、それが最新であることを確信できる（手動で管理されたドキュメントは現実との乖離が容易に生じるのに対し）。
- スキーマのデータベースを保持することで、何かがデプロイされる前にスキーマの変更の**前方互換性と後方互換性をチェック**できる。
- 静的型付けプログラミング言語のユーザーにとって、スキーマからの**コード生成**の能力は有用である。コンパイル時の型チェックを可能にするからだ。

要約すると、スキーマの進化はスキーマレス/schema-on-readのJSONデータベースが提供するのと同じ柔軟性を可能にしながら（「ドキュメントモデルにおけるスキーマの柔軟性」39ページ参照）、データに関するより良い保証とより良いツーリングも提供する。

## データフローのモード

本章の冒頭で、メモリを共有しない別のプロセスにデータを送信したいときは——たとえばネットワーク経由でデータを送信したりファイルに書き込みたいときは——バイト列としてエンコードする必要があると述べた。次にこれを行うためのさまざまな異なるエンコーディングについて議論した。

前方互換性と後方互換性について話した。これは進化可能性にとって重要である（システムの異なる部分を独立してアップグレードでき、すべてを一度に変更する必要がないことにより、変更を容易にする）。互換性は、データをエンコードする1つのプロセスと、それをデコードする別のプロセスの間の関係である。

これはかなり抽象的なアイデアである——データが1つのプロセスから別のプロセスに流れる方法はたくさんある。誰がデータをエンコードし、誰がデコードするのか？ 本章の残りでは、プロセス間でデータが流れる最も一般的な方法のいくつかを探求する：

- データベース経由（「データベース経由のデータフロー」129ページ参照）
- サービス呼び出し経由（「サービス経由のデータフロー：RESTとRPC」131ページ参照）
- 非同期メッセージパッシング経由（「メッセージパッシングのデータフロー」136ページ参照）

### Dataflow Through Databases（データベース経由のデータフロー）

データベースでは、データベースに書き込むプロセスがデータをエンコードし、データベースから読み取るプロセスがそれをデコードする。データベースにアクセスするプロセスが1つだけかもしれない。その場合リーダーは単に同じプロセスの後のバージョンである——この場合、データベースに何かを格納することは未来の自分へのメッセージ送信と考えることができる。

ここでは後方互換性が明らかに必要である。さもなければ未来の自分が以前に書いたものをデコードできなくなる。

一般的に、同時に複数の異なるプロセスがデータベースにアクセスしていることはよくある。それらのプロセスは複数の異なるアプリケーションやサービスかもしれないし、同じサービスの複数のインスタンス（スケーラビリティまたはフォールトトレランスのために並行して実行されている）かもしれない。いずれにしても、アプリケーションが変化している環境では、データベースにアクセスするプロセスの一部が新しいコードを実行し、一部が古いコードを実行している可能性がある——たとえば新しいバージョンがローリングアップグレードで現在デプロイ中であり、一部のインスタンスは更新済みだがまだ更新されていないものもある。

これは、データベース内の値が新しいバージョンのコードによって書かれ、その後まだ実行中の古いバージョンのコードによって読まれる可能性があることを意味する。したがって、データベースでは前方互換性もしばしば必要となる。

しかし、追加の落とし穴がある。レコードスキーマにフィールドを追加し、新しいコードがその新しいフィールドの値をデータベースに書き込んだとしよう。その後、古いバージョンのコード（新しいフィールドについてまだ知らない）がそのレコードを読み取り、更新し、書き戻す。この状況で望ましい動作は通常、古いコードが新しいフィールドを解釈できなくても、そのまま保持することである。

前に議論したエンコーディングフォーマットはそのような未知のフィールドの保持をサポートするが、Figure 4-7に示されるように、アプリケーションレベルで注意が必要な場合もある。たとえば、データベースの値をアプリケーション内のモデルオブジェクトにデコードし、後でそれらのモデルオブジェクトを再エンコードすると、その変換プロセスで未知のフィールドが失われるかもしれない。これを解決するのは難しい問題ではない。ただ認識している必要がある。

#### Different values written at different times（異なる時期に書かれた異なる値）

データベースは一般的に任意の時点で任意の値を更新することを許可する。これは、1つのデータベース内に5ミリ秒前に書かれた値と5年前に書かれた値が混在しうることを意味する。

アプリケーションの新しいバージョンをデプロイする場合（少なくともサーバーサイドアプリケーションの場合）、数分以内に古いバージョンを新しいバージョンに完全に置き換えるかもしれない。データベースの内容についてはそうではない：5年前のデータは、明示的に書き直していない限り、元のエンコーディングのままそこにある。この観察は「**data outlives code**（データはコードより長生きする）」と要約されることがある。

データを新しいスキーマにリライト（マイグレーション）することは確かに可能だが、大規模なデータセットに対してはコストのかかる操作であるため、ほとんどのデータベースは可能であれば避ける。ほとんどのリレーショナルデータベースはnullデフォルト値を持つ新しい列の追加のようなシンプルなスキーマ変更を、既存のデータを書き直すことなく許可する。^v 古い行が読まれるとき、ディスク上のエンコードされたデータに欠けている列にはnullが埋め込まれる。LinkedInのドキュメントデータベースEspressoはストレージにAvroを使い、Avroのスキーマ進化ルールを利用できる [23]。

スキーマの進化により、基盤となるストレージには歴史的なさまざまなバージョンのスキーマでエンコードされたレコードが含まれていても、**データベース全体が単一のスキーマでエンコードされたかのように**見える。

^v MySQLは例外で、「ドキュメントモデルにおけるスキーマの柔軟性」（39ページ）で述べたように、厳密に必要でないにもかかわらずテーブル全体を書き直すことが多い。

#### Archival storage（アーカイブストレージ）

おそらく時折データベースのスナップショットを取得するだろう。バックアップ目的やデータウェアハウスへのロードのために（「データウェアハウジング」91ページ参照）。この場合、データダンプは通常、ソースデータベースの元のエンコーディングが異なる時代のスキーマバージョンの混合を含んでいたとしても、最新のスキーマを使ってエンコードされる。いずれにしてもデータをコピーするのだから、データのコピーを一貫してエンコードしてもよいだろう。

データダンプは一度に書かれ、その後は不変であるため、Avroオブジェクトコンテナファイルのようなフォーマットが適している。これはデータをParquetのような分析に適した列指向フォーマットでエンコードする良い機会でもある（「列の圧縮」97ページ参照）。

第10章ではアーカイブストレージでのデータの使用についてさらに話す。

### Dataflow Through Services: REST and RPC（サービス経由のデータフロー：RESTとRPC）

ネットワーク上で通信する必要があるプロセスがある場合、その通信を配置するいくつかの異なる方法がある。最も一般的な配置はクライアントとサーバーの2つの役割を持つことである。サーバーはネットワーク上でAPIを公開し、クライアントはそのAPIにリクエストを行うためにサーバーに接続できる。サーバーが公開するAPIはservice（サービス）として知られている。

Webはこのように動作する：クライアント（Webブラウザ）がWebサーバーにリクエストを行い、GETリクエストでHTML、CSS、JavaScript、画像などをダウンロードし、POSTリクエストでデータをサーバーに送信する。APIはHTTP、URL、SSL/TLS、HTMLなどの標準化されたプロトコルとデータフォーマットのセットで構成される。

Webブラウザだけがクライアントの種類ではない。モバイルデバイスやデスクトップコンピュータで動作するネイティブアプリ、あるいはWebブラウザ内で動作するクライアントサイドJavaScriptアプリケーションもXMLHttpRequestを使ってHTTPクライアントになれる（この技法はAjax [30] として知られている）。この場合、サーバーの応答は通常、人間に表示するためのHTMLではなく、クライアントサイドのアプリケーションコードでさらに処理するのに便利なエンコーディング（JSONなど）のデータである。

さらに、サーバー自体が別のサービスのクライアントになりうる（たとえば、典型的なWebアプリサーバーはデータベースのクライアントとして振る舞う）。このアプローチは大規模なアプリケーションを機能の領域ごとに小さなサービスに分解するためによく使われ、1つのサービスが別のサービスの機能やデータを必要とするときにリクエストを行う。このアプリケーション構築方法は伝統的にservice-oriented architecture（SOA、サービス指向アーキテクチャ）と呼ばれ、より最近ではmicroservices architecture（マイクロサービスアーキテクチャ）[31, 32] として洗練・リブランドされている。

いくつかの点でサービスはデータベースに似ている：通常クライアントがデータを送信しクエリすることを許可する。しかし、データベースが第2章で議論したクエリ言語を使った任意のクエリを許可するのに対し、サービスはサービスのビジネスロジック（アプリケーションコード）によって事前に決められた入出力のみを許可するアプリケーション固有のAPIを公開する [33]。この制限により、一定のカプセル化が提供される：サービスはクライアントが何をできて何をできないかに対してきめ細かい制限を課すことができる。

サービス指向/マイクロサービスアーキテクチャの重要な設計目標は、サービスを独立してデプロイ可能かつ進化可能にすることで、アプリケーションの変更と保守を容易にすることである。たとえば、各サービスは1つのチームが所有すべきであり、そのチームは他のチームと調整することなく頻繁にサービスの新バージョンをリリースできるべきである。言い換えれば、サーバーとクライアントの古いバージョンと新しいバージョンが同時に動作していることを想定すべきであり、したがってサーバーとクライアントが使用するデータエンコーディングはサービスAPIのバージョン間で互換性がなければならない——まさに本章で話してきたことである。

#### Web services（Webサービス）

HTTPがサービスとの通信の基盤プロトコルとして使われる場合、それはWeb service（Webサービス）と呼ばれる。Webサービスはウェブ上だけでなく、いくつかの異なるコンテキストで使われる：

1. ユーザーのデバイス上で動作するクライアントアプリケーション（モバイルデバイスのネイティブアプリ、AjaxによるJavaScript Webアプリなど）がHTTP経由でサービスにリクエストを行う。これらのリクエストは通常パブリックインターネットを経由する。
2. 同じ組織が所有する別のサービスにリクエストを行うサービスで、しばしば同じデータセンター内に位置し、サービス指向/マイクロサービスアーキテクチャの一部である（このユースケースをサポートするソフトウェアはmiddlewareと呼ばれることがある）。
3. 異なる組織が所有するサービスにリクエストを行うサービスで、通常はインターネット経由。異なる組織のバックエンドシステム間のデータ交換に使われる。クレジットカード処理システムのようなオンラインサービスが提供するパブリックAPI、またはユーザーデータへの共有アクセスのためのOAuthなどが含まれる。

Webサービスには2つの人気のあるアプローチがある：**REST**と**SOAP**である。これらは哲学の面でほぼ正反対であり、しばしばそれぞれの支持者間で熱い議論の対象となる。

RESTはプロトコルではなく、HTTPの原則の上に構築された設計哲学である [34, 35]。シンプルなデータフォーマット、リソースの識別にURLを使用し、キャッシュ制御、認証、コンテンツタイプのネゴシエーションにHTTPの機能を使用することを強調する。RESTは少なくともクロス組織のサービス統合のコンテキストでは、SOAPと比較して人気が高まっており [36]、マイクロサービスと関連付けられることが多い [31]。RESTの原則に従って設計されたAPIはRESTfulと呼ばれる。

対照的に、SOAPはネットワークAPIリクエストを行うためのXMLベースのプロトコルである。HTTPの上で最も一般的に使われるが、HTTPから独立することを目指し、ほとんどのHTTP機能の使用を避ける。代わりに、さまざまな機能を追加する関連標準の膨大で複雑な集合（WS-*として知られるWebサービスフレームワーク）が付属する [37]。

SOAPのWebサービスのAPIは、WSDL（Web Services Description Language）と呼ばれるXMLベースの言語を使って記述される。WSDLはコード生成を可能にし、クライアントがローカルクラスとメソッド呼び出しを使ってリモートサービスにアクセスできる（XMLメッセージにエンコードされ、フレームワークによって再デコードされる）。これは静的型付けプログラミング言語で有用だが、動的型付け言語ではそれほど有用ではない。

WSDLは人間が読むことを想定して設計されておらず、SOAPメッセージは手動で構築するには複雑すぎることが多いため、SOAPのユーザーはツールサポート、コード生成、IDEに大きく依存する [38]。SOAPベンダーがサポートしないプログラミング言語のユーザーにとって、SOAPサービスとの統合は困難である。

SOAPとそのさまざまな拡張は表向き標準化されているが、異なるベンダーの実装間の相互運用性はしばしば問題を引き起こす [39]。これらすべての理由から、SOAPは多くの大企業でまだ使用されているが、ほとんどの小企業では支持を失っている。

RESTful APIはよりシンプルなアプローチを好む傾向があり、通常はより少ないコード生成と自動化ツーリングが関わる。OpenAPI（Swagger [40] としても知られる）のような定義フォーマットを使って、RESTful APIを記述しドキュメントを生成できる。

#### The problems with remote procedure calls (RPCs)（RPCの問題点）

Webサービスは、ネットワーク上でAPIリクエストを行うための長い一連のテクノロジーの最新の化身にすぎない。その多くは大きな注目を集めたが、深刻な問題を抱えていた。Enterprise JavaBeans（EJB）とJavaのRemote Method Invocation（RMI）はJavaに限定される。Distributed Component Object Model（DCOM）はMicrosoftのプラットフォームに限定される。Common Object Request Broker Architecture（CORBA）は過度に複雑であり、後方・前方互換性を提供しない [41]。

これらはすべて、1970年代から存在するremote procedure call（RPC、リモートプロシージャコール）[42] のアイデアに基づいている。RPCモデルは、リモートのネットワークサービスへのリクエストを、同じプロセス内のプログラミング言語の関数やメソッドの呼び出しと同じに見せようとする（この抽象化はlocation transparency〈ロケーション透過性〉と呼ばれる）。

RPCは最初は便利に見えるが、このアプローチは根本的に欠陥がある [43, 44]。ネットワークリクエストはローカルの関数呼び出しとは非常に異なる：

- ローカル関数呼び出しは予測可能で、制御下にあるパラメータにのみ依存して成功か失敗する。ネットワークリクエストは予測不可能である：ネットワーク問題によりリクエストまたは応答が消失するかもしれないし、リモートマシンが遅いか利用不可能かもしれず、そのような問題は完全に制御外である。
- ローカル関数呼び出しは結果を返すか例外をスローするか返らないかのいずれかである。ネットワークリクエストにはもう1つの可能な結果がある：タイムアウトにより結果なしで返ることがある。その場合、何が起きたかわからない（第8章でさらに議論する）。
- 失敗したネットワークリクエストをリトライすると、リクエストは実際には届いており応答だけが失われているということがありうる。その場合、リトライによりアクションが複数回実行される。プロトコルにべき等性のメカニズムを組み込まない限り（第11章でさらに議論する）。
- ローカル関数を呼ぶたびに、通常ほぼ同じ実行時間がかかる。ネットワークリクエストは関数呼び出しよりはるかに遅く、そのレイテンシも大幅に変動する。
- ローカル関数を呼ぶとき、ローカルメモリ内のオブジェクトへの参照（ポインタ）を効率的に渡せる。ネットワークリクエストを行うとき、それらのパラメータはすべてバイト列にエンコードされる必要がある。
- クライアントとサービスは異なるプログラミング言語で実装されているかもしれないため、RPCフレームワークはデータ型をある言語から別の言語に変換しなければならない。

これらすべての要因は、リモートサービスをプログラミング言語のローカルオブジェクトのように見せすぎようとすることに意味がないことを示している。根本的に異なるものだからだ。RESTの魅力の一部は、それがネットワークプロトコルであることを隠そうとしないことにある。

#### Current directions for RPC（RPCの現在の方向性）

これらすべての問題にもかかわらず、RPCはなくなりつつはない。本章で言及したすべてのエンコーディングの上にさまざまなRPCフレームワークが構築されている：たとえばThriftとAvroにはRPCサポートが含まれ、gRPCはProtocol Buffersを使用するRPC実装であり、FinagleもThriftを使用し、Rest.liはHTTP上のJSONを使用する。

この新世代のRPCフレームワークは、リモートリクエストがローカル関数呼び出しとは異なるという事実についてより明示的である。たとえばFinagleとRest.liはfutures（promises）を使用する。gRPCはstreams（ストリーム）をサポートする [46]。

バイナリエンコーディングフォーマットを持つカスタムRPCプロトコルは、JSON over RESTのような汎用的なものよりも良いパフォーマンスを達成できる。しかしRESTful APIには他の重要な利点がある：実験とデバッグに優れており、すべての主流のプログラミング言語とプラットフォームでサポートされ、ツールの膨大なエコシステムが利用可能である。

これらの理由から、RESTはパブリックAPIの主流スタイルである。RPCフレームワークの主な焦点は、同じ組織が所有するサービス間のリクエスト、通常は同じデータセンター内にある。

#### Data encoding and evolution for RPC（RPCのためのデータエンコーディングと進化）

進化可能性のためには、RPCのクライアントとサーバーが独立して変更・デプロイできることが重要である。サービスを通じたデータフローの場合は簡素化する仮定ができる：すべてのサーバーが最初に更新され、すべてのクライアントが次に更新されると仮定するのが合理的である。したがって、リクエストには後方互換性、レスポンスには前方互換性のみが必要となる。

RPCスキームの後方・前方互換性の特性は、使用するエンコーディングから継承される：

- Thrift、gRPC（Protocol Buffers）、Avro RPCは、それぞれのエンコーディングフォーマットの互換性ルールに従って進化できる。
- SOAPでは、リクエストとレスポンスはXMLスキーマで指定される。進化は可能だがいくつかの微妙な落とし穴がある [47]。
- RESTful APIは最も一般的にレスポンスにJSON（正式なスキーマなし）を使い、リクエストにはJSONまたはURIエンコード/フォームエンコードのパラメータを使う。オプションのパラメータ追加やレスポンスへの新フィールド追加は通常互換性を維持する変更と考えられる。

サービスの互換性は、RPCがしばしば組織の境界を越えた通信に使用されるという事実によりさらに困難になる。互換性は長期間、おそらく無期限に維持される必要がある。互換性を壊す変更が必要な場合、サービス提供者はしばしば複数のバージョンのサービスAPIを並行して維持することになる。

APIバージョニングについては合意がない。RESTful APIでは、URLにバージョン番号を含めるかHTTP Acceptヘッダーに含めるのが一般的なアプローチである。APIキーを使って特定のクライアントを識別するサービスでは、クライアントが要求するAPIバージョンをサーバーに格納し、別の管理インターフェースを通じて更新可能にする方法もある [49]。

### Message-Passing Dataflow（メッセージパッシングのデータフロー）

エンコードされたデータがプロセス間を流れるさまざまな方法を見てきた。ここまでRESTとRPC（1つのプロセスがネットワーク経由で別のプロセスにリクエストを送り、できるだけ早い応答を期待する）、そしてデータベース（1つのプロセスがエンコードされたデータを書き込み、別のプロセスが将来それを読み取る）について議論した。

この最後のセクションでは、asynchronous message-passing（非同期メッセージパッシング）システムを簡潔に見る。これはRPCとデータベースの中間に位置する。クライアントのリクエスト（通常message〈メッセージ〉と呼ばれる）が低レイテンシで別のプロセスに配信されるという点でRPCに似ている。メッセージが直接のネットワーク接続ではなくmessage broker（メッセージブローカー、message queue〈メッセージキュー〉またはmessage-oriented middleware〈メッセージ指向ミドルウェア〉とも呼ばれる）という仲介者を経由して送信され、一時的に格納されるという点でデータベースに似ている。

メッセージブローカーの使用は直接のRPCと比較していくつかの利点がある：

- 受信者が利用不可能または過負荷の場合にバッファとして機能でき、システムの信頼性を向上させる。
- クラッシュしたプロセスにメッセージを自動的に再配信でき、メッセージの喪失を防ぐ。
- 送信者が受信者のIPアドレスとポート番号を知る必要がない（仮想マシンが頻繁に出現・消失するクラウドデプロイメントで特に有用）。
- 1つのメッセージを複数の受信者に送信できる。
- 送信者を受信者から論理的にデカップリングする（送信者はメッセージを発行するだけで、誰が消費するかは気にしない）。

しかしRPCとの違いは、メッセージパッシング通信は通常一方向であることだ：送信者は通常メッセージへの応答を受け取ることを期待しない。プロセスが応答を送ることは可能だが、通常は別のチャネルで行われる。この通信パターンはasynchronous（非同期）である：送信者はメッセージの配信を待たず、単に送信してそのことを忘れる。

#### Message brokers（メッセージブローカー）

過去には、メッセージブローカーの領域はTIBCO、IBM WebSphere、webMethodsのような企業の商用エンタープライズソフトウェアが支配していた。より最近では、RabbitMQ、ActiveMQ、HornetQ、NATS、Apache Kafkaのようなオープンソース実装が人気を得ている。第11章でこれらをより詳しく比較する。

詳細な配信セマンティクスは実装と設定によって異なるが、一般的にメッセージブローカーは次のように使われる：1つのプロセスが名前付きのqueueまたはtopicにメッセージを送信し、ブローカーはそのqueueまたはtopicの1つまたは複数のconsumer（コンシューマー）またはsubscriber（サブスクライバー）にメッセージが配信されることを保証する。同じtopicに多くのプロデューサーと多くのコンシューマーが存在できる。

topicは一方向のデータフローのみを提供する。しかしコンシューマー自体が別のtopicにメッセージを発行することができ（第11章で見るようにチェーンできる）、または元のメッセージの送信者が消費するリプライキューにメッセージを発行することもできる（RPCに似たリクエスト/レスポンスのデータフローを可能にする）。

メッセージブローカーは通常特定のデータモデルを強制しない——メッセージは何らかのメタデータを持つ単なるバイト列であるため、任意のエンコーディングフォーマットを使用できる。エンコーディングが後方互換性と前方互換性を持つならば、パブリッシャーとコンシューマーを独立して変更し、任意の順序でデプロイする最大の柔軟性が得られる。

コンシューマーが別のtopicにメッセージを再発行する場合、データベースのコンテキストで前述した問題（Figure 4-7）を防ぐため、未知のフィールドの保持に注意が必要かもしれない。

#### Distributed actor frameworks（分散アクターフレームワーク）

actor model（アクターモデル）は単一プロセス内の並行性のためのプログラミングモデルである。スレッドを直接扱う（そしてレースコンディション、ロック、デッドロックの関連問題に対処する）のではなく、ロジックはactor（アクター）にカプセル化される。各アクターは通常1つのクライアントまたはエンティティを表し、何らかのローカルステート（他のアクターとは共有されない）を持ち、非同期メッセージの送受信によって他のアクターと通信する。メッセージの配信は保証されない：特定のエラーシナリオではメッセージが失われる。各アクターは一度に1つのメッセージのみを処理するため、スレッドについて心配する必要がなく、各アクターはフレームワークによって独立してスケジュールできる。

distributed actor framework（分散アクターフレームワーク）では、このプログラミングモデルを使ってアプリケーションを複数のノードにスケールする。送信者と受信者が同じノード上にあっても異なるノード上にあっても、同じメッセージパッシングメカニズムが使われる。異なるノード上にある場合、メッセージは透過的にバイト列にエンコードされ、ネットワーク経由で送信され、相手側でデコードされる。

ロケーション透過性はRPCよりもアクターモデルでより良く機能する。アクターモデルは単一プロセス内であってもメッセージが失われる可能性をすでに仮定しているからだ。

分散アクターフレームワークは本質的にメッセージブローカーとアクタープログラミングモデルを単一のフレームワークに統合する。しかし、アクターベースのアプリケーションのローリングアップグレードを実行したい場合、前方互換性と後方互換性について心配する必要がある。

3つの人気のある分散アクターフレームワークはメッセージエンコーディングを次のように扱う：

- **Akka**はデフォルトでJavaの組み込みシリアライゼーションを使用するが、前方・後方互換性を提供しない。Protocol Buffersのようなもので置換でき、ローリングアップグレードの能力を得られる [50]。
- **Orleans**はデフォルトでカスタムデータエンコーディングフォーマットを使用し、ローリングアップグレードデプロイメントをサポートしない。新バージョンをデプロイするには、新しいクラスタをセットアップし、トラフィックを移動し、古いクラスタをシャットダウンする必要がある [51, 52]。カスタムシリアライゼーションプラグインも使用できる。
- **Erlang OTP**ではレコードスキーマの変更を行うのが驚くほど困難である（高可用性のために設計された多くの機能を持つシステムであるにもかかわらず）。ローリングアップグレードは可能だが注意深く計画する必要がある [53]。実験的な新しいmapsデータ型（2014年のErlang R17で導入）がこれを将来的に容易にするかもしれない [54]。

## Summary（まとめ）

本章では、データ構造をネットワーク上のバイトやディスク上のバイトに変換するいくつかの方法を見た。これらのエンコーディングの詳細がその効率性にだけでなく、より重要なことに、アプリケーションのアーキテクチャとデプロイの選択肢にどのように影響するかを見た。

特に、多くのサービスはrolling upgrade（ローリングアップグレード）をサポートする必要がある。ローリングアップグレードにより、ダウンタイムなしにサービスの新バージョンをリリースでき、デプロイメントのリスクを減らす。これらの特性は進化可能性にとって非常に有益である。

ローリングアップグレード中、またはその他のさまざまな理由により、異なるノードがアプリケーションコードの異なるバージョンを実行していることを前提としなければならない。したがって、システム内を流れるすべてのデータが後方互換性（新しいコードが古いデータを読める）と前方互換性（古いコードが新しいデータを読める）を提供する方法でエンコードされることが重要である。

いくつかのデータエンコーディングフォーマットとその互換性の特性について議論した：

- **プログラミング言語固有のエンコーディング**は単一のプログラミング言語に制限され、前方・後方互換性の提供にしばしば失敗する。
- **JSON、XML、CSVのようなテキスト形式**は広く普及しており、その互換性は使い方に依存する。オプションのスキーマ言語があり、時に有用で時に邪魔になる。これらのフォーマットはデータ型についてやや曖昧であるため、数値やバイナリ文字列のようなものに注意が必要である。
- **Thrift、Protocol Buffers、Avroのようなバイナリのスキーマ駆動フォーマット**は、明確に定義された前方・後方互換性のセマンティクスを持つコンパクトで効率的なエンコーディングを可能にする。スキーマはドキュメントや静的型付け言語でのコード生成に有用でありうる。しかし、データを人間が読めるようにするにはデコードが必要であるという欠点がある。

いくつかのデータフローのモードについても議論した：

- **データベース**。データベースに書き込むプロセスがデータをエンコードし、データベースから読み取るプロセスがデコードする。
- **RPCとREST API**。クライアントがリクエストをエンコードし、サーバーがリクエストをデコードしてレスポンスをエンコードし、クライアントが最終的にレスポンスをデコードする。
- **非同期メッセージパッシング**（メッセージブローカーまたはアクターを使用）。ノードが互いにメッセージを送信し、送信者がエンコードし受信者がデコードする。

少しの注意を払えば、後方/前方互換性とローリングアップグレードはかなり達成可能であると結論できる。アプリケーションの進化が迅速であり、デプロイメントが頻繁であることを願っている。
