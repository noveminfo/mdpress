# Chapter 7: Transactions（トランザクション）

> 「いくつかの著者は、汎用のtwo-phase commit（2相コミット）をサポートすることは価値以上にコストがかかるため避けるべきだと主張してきた。彼らをどのように評価するかは、その著者がすでにtwo-phase commitを実装したことがあるかどうかに大きく依存する。」
> — ジェームズ・コーベット他、*Spanner: Google's Globally-Distributed Database*（2012年）

現実世界のデータシステムでは、多くの問題が発生しうる:

- データベースのソフトウェアまたはハードウェアが、書き込みの途中を含め、いつでも障害を起こす可能性がある
- アプリケーションが、一連の操作の途中を含め、いつでもクラッシュする可能性がある
- ネットワークの中断により、アプリケーションとデータベースの間、またはデータベースノード間の通信が突然途切れる可能性がある
- 複数のクライアントが同時にデータベースに書き込み、互いの変更を上書きする可能性がある
- クライアントが部分的にしか更新されていない、意味のないデータを読み取る可能性がある
- クライアント間の競合条件により、予期しないバグが発生する可能性がある

システムが信頼性を持つためには、これらの障害に対処し、システム全体の障害に陥らないようにしなければならない。しかし、フォールトトレラントなメカニズムの実装には多大な作業が必要である。エラーの可能性をすべて慎重に考慮し、何度もテストして、ソリューションが実際に機能することを確認する必要がある。

**Transaction（トランザクション）** は、何十年もの間、これらの問題を簡素化するためのメカニズムとして存在してきた。トランザクションとは、アプリケーションが複数の読み書きを1つの論理的な単位にまとめる方法である。概念的には、トランザクション内のすべての読み書きは単一の操作として実行される。トランザクション全体が成功（**commit**）するか、失敗（**abort**, **rollback**）するかのどちらかである。トランザクションが失敗した場合、アプリケーションは安全にリトライできる。トランザクションを使えば、部分的な障害を心配する必要がなくなるため、エラー処理がはるかに簡単になる。

トランザクションは天から降ってきたものではない。1975年にIBM System Rで導入され、最初のSQLデータベースによって普及した。今日では、ほとんどすべてのリレーショナルデータベースと一部の非リレーショナルデータベースがトランザクションをサポートしている。

この章では、特にconcurrency control（同時実行制御）の文脈でうまくいかない可能性のあるさまざまな例を検討し、データベースがそれらを防ぐために使用するアルゴリズムについて議論する。具体的には、**read committed**、**snapshot isolation（スナップショット分離）**、および**serializability（直列化可能性）** という分離レベルについて詳しく検討する。

この章は主に単一マシン上のデータベースに適用される。Chapter 8では分散システム固有の課題を、Chapter 9では分散トランザクションにおけるconsistency（一貫性）保証について議論する。

## The Slippery Concept of a Transaction（トランザクションという曖昧な概念）

ほとんどすべてのリレーショナルデータベースと一部の非リレーショナルデータベースがトランザクションをサポートしている。それらのほとんどは、1975年にIBM System Rで導入されたスタイルに従っている。40年以上の間に多くの変化があったが、基本的な設計はほとんど変わっていない。MySQL, PostgreSQL, Oracle, SQL ServerなどのSQLデータベースや、一部のNoSQLデータベースも同様のトランザクションモデルを持っている。

2000年代後半にNoSQLデータベースが普及し始めると、多くの新世代データベースはトランザクションを完全に放棄するか、以前よりもはるかに弱い保証セットに再定義して、トランザクションをスケーラビリティの障害として捉えた。大規模なシステムではトランザクションのコストが高すぎるため、高可用性や高性能を維持するには放棄せざるを得ないという主張がなされた。

一方で、トランザクションの支持者は、トランザクションはデータベースが提供できる重要な安全性保証であり、価値ある意味データを扱う真剣なデータベースにはトランザクションが必要だと主張した。

どちらの観点も誇張されている。真実はそれほど単純ではない。他の技術的設計上の選択と同様に、トランザクションには利点と制限がある。これらのトレードオフを理解するために、トランザクションが提供する保証と、それに伴うコストについて詳しく見ていこう。

### The Meaning of ACID（ACIDの意味）

トランザクションが提供する安全性保証は、**ACID** として知られている。ACIDは1983年にTheo HärderとAndreas Reuterによって造語され^i、**Atomicity（原子性）**、**Consistency（一貫性）**、**Isolation（分離性）**、**Durability（永続性）** を表す。

しかし実際には、あるデータベースの「ACIDコンプライアント」と別のデータベースの実装は大きく異なる場合がある。例えば、後述するように、**isolation（分離性）** の意味には多くの曖昧さがある。高レベルの概念は健全だが、悪魔は細部に宿る。今日、ACIDという用語はほぼマーケティング用語と化しており、正確な意味は不明確になっている。

ACIDに準拠しないシステムは、時に**BASE**（**B**asically **A**vailable, **S**oft state, **E**ventual consistency）と呼ばれる^ii。これはACIDよりもさらに曖昧で、「ACIDではない」以上の意味はほとんどない。

データベースの一般的な実装を考慮しながら、ACIDの各用語を掘り下げていこう。

#### Atomicity（原子性）

一般的に、**atomic（原子的）** とは、より小さな部分に分解できないものを指す。この言葉は、コンピューティングのさまざまな分野で、類似してはいるが完全には同一でない意味で使われる。例えば、マルチスレッドプログラミングでは、あるスレッドがatomic（原子的）な操作を実行する場合、他のスレッドがその操作を途中の状態で見ることはない。システムは操作の前の状態か後の状態のいずれかにしかなり得ない。

ACIDの文脈でのatomicity（原子性）は、concurrency（同時実行性）には関係ない。atomicity は複数の操作を同時に実行しようとしたときに何が起こるかを記述するものではない — それはisolation（分離性）でカバーされる（本章で後述）。

むしろ、ACIDのatomicity は、クライアントが複数の書き込みを行いたいが、それらの一部が処理された後に障害（プロセスクラッシュ、ネットワーク接続の中断、ディスク満杯、整合性制約違反等）が発生した場合に何が起こるかを記述する。これらの書き込みがatomic transaction（原子的トランザクション）にまとめられており、障害のためにトランザクションを完了（commit）できない場合、トランザクションは中止（abort）され、データベースはそのトランザクションでこれまでに行った書き込みを破棄またはアンドゥしなければならない。

atomicityなしでは、複数の変更を行う途中でエラーが発生した場合、どの変更が適用されどの変更が適用されていないかを知ることが困難である。アプリケーションは再試行できるが、同じ変更が二重に適用されるリスクがあり、データの重複や不正確さにつながる可能性がある。

atomicityは、中止されたトランザクションによる変更が存在しないことを保証するため、アプリケーションは安全にリトライできる。abortability（中止可能性）のほうがatomicityよりも適切な言葉かもしれないが、ここでは慣例に従ってatomicityという用語を使い続ける。

#### Consistency（一貫性）

**consistency（一貫性）** という言葉は、非常に多くの異なる意味で使われる:

- Chapter 5: replica consistency（レプリカ一貫性）、eventual consistency（結果整合性）— レプリケーションラグの問題
- Chapter 9: consistent hashing（一貫性ハッシュ）— パーティショニングのリバランシング方法
- CAP定理: consistency（一貫性）= linearizability（線形化可能性）
- ACID: consistency（一貫性）= データベースが「良い状態」にあること

ACIDにおけるconsistency（一貫性）とは、データに関する特定の記述（**invariants（不変条件）**）が常に真でなければならないという考え方である。例えば、会計システムでは、すべての口座の貸借が常にバランスしなければならない。

しかし、consistency のこの概念は、データベースが提供する特性ではなく、**アプリケーションの特性**である。アプリケーションが一貫した不変条件を維持するような方法でデータを書き込むことはアプリケーションの責任である。これはデータベースが保証できるものではない — 不変条件を破るデータを書き込んだ場合、データベースはそれを阻止できない（特定の種類の不変条件はデータベースがチェックできる。例えば、外部キー制約やuniqueness制約。しかし一般的には、一貫性を定義し維持するのはアプリケーションの仕事である）。

Atomicity、isolation、durabilityはデータベースの特性であるが、consistency（ACID的な意味での）はアプリケーションの特性である。したがって「C」は実際にはACIDに属さない^iii。

#### Isolation（分離性）

ほとんどのデータベースは、複数のクライアントが同時にアクセスする。同じデータベースレコードにアクセスする場合、concurrency（同時実行性）の問題（**race condition（競合条件）**）が発生する可能性がある。

**Isolation（分離性）** とは、ACIDの文脈では、同時に実行されるトランザクションが互いに分離されており、互いに干渉しないことを意味する。従来の教科書ではisolation を **serializability（直列化可能性）** として形式化している。これは、各トランザクションがデータベース全体で唯一実行されているかのように振る舞えることを意味する。データベースは、実際に複数のトランザクションが同時に実行されていたとしても、トランザクションがコミットされると、その結果が直列的（1つずつ順番に）に実行されたのと同じであることを保証する。

しかし実際には、serializableなisolation はパフォーマンスのペナルティがあるため、ほとんど使われない。代わりに多くのデータベースは**弱い（weak）** 分離レベルを使用しており、それらはある種のconcurrency（同時実行）の問題からは保護するが、すべてではない。これらの弱い分離レベルは実際に多くの微妙なバグの原因となっている。これについて本章で詳しく議論する。

#### Durability（永続性）

**Durability（永続性）** は、トランザクションが正常にコミットされたら、書き込まれたデータは、ハードウェア障害やデータベースクラッシュが発生しても失われないという約束である。

単一ノードデータベースでは、durability は通常、データがハードディスクやSSDなどの不揮発性ストレージに書き込まれたことを意味する。通常、write-ahead log（先行書き込みログ）やそれに類するものも含まれ、データ構造がディスク上で破損した場合のリカバリを可能にする（Chapter 3参照）。レプリケーションを持つデータベースでは、durability はデータが一定数のノードに正常にコピーされたことを意味する場合もある。

**完璧なdurability は存在しない**。すべてのハードディスクとバックアップが同時に破壊される可能性がある（理論的にはありえないが、実際には起こりうる^iv）。実際には、完全に保証する方法は存在しない — 絶対的な保証を提供する技術は存在しない。リスクを軽減するアプローチ（ディスクへの書き込み、リモートマシンへのレプリケーション、バックアップ等）の組み合わせがあるのみである。

### Single-Object and Multi-Object Operations（単一オブジェクトと複数オブジェクトの操作）

ACIDのatomicity とisolation は、クライアントが1つのトランザクション内で複数の書き込みを行う場合に、データベースがどのように振る舞うべきかを記述する:

- **Atomicity**: 途中でエラーが発生した場合、これまでの書き込みをすべてアンドゥ
- **Isolation**: 同時実行のトランザクションは互いの部分的な書き込みを見ない

これらの定義は、**複数のオブジェクト（行、ドキュメント、レコード）** に対する変更を1つのatomic unit（原子的な単位）としてまとめる必要がある場面を前提としている。

**Multi-object transactions（複数オブジェクトトランザクション）** は、一般にどのトランザクションの中でどの読み書きが属するかを決定する方法が必要である。リレーショナルデータベースでは通常、`BEGIN TRANSACTION` と `COMMIT` の間のすべてがトランザクションの一部とみなされる。

多くの非リレーショナルデータベースでは、このような操作をグループ化する方法がない。たとえ multi-object API が存在しても（例えば key-value ストアに1回のオペレーションで複数のキーを設定できるとしても）、それが必ずしもトランザクションを意味するわけではない。コマンドの途中でネットワーク接続が切れた場合に、一部のキーだけが更新され、他のキーは更新されないまま残る可能性がある。

#### The need for multi-object transactions（複数オブジェクトトランザクションの必要性）

多くの分散データストアはmulti-object transactionsのサポートを放棄した。クロスパーティションのトランザクションは実装が困難で、高いavailability（可用性）やperformance（性能）が必要な場面では邪魔になることがある。しかし、multi-object transactionsが本当に必要とされるケースはいくつか存在する:

- リレーショナルデータモデルでは、あるテーブルの行が他のテーブルの外部キー参照を持つことが多い。multi-object transactions を使えば、参照が常に有効であることを保証できる。
- ドキュメントデータモデルでは、一緒に更新されるべきフィールドが1つのドキュメント内にある場合、単一のドキュメントは単一オブジェクトとして扱われるため問題ない。しかし、joinの不足を非正規化で補う場合、複数のドキュメントを同時に更新する必要がある。
- セカンダリインデックスを持つデータベースでは、値を変更するたびにインデックスも更新する必要がある。isolation がなければ、インデックスと実データの間に不整合が発生する可能性がある。

これらのアプリケーションはトランザクションなしでも実装できるが、atomicity がなければエラー処理がはるかに複雑になり、isolation がなければ同時実行に起因するバグが発生しやすくなる。

#### Handling errors and aborts（エラー処理と中止）

トランザクションの主要な機能はエラー時に中止して安全にリトライできることだが、すべてのシステムがこの哲学に従っているわけではない。例えば、leaderless replication（リーダーレスレプリケーション）の多くの実装は「ベストエフォート」ベースで動作する。

エラーは発生するものであり、リトライはエラーを処理する簡単で効果的なメカニズムだが、完璧ではない:

- トランザクションが実際には成功したが、成功をクライアントに通知するネットワークが失敗した場合、リトライすると同じ操作が二重に実行される（アプリケーションレベルのde-deduplicationメカニズムが必要）
- エラーが過負荷に起因する場合、リトライは問題を悪化させる。リトライ回数の制限、exponential backoff（指数的バックオフ）の使用、過負荷関連のエラーと他のエラーの分離処理が必要
- 一時的なエラー（デッドロック、isolation違反、一時的なネットワーク中断、フェイルオーバー）のみリトライする価値がある。永続的なエラー（制約違反など）のリトライは無意味
- トランザクションがデータベース外に副作用を持つ場合（メール送信など）、トランザクションが中止されてもその副作用は発生しうる。two-phase commit（2相コミット）が助けになる場合がある（Chapter 9）
- クライアント自体がリトライ中に失敗した場合、DBへの書き込みを試みていたデータが失われる

## Weak Isolation Levels（弱い分離レベル）

2つのトランザクションが異なるデータに触れる場合、安全に並行実行できる。concurrency の問題が発生するのは、あるトランザクションが別のトランザクションが同時に修正しているデータを読み取る場合、または2つのトランザクションが同じデータを同時に修正しようとする場合のみである。

concurrency のバグはテストで発見するのが難しい。なぜなら、そのようなバグはタイミングが不運な場合にのみ発生するからである。このようなタイミングは通常再現が困難で、特にテストでは問題なく動作するが、本番環境で並行にアクセスされると問題が発生する場合がある。

このため、データベースは長い間 **transaction isolation（トランザクション分離）** を提供することで、アプリケーション開発者にconcurrency の問題を隠そうとしてきた。理論上は、isolation はconcurrency がないかのように振る舞うことを意味する。**serializable（直列化可能）** なisolation はトランザクションが直列的に実行されたのと同じ結果を保証する。

しかし、serializable isolation にはperformance のコストがあり、多くのデータベースはそのコストを支払いたくない。そのため、より弱い分離レベルが一般的に使用されている。これらの弱い分離レベルは、いくつかのconcurrency 問題からは保護するが、すべてからは保護しない。

弱い分離レベルに起因する並行性のバグは、単に理論的な問題ではなく、金銭的な損失を含む大きな実害を引き起こしてきた^v。このセクションでは、実際に使用されている弱い分離レベルとその詳細を見ていく。

### Read Committed（読み取りコミット）

最も基本的なトランザクション分離レベルは **read committed（読み取りコミット）** である。これは2つの保証を提供する:

1. データベースから読み取るとき、コミットされたデータのみが見える（**no dirty reads（ダーティリードなし）**）
2. データベースに書き込むとき、コミットされたデータのみを上書きする（**no dirty writes（ダーティライトなし）**）

#### No dirty reads（ダーティリードなし）

トランザクションが別のトランザクションのまだコミットされていないデータを読み取れてしまう場合を **dirty read（ダーティリード）** という。read committed 分離レベルで動作するトランザクションは、ダーティリードを防止しなければならない。他のトランザクションの書き込みは、コミットされた後にのみ見える。

ダーティリードを防止すべき理由:

- トランザクションが複数のオブジェクトを更新する場合、ダーティリードは部分的に更新されたデータを他のトランザクションに見せてしまう。ユーザーにとって混乱の原因となる。
- トランザクションが中止された場合、そのトランザクションが行ったすべての書き込みはロールバックされる。ダーティリードを許可すると、後にロールバックされるデータを他のトランザクションが見てしまう。

#### No dirty writes（ダーティライトなし）

2つのトランザクションが同じオブジェクトを同時に更新しようとする場合はどうなるか？ どちらのトランザクションの書き込みが先に行われるかはわからないが、通常、後の書き込みが先の書き込みを上書きすると想定する。

しかし、先の書き込みがまだコミットされていないトランザクションの一部であり、後の書き込みがコミット前の値を上書きする場合、それは **dirty write（ダーティライト）** と呼ばれる。read committed 分離レベルでは、ダーティライトを防止しなければならない。通常は、先のトランザクションがコミットまたは中止されるまで後の書き込みを遅延させることで実現する。

ダーティライトの防止により、いくつかのconcurrency の問題を回避できる:

- 異なるトランザクションからの書き込みが混ざるのを防ぐ。例えば、Aliceのトランザクションのlistingsテーブル更新とBobのトランザクションのinvoicesテーブル更新が混在すると、車がBobに登録されるが請求書がAliceに送られるといった問題が発生しうる。
- ただし、ダーティライトの防止だけではcounterの競合条件（2つのトランザクションが同時にread-modify-writeサイクルを実行する場合）を防げない。これは後の「Lost Updates」セクションで議論する。

#### Implementing read committed（read committedの実装）

Read committed は非常に一般的な分離レベルであり、Oracle 11g、PostgreSQL、SQL Server 2012、MemSQL等のデフォルトである。

最も一般的な実装方法:

- **ダーティライト防止**: 行レベルのロックを使用。トランザクションがオブジェクトを変更したい場合、まずそのオブジェクトのロックを取得し、コミットまたは中止されるまで保持する。
- **ダーティリード防止**: ロックを使う方法もあるが、読み取りのロックは実際にはうまく機能しない。1つの長時間実行トランザクションが複数のオブジェクトのロックを保持すると、他のトランザクションの読み取りが長時間ブロックされる。代わりに、ほとんどのデータベースは以下のアプローチを使用する: オブジェクトが書き込みロック下にある間、そのオブジェクトの古いコミット済みの値を記憶し、他のトランザクションの読み取りにはその古い値を返す。新しい値は書き込みトランザクションがコミットされた後にのみ他のトランザクションに見えるようになる。

### Snapshot Isolation and Repeatable Read（スナップショット分離とリピータブルリード）

read committed 分離レベルを使用していても、concurrency のバグは依然として発生しうる。

例えば、Aliceが2つの銀行口座を持ち、それぞれに500ドル入っていたとする。あるトランザクションが口座1から口座2に100ドルを送金する。ちょうど同じ頃にAliceが残高を確認する際、転送処理の途中で口座1を見ると（転送前の）500ドル、口座2を見ると（転送が口座2に適用される前の）400ドルが表示される。Aliceには合計900ドルしかないように見え、100ドルが消えたように見える。

この異常は **nonrepeatable read（反復不能読み取り）** または **read skew（読み取りスキュー）** と呼ばれる。read committed で一時的にこの不整合を見ることが許容される場合もあるが、問題となる場面がある:

- **バックアップ**: バックアップは数時間かかることがあり、その間にデータベースへの書き込みは続く。バックアップの一部が古いバージョン、一部が新しいバージョンのデータを含む場合、そのバックアップには永続的な不整合が含まれる。
- **分析クエリとインテグリティチェック**: 大量のデータをスキャンするクエリは、異なる時点のデータの一部を見ると、意味のない結果を返す可能性がある。

**Snapshot isolation（スナップショット分離）** は、この問題に対する最も一般的な解決策である。各トランザクションは、データベースの**consistent snapshot（一貫したスナップショット）** から読み取る。つまり、トランザクションはそのトランザクションの開始時にデータベースにコミットされていたすべてのデータを見る。データがその後他のトランザクションによって変更されても、各トランザクションは特定の時点の古いデータのみを見る。

Snapshot isolation は特にバックアップや分析などの長時間実行の読み取り専用クエリに有用である。snapshot上でクエリが実行されている間にデータが変更されても、クエリの結果は変わらない。

#### Implementing snapshot isolation（スナップショット分離の実装）

read committed の実装と同様に、snapshot isolation の実装は通常、ダーティライトを防止するために書き込みロックを使用する。しかし、読み取りにはロックを必要としない。パフォーマンスの鍵となるのは、**readers never block writers, and writers never block readers（読み取りは書き込みをブロックせず、書き込みは読み取りをブロックしない）** という原則である。

snapshot isolation を実装するために、データベースは**MVCC（Multi-Version Concurrency Control: 多版同時実行制御）** と呼ばれる技術を使用する。データベースはオブジェクトの異なるコミット済みバージョンを保持する。なぜなら、さまざまな進行中のトランザクションが異なる時点のデータベースの状態を見る必要があるためである。

read committed isolationもMVCCを使用しているが、より単純な形でのみ必要とする。read committed はクエリごとに別のスナップショットを使用するが、snapshot isolation はトランザクション全体に対して同じスナップショットを使用する。

MVCCベースのsnapshot isolationの典型的な実装方法: テーブルの各行に、`created_by` フィールド（その行を挿入したトランザクションのID）と `deleted_by` フィールド（その行を削除としてマークしたトランザクションのID）がある。行が削除される場合、即座にテーブルから削除されるわけではなく、`deleted_by` フィールドにトランザクションIDが設定される。ある時点で、削除された行がどのトランザクションからも見えなくなった時に、ガベージコレクションプロセスがその行を安全に削除する。更新は内部的には削除と作成に変換される。

#### Visibility rules for observing a consistent snapshot（一貫したスナップショットの可視性ルール）

トランザクションがデータベースから読み取る際、トランザクションIDを使用してどのオブジェクトが可視でどのオブジェクトが不可視かを決定する。これらの可視性ルールを慎重に定義することで、データベースの一貫したスナップショットをアプリケーションに提示する:

1. トランザクション開始時に、進行中の（まだコミットも中止もされていない）他のすべてのトランザクションのリストを作成する。これらのトランザクションが行った書き込みは、たとえ後でコミットされても無視される。
2. 中止されたトランザクションが行った書き込みはすべて無視される。
3. 現在のトランザクションよりも大きいトランザクションID（つまり、現在のトランザクションの開始後に開始されたトランザクション）が行った書き込みは、コミット済みかどうかにかかわらず無視される。
4. それ以外のすべての書き込みはアプリケーションのクエリに対して可視である。

つまり、以下の両方の条件が満たされる場合にオブジェクトが可視となる:

- そのオブジェクトを作成したトランザクションが、読み取りトランザクションの開始前にすでにコミットされている
- そのオブジェクトが削除済みとマークされていないか、削除としてマークしたトランザクションが読み取りトランザクションの開始時にまだコミットされていない

#### Indexes and snapshot isolation（インデックスとスナップショット分離）

マルチバージョンデータベースでインデックスはどのように機能するか？1つのオプションは、インデックスがオブジェクトのすべてのバージョンを指すようにし、インデックスクエリで現在のトランザクションに見えないオブジェクトバージョンをフィルタリングすることである。ガベージコレクションで古いオブジェクトバージョンが削除されると、対応するインデックスエントリも削除される。

もう1つのアプローチは、CouchDBやDatomic、LMDBで使用されている、append-only/copy-on-writeの変種であるB-treeを使用することである。ページが更新されるたびに新しいコピーが作成され、親ページも新しいバージョンの子を指すように更新される。影響を受けないページはコピーされない。

#### Repeatable read and naming confusion（リピータブルリードと命名の混乱）

Snapshot isolation は有用な分離レベルであり、特に読み取り専用トランザクションに便利である。しかし、多くのデータベースがこれを異なる名前で実装している。Oracleでは**serializable**、PostgreSQLとMySQLでは**repeatable read**と呼ばれている。

この名前の混乱の理由は、SQL標準にsnapshot isolationの概念がないことにある。SQL標準は、SQL標準が定義される前のSystem R (1975年) のisolation レベルに基づいている。SQL-92標準で定義されている分離レベルは曖昧で、不正確で、標準的ではない — いくつかのデータベースが「repeatable read」を実装しているが、それらが提供する保証は実際には大きく異なる^vi。

結果として、SQL標準やデータベースの実装がrepeatable readと言った場合に何を意味するのかについて正式な定義はない。

## Preventing Lost Updates（更新ロストの防止）

ここまでに議論した read committed と snapshot isolation 分離レベルは、主に、同時に実行中の書き込みがある場合に **読み取りトランザクション** が何を見るかについてであった。しかし、2つのトランザクションが同時にデータに書き込む場合の競合については、ダーティライトの防止（先述）のみ議論した。

同時書き込みトランザクション間で発生しうるその他の興味深い競合がある。最もよく知られているのが **lost update（更新ロスト）** 問題である。

**lost update** 問題は、アプリケーションがデータベースから値を読み取り、値を変更し、変更した値を書き戻す場合（**read-modify-write cycle（読み取り-修正-書き込みサイクル）**）に発生しうる。2つのトランザクションがこれを並行して行った場合、一方の変更が失われる可能性がある。なぜなら、2番目の書き込みが最初の変更を含んでいないためである。

このパターンはさまざまなシナリオで発生する:

- カウンターのインクリメントやアカウント残高の更新（現在の値を読み取り、新しい値を計算し、更新された値を書き戻す）
- 複雑な値へのローカルな変更（JSONドキュメントのリスト要素の追加など — 現在のドキュメントを読み取り、ローカルで変更し、修正されたドキュメントを書き戻す）
- 2人のユーザーが同じwikiページを編集し、各自がページ全体の内容をサーバーに送信して、現在のデータベースの内容を置き換える

### Solutions for lost updates（更新ロストへの解決策）

#### Atomic write operations（アトミック書き込み操作）

多くのデータベースはatomic update（原子的更新）操作を提供しており、アプリケーションコードでread-modify-writeサイクルを実行する必要をなくす。例えば、以下のSQLはほとんどのリレーショナルデータベースで安全に使える:

```sql
UPDATE counters SET value = value + 1 WHERE key = 'foo';
```

同様に、MongoDBなどのドキュメントデータベースは、JSONドキュメントの一部に対するlocal modificationのためのアトミック操作を提供する。Redisは優先度キューなどのデータ構造を変更するためのアトミック操作を提供する。

アトミック操作は通常、オブジェクトの読み取り時にexclusive lock（排他ロック）を取得し、更新が適用されるまでそのロックを保持することで実装される。このテクニックは **cursor stability（カーソル安定性）** と呼ばれることがある。もう1つのオプションは、すべてのアトミック操作を単一のスレッドで強制的に実行することである。

残念ながら、ORM（Object-Relational Mapping）フレームワークはread-modify-writeサイクルを実行するコードを簡単に生成してしまい、データベース提供のアトミック操作を使用しない不安全なコードになりがちである。

#### Explicit locking（明示的ロック）

データベースの組み込みアトミック操作が必要な機能を提供しない場合、もう1つのオプションはアプリケーションが更新されるオブジェクトに明示的にロックを取得することである。その後、アプリケーションはread-modify-writeサイクルを実行し、他のトランザクションが同じオブジェクトを同時に読み取ろうとしても、最初のread-modify-writeサイクルが完了するまで強制的に待たされる。

例えば、マルチプレイヤーゲームで、複数のプレイヤーが同じ駒を動かせる場合:

```sql
BEGIN TRANSACTION;

SELECT * FROM figures
  WHERE name = 'robot' AND game_id = 222
  FOR UPDATE;

-- 移動が有効かどうかチェックし、新しい位置を計算
UPDATE figures SET position = 'c4' WHERE id = 1234;

COMMIT;
```

`FOR UPDATE` 句は、返されるすべての行にロックを取得することをデータベースに指示する。

#### Automatically detecting lost updates（更新ロストの自動検出）

アトミック操作や明示的ロックの代わりに、トランザクションマネージャがlost updateを検出し、そのトランザクションを中止して強制的にread-modify-writeサイクルをリトライさせるという方法もある。

この方法の利点は、データベースがこのチェックをsnapshot isolationと組み合わせて効率的に実行できることである。PostgreSQL のrepeatable read、Oracle のserializable、SQL Server のsnapshot isolationレベルは自動的にlost updateを検出し、それを引き起こしたトランザクションを中止する。

しかし、MySQL/InnoDBのrepeatable readはlost updateを検出しない^vii。一部の著者は、MySQLはsnapshot isolationを提供していないと主張している。

#### Compare-and-set（比較と設定）

トランザクションを提供しないデータベースでは、**compare-and-set** 操作が見られることがある。最後に読んだ時点から値が変更されていない場合にのみ更新を許可する:

```sql
UPDATE wiki_pages SET content = '新しい内容'
  WHERE id = 1234 AND content = '以前の内容';
```

ただし、データベースがWHERE条件を古いスナップショットから読み取る場合、このステートメントはlost updateを防止できない可能性がある。compare-and-setを使用する前に、データベースの実装が安全かどうかを確認する必要がある。

#### Conflict resolution and replication（競合解決とレプリケーション）

ロックやcompare-and-setは、データの最新コピーが1つだけ存在することを前提としている。しかし、multi-leader（マルチリーダー）やleaderless（リーダーレス）レプリケーションを持つデータベースでは、データの複数のコピーに対する書き込みが同時に発生しうるため、ロックやcompare-and-setは適用できない。

レプリケーション環境でのlost updateの一般的なアプローチは、同一のデータの複数の競合するバージョンを作成することを許可し、事後的にアプリケーションコードまたは特別なデータ構造を使用して競合を解決・マージすることである。

アトミック操作は、特にcommutative（交換可能）な操作であれば、レプリケーション環境でもうまく機能する。例えばカウンターのインクリメントやセットへの要素追加は、レプリカの順序に関係なく正しい結果を得られる。Riakのデータ型はこの考え方で動作する。

一方、**last write wins (LWW)** という競合解決方法はlost updateを引き起こしやすいが、多くのレプリケーションデータベースのデフォルトである。

## Write Skew and Phantoms（書き込みスキューとファントム）

ここまでで dirty write と lost update という2種類の競合条件を見てきた。いずれも、2つのトランザクションが同じオブジェクトに同時に書き込む場合に発生する。

しかし、競合条件はそれだけではない。もっと微妙な例を考えてみよう。

### Characterizing write skew（書き込みスキューの特徴づけ）

例として、病院の当直システムを考える。少なくとも1人の医師が常に当直であることが求められる。2人の医師、AliceとBobがたまたま同時に当直であり、2人とも体調を崩したので休みたいと考えた。2人が「当直中の医師の数を確認 → 2人以上いるので安全に休める」と判断して同時に休みを申請すると、結果として誰も当直でない状態になってしまう:

```sql
-- Alice と Bob がそれぞれ実行:
BEGIN TRANSACTION;
SELECT COUNT(*) FROM doctors WHERE on_call = true AND shift_id = 1234;
-- 結果: 2人 → 安全に休めると判断
UPDATE doctors SET on_call = false WHERE name = 'Alice' AND shift_id = 1234;
COMMIT;
```

この異常は **write skew（書き込みスキュー）** と呼ばれる。dirty write でも lost update でもない — 2つのトランザクションが異なるオブジェクトを更新しているため。しかし、明らかに競合条件である。2人のうち一方だけが休みを申請すれば問題ないが、2人同時に実行されると、要件が破られる。

write skew は lost update を一般化したものと考えることができる。2つのトランザクションが同じオブジェクトを読み取り、次にそのオブジェクトの一部を更新する（各トランザクションが異なるオブジェクトを更新する）場合、write skew が発生しうる。特殊なケースとして、異なるトランザクションが同じオブジェクトを更新する場合は dirty write か lost update になる。

write skew への対処方法は限られている:

- アトミック単一オブジェクト操作は使えない（複数のオブジェクトが関係するため）
- 一部のデータベースの自動lost update検出は助けにならない（PostgreSQLのrepeatable read, MySQL/InnoDBのrepeatable read, Oracleのserializable はいずれもwrite skewを自動検出しない）
- 制約として表現できる場合もあるが（外部キーやuniqueness制約）、multi-object の制約はほとんどのデータベースでサポートされていない
- Serializable isolation レベルを使用するのが最善策
- 次善策として、トランザクションが依存する行に `SELECT ... FOR UPDATE` で明示的にロックを取得する

### More examples of write skew（書き込みスキューのさらなる例）

- **ミーティングルームの予約システム**: 同じ時間帯に同じ部屋が二重予約される
- **マルチプレイヤーゲーム**: 2人のプレイヤーが同じポジションに同時に駒を移動
- **ユーザー名の一意性**: 2人のユーザーが同時に同じユーザー名を取得
- **二重支出の防止**: 同じアカウントから2つの同時トランザクションが支出し、残高がマイナスに

### Phantoms causing write skew（ファントムによる書き込みスキュー）

これらの例はすべて同じパターンに従う:

1. SELECTクエリが何らかの検索条件に一致する行を検索
2. 最初のクエリの結果に基づいて、アプリケーションコードが何をすべきかを決定
3. アプリケーションが（INSERT, UPDATE, DELETEにより）書き込みを行い、データベースにコミットする

ステップ3の書き込みがステップ1の検索条件の結果を変更してしまう — もう一度ステップ1を実行すると、異なる結果が得られるだろう。

ある トランザクションの書き込みが別のトランザクションの検索クエリの結果を変更するこの効果を **phantom（ファントム）** と呼ぶ。Snapshot isolation は単純な phantom reads を防止する（読み取り専用のクエリでは）が、write skew の文脈でのphantomは特に扱いにくい。

### Materializing conflicts（衝突の実体化）

ロックを取得する対象の行が存在しない場合は困難である（INSERT で行が追加されて初めて存在する場合）。対策として、ロック対象となる人工的な行を事前に作成する方法がある。

例えば、ミーティングルーム予約の場合、部屋とタイムスロットの組み合わせごとに行を作成したテーブルを用意しておく。これにより、SELECT ... FOR UPDATE でその行にロックを取得し、予約の競合を防止できる。

このアプローチは **materializing conflicts（衝突の実体化）** と呼ばれる。concurrency control のメカニズムのリークをアプリケーションのデータモデルに持ち込むため、**最終手段として**のみ使用すべきである。ほとんどの場合、serializable isolation levelを使用するほうが望ましい。

## Serializability（直列化可能性）

この章では、各分離レベルと、各レベルが防止する競合条件、そして防止しないものについて見てきた。複雑で多くの微妙な判断が必要である。

- Read committed や snapshot isolation レベルでどの競合条件から保護されているかを直感的に理解するのは困難
- 特にアプリケーションコードでwrite skew のようなバグを見つけるのは非常に難しい

**Serializable（直列化可能）** isolation は通常、最も強い分離レベルとみなされる。同時に実行される複数のトランザクションがあったとしても、結果は各トランザクションが1つずつ直列に実行されたかのようになることを保証する。したがって、データベースはすべての可能な競合条件を防止する。

現在、serializable isolationを提供するほとんどのデータベースは、3つの技術のいずれかを使用している:

1. 文字通りの直列実行（Actual serial execution）
2. Two-phase locking（2相ロック、2PL）
3. Serializable snapshot isolation（SSI）

### Actual Serial Execution（実際の直列実行）

concurrency 問題を回避する最も単純な方法は、concurrency を完全に排除すること — すなわち、1つのスレッドで1つずつトランザクションを実行することである。これにより、serializability が定義上達成される。

このアイデアは明白だが、データベース設計者は約30年にわたってこのアプローチを実用的ではないとして却下していた。しかし、2007年頃に、このアプローチを実用化した2つのシステム（VoltDB/H-StoreとRedis）が登場した。単一スレッドの実行が実用的になったのは、2つの進展によるものである:

- RAM が安価になり、多くのユースケースでアクティブなデータセット全体をメモリに保持できるようになった。メモリ内のトランザクションは、ディスクI/Oを待つ必要がないため非常に高速に実行できる。
- データベース設計者は、OLTP トランザクションは通常短く、少数の読み書きしか行わないことを認識した。対照的に、長時間実行の分析クエリは読み取り専用であるため、snapshot isolation 上で直列実行の外で実行できる。

#### Encapsulating transactions in stored procedures（ストアドプロシージャでのトランザクションのカプセル化）

データベースの初期の頃、トランザクションはユーザーアクティビティのフロー全体を包含することが想定されていた。例えば、フライトの予約では、座席検索から支払いまでの全プロセスが1つのトランザクションであった。しかし、ユーザーの応答を待っている間データベースのトランザクションをオープンにしておくと、多くの同時トランザクションをサポートする必要があり、パフォーマンスがひどいことになる。

このため、webアプリケーションではユーザーの各アクション（HTTPリクエスト）が個別のトランザクションになるように設計されている。しかし、単一スレッドの直列実行では、アプリケーションとデータベース間のインタラクティブなマルチステートメントトランザクションは使えない — ネットワーク通信がボトルネックになるためである。

代わりに、アプリケーションはトランザクションコード全体を**stored procedure（ストアドプロシージャ）** としてデータベースに送信する。データベースは必要なすべてのデータが事前にメモリ内にある限り、ストアドプロシージャを非常に高速に実行できる。

ストアドプロシージャには歴史的に悪い評判がある:

- 各データベースベンダーが独自の言語を持っており（PL/SQL(Oracle), T-SQL(SQL Server), PL/pgSQL(PostgreSQL)）、一般的なプログラミング言語に比べて使いにくい
- データベース内で実行されるコードの管理（デプロイ、テスト、デバッグ、バージョン管理、メトリクス）が困難
- ストアドプロシージャのパフォーマンスの悪さがデータベースの他の部分にも影響する

しかし、最近のシステムではこれらの問題を克服している。VoltDB はJava またはGroovy を使用し、Datomic はJava またはClojure を使用し、Redis はLua を使用する。

#### Partitioning（パーティショニング）

直列的にトランザクションを実行するのは、すべてのトランザクションが小さく高速である場合には実行可能だが、単一CPU コアのスループットに制限される。読み書きの多いワークロードでは、単一スレッドのトランザクション処理が深刻なボトルネックになりうる。

書き込みスループットを複数のCPUコアにスケールするために、データを**partition（パーティション）** し、各パーティションに専用のCPUコアでトランザクション処理を行わせることが可能である。各パーティション内ではトランザクションは直列的に実行され、異なるパーティション間では並列に実行できる。

ただし、複数のパーティションにまたがるトランザクションの場合、データベースはそのトランザクションに関わるすべてのパーティション間で調整を行う必要がある。ストアドプロシージャは関係するすべてのパーティションをロックしながら直列的に実行される。クロスパーティションの調整にはオーバーヘッドが大きいため、クロスパーティションのトランザクションが全トランザクションの小さな割合でない限り、このアプローチのメリットは限定的である。

#### Summary of serial execution（直列実行のまとめ）

直列的にトランザクションを実行することは、実現可能なserializableの選択肢であるが、以下の制約の中での使用に限られる:

- すべてのトランザクションが小さく高速であること。遅いトランザクションが1つあるだけで、他のすべての処理が停止する
- アクティブなデータセットがメモリに収まること。滅多にアクセスされないデータはディスクに置けるが、単一スレッドのトランザクションでディスクアクセスが必要になると非常に遅くなる
- 書き込みスループットが単一CPUコアで処理可能か、クロスパーティショントランザクションが少数であること
- クロスパーティショントランザクションは可能だが、使用可能な範囲に厳しい制限がある

### Two-Phase Locking (2PL)（2相ロック）

約30年間、データベースでserializableを実現する唯一広く使われた方法は **two-phase locking（2PL: 2相ロック）** であった。

2PL は read committed（ダーティライトを防止するためにロックを使用する）に似ているが、ロックの要件がはるかに強い。2PL の重要なルールは以下の通り:

- 書き込みトランザクションは読み取りトランザクションもブロックし、その逆も成り立つ（read committed では writers は readers をブロックしない）

このルールにより、write skew やその他のrace condition が防止され、isolation がserializableになる。

#### Implementation of two-phase locking（2PLの実装）

2PL はMySQL (InnoDB) とSQL Server のserializable分離レベル、DB2のrepeatable read 分離レベルで使用されている。

ロックは **shared mode（共有モード）** または **exclusive mode（排他モード）** のいずれかで取得できる:

- オブジェクトを**読み取る**場合、まずshared mode でロックを取得する。複数のトランザクションが同時にshared lock を保持できる。ただし、他のトランザクションがexclusive lockを保持している場合、そのトランザクションの完了を待つ必要がある。
- オブジェクトに**書き込む**場合、まずexclusive mode でロックを取得する。他のトランザクションがそのオブジェクトにいかなるロックも保持していない場合にのみ取得できる。
- オブジェクトを読み取った後に書き込む場合、shared lock をexclusive lock にアップグレードする。
- ロックを取得したトランザクションは、**トランザクションの終了（コミットまたは中止）まで**そのロックを保持し続ける。

ロックが多く使用されるため、同じオブジェクトにアクセスしようとするトランザクション間で**deadlock（デッドロック）** が容易に発生する。データベースは自動的にデッドロックを検出し、いずれかのトランザクションを中止する。

#### Performance of two-phase locking（2PLのパフォーマンス）

2PL のパフォーマンスが弱い分離レベルに比べて大幅に悪い理由は、ロックの取得と解放のオーバーヘッドそのものよりも、**concurrency の低下**にある。設計上、あるトランザクションが別のトランザクションの完了を待たなければならない場合、トランザクションの遅延時間に制限がない。

2PL 下では、デッドロックがはるかに頻繁に発生する可能性がある。これはパフォーマンスのさらなる問題につながる。トランザクションが中止されデッドロックのため再試行されると、最初からやり直す必要があるためである。

#### Predicate locks（述語ロック）

ファントムについて考えると — 1つのトランザクションが別のトランザクションの検索クエリの結果を変更する — 2PL を使ったserializable isolation ではファントムも防止する必要がある。

概念的には、**predicate lock（述語ロック）** が必要である。これは特定のオブジェクトに属するのではなく、何らかの検索条件（例えば`WHERE room_id = 123 AND start_time < '2025-01-01' AND end_time > '2024-12-31'`）に一致するすべてのオブジェクトに属するロックである。

predicate lock はまだ存在しないオブジェクト（将来挿入される可能性のある行）にも適用される。2PL にpredicate lock を含む場合、あらゆる形式のwrite skew やその他のrace condition を防止でき、isolation がserializable になる。

#### Index-range locks（インデックス範囲ロック）

残念ながら、predicate locks はパフォーマンスが悪い傾向がある。アクティブなトランザクションが保持するロックの数が多い場合、条件の一致チェックに時間がかかる。このため、ほとんどの2PLを使用するデータベースは、実際には **index-range locking（インデックス範囲ロック）** を実装している。これはpredicate lock の簡略化された近似である。

安全な簡略化は、述語をより広い条件に拡大すること。例えば、12:00から13:00の部屋123への予約のpredicate lock の代わりに、12:00から13:00のすべての部屋、または部屋123のすべての時間帯にロックを取得することができる。

この近似はprecise（正確）ではないが — ロックされるべきでないトランザクションがロック待ちになる可能性がある — 、overheadが小さいため良い妥協点である。

適切なインデックスがない場合、データベースはテーブル全体にshared lockを設定するフォールバックが可能である。これはパフォーマンスには悪いが、安全な代替手段である。

### Serializable Snapshot Isolation (SSI)（直列化可能スナップショット分離）

この章では、serializable isolationの暗い絵を描いてきた — 直列実行は制約が多く、2PL はパフォーマンスが悪い。弱い分離レベルは高性能だが、さまざまな race condition に対して脆弱である。

**Serializable snapshot isolation (SSI)** は、まだ非常に新しいアルゴリズムであるが、非常に有望な方法である。完全な serializability を提供しつつ、snapshot isolation に比べてわずかなperformance のペナルティしかない。2008 年にMichael Cahill の博士論文で初めて記述され^viii、PostgreSQL 9.1以降では serializable isolation level として使用されている。FoundationDB もSSI に基づくアルゴリズムを使用している。

#### Pessimistic versus optimistic concurrency control（悲観的 vs 楽観的同時実行制御）

2PL は **pessimistic（悲観的）** な同時実行制御メカニズムである。問題が発生する可能性がある場合（他のトランザクションが現在使用しているオブジェクトにロックが保持されている場合）、状況が安全になるまで待つことを選ぶ。直列実行は究極的に悲観的 — 各トランザクションはデータベース全体の排他ロックを持っているのと同等である。

対照的に、SSI は **optimistic（楽観的）** な同時実行制御技術である。楽観的とは、危険な可能性があっても、トランザクションをブロックする代わりに続行させることを意味する。トランザクションがコミットしようとする時点で、何か問題が発生したかどうかをチェックし（isolation が違反されたかどうか）、もしそうであればトランザクションを中止してリトライする。

楽観的同時実行制御は古い考え方であるが、contention（競合）が高い場合にパフォーマンスが悪化する傾向があった — 多くのトランザクションが同じオブジェクトにアクセスしようとする場合、中止されるトランザクションの割合が高くなるためである。しかし、contention がそれほど高くなく、予備容量が十分にある場合、楽観的同時実行制御は悲観的なものよりも優れたパフォーマンスを発揮する傾向がある。

SSI はsnapshot isolation の上に構築されている。トランザクションのすべての読み取りはデータベースの一貫したスナップショットから行われる。snapshot isolationとの重要な違いは、SSI がスナップショットに対する読み取りと書き込みの間のserializability conflicts を検出するアルゴリズムを追加し、どのトランザクションを中止すべきかを決定する点である。

#### Decisions based on an outdated premise（古い前提に基づく判断）

snapshot isolation でのwrite skew は、トランザクションがスナップショットからデータを読み取り、読み取った値に基づいて何らかのアクションを取り、データベースに書き込むことで発生する。しかし、snapshot isolation の下では、書き込みがコミットされる時点で、元の読み取りのデータが変更されている可能性がある。

言い換えると、トランザクションはある**premise（前提）**（トランザクション開始時に真であった事実）に基づいて行動する。後で、トランザクションがコミットしたい時点で、元のデータが変更されている可能性があり、前提がもはや真でない可能性がある。

データベースがクエリの結果が変更されたかどうかを知るためには、2つのケースを検出する必要がある:

1. **古いMVCC読み取り（stale MVCC reads）の検出**: スナップショットの時点ではコミットされていなかったが、読み取りトランザクションがコミットしようとする時点ではコミットされている書き込みの検出
2. **過去の読み取りに影響する書き込みの検出**: トランザクションが書き込もうとした時、以前の読み取りに影響する可能性がある他のトランザクションの書き込みの検出

**ケース1: stale MVCC read の検出**

snapshot isolation では、トランザクションはMVCCの可視性ルールに基づいて他のトランザクションの書き込みを無視する。トランザクションがコミットしようとする時、データベースは無視された書き込みのいずれかがコミットされているかどうかをチェックする。もしそうであれば、トランザクションを中止しなければならない。

なぜ読み取り時点ではなくコミット時点まで中止を待つのか？ 読み取り専用トランザクションの場合、この最適化により不必要な中止を回避できる。読み取りトランザクションが十分に長い間生き残ってコミットされれば、読み取りは確かに古かったが、そのstale read に基づいてトランザクションがアクションを取らなかったため、中止する必要がない。

**ケース2: 過去の読み取りに影響する書き込みの検出**

2つのトランザクションが同時にデータベースに読み取りと書き込みを行う場合、2PL のindex-range lock と同様に、SSI はインデックスを使用する。しかし SSI はロックをブロックメカニズムとしてではなく、他のトランザクションが現在のトランザクションの読み取り結果に影響しうることを単に通知するために使用する。

トランザクションが別のトランザクションの以前の読み取りに影響する可能性のあるデータを書き込む場合、影響を受けるトランザクションに通知する。コミット時に、トランザクションは通知を受け取ったかどうかをチェックし、受け取った場合はトランザクションを中止する。

#### Performance of serializable snapshot isolation（SSIのパフォーマンス）

他のserializable isolation技術と比較して、SSI には大きな利点がある: トランザクションはロックの解放を待つ必要がない。snapshot isolation と同様に、writers は readers をブロックせず、readers は writers をブロックしない。これにより、クエリの遅延がはるかに予測しやすくなり、変動が少なくなる。特に、読み取り専用のクエリはロックなしで一貫したスナップショット上で実行できるため、読み取りの多いワークロードに非常に魅力的である。

2PL と比較して、SSI の中止率はcontention（競合）が高くなると上がるが、2PL のようにロック待ちによるスループット低下は発生しない。直列実行と比較して、SSI は単一のCPU コアのスループットに制限されない。

中止率はSSI のパフォーマンスに大きな影響を与える。長い期間にわたってデータの読み書きを行うトランザクションは、conflict が発生しやすく中止される可能性が高い。そのため、SSI では読み書きトランザクションがかなり短いことが求められる（長時間実行の読み取り専用トランザクションは問題ない）。ただし、SSI は2PL や直列実行よりも遅いトランザクションに対する影響は少ないと考えられる。

## Summary（まとめ）

トランザクションは、特定のconcurrency 問題や特定のハードウェアおよびソフトウェア障害が存在しないかのようにアプリケーションが振る舞えるようにする抽象化レイヤーである。多くのエラーが単純なトランザクションのabort に集約され、アプリケーションは再試行するだけでよい。

この章では、トランザクションが防止する多くの問題の例を見た。すべてのアプリケーションがすべての問題の影響を受けるわけではない — 単一のレコードの読み書きのみを行うような単純なアクセスパターンのアプリケーションは、トランザクションなしでも問題ない可能性がある。しかし、より複雑なアクセスパターンの場合、トランザクションは考慮すべき潜在的なエラーケースの数を大幅に削減できる。

トランザクションなしでは、さまざまなエラーシナリオ（プロセスクラッシュ、ネットワーク中断、停電、ディスク満杯、予期しないconcurrency 等）により、データがさまざまな方法で不整合になりうる。例えば、非正規化データがソースデータと容易に同期しなくなる。トランザクションなしでは、複雑な相互作用するアクセスがデータベースに与える影響について推論することが非常に困難になる。

この章では、特にconcurrency control のトピックを深く掘り下げた。広く使用されているいくつかの分離レベル、特に **read committed**、**snapshot isolation（repeatable read とも呼ばれる）**、および **serializable** について議論した。これらの分離レベルを、さまざまな race condition の例を通じて特徴づけた:

- **Dirty reads（ダーティリード）**: 1つのクライアントが、まだコミットされていない別のクライアントの書き込みを読み取る。read committed 分離レベル以上で防止される。
- **Dirty writes（ダーティライト）**: 1つのクライアントが、まだコミットされていない別のクライアントが書き込んだデータを上書きする。ほぼすべてのトランザクション実装で防止される。
- **Read skew（nonrepeatable reads、読み取りスキュー）**: クライアントが異なる時点のデータベースの異なる部分を見る。最も一般的にはsnapshot isolation で防止され、MVCC（Multi-Version Concurrency Control）で実装される。
- **Lost updates（更新ロスト）**: 2つのクライアントが同時にread-modify-writeサイクルを実行し、一方の書き込みがもう一方の変更を組み込まずに上書きしてデータが失われる。snapshot isolation の一部の実装はこの異常を自動的に防止し、他の実装ではmanual lock（`SELECT FOR UPDATE`）が必要である。
- **Write skew（書き込みスキュー）**: トランザクションが何かを読み取り、読み取った値に基づいて判断を下し、その判断をデータベースに書き込む。しかし、書き込みが行われる時点で、判断の前提がもはや真でない。serializable isolation のみがこの異常を防止する。
- **Phantom reads（ファントムリード）**: トランザクションが何らかの検索条件に一致するオブジェクトを読み取る。別のクライアントがその検索結果に影響する書き込みを行う。snapshot isolation は単純なphantom reads を防止するが、write skew の文脈でのphantom にはindex-range locks などの特別な対処が必要である。

弱い分離レベルはこれらの異常の一部からは保護するが、残りはアプリケーション開発者が手動で対処する必要がある（明示的なロック等）。serializable isolation のみがこれらすべての問題から保護する。serializable transaction の3つの異なる実装アプローチについて議論した:

- **文字通りの直列実行**: 各トランザクションの実行を非常に高速にでき、トランザクションスループットが単一CPU コアで処理可能なほど低い場合、シンプルかつ効果的な選択肢である。
- **Two-phase locking（2PL）**: 数十年にわたりserializability を実装する標準的な方法であったが、パフォーマンス特性のために多くのアプリケーションで使用が避けられている。
- **Serializable snapshot isolation (SSI)**: これまでのアプローチの欠点のほとんどを回避する、かなり新しいアルゴリズムである。楽観的なアプローチを使い、トランザクションをブロックせずに進行させる。トランザクションがコミットしようとする時にチェックし、実行がserializable でなかった場合は中止する。

この章の例はリレーショナルデータモデルを使用した。しかし、「複数オブジェクトトランザクションの必要性」で議論したように、トランザクションは使用されるデータモデルに関係なく、有用なデータベース機能である。

この章では、主に単一マシンで動作するデータベースのコンテキストで、アイデアとアルゴリズムを探求した。分散データベースにおけるトランザクションは、新たな困難な課題を開くものであり、次の2つの章で議論する。
